## R01

Voronoi 图将平面划分为若干区域：每个区域对应一个站点（site），区域内任一点到该站点的距离不大于到其他站点的距离。Fortune 算法是构造 2D Voronoi 图的经典扫描线算法，核心结构包括：
- 扫描线（directrix）从上向下移动；
- beachline（抛物线弧组成的前沿）；
- 事件队列（site event 与 circle event）；
- 输出结构（Voronoi 顶点与边）。

在一般位置（无退化）下，Fortune 算法时间复杂度为 `O(n log n)`。

## R02

Fortune 算法由 Steven Fortune 在 1986 年提出，用于高效构造 Voronoi 图。它相较于基于网格、增量插入但无复杂数据结构优化的方法，显著降低了复杂度，并成为计算几何课程和工程实现中的标准方法之一。

其关键贡献是把“几何构造”转成“事件驱动 + 平衡维护”的流程：
- site event 插入新的抛物线弧；
- circle event 删除即将消失的弧并产生 Voronoi 顶点。

## R03

要解决的问题：给定 `n` 个二维点，构造它们的 Voronoi 图（顶点、边、射线关系）。

难点在于：
1. 边是连续几何对象，不是离散网格可直接枚举；
2. beachline 会动态变化，局部更新会引发全局拓扑变化；
3. circle event 需要正确判定“何时某个弧消失”。

Fortune 算法通过优先队列 + 局部邻接更新，将复杂几何变化转为可维护的数据结构操作。

## R04

标准 Fortune 流程可概括为：
1. 将所有 site 按 `y` 坐标加入事件队列（大 `y` 先处理）。
2. 初始化空 beachline。
3. 处理 site event：找到站点 `x` 正上方的弧，分裂为三段（左旧弧-新弧-右旧弧），并创建两条新 Voronoi 半边。
4. 在受影响的局部三元弧上检查 circle event 候选。
5. 处理 circle event：删除中间弧，在圆心处产生 Voronoi 顶点，并把相关半边闭合到该顶点。
6. 重复直到事件队列为空。
7. 将未闭合的无限边向外延伸并裁剪到边界盒。

## R05

理论复杂度（一般位置）：
- 时间复杂度：`O(n log n)`
  - 事件队列操作 `log n`
  - 总事件数线性级别（site `n` + circle `O(n)`）
- 空间复杂度：`O(n)`
  - beachline 节点、事件、边、顶点均线性规模。

本目录 MVP 使用双向链表维护 beachline（未用平衡树），在最坏实现细节上可能退化；但整体仍体现 Fortune 的事件驱动核心。

## R06

简化例子（概念层）：
- 先处理最高点 `s1`：beachline 只有一个弧。
- 再处理 `s2`：`s1` 对应弧被分裂，出现第一对 breakpoints，生成一条 Voronoi 边的两条半边。
- 再处理 `s3`：若三弧构成顺时针结构，将出现 circle event。
- 当扫描线降到该事件高度时，中间弧消失，圆心成为一个 Voronoi 顶点，相关边在此汇合。

这样不断迭代，最终形成完整 Voronoi 骨架。

## R07

优点：
- 理论复杂度优秀（`O(n log n)`）。
- 是“直接构造”方法，几何含义清晰。
- 适合生成 Voronoi 顶点/边拓扑，不依赖离散化分辨率。

局限：
- 代码实现复杂，退化情形（共圆、共线、重复点、同 `y`）处理容易出错。
- 需要较细致的数值稳健性设计（容差、事件失效标记、边界裁剪）。
- 工程中常用成熟库以减少维护成本。

## R09

适用场景：
- 平面点集 Voronoi 分区、最近邻区域、设施选址可视化；
- 需要 Voronoi 顶点和边结构而非仅查询某一点最近站点。

不适合或需改造的场景：
- 高维 Voronoi（算法与数据结构需变化）；
- 要求工业级鲁棒性但未处理退化输入；
- 含大量重复点/强退化数据且必须严格拓扑正确。

## R10

实现正确性关键点：
1. 事件按 `y` 递减处理；同层时站点事件优先。
2. 插入站点时必须找到正确“正上方弧”并做三段分裂。
3. 旧 circle event 必须失效（lazy invalidation），否则会处理过期事件。
4. circle event 仅在局部三弧满足方向条件时成立（本实现使用顺时针判定）。
5. 弧消失时，左右边要在同一 Voronoi 顶点闭合，并创建新 breakpoint 对应边。

## R11

数值稳定性处理：
- 使用 `EPS=1e-9` 统一容差；
- breakpoint 计算时对“焦点接近准线”做分支，避免除零；
- 判别式若出现微小负值，截断到 0 以避免 `sqrt` 域错误；
- 事件判断使用 `>= current_y - EPS` 规避临界抖动；
- 构造样例点时加入极小扰动，降低同 `x/y` 退化风险。

## R14

常见失效模式与防护：
- 失效：过期 circle event 被误处理，导致错误拓扑。
  - 防护：事件对象携带 `valid` 标记，并与 `arc.circle_event` 指针一致性校验。
- 失效：breakpoint 根选择错误，导致弧定位错乱。
  - 防护：按左右弧与焦点高度关系选择二次方程根。
- 失效：无限边未裁剪，输出不可用。
  - 防护：统一与 bbox 求交，输出有限线段。
- 失效：浮点边界抖动导致频繁误判。
  - 防护：比较使用 `EPS` 容差，临界值做稳定分支。

## R16

相关算法与前后关系：
- Voronoi 与 Delaunay 互为对偶。
- 同类构造思路：
  - 增量插入 + 局部翻转（常见于 Delaunay 先行再转 Voronoi）；
  - 分治构造 Voronoi；
  - 调用 Qhull/CGAL 等成熟几何库。

Fortune 的优势在于二维静态点集下的经典理论复杂度与清晰事件模型。

## R17

本目录 `demo.py` 的实现说明：
- 从零实现 Fortune MVP，不依赖黑盒 Voronoi 计算。
- 数据结构：
  - `Event`（site/circle）+ `heapq` 事件队列；
  - `Arc` 双向链表 beachline；

- 输出：
  - circle event 生成的有限 Voronoi 顶点；
  - 经过 bbox 裁剪后的 Voronoi 线段。
- 脚本包含 `optional_scipy_check`，若环境有 SciPy，会做结果对照（仅自检，不依赖其求解）。

## R18

源码级流程拆解（对应 `demo.py`，9 步）：
1. `make_demo_points` 生成固定随机站点并添加微小扰动，减少退化。  
2. `FortuneVoronoi.__init__` 把输入点转为 `Site`，初始化事件堆、beachline、边界盒。  
3. `run` 将每个站点封装为 `site event` 入堆，按 `(-y, type, x)` 顺序弹出事件。  
4. `site event` 触发 `_handle_site_event`：通过 `_find_arc_above` 找到目标弧并三分裂（左旧弧/新弧/右旧弧）。  
5. 站点插入后创建两条新半边（`_new_edge`），并在局部调用 `_check_circle_event` 生成圆事件候选。  
6. `_check_circle_event` 对相邻三弧做方向判定、外接圆计算（`_circumcenter`），把有效圆事件压入堆。  
7. `circle event` 触发 `_handle_circle_event`：删除中间弧、记录 Voronoi 顶点、闭合左右边，并创建新 breakpoint 对应边。  
8. 全部事件处理后，`_clip_edge` + `_line_box_intersections` 将无限半边裁剪到 bbox，得到可输出有限线段。  
9. `main` 打印统计信息，并可选执行 `optional_scipy_check` 做参考实现一致性诊断。  

该流程覆盖了 Fortune 算法的核心机制：事件队列驱动、beachline 动态维护、circle vertex 生成与边连接。
