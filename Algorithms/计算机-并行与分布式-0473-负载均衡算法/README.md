# 负载均衡算法

## R01

本条目实现的是“负载均衡算法”在并行与分布式系统中的最小可运行版本。  
目标不是构建完整网关，而是通过离散事件仿真复现常见策略在同一工作负载下的差异。

本 MVP 对比 4 类经典策略：

- 轮询（Round Robin）
- 加权轮询（Weighted Round Robin）
- 最少连接（Least Connections）
- 二选一随机（Power of Two Choices, P2C）

## R02

问题定义（本条目约定）：

- 输入：后端节点集合（含权重与容量）、请求到达流、请求服务时长分布
- 输出：每种策略下的分配结果与指标
  - 每节点请求数
  - 平均等待时间、P95 等待时间、最大等待时间
  - 负载不均衡系数（请求计数的变异系数）
  - 各节点近似利用率

我们关注的是“调度策略对系统行为的影响”，而非网络协议细节。

## R03

算法直觉：

- 轮询：追求实现简单与分配均匀，但忽略节点性能差异。
- 加权轮询：通过权重表达节点处理能力，更贴近异构集群。
- 最少连接：实时查看当前活跃连接数，优先把请求发到更空闲节点。
- P2C：随机挑 2 个节点再选更轻载者，用很小开销换取显著降拥塞效果。

其中 P2C 常用于大规模系统，因为它接近全局最优但只需局部比较。

## R04

MVP 的系统模型：

- 每个后端节点有固定容量 `capacity`（可并行处理的请求槽位数）。
- 请求到达后先按策略选节点，再进入该节点局部队列。
- 若节点未满，请求立即开始；若满，请求等待到最早空闲时刻。
- 不考虑请求迁移、预抢占、失败重试，仅保留调度核心行为。

该模型足够小，但可以真实体现“同样流量下策略优劣差异”。

## R05

关键数据结构（`demo.py`）：

- `Backend(name, weight, capacity)`：节点元信息。
- `LoadBalancer`：封装 4 种策略的选择逻辑。
- `completion_heaps[i]`：第 `i` 个节点的最小堆，维护该节点各处理槽位的完成时间。
- `assigned_counts / waiting_time / busy_time`：指标统计数组。

最小堆用于快速获得“该节点最早可用时间”，是离散事件仿真的核心。

## R06

请求处理流程（单请求）：

1. 清理每个节点堆中 `<= 当前到达时刻` 的完成事件，得到实时活跃连接数。  
2. 调用策略函数选目标节点。  
3. 若目标节点未满：`start = arrival`。  
4. 若目标节点已满：弹出该节点最早完成时刻 `t_free`，令 `start = t_free`。  
5. 计算 `finish = start + service_time`，回写到该节点堆。  
6. 更新计数、等待时间和忙碌时间统计。

该流程完全无交互输入，适合自动验证环境直接运行。

## R07

时间复杂度（设请求数为 `N`，节点数为 `M`）：

- 每个请求都要扫描节点活跃连接：`O(M)`。
- 在目标节点做堆操作：`O(log C)`，`C` 为该节点容量上界。

总复杂度近似：`O(N * (M + log C))`。  
本条目 `M` 很小（3 个节点），重点是策略可比性与代码可读性。

## R08

空间复杂度：

- 各节点完成时间堆总元素上界约为总容量 `sum(capacity)`。
- 统计数组规模为 `O(N)`（等待时间/完成时间）与 `O(M)`（计数/利用率）。

整体空间复杂度：`O(N + M + sum(capacity))`。

## R09

策略差异的常见现象：

- 轮询在异构节点上可能“表面公平、实际拥塞”。
- 加权轮询能更接近节点能力比例，但遇到瞬时抖动不一定最优。
- 最少连接对短时拥塞响应快，但需要实时状态采集。
- P2C 在状态开销和效果之间通常有很好的折中。

因此工程里经常是“权重 + 动态反馈”组合，而非单一静态策略。

## R10

边界与异常处理：

- `backends` 不能为空。
- 权重全部为 0 会报错（无法构建加权轮询表）。
- 仅 1 个节点时，P2C 直接返回该节点。
- 策略名非法时抛出 `ValueError`。

这些校验保证脚本即使在错误输入下也能快速失败并给出明确信息。

## R11

正确性与可解释性：

- 轮询/加权轮询的行为是确定性的，便于基准复现实验。
- 最少连接与 P2C 的选点逻辑均可由实时活跃连接数解释。
- 输出直接给出“分配计数 + 等待时间 + 利用率”，可对照业务目标（延迟优先或吞吐优先）做决策。

MVP 不追求所有生产细节，但每个指标都可回溯到具体调度动作。

## R12

参数选择（`main()`）：

- 节点配置：`node-A(5,6)`、`node-B(3,4)`、`node-C(2,3)`，体现异构能力。
- 请求数量：`5000`，可在数秒内完成且统计稳定。
- 到达过程：指数分布（泊松到达）。
- 服务时长：对数正态分布，模拟真实系统里“短请求多、长尾存在”。

这些参数让策略差异更容易被观察到，而无需大型框架。

## R14

指标解释建议：

- `mean_wait / p95_wait / max_wait`：反映延迟体验与尾部风险。
- `imbalance_cv`：反映分配计数离散程度，越小越均衡。
- `counts[...]`：显示策略是否按预期体现权重或动态均衡。
- `util[...]`：观察是否出现“某些节点明显过载、某些节点明显闲置”。

不要只看单一指标，需结合业务目标综合判断。

## R18

`demo.py` 源码级算法流程拆解（8 步）：

1. 在 `main()` 中定义 3 个异构后端节点（权重、容量）并生成固定随机种子的请求流。  
2. `generate_workload()` 产生 `arrivals`（指数分布累积）和 `service_times`（对数正态）。  
3. `simulate_strategy()` 为每个节点初始化完成时间最小堆，作为其并发处理槽位的占用状态。  
4. 对每个请求，先清理所有节点堆中已完成任务，得到实时 `active_connections`。  
5. `LoadBalancer.choose()` 按策略选节点：轮询、加权轮询、最少连接或 P2C。  
6. 若目标节点未满则立即开始；若已满则弹出最早完成时间作为排队后的开始时刻。  
7. 计算完成时刻并回写堆，同时累计请求计数、等待时间、忙碌时间。  
8. 全部请求完成后计算 `mean/p95/max wait`、`imbalance_cv`、各节点利用率并打印对比结果。  

以上流程完整展示了负载均衡从“选点”到“排队与指标统计”的可执行路径，不依赖黑盒调度器。
