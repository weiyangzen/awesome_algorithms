## R01

并行快速排序（Parallel Quick Sort）是在经典快速排序基础上，把可独立处理的子问题交给多个计算单元并发执行。  
快速排序本身是分治结构：一次分区后，左侧和右侧子数组互不依赖，这为并行化提供了天然入口。

本条目提供一个可运行的 Python MVP：
- 顺序三路快排作为基线；
- 主进程有限深度展开分区树；
- 子分区交给多进程并行排序；
- 自动进行正确性校验和耗时对比。

## R02

为什么需要并行快速排序：
- 在多核 CPU 环境中，大规模排序是常见预处理瓶颈（日志清洗、离线特征工程、索引构建）；
- 快排分区后的左右子问题独立，能映射为并发任务；
- 对大输入，合理控制分区粒度后可明显降低 wall-clock 时间。

需要注意，并行化并不总是收益。任务拆分、进程通信、序列化都会带来额外成本。

## R03

本实现的核心思路：
1. 先使用三路分区（`< pivot`、`== pivot`、`> pivot`）实现顺序快排；
2. 在主进程将分区树展开到有限深度，得到“有序拼接顺序”的多个子分区；
3. 使用进程池并行执行每个子分区的顺序快排；
4. 按分区顺序拼接结果，得到全局有序数组。

这里不是直接调用黑盒并行排序库，而是把分区、展开、并行执行、拼接全过程显式写在源码里。

## R04

与传统单线程快排的差异：
- 单线程：一次分区后递归处理左右区间，整个递归树都在一个线程里执行；
- 本并行版：仅在主进程做前几层分区展开，随后把独立子分区并发提交到进程池；
- 通过 `min_partition_size` 与 `partition_depth` 控制并行粒度；
- 当数据过小或并行度不足时自动回退顺序快排，避免并行开销反噬。

## R05

复杂度分析：
- 顺序快排平均时间复杂度 `O(n log n)`，最坏 `O(n^2)`；
- 三路分区在重复值较多场景可减少无效递归，实践中更稳定；
- 并行版本总工作量仍近似 `O(n log n)`，但临界路径在理想负载均衡下可缩短；
- 额外代价主要来自进程调度与数据序列化。

因此并行效果依赖于数据规模、分区均衡程度和硬件核数。

## R06

`demo.py` 的实验设置：
- 随机数据规模：`n=140000`；
- 数据生成：`numpy.random.default_rng(seed=20260407)`，可复现；
- 进程数：`workers=min(4, os.cpu_count())`；
- 分区阈值：`min_partition_size=20000`；
- 分区深度：`partition_depth=ceil(log2(workers))`。

脚本无需交互输入，直接执行即可产出配置、耗时和一致性检查结果。

## R07

MVP 的并行策略优势：
- 依赖简单，仅使用 `numpy` + Python 标准库 `concurrent.futures`；
- 并行边界清晰：主进程负责结构化拆分，工作进程负责局部排序；
- 结果确定性强：同一输入和参数下输出稳定一致；
- 容易扩展到分布式场景（把“子分区任务”映射到远端 worker 即可）。

## R09

适用场景：
- 多核机器上的中大规模整数排序；
- 离线任务里“先排序再去重/聚合/分桶”的前置步骤；
- 教学和算法实验：理解快排如何映射到并行执行模型。

不适用场景：
- 极小数据集（并行调度开销可能超过收益）；
- 对最坏时间复杂度极端敏感且需严格上界保证的业务；
- 进程启动受限或 IPC 成本极高的运行环境。

## R10

正确性要点（简版）：
1. 三路分区保证 `less` 中元素都小于主元，`equal` 全等于主元，`greater` 都大于主元；
2. 顺序快排递归地分别排序 `less/greater` 后拼接，结果有序；
3. 并行版本只改变“子问题何时在哪个进程执行”，不改变子问题语义；
4. 主进程展开分区树时保留了分区的中序拼接顺序，故并行子结果拼接后全局有序。

`demo.py` 同时用 `_assert_sorted` 和与顺序版逐元素比较来做实证校验。

## R11

边界与鲁棒性处理：
- 空数组、单元素数组直接返回副本；
- 全相等数组由三路分区快速识别并直接返回；
- `max_workers` 自动规整到 `>=1`；
- 并行条件不满足时回退顺序版；
- `_self_test` 覆盖空输入、重复值、负数、逆序等典型样例。

## R12

关键函数职责：
- `_choose_pivot`：中位三数法选主元；
- `_partition_three_way`：执行三路分区；
- `sequential_quick_sort`：顺序三路快排；
- `_expand_partitions`：有限深度展开分区树；
- `parallel_quick_sort`：并行调度与拼接结果；
- `_assert_sorted`：单调性校验；
- `_self_test`：快速回归测试；
- `_benchmark`：耗时测量；
- `main`：实验入口与结果输出。

## R13

性能观察建议：
- 增大 `n`（例如 20 万、50 万）更容易看到并行收益；
- `workers` 通常不宜超过物理核数；
- `min_partition_size` 太小会产生大量短任务，IPC 成本上升；
- `partition_depth` 太浅并行不足，太深又会导致子任务过碎。

该 MVP 更强调算法流程可追踪与可复现，不追求极限吞吐。

## R14

常见实现错误：
1. 直接把每层递归都提交进程池，造成任务爆炸和严重调度开销；
2. 在进程池中使用不可 pickle 的局部函数/闭包，导致运行时失败；
3. 分区后拼接顺序错误（把 `equal` 放到两侧之外）导致结果失序；
4. 忽略全重复值场景，导致递归深度和开销异常；
5. 未设置小数组回退策略，导致并行版常常比顺序版慢。

## R15

可扩展方向：
- 使用共享内存结构减少跨进程序列化成本；
- 引入更稳健的主元采样策略（如随机采样中位数）；
- 增加多轮 benchmark（均值/方差）提升性能评估可信度；
- 将子分区任务分发到分布式执行框架（Ray/Dask/Spark）；
- 为对象排序增加 `key` 函数与稳定性分析。

## R16

相关算法与技术：
- 顺序快速排序（基础算法）；
- 并行归并排序（更稳定但依赖归并缓冲）；
- Sample Sort、Bitonic Sort（并行排序常见方案）；
- Work Stealing 调度（并行递归任务平衡）；
- MapReduce 风格分区排序与全局拼接。

## R18

`demo.py` 源码级算法流程（8 步）：
1. `main` 先调用 `_self_test`，对空数组、逆序、重复值等样例分别比较顺序版/并行版与 `sorted(case)`。  
2. `main` 使用固定随机种子生成输入数组，并计算 `workers`、`min_partition_size`、`partition_depth`。  
3. 顺序路径进入 `sequential_quick_sort`：先 `_choose_pivot` 选主元，再 `_partition_three_way` 把数组拆为 `less/equal/greater`。  
4. 若分区结果全在 `equal`（全重复值），顺序快排直接返回，避免无效递归；否则递归排序两侧并拼接。  
5. 并行路径进入 `parallel_quick_sort`，调用 `_expand_partitions` 在主进程把分区树展开到有限深度，生成保持大小关系顺序的子分区列表。  
6. 进程池执行 `pool.map(sequential_quick_sort, partitions)`，每个子分区在独立进程内完成局部排序。  
7. 主进程按分区顺序线性拼接 `sorted_parts`，得到全局有序结果，并用 `_assert_sorted` 与顺序结果一致性检查。  
8. `main` 打印顺序耗时、并行耗时和加速比，形成“实现-验证-度量”的可复现实验闭环。  

实现中没有把第三方并行排序当黑箱直接调用，分区、并行拆解、排序与拼接流程都可以在源码中逐步追踪。
