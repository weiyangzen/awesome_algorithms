## R01

语法制导翻译（Syntax-Directed Translation, SDT）是在语法分析过程中执行语义动作，把“句法结构”同步翻译成“中间表示”。
本条目给出一个教学级最小可运行 MVP：
- 输入算术表达式（`id/num/+/*/()`）
- 使用递归下降分析（基于消左递归后的文法）
- 在产生式处执行语义动作，生成三地址码（TAC）
- 执行 TAC 并与直接求值结果做断言比对

## R02

本实现解决的问题可以表述为：
- 输入：表达式字符串 + 变量环境（符号表绑定）
- 输出：
  - 词法 token 序列
  - 三地址码四元式序列 `result = arg1 op arg2`
  - 最终结果位置 `result_place`
  - TAC 执行值（并验证与表达式语义一致）

目标不是构建完整编译器，而是把 SDT 的关键链路清晰展开。

## R03

采用的文法（消左递归后）为：

- `E  -> T E'`
- `E' -> + T E' | ε`
- `T  -> F T'`
- `T' -> * F T' | ε`
- `F  -> ( E ) | id | num`

该文法保持了运算语义：
- `*` 优先级高于 `+`
- 两者都按左结合解释

## R04

属性与语义动作（简化版）如下：

- 综合属性：`X.place`（当前子表达式计算结果所在地址）
- 继承属性：`E'.inh`, `T'.inh`（累计的左侧结果）
- 语义动作核心：
  - `E -> T E'`：令 `E'.inh = T.place`，最终 `E.place = E'.syn`
  - `E' -> + T E'1`：`tmp = new_temp(); emit(tmp = E'.inh + T.place); E'1.inh = tmp`
  - `E' -> ε`：`E'.syn = E'.inh`
  - `T'` 同理处理乘法

在代码里，这些动作直接体现在 `parse_E_prime/parse_T_prime` 中的 `emit_binary` 调用。

## R05

词法分析器 `tokenize` 支持：
- `ID`: 标识符（`[A-Za-z_]\w*`）
- `NUM`: 十进制整数
- 运算与括号：`+`, `*`, `(`, `)`
- 末尾哨兵：`EOF($)`

若遇到不合法字符，立即抛出 `SyntaxError`，保证后续语法阶段输入可靠。

## R06

递归下降翻译器 `Parser` 的结构：
- `parse_E -> parse_T + parse_E_prime`
- `parse_T -> parse_F + parse_T_prime`
- `parse_F` 处理原子和括号

区别于“先建 AST 再遍历”的两段式流程，本实现是“边分析边翻译”——每次识别到二元操作就立刻发射 TAC。

## R07

中间代码采用四元式结构 `Quad(op, arg1, arg2, result)`，例如：
- `t1 = b * c`
- `t2 = a + t1`

`Emitter` 负责：
- 分配临时变量名 `t1, t2, ...`
- 追加四元式到线性代码流

这是经典 SDT 里“语义动作生成中间代码”的最小映射。

## R08

为了避免“只生成代码不验证语义”，`demo.py` 还实现了 TAC 执行器：
- `execute_three_address_code` 顺序解释四元式
- `eval_operand` 从符号表或常量读取操作数
- 得到 `final_place` 的运行结果

脚本把该结果与 `eval(expr, ..., env)` 的值做断言，形成闭环验证。

## R09

内置正例：
- `a + b * c`
- `(a + b) * (c + 2)`
- `x * y + z * 3 + 1`
- `42`
- `m * (n + 8) * p`

内置反例：
- `a + * b`（应被语法阶段拒绝）

运行时会打印 token、TAC、执行结果和统计信息（指令条数与均值）。

## R10

正确性要点：
- 语法正确性：`expect` 保证终结符匹配，错误位置可追踪
- 翻译正确性：语义动作只在匹配产生式时执行，保持文法驱动
- 语义正确性：通过“执行 TAC 与直接求值一致”断言校验
- 失败可见性：反例必须触发 `SyntaxError`，否则脚本显式失败

## R11

时间复杂度（设 token 数为 `n`，TAC 条数为 `k`）：
- 词法分析：`O(n)`
- 递归下降翻译：`O(n)`（每个 token 被消费常数次）
- TAC 执行：`O(k)`，且 `k <= n`
- 总体：`O(n)`

## R12

空间复杂度：
- token 列表：`O(n)`
- 四元式序列：`O(k)`
- 递归栈深度：与括号嵌套深度相关，最坏 `O(n)`

对教学规模输入，这个成本很低且足够透明。

## R14

工程边界与取舍：
- 仅覆盖整数与 `+/*/()` 的表达式子集
- 不含一元负号、除法、函数调用、类型系统
- 不实现错误恢复（遇错立即报错）
- 不接入真实词法/语法生成器工具链

这些限制是为了把 SDT 主线压缩成可快速审计的 MVP。

## R15

## R18

`demo.py` 源码级算法流程（8 步）：
1. `tokenize` 将输入字符串切分为 `ID/NUM/PLUS/MUL/LPAREN/RPAREN/EOF` token。  
2. `Parser.parse_E/parse_T/parse_F` 按消左递归文法执行递归下降语法分析。  
3. 在 `parse_E_prime` 中遇到 `+ T` 时，执行语义动作：调用 `emit_binary("+", inh, t_place)` 发射一条加法四元式。  
4. 在 `parse_T_prime` 中遇到 `* F` 时，执行语义动作：调用 `emit_binary("*", inh, f_place)` 发射一条乘法四元式。  
5. `Emitter.new_temp` 生成临时变量 `t1,t2,...`，把复杂表达式拆成线性三地址码。  
6. `Parser.translate` 返回 `result_place + quads`，即语法制导翻译结果。  
7. `execute_three_address_code` 顺序执行四元式，得到最终值并与 Python 直接求值结果断言一致。  
8. `run_negative_case` 验证错误输入会在语法阶段被拒绝，确认翻译器具备基本错误检测能力。

说明：实现没有把“翻译”交给第三方函数一步完成，而是将词法、语法、属性传播、语义动作、代码执行全部拆解为可审计源码步骤。
