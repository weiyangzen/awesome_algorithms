## R01

四叉树（Quadtree）与八叉树（Octree）是分层空间划分结构：

- 四叉树：递归把二维轴对齐正方形区域切成 4 个子象限。
- 八叉树：递归把三维轴对齐立方体区域切成 8 个子立方体。

它们适用于“点集或体素在空间中分布不均匀”的场景，通过只细分密集区域来减少查询时的扫描范围。

## R02

历史脉络简述：

- 四叉树在 1970 年代由 Raphael Finkel 与 J. L. Bentley 等人在分层文件结构与区域检索问题中系统化。
- 八叉树可视为四叉树向三维的自然推广，随后广泛用于计算机图形学、体绘制、三维碰撞检测和机器人建图。

核心思想始终一致：用树结构编码“空间分治”，让时间复杂度从全量扫描转向“与局部密度相关”。

## R03

本题关注的核心问题：

- 输入：二维或三维点集，及若干轴对齐范围查询（AABB 查询框）。
- 输出：每个查询框内的所有点。

若暴力做法每次查询都扫全体点，复杂度是 `O(n)`；当查询次数多时，总成本高。目标是构建索引，把单次查询降到“访问少量相关节点 + 少量候选点”。

## R04

四叉树/八叉树的标准流程可以拆成两段。

构建（插入）流程：
1. 给定根边界（2D 正方形或 3D 立方体）、叶子容量 `capacity`、最大深度 `max_depth`。
2. 插入点时，若当前节点未超容量，直接放入当前节点。
3. 若超容量且未达最大深度，执行一次分裂（2D 拆 4 子节点，3D 拆 8 子节点）。
4. 将当前节点存量点重分配到子节点，再继续插入新点。

查询（范围检索）流程：
1. 从根节点开始，若“节点边界与查询框不相交”，整棵子树直接剪枝。
2. 若相交，先检查当前节点持有的点是否落入查询框。
3. 再递归访问可能相交的子节点。
4. 汇总命中的点集作为结果。

## R05

设点数为 `n`，查询返回 `k` 个点。

- 构建：平均常见为 `O(n log n)`，最坏在病态分布下可能退化接近 `O(n^2)`（若无限细分且点高度集中）。
- 查询：平均常见为 `O(log n + k)`，最坏可达 `O(n)`（查询框很大或树退化时）。
- 空间：`O(n + 节点数)`，节点数与分裂策略和数据分布相关。

与暴力法对比：当查询多、空间分布稀疏且查询框局部化时，树结构通常显著更快。

## R06

一个直观二维例子：

- 在 `[-1,1] x [-1,1]` 中放入 2000 个随机点。
- 构建四叉树，`capacity=16, max_depth=8`。
- 查询框 `center=(0,0), half=0.35`。

该查询只需访问与中心区域相交的少量节点，不必扫描全部 2000 点。

三维版本同理：在 `[-1,1]^3` 中放入 4000 个点，用八叉树做立方查询框检索。

## R07

优点：

- 实现直观，工程可控，适合点云/空间索引入门与中小规模系统。
- 支持高效范围查询、邻近搜索预筛、碰撞检测粗筛。
- 非均匀分布下通常优于均匀网格（不会在稀疏区浪费太多桶）。

局限：

- 对参数敏感（`capacity`、`max_depth`）。
- 极端聚簇或大量共边界点时可能出现树不平衡或重复判断增多。
- 仅支持轴对齐边界盒时，对任意旋转查询体需要额外变换或二次过滤。

前置知识：

- 轴对齐包围盒（AABB）包含/相交判定。
- 均摊复杂度与最坏复杂度区别。

实现层面建议先写清楚不变量：
- 任一点应落在其祖先节点边界内。
- 非相交节点必须可被剪枝。

## R09

适用场景：

- 2D 地理对象粗筛、游戏可见性区域检索、UI 命中测试。
- 3D 点云局部检索、体素加速结构、场景碰撞前置筛选。

不适用或需谨慎：

- 数据维度继续升高（>3）时，分叉因子爆炸，结构收益下降。
- 数据持续高速动态更新且重建成本敏感时，可能需要 BVH、k-d tree 或网格哈希替代。

## R10

正确性可由两条不变量给出：

1. `insert` 不变量：每个成功插入点最终被存储在某个节点中，且该节点边界包含该点。
2. `range_query` 不变量：若节点与查询框不相交，则该节点及其后代不可能包含命中点。

因此查询只会漏掉“不相交子树中的点”，而这些点本就不在查询框内；相交子树中的点会被逐一判定并纳入结果，故结果与暴力筛选一致。

## R11

数值稳定性与边界策略：

- 本实现对包含判定使用闭区间 `<=`，保证边界点可被接纳。
- 在子节点分配时采用“顺序尝试 + 首个命中”策略，避免边界点重复插入多个子节点。
- 提供回退：若数值边界导致未进入任何子节点，点暂存当前节点，保证不丢点。
- 结果校验时用小数截断形成集合键，减少浮点显示误差影响。

## R12

关键参数与调优建议：

- `capacity`（叶节点容量）
  - 太小：树更深，节点与递归开销增加。
  - 太大：叶节点候选点过多，查询退化。
- `max_depth`（最大深度）
  - 太小：难以细分热点区域。
  - 太大：内存与遍历开销上升，病态数据下收益变差。

实践可先从：
- 2D：`capacity=8~32`，`max_depth=6~10`
- 3D：`capacity=16~64`，`max_depth=6~9`
再按命中率和访问节点数回归调参。

## R14

常见失效模式：

1. 根边界过小，导致部分点无法插入。
2. 忘记“分裂后重分配旧点”，造成查询漏点。
3. 查询只检查叶节点，忽略内部节点暂存点。
4. 相交判定写错（例如 `<` 与 `<=` 混淆）导致剪枝错误。
5. 参数设置不当导致退化（过深或过浅）。

防护方式：

- 构建后校验 `stored_points == n`。
- 每个查询都可与暴力法抽样比对。
- 打印访问节点比例，持续监控索引是否失效。

## R15

工程落地建议：

- 把 AABB 几何判定从树逻辑中解耦，便于单测。
- 明确 API 契约：`insert` 返回布尔值，`range_query` 返回命中点与访问节点数。
- 对动态场景预留“批量重建”开关；当点分布大改时，重建常优于局部修补。
- 对高并发读取场景可采用“构建后只读”模型，减少锁竞争。

## R16

相关结构与扩展方向：

- `k-d tree`：对最近邻查询常更强，尤其在中低维欧氏空间。
- `R-tree`：更适合矩形对象集合和磁盘索引。
- `BVH`：图形学射线追踪与碰撞检测常用层次包围结构。
- 稀疏体素八叉树（SVO）：在体渲染与 3D 地图压缩中常见。

## R18

`demo.py` 的源码级算法流（8 步）：

1. `AABB2D/AABB3D` 定义 `contains` 与 `intersects`，提供几何判定基础。  
2. `QuadtreeNode.insert` / `OctreeNode.insert` 先做边界检查，容量未满则直接存点。  
3. 超容量且未达深度上限时触发 `subdivide`，创建 4 或 8 个子节点。  
4. 分裂后把旧点逐个重新分配到子节点，保证“每点可达且不丢失”。  
5. `range_query` 递归前先做“节点-查询框相交测试”，不相交直接剪枝返回。  
6. 对相交节点检查当前节点持有点，再递归访问子节点并累计命中点。  
7. `run_quadtree_demo` 与 `run_octree_demo` 分别构造随机数据、建树、执行多组查询。  
8. 用 `brute_force_query_2d/3d` 做基线，并通过 `canonical_set` 比较结果集合一致性；全部通过后输出统计信息。  

这 8 步完整覆盖“建索引 -> 剪枝查询 -> 基线校验”的闭环，实现细节可直接在源码中逐函数追踪。
