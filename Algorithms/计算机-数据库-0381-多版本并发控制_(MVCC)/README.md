# 多版本并发控制 (MVCC)

- UID: `CS-0227`

## R01

多版本并发控制（MVCC, Multi-Version Concurrency Control）是一种通过“同一数据保留多个已提交版本”来提升并发能力的机制。

核心思想：

- 写事务提交时不覆盖旧值，而是生成新版本；
- 读事务按自己的快照时间点读取“对自己可见”的历史版本；
- 读写互不阻塞（在多数实现中），把冲突处理从“等待锁”转为“版本可见性判断 + 提交时冲突检查”。

## R02

MVCC 主要解决的问题：

- 在高并发读场景中降低读请求被写请求阻塞的概率；
- 让事务读取一致快照，避免脏读和中间状态暴露；
- 在保持隔离语义的同时，减少长时间持锁导致的吞吐下降。

本条目 MVP 采用“快照读 + 首提交者获胜（First-Committer-Wins）”的简化模型来演示核心行为。

## R03

本实现的数据模型：

- 对每个数据项 `X` 维护版本链 `[(commit_ts, value, writer), ...]`；
- 引导版本（初始值）统一记作 `commit_ts=0`；
- 每个事务 `T` 在 `BEGIN` 时拿到 `start_ts`，它定义了事务快照；
- `READ(X)` 选择 `commit_ts <= start_ts` 的最新版本；
- `WRITE(X, v)` 只写事务私有缓冲区，不立即改全局版本链。

## R04

提交规则（Snapshot Isolation 风格，简化版）：

1. 事务执行阶段：
   - 读按快照选可见版本；
   - 写仅进入 `write_buffer`。
2. 提交阶段：
   - 对每个待写键 `k`，取当前最新已提交版本时间戳 `latest_ts(k)`；
   - 若存在 `latest_ts(k) > start_ts(T)`，判定写写冲突，事务回滚；
   - 否则分配 `commit_ts`，把缓冲区中的键值作为新版本追加到版本链。

该规则等价于常见的“先提交者获胜”策略，可阻止并发写覆盖。

## R05

`demo.py` 提供两个确定性场景：

- 场景 A（快照稳定读）：
  - `T1` 先开始并读 `X=10`；
  - `T2` 更新 `X=20` 并提交；
  - `T1` 再读仍看到旧值 `10`（快照稳定），晚启动的 `T3` 读到 `20`。
- 场景 B（写写冲突 + 重试）：
  - `T1` 与 `T2` 并发写同一键 `X`；
  - `T2` 先提交，`T1` 提交时检测到冲突并回滚；
  - `T1` 自动重试并在新快照下成功提交。

## R06

复杂度分析（设单事务读写键数为 `K`，总操作数为 `N`）：

- 单次快照读：最坏 `O(Vk)`，`Vk` 为该键版本数；本实现从链尾逆序查找；
- 单次写缓冲：`O(1)`；
- 单次提交冲突检查：`O(K)`（每个写键检查最新版本时间戳）；
- 总体执行开销约 `O(N + total_commit_check)`。

空间复杂度由版本链主导：

- `O(total_versions)`，每次成功写提交都会新增版本。

## R09

适用场景：

- 读多写少或读密集型业务；
- 希望提升并发读吞吐、降低读阻塞；
- 需要快照一致读语义（分析查询、报表类读负载）。

不适用场景（仅针对本简化实现）：

- 需要完整可串行化而非快照隔离；
- 超高写热点且版本膨胀明显；
- 需要完整垃圾回收（GC/VACUUM）与长事务治理策略。

## R10

正确性直觉（本实现层面）：

1. 读操作只读已提交版本，且受 `start_ts` 限制，不会看到未来提交；
2. 事务私有写缓冲避免未提交写泄露；
3. 写写冲突在提交点统一判定，阻止“后提交覆盖先提交”；
4. 提交后版本以单调 `commit_ts` 追加，形成可重放历史。

因此，示例调度下可得到与快照隔离一致的可见性行为。

## R11

鲁棒性细节：

- 显式事务状态：`active / committed / aborted_retry / aborted_final`；
- 逻辑时钟同时驱动 `start_ts` 与 `commit_ts`，便于复现实验；
- 未出现的键会懒初始化为引导版本（`value=0, ts=0`）；
- 调度结束时若仍有未终止事务会抛错，防止“静默半完成”；
- 用断言检查关键语义（快照稳定读、冲突回滚、重试成功）。

## R12

关键参数与调优：

- `max_retries`：
  - 值大：冲突场景成功率更高，但尾延迟可能变大；
  - 值小：更快失败，便于上层补偿或降级。
- `schedule`：
  - 通过调度顺序控制冲突出现位置；
  - 教学和测试中可复现特定交错。
- 版本增长控制（工程扩展项）：
  - 可引入“最小活跃快照”来做旧版本回收。

## R13

理论性质说明：

- 本实现对应快照隔离常见子集，不保证完全可串行化；
- 近似比/随机化成功率：N/A（并发控制协议，不是近似优化问题）；
- 在有限调度长度和有限重试预算下，执行可终止；
- 若冲突持续，事务可能落入 `aborted_final`。

## R14

常见失效模式与边界：

1. 只做写写冲突检查，可能出现写偏斜（write skew）；
2. 无版本回收会导致历史版本无限增长；
3. 长事务会拉大可见性窗口，增加冲突和存储成本；
4. 若重试预算过低，热点键下事务成功率会下降。

排查建议：

- 检查日志中的 `snapshot_ts`、`commit_ts` 与冲突键；
- 先用两事务最小样例复现，再扩大调度规模；
- 明确业务是否需要 Serializable，而不是仅 Snapshot Isolation。

## R15

工程化落地建议：

- 把“版本存储”“可见性判断”“冲突检测”“调度器”拆分为独立模块；
- 增加指标：版本链长度分布、冲突率、平均重试次数、长事务占比；
- 引入后台 GC（依据最小活跃事务时间戳）；
- 对热点键可混合悲观锁策略，降低重试风暴。

## R16

与其他并发控制方案对比：

- 对比 2PL：
  - MVCC 读通常不阻塞写，吞吐更稳；
  - 2PL 逻辑更直观，但锁等待和死锁处理成本更高。
- 对比 OCC：
  - MVCC 读路径更友好（直接读快照版本）；
  - OCC 常在提交阶段集中验证，冲突时回滚成本可能更集中。
- 对比时间戳排序（TO）：
  - MVCC 通过多版本满足读可见性；
  - TO 常基于 `RTS/WTS` 直接拒绝逆序访问，版本管理相对简单。

## R18

`demo.py` 源码级算法流程（8 步）：

1. `run_workload` 按调度首次遇到事务名时调用 `MVCCEngine.begin`，分配 `snapshot_ts(start_ts)`。  
2. `execute_step` 按事务程序计数器推进单步 `R/W/C` 操作。  
3. `READ` 时，若键在本事务 `write_buffer`，优先读缓冲；否则在版本链中从新到旧找 `commit_ts <= start_ts` 的最新可见版本。  
4. `WRITE` 时仅写入 `write_buffer`，不立即修改全局版本链。  
5. `COMMIT` 时执行 `_has_write_write_conflict`：逐个待写键检查其最新已提交版本时间戳是否大于事务快照时间戳。  
6. 若存在冲突，事务标记 `aborted_retry`，记录冲突原因，不追加新版本。  
7. 若无冲突，分配新 `commit_ts`，把缓冲区写集逐键追加为新版本，并记入 `commit_log`。  
8. 调度器在后续步对 `aborted_retry` 事务按 `max_retries` 决定是否重启（新快照）；最终输出可见值、版本链、日志并通过断言校验。  

该流程完整展示了 MVCC 的可见性判定和提交冲突检测，没有依赖外部数据库黑盒接口。
