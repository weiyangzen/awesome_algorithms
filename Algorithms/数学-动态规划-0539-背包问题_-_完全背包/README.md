## R01

完全背包（Complete / Unbounded Knapsack）是经典动态规划问题：每种物品可取无限件，在容量不超过 `W` 的约束下最大化总价值。  
它和 0/1 背包的关键区别是“可重复选取同一物品”，这会直接改变状态转移方向。

## R02

形式化定义：
- 输入：`n` 种物品，第 `i` 种重量 `w_i > 0`、价值 `v_i >= 0`，背包容量 `W >= 0`。
- 决策变量：`x_i ∈ {0,1,2,...}`（可重复选）。
- 目标：
\[
\max \sum_{i=1}^{n} x_i v_i
\]
- 约束：
\[
\sum_{i=1}^{n} x_i w_i \le W.
\]

## R03

一维 DP 状态定义：`dp[c]` 表示容量恰为/不超过 `c` 时能取得的最大价值。  
完全背包递推：
\[
dp[c] = \max\big(dp[c],\ dp[c-w_i] + v_i\big),\quad c \in [w_i, W].
\]
因为 `dp[c-w_i]` 在同一轮 `i` 内已可包含物品 `i`，所以天然支持“无限次取第 i 种物品”。

## R04

迭代顺序是本题最容易错的点：
- 完全背包：容量 `c` 必须 **从小到大**（`w_i -> W`）。
- 0/1 背包：容量 `c` 通常 **从大到小**。

若把完全背包写成降序，会错误限制“同一轮只能取一次”，结果退化为 0/1 背包。

## R05

初始化与边界：
- `dp[0] = 0`，其余初值设为 `0`（本示例价值非负，初值 0 合理）。
- 容量 `W = 0` 时答案应为 `0`。
- 若某些物品重量都大于 `W`，答案保持 `0`。
- 要求 `w_i > 0`，否则可能出现无限取值导致模型失真。

## R06

为了输出“具体取了哪些物品”，`demo.py` 额外维护：
- `parent_item[c]`：达到 `dp[c]` 时最后加入的物品索引。
- `parent_capacity[c]`：上一个容量状态 `c - w_i`。

从 `capacity=W` 逆向回溯到 `0`，即可恢复一组最优解的物品件数向量 `counts`。

## R07

伪代码：

```text
input: items[(w_i, v_i)], capacity W
initialize dp[0..W] = 0
initialize parent_item[0..W] = -1, parent_capacity[0..W] = -1

for each item i:
    for c from w_i to W:   # ascending for complete knapsack
        candidate = dp[c - w_i] + v_i
        if candidate > dp[c]:
            dp[c] = candidate
            parent_item[c] = i
            parent_capacity[c] = c - w_i

reconstruct counts by tracing parent arrays from c = W
return dp[W], counts
```

## R08

正确性直观说明：
1. 最优子结构：若最优解在容量 `c` 最后使用了物品 `i`，其余部分必是容量 `c-w_i` 的最优解，否则可替换得到更优值。
2. 无后效性：`dp[c]` 只依赖更小容量状态，不依赖历史路径细节。
3. 升序容量保证可重复：同一物品处理阶段里，`dp[c-w_i]` 已包含“再选 i”的结果，因此可无限叠加。

## R09

复杂度：
- 时间复杂度：`O(nW)`。
- 空间复杂度：`O(W)`（主 DP 数组），加上回溯数组仍是 `O(W)` 级别。

这是完全背包最常用、工程上也最稳妥的基线实现。

## R10

常见错误与规避：
- 错把容量循环写成降序，导致结果变成 0/1 背包。
- 忘记检查 `weight <= 0`，会导致非法模型或死循环式解释。
- 只算最优值不做回溯，难以排查方案正确性。
- 未做小规模真值校验，DP 代码可能“看起来能跑”但逻辑有误。

本 MVP 提供 `brute_force_check` 做小规模穷举校验，防止静默错误。

## R11

`demo.py` 模块划分：
- `Item`：物品数据结构。
- `validate_inputs`：输入合法性校验。
- `solve_complete_knapsack`：一维 DP + 父指针记录 + 方案回溯。
- `brute_force_check`：小规模递归穷举最优值（仅用于校验）。
- `solution_summary`：把件数方案整理成 `pandas.DataFrame`。
- `main`：构造样例、运行算法、打印 DP 表与摘要、做一致性断言。

## R14

参数与可调项：
- 可替换 `items` 与 `capacity` 做不同规模实验。
- 若容量较大，`O(nW)` 运行时间线性增长，应按需求控制 `W`。
- `brute_force_check` 仅适合小规模验证；大规模时应关闭或限制样例。

## R15

与 0/1 背包对比：
- 0/1 背包：每种物品最多 1 件，容量循环降序。
- 完全背包：每种物品无限件，容量循环升序。
- 二者转移式看似相似，但循环方向不同，语义完全不同。

这也是动态规划入门中最关键的“同形异义”案例之一。

## R16

应用场景：
- 资源采购：物料可重复购买，预算/容量受限。
- 切割与配料：标准件可多次使用以最大化收益。
- 云资源配置：同规格实例可重复开通，约束总成本或配额。
- 金额凑整/硬币系统：硬币可重复使用，优化金额或价值目标。

## R17

可扩展方向：
- 多约束完全背包（体积、重量、预算等多维约束）。
- 分组 + 完全背包混合模型。
- 最优值之外再优化次级目标（如件数最少、剩余容量最小）。
- 把 `brute_force_check` 替换为随机测试对拍框架，做回归验证。

## R18

`demo.py` 源码级算法流程（8 步）：
1. `main` 构造固定样例 `items` 和 `capacity=17`，调用 `solve_complete_knapsack`。
2. `solve_complete_knapsack` 先用 `validate_inputs` 检查容量、物品非空、`weight>0`、`value>=0`。
3. 初始化 `dp` 为长度 `W+1` 的 0 向量，并初始化 `parent_item`、`parent_capacity` 为 `-1`。
4. 外层遍历每个物品 `i`，内层按 `c = w_i..W` 升序遍历容量，计算候选值 `dp[c-w_i] + v_i`。
5. 若候选值优于当前 `dp[c]`，就更新 `dp[c]`，并记录本次转移来源到父指针数组。
6. DP 结束后从 `c=W` 开始按父指针回溯，累计每种物品件数 `counts`，得到一组最优方案。
7. `main` 额外调用 `brute_force_check` 做小规模穷举真值，并调用 `solution_summary` 生成方案表。
8. 打印 DP 表、方案表和摘要指标；若“回溯值 != DP 最优值”或“DP != 穷举真值”则抛错，保证结果可信。
