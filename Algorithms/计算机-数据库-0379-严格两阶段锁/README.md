## R01

严格两阶段锁（Strict Two-Phase Locking, Strict 2PL）是并发控制中的经典协议。

核心目标：

- 通过锁保证冲突可串行化（conflict-serializable）；
- 通过“提交前不释放写相关锁”避免脏读/脏写；
- 让恢复流程更简单（未提交事务的影响不会暴露给其他事务）。

## R02

直觉上，2PL 把事务分成两个阶段：

- 扩展阶段（growing）：只加锁，不解锁；
- 收缩阶段（shrinking）：只解锁，不再加锁。

“严格”版本在工程里常实现为：事务在 `COMMIT/ABORT` 前不释放锁（本 MVP 使用这一更强实现，等价于常见教学中的严格做法），从而直接阻断读取未提交写入。

## R03

本条目要解决的问题：

- 输入：多个事务程序（`R/W/C` 操作序列）与初始数据库状态；
- 输出：调度执行日志、事务最终状态、提交后数据库值；
- 约束：遵守严格两阶段锁语义，遇到死锁可检测并回滚受害者。

`demo.py` 不依赖数据库内核，直接在 Python 源码中实现锁管理与调度。

## R04

本目录的实现策略：

1. 定义 `LockManager` 管理 `S/X` 锁、锁升级（`S->X`）与每个数据项的等待队列；
2. 调度器按轮询执行事务，每轮每个事务最多前进一步；
3. 读操作申请 `S` 锁，写操作申请 `X` 锁，拿不到锁则进入等待；
4. 提交时一次性应用写缓冲并释放全部锁（严格 2PL 的关键）；
5. 维护等待图（wait-for graph），检测环路后选择受害者回滚；
6. 回滚只丢弃该事务未提交写缓冲，不污染已提交状态。

## R05

复杂度分析（设事务数 `T`，锁请求总数 `L`，等待边数 `E`）：

- 单次加锁/解锁检查：均摊近似 `O(1)`（哈希表 + 小队列场景）；
- 一轮调度：`O(T)` 次事务尝试；
- 一次死锁检测（DFS）：`O(T + E)`；
- 总体取决于冲突强度、等待长度与回滚次数。

严格 2PL 更偏“正确性优先”，吞吐不一定最优。

## R06

`demo.py` 内置两个场景：

- 场景 1（无死锁）：
  - `T1` 先写 `A/B`，`T2` 尝试读 `A` 被阻塞，直到 `T1` 提交后才读到 `A=100`；
  - 用于验证“无脏读 + 提交后可见”。
- 场景 2（有死锁）：
  - `T1` 先锁 `X` 再要 `Y`，`T2` 先锁 `Y` 再要 `X`；
  - 等待图成环后触发死锁检测，回滚年轻事务。

## R07

优点：

- 能保证冲突可串行化；
- 严格语义天然避免脏读/脏写；
- 恢复逻辑清晰（未提交更改不外泄）。

局限：

- 容易产生等待与死锁；
- 长事务会放大锁持有时间；
- 在高并发读场景中可能弱于 MVCC 方案。

## R09

适用场景：

- OLTP 中对一致性与恢复可控性要求高；
- 写冲突不可忽略、需要强隔离语义；
- 教学/实验环境下演示并发控制核心机制。

不适用场景：

- 超高并发读多写少场景（MVCC 常更友好）；
- 极长事务或热点数据极高冲突场景；
- 对等待延迟极其敏感的业务路径。

## R10

正确性直觉（关键不变式）：

1. 事务提交前不释放持有锁，其他事务无法越过冲突锁读取未提交写入；
2. 每个冲突访问都被锁兼容规则序列化；
3. 提交时原子应用写缓冲，回滚时丢弃写缓冲；
4. 因此系统执行历史可映射到某个串行顺序（至少在冲突图意义上）。

## R11

鲁棒性实现细节：

- 等待队列去重，避免同事务重复排队；
- 支持 `S->X` 升级（仅当无其他冲突持有者）；
- 释放锁后立即尝试唤醒队列，保持进度；
- 若出现“无进展且无死锁环”直接抛错，防止静默卡住；
- 通过断言验证事务状态、最终值和关键日志顺序。

## R12

关键参数：

- `max_rounds`：调度轮数上限，防止无限循环；
- 事务顺序策略：本实现按事务编号轮询（确定性）；
- 受害者选择：死锁时回滚“编号更大（更年轻）”事务。

调参建议：

- 教学演示可用 2-4 个事务构造清晰冲突；
- 工程场景可替换成时间戳/代价驱动的受害者策略；
- 热点项多时可加入超时或优先级机制降低长等待。

## R14

常见失效模式：

1. 忘记在提交前持有关键锁，导致脏读；
2. 锁升级实现错误，可能出现“共享锁下并发写”；
3. 等待图构建不完整，导致死锁漏检；
4. 回滚后未清理队列/锁表，造成幽灵阻塞。

排查方式：

- 打印逐步事件日志（加锁、等待、唤醒、提交、回滚）；
- 对照兼容矩阵逐条核对冲突点；
- 用最小两事务死锁样例做回归测试。

## R15

- 把事件日志结构化（JSON）以支持观测与回放；
- 增加 metrics：平均等待时长、死锁率、回滚率；
- 引入意向锁、多粒度锁和锁升级策略以贴近真实数据库。

## R16

与其他并发控制方案对比：

- 基础 2PL：可串行化但不一定严格，恢复复杂度更高；
- 时间戳排序（TO）：无锁但可能频繁回滚；
- 乐观并发控制（OCC）：低冲突场景性能好，高冲突下验证失败率高；
- MVCC：读写分离友好，但版本管理与垃圾回收复杂。

Strict 2PL 的定位是“保守但稳健”的一致性方案。

## R18

`demo.py` 源码级算法流程（8 步）：

1. `LockManager.request_lock` 根据兼容矩阵判断是否可授予 `S/X` 锁，不可授予则入等待队列。  
2. 调度器 `run_workload` 轮询事务；每次 `execute_step` 只执行当前事务的一条操作。  
3. `R` 操作先申请 `S` 锁，成功后读取“自己的写缓冲优先，否则读已提交值”；失败则记为等待。  
4. `W` 操作先申请 `X` 锁，成功后只写入事务私有 `write_buffer`，不立即改动全局数据库。  
5. `C` 操作把 `write_buffer` 原子刷入 `committed_db`，随后 `release_all` 一次性释放该事务全部锁（严格 2PL 核心）。  
6. 释放锁后 `_drain_queue` 尝试按队列顺序授予后继请求，生成 “LOCK-GRANT after release” 事件。  
7. 每轮结束构建等待图 `build_wait_for_graph`，`find_cycle` 用 DFS 找环；若成环则 `choose_victim` 选受害者。  
8. `abort_transaction` 清空受害者写缓冲并释放其锁，让其余事务继续推进，直到全部 `committed/aborted`。  

以上流程把加锁、等待、提交、回滚、死锁处理全部显式展开为源码步骤，没有依赖外部数据库黑盒。
