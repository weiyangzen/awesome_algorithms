## R01

“为运算表达式设计优先级”对应的问题是：给定一个只含整数与二元运算符（`+`、`-`、`*`）的表达式，枚举所有可能的加括号方式，并返回每种方式的计算结果。

核心是“不同括号结构会改变运算顺序”，因此输出不是单个值，而是一个结果多重集（允许重复值）。

## R02

该问题常作为分治算法教学案例出现，也常见于在线题库（英文名常见为 `Different Ways to Add Parentheses`）。

历史语境上，它与以下经典主题直接相关：
- 全括号化（full parenthesization）与 Catalan 结构计数；
- 矩阵链乘法的“切分点”思想；
- 编译原理中的表达式语法树（AST）枚举。

## R03

形式化定义：
- 输入：字符串 `expr`，由非负整数和运算符 `+ - *` 组成，不含变量。
- 输出：列表 `R`，其中每个元素是对 `expr` 的一种合法全括号化求值结果。

例如：
- `expr = "2-1-1"`
- 合法括号化有两种：`(2-(1-1)) = 2`，`((2-1)-1) = 0`
- 输出可为 `[2, 0]`（顺序可不同）

## R04

分治递推思想：
- 对任一子表达式区间 `[l, r]`，若只含一个数字，则结果集合就是该数字本身。
- 否则枚举每个运算符位置 `k` 作为“最后一次计算”的切分点：
  - 左边 `[l, k]` 递归求所有值；
  - 右边 `[k+1, r]` 递归求所有值；
  - 对左右结果做笛卡尔积并应用运算符。

这正是典型“分割点枚举 + 子问题组合”的分治框架。

## R05

MVP 中的关键状态：
- `nums`: 表达式中的数字序列。
- `ops`: 运算符序列，长度为 `len(nums)-1`。
- `solve(i, j)`: 返回子区间 `nums[i..j]` 对应子表达式的全部结果。
- `memo[(i, j)]`: 记忆化缓存，避免重复计算同一子区间。
- `stats`: 记录递归调用、缓存命中、结果组合次数。

## R06

示例 `2*3-4*5`：

1. 运算符切分点可在 `*`、`-`、`*` 三个位置。
2. 以中间 `-` 切分时：
   - 左子式 `2*3 -> [6]`
   - 右子式 `4*5 -> [20]`
   - 合并得 `6-20=-14`
3. 以左侧 `*` 切分时：
   - 左子式 `2 -> [2]`
   - 右子式 `3-4*5` 会递归产生多个值
4. 以右侧 `*` 切分同理。
5. 汇总可得到 `[-34, -14, -10, -10, 10]`（含重复值，代表不同括号结构）。

## R07

该算法的价值：
- 理论上：直观展示分治与“结构枚举”问题的结合。
- 工程上：可扩展到表达式搜索、程序合成、符号计算中的候选树生成。
- 教学上：非常适合讲解“为什么要记忆化”以及“重复子问题”。

## R08

前置知识：
- 递归与分治基本模式。
- 字符串解析（把表达式拆成数字与运算符）。
- 笛卡尔积组合与复杂度概念。
- 动态规划/记忆化搜索思想。

## R09

复杂度（令数字个数为 `n`）：
- 不带记忆化时，重复子问题很多，时间开销接近指数级。
- 带记忆化后，子区间状态数为 `O(n^2)`；但每个状态要做结果集合组合，整体仍可能很大。
- 输出规模本身与括号化数量相关，数量级与 Catalan 数同阶，故总时间下界受“必须生成全部结果”限制。
- 额外空间来自缓存与结果集合，通常也随输出规模增长。

## R10

正确性要点（归纳）：

1. 基础情形：`i == j` 时子表达式仅一个数字，返回该数字，正确。
2. 归纳假设：任意更短区间 `solve` 都能返回其全部合法结果。
3. 对区间 `[i, j]`，任一合法全括号化必有唯一“最后运算符”切分点 `k`。
4. 左右子树结果分别由 `solve(i, k)`、`solve(k+1, j)` 完整覆盖。
5. 枚举所有 `k` 并做左右结果笛卡尔积，恰好覆盖且不遗漏所有括号化结果。

因此算法返回值与问题定义一致。

## R11

数值与语义稳定性：
- 本题在整数域做 `+ - *`，无浮点舍入误差传播问题。
- 主要风险在语义约束：
  - 若支持一元负号却未正确词法处理，会误分割表达式；
  - 若结果去重会丢失“不同括号结构得到相同数值”的信息。

本目录 MVP 采用“多重集语义”（保留重复结果）。

## R12

工程成本（超越大 O）：
- 子问题结果列表在组合时会产生大量中间对象；
- Python 递归与列表扩展有明显常数开销；
- 记忆化可显著减少重复计算，但内存占用会增加；
- 输入稍大时，瓶颈通常在“输出爆炸”，而不是单次运算符计算。

## R14

常见失效模式：
- 解析错误：未正确处理多位整数或空白字符。
- 边界错误：表达式开头/结尾是运算符时未拦截。
- 语义偏差：误把结果去重，导致与题意不一致。
- 性能问题：未做记忆化导致重复递归爆炸。

## R15

实现建议：
- 先做词法拆分，得到 `nums` 和 `ops` 再递归，避免在递归中重复解析字符串。
- 用 `(i, j)` 做缓存键，保证子问题唯一。
- 保持结果列表不去重，必要时在展示阶段再统计频次。
- 测试至少覆盖：
  - 单数字表达式；
  - 多位数字；
  - 典型样例 `2*3-4*5`；
  - 随机表达式与无记忆化版本对拍。

## R18

`demo.py` 源码级流程（8 步）：

1. `main()` 先运行固定样例校验，再做随机对拍和小基准，整个过程无交互输入。  
2. `tokenize_expression(expr)` 扫描字符串，把多位数字解析到 `nums`，把 `+ - *` 解析到 `ops`，并校验表达式合法性。  
3. `different_ways_to_compute` 初始化 `memo` 与 `DivideConquerStats`，定义递归函数 `solve(i, j)`。  
4. `solve(i, j)` 先查缓存；若命中直接返回，避免重复展开同一子区间。  
5. 若 `i == j`，返回 `[nums[i]]` 作为递归基。  
6. 否则枚举每个切分点 `k`：递归得到左结果集与右结果集，调用 `apply_op` 做笛卡尔积组合并累计到当前区间结果。  
7. 当前区间结果写入 `memo[(i, j)]` 后返回，逐层汇总到顶层得到表达式全部结果（含重复值）。  
8. `run_fixed_cases` 与 `run_random_regression` 用无缓存基线对拍；`run_benchmark` 再用 `pandas.DataFrame` 输出 memo 与 naive 的耗时对比表。
