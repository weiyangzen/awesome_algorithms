## R01

归并排序（Merge Sort）是一种典型的分治排序算法，核心思想是“先分后治再合并”。

- 输入: 一个可比较元素序列 `A`。
- 输出: 按非降序排列的新序列。
- 分治结构:
  1. 将序列二分为左右两半。
  2. 递归地分别排序左右子序列。
  3. 线性合并两个有序子序列。
- 关键性质: 稳定排序（当比较结果相等时，原相对顺序不变）。

## R02

- 主要提出者: John von Neumann。
- 时间: 1945 年前后在 EDVAC 报告相关工作中系统描述。
- 历史意义: 是最早被广泛分析并在工程中落地的 `O(n log n)` 比较排序之一，对后续外部排序与并行排序影响深远。

## R03

朴素排序（如冒泡、选择、插入）在平均或最坏情况下通常是 `O(n^2)`，数据规模稍大时开销迅速上升。归并排序通过分治把排序问题拆成更小子问题，并在合并阶段只做线性扫描，从而把整体复杂度压到 `O(n log n)`，特别适合以下场景:

- 需要稳定排序（例如按次关键字排序后再按主关键字排序）。
- 链表排序（可做到接近原地、合并高效）。
- 外部排序（磁盘/网络分块排序后多路归并）。

## R04

时间复杂度（长度为 `n`）:

- 递推关系: `T(n) = 2T(n/2) + O(n)`。
- 最好/平均/最坏: 均为 `O(n log n)`。
- 结论: 复杂度稳定，不会退化到 `O(n^2)`。

这也是归并排序相对快速排序（最坏可能退化）的一大优势。

## R05

空间复杂度:

- 经典数组版（本目录 demo 采用）: 需要辅助数组，额外空间 `O(n)`。
- 递归调用栈: `O(log n)`。
- 链表版可把额外数组空间降到 `O(1)`（不计递归栈）。

总的工程观感是“时间稳、空间换时间”。

## R06

示例: 对 `[38, 27, 43, 3, 9, 82, 10]` 排序。

1. 拆分为 `[38, 27, 43]` 与 `[3, 9, 82, 10]`。
2. 左侧继续拆分并排序得到 `[27, 38, 43]`。
3. 右侧继续拆分并排序得到 `[3, 9, 10, 82]`。
4. 合并两个有序序列:
   - 比较 `27` 与 `3`，取 `3`
   - 比较 `27` 与 `9`，取 `9`
   - 比较 `27` 与 `10`，取 `10`
   - 比较 `27` 与 `82`，取 `27`
   - 比较 `38` 与 `82`，取 `38`
   - 比较 `43` 与 `82`，取 `43`
   - 追加剩余 `82`
5. 结果: `[3, 9, 10, 27, 38, 43, 82]`。

## R07

归并排序的核心价值:

- 理论层: 经典分治范式，便于复杂度证明与教学。
- 算法层: 稳定 `O(n log n)`，性能上界可靠。
- 工程层: 是外部排序和分布式排序中“归并阶段”的基础操作。
- 系统层: 便于并行化（子问题可并发排序，最后归并）。

## R08

前置知识与依赖:

- 比较模型排序基础（元素间全序或可比较）。
- 分治思想与递归。
- 线性时间有序合并过程。
- 复杂度分析工具（递推式、主定理/递归树）。

## R09

适用条件与边界:

- 输入元素需要可比较；若是对象，需提供 `key`。
- 适合对最坏复杂度有明确要求的场景。
- 对内存非常紧张的数组场景不一定最优（额外 `O(n)` 空间）。
- 若只追求平均常数性能、且可接受最坏退化风险，快速排序可能更快。

## R10

正确性可由归纳与循环不变式说明:

1. 基础情形: 长度 `0/1` 的序列天然有序。
2. 归纳假设: 递归调用能正确返回左右子序列的有序结果。
3. 合并不变式: 合并时维护“输出前缀始终有序，且由左右已消费部分的最小元素构成”。
4. 终止时左右元素全部输出，得到整体有序序列。
5. 因每层仅做正确合并，整体排序正确。

## R11

数值稳定性与误差传播:

- 归并排序是比较排序，不涉及浮点迭代误差传播模型。
- 主要风险不在“数值不稳定”，而在“比较器定义不一致”:
  - 若比较关系非传递，排序结果可能不符合预期。
  - 若 `key` 函数有副作用，可能导致不可复现行为。
- 稳定性由实现策略保证: 合并时相等元素优先取左序列。

## R12

真实工程成本（超越大 O）:

- 常数项来源:
  - 辅助数组分配与拷贝成本。
  - 递归函数调用开销。
- 缓存局部性:
  - 合并阶段线性扫描通常较友好。
- 并行潜力:
  - 两个子问题可并行，但最终归并仍是关键同步点。
- 外存场景:
  - 多路归并可显著减少随机 I/O，是归并思想在工程中的关键延展。

- 近似比/概率保证: N/A。
- 原因: 归并排序是确定性精确排序算法，不是随机近似算法。

## R14

鲁棒性与常见失效模式:

- 递归深度问题: 极大输入在某些语言可能触发栈限制（Python 可考虑迭代版）。
- 辅助空间压力: 大规模数据可能因 `O(n)` 额外内存触发内存瓶颈。
- 非法比较器: 若 `key` 返回不可比较对象会抛异常。
- 稳定性被破坏: 如果合并时把“相等时取右”写错，会失去稳定排序性质。

## R15

实现建议:

- 明确 API 语义: 返回新列表还是原地排序。
- 合并时使用双指针并在“相等时优先左侧”保证稳定性。
- 对小数组可切换到插入排序以改善常数（可选优化）。
- 测试覆盖至少包含:
  - 空数组/单元素
  - 重复值
  - 已排序与逆序
  - 自定义 `key` 与稳定性验证

## R17

本目录 `demo.py` 的 Python MVP 设计:

- 依赖: Python 标准库（`typing`、`random`、`time`），无第三方依赖。
- 核心函数:
  - `merge(left, right, key)`: 线性合并两个有序列表，保持稳定性。

  - 小规模性能对照（`merge_sort` vs `sorted`）。
- 运行方式:
  - `uv run python demo.py`

## R18

源码级流程拆解（对应 `demo.py`，不依赖第三方黑箱）:

1. `main()` 准备固定样例与随机样例，并调用 `merge_sort`。
2. `merge_sort` 先把输入转为列表；若长度 `<= 1` 直接返回（递归基）。
3. 否则按中点切分为左右子序列，分别递归调用 `merge_sort`。
4. 两个递归分支返回后，左右列表已经各自有序。
5. 调用 `merge(left_sorted, right_sorted, key)` 进入双指针合并。
6. 在 `merge` 中比较 `key(left[i])` 与 `key(right[j])`，较小者追加到结果；相等时优先左侧以保证稳定性。
7. 某一侧耗尽后，将另一侧剩余元素整体追加，返回最终有序结果给上一层，直至顶层返回完整排序序列。
