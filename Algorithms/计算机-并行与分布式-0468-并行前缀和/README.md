## R01

并行前缀和（Parallel Prefix Sum / Parallel Scan）用于计算序列的累积和。  
给定输入 `x0, x1, ..., xn-1`，其包含型前缀和定义为：

`y[i] = x0 + x1 + ... + xi`

该问题是并行算法里的经典基础模块，常作为更复杂并行任务（排序、图算法、流处理聚合）的子步骤。

## R02

为什么它重要：
- 前缀和本身是数据并行任务，天然适合 GPU/SIMD/多核架构；
- 许多并行算法都要先做“局部统计 + 全局偏移修正”，其核心即扫描操作；
- 在大规模数据处理中，scan 经常出现在“索引构建、桶边界计算、稀疏压缩”等流程。

换句话说，掌握并行前缀和，基本等于掌握一类并行程序的通用骨架。

## R03

本条目给出一个可运行的 Python MVP，核心算法是 **Hillis-Steele 并行扫描**：
1. 初始化输出数组 `out = input`；
2. 每一轮步长 `offset` 翻倍（`1, 2, 4, ...`）；
3. 该轮中所有满足 `i >= offset` 的位置并行更新：
   `out[i] = prev[i] + prev[i - offset]`；
4. 循环直到 `offset >= n`。

代码中显式实现上述流程，不把 `numpy.cumsum` 作为核心黑盒。

## R04

本实现选择“**包含型**前缀和（inclusive scan）”。

- 包含型：`y[i]` 包含 `x[i]`；
- 排他型（exclusive）可由包含型平移得到：
  `exclusive[0] = 0`，`exclusive[i] = inclusive[i-1]`（`i > 0`）。

在工程中，两种定义都常见，重点是接口语义要明确。

## R05

复杂度分析（Hillis-Steele）：
- 总工作量（work）：`O(n log n)`；
- 并行深度（span）：`O(log n)`；
- 额外空间：每轮保留 `prev` 快照，单轮 `O(n)`。

与顺序扫描 `O(n)` 工作量相比，它牺牲一些总运算，换取低并行深度，这正是并行算法的典型折中。

## R09

适用场景：
- 并行算法教学：理解“步长翻倍”式扫描；
- 需要可复现的小型实验；
- 作为后续并行算法（桶划分、压缩写入索引）的基础组件。

不适用场景：
- 只追求 CPU 上极致吞吐（应考虑更底层并行实现）；
- 数据规模很小（并行结构优势不明显）；
- 强内存受限场景（该实现每轮复制数组）。

## R10

正确性直觉：
1. 第 `k` 轮结束后，每个位置 `i` 都聚合了长度 `2^k` 范围内可达的历史贡献；
2. 经过 `ceil(log2(n))` 轮后，`i` 能覆盖其全部前缀区间 `[0, i]`；
3. 因此输出即包含型前缀和。

`demo.py` 既做结构性断言（相邻差分还原原数组），也做结果对照断言（与顺序版和 `numpy.cumsum` 完全一致）。

## R11

边界处理：
- 空数组：直接返回空副本；
- 单元素：直接返回自身；
- 负数与混合符号：允许前缀和非单调；
- 通过 `dtype=np.int64` 降低常见整数溢出风险（但极端大数仍需业务侧约束）。

## R13

性能观察建议：
- 扩大 `n` 后，Hillis-Steele 的并行轮次优势更直观；
- 数据类型会影响速度和内存占用；
- `numpy` 后端/硬件向量能力会影响每轮切片加法效率；
- 该 MVP 重点是算法透明性，不追求最优常数。

## R14

常见错误：
1. 原地直接更新且不保留 `prev`，会污染同轮尚未计算的位置；
2. 步长不翻倍（或翻倍逻辑写错），导致收敛失败；
3. 忘记处理空数组边界；
4. 把 `numpy.cumsum` 当主算法而无法解释源码级流程；
5. 将“前缀和必须单调”误当断言（含负数时不成立）。

## R15

可扩展方向：
- 改写为 Blelloch（up-sweep/down-sweep）扫描，降低总工作量到 `O(n)`；
- 加入分块多进程版本（局部前缀 + 块偏移修正）；
- 迁移到 GPU（如 CUDA）并比较吞吐；
- 支持其它结合律运算（`max`、`min`、逻辑运算等通用 scan）。

## R18

`demo.py` 源码级算法流程（8 步）：
1. `main` 调用 `_self_test`，用空数组、重复值、负数混合数组校验顺序版与并行版结果。  
2. `main` 固定随机种子生成输入数组 `data`，打印规模 `n` 与预计轮次 `ceil(log2(n))`。  
3. 顺序路径进入 `sequential_prefix_sum`：维护 `running`，逐元素累加并写入输出。  
4. 并行路径进入 `hillis_steele_prefix_sum`：初始化 `out = input`，设置 `offset = 1`。  
5. 每轮先复制 `prev = out.copy()`，再对切片执行同轮并行更新：`out[offset:] = prev[offset:] + prev[:-offset]`。  
6. `offset <<= 1` 后继续下一轮，直到 `offset >= n`，得到包含型前缀和结果。  
7. 结果检查阶段调用 `_assert_prefix_sum(data, hs)`，验证 `prefix[0]=x0` 且 `prefix[i]-prefix[i-1]=x[i]`。  
8. 最后将 `hillis_steele_prefix_sum` 与 `sequential_prefix_sum`、`numpy.cumsum` 做逐元素一致性比较并输出耗时。  

以上 8 步完整对应源码中的函数调用和数组更新逻辑，核心并行扫描过程可直接追踪。
