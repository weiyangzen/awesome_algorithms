## R01

排序合并连接（Sort-Merge Join）是数据库等值连接中的经典算法。

核心思想：

- 先按连接键对左右两侧数据排序；
- 再用两个指针线性扫描；
- 键小的一侧前进；
- 键相等时按“键分组”做笛卡尔配对并输出。

它把连接问题从“全量两两比较”转化为“有序流对齐”。

## R02

算法直觉：利用有序性避免无效比较。

- 若 `left_key < right_key`，当前左行不可能再匹配未来更大的右键；
- 若 `left_key > right_key`，同理当前右行可直接推进；
- 只有键相等时才产生结果。

因此，排序后主循环是近似线性的扫描过程，通常优于朴素嵌套循环连接。

## R03

本条目要解决的问题：

- 输入：左表、右表（`list[dict]`），连接键（单列或多列），连接类型；
- 输出：前缀化后的连接结果行与执行统计信息。

`demo.py` 支持：

- `inner`（内连接）
- `left`（左外连接）
- `right`（右外连接）
- `full`（全外连接）

## R04

本目录实现策略：

1. 统一连接键规格（字符串或列名序列）；
2. 提取每行连接键并转为“可比较的规范键”（处理 `None`、数值、字符串等类型）；
3. 分别对左右表按规范键排序；
4. 双指针线性扫描，按 `<`、`>`、`==` 三种关系推进；
5. 键相等时提取左右键分组并输出分组间笛卡尔积；
6. 对外连接补齐未匹配侧 `NULL` 模板；
7. 返回结果行、键比较次数、匹配对数。

## R05

复杂度分析（左表 `N` 行，右表 `M` 行）：

- 排序阶段：`O(N log N + M log M)`；
- 合并扫描阶段：`O(N + M + K)`，`K` 为相等键分组产出的匹配对数；
- 总体时间复杂度：`O(N log N + M log M + K)`；
- 额外空间：`O(N + M)`（排序准备与结果输出之外的中间结构）。

当输入已按连接键有序时，可省去排序成本，主流程接近线性。

## R06

`demo.py` 的样例数据特意覆盖真实边界：

- 输入是**无序**的（验证“先排序再连接”）；
- 存在重复键（`101`、`102`）验证 many-to-many 输出；

这样可以一次性验证 `inner/left/right/full` 的语义。

## R07

优点：

- 对已排序数据非常高效；
- 可自然支持流式顺序读（尤其外存场景）；
- 不依赖哈希桶，内存峰值可控。

局限：

- 若输入无序，排序成本可能成为瓶颈；
- 对大量重复键，分组笛卡尔积会放大输出；
- 非等值连接通常不如专门算法直观。

## R09

适用场景：

- 双方数据已按键排序，或排序代价可接受；
- 需要稳定、可解释的连接流程；
- 数据仓库中按主键/外键批处理连接。

不适用场景：

- 小表与大表随机访问且可直接命中索引（索引 NLJ 更合适）；
- 高基数无序数据且内存充足（哈希连接常更快）；
- 连接条件非等值且高度复杂。

## R10

正确性直觉（不变式）：

1. 指针 `i, j` 前面的行要么已匹配输出，要么被证明不可能再匹配；
2. 当 `left_key < right_key` 时，当前左行与右侧未来键不可能相等；
3. 当 `left_key > right_key` 时，对称结论成立；
4. 当键相等时，必须完整输出该键的左右分组笛卡尔积，才能不漏解。

因此扫描结束后，结果恰好满足连接定义，不多不少。

## R11

鲁棒性实现细节：

- `normalize_key_spec` 约束连接键非空且结构统一；
- `canonicalize_value` 把不同类型映射到可比较秩，避免 Python3 跨类型比较异常；
- 缺失列通过 `row.get()` 读取，避免 `KeyError`；
- 输出统一加 `l_` / `r_` 前缀，避免同名列冲突；
- 通过断言校验输出规模与未匹配计数，防止语义回归。

## R12

关键参数：

- `left_on` / `right_on`：连接键（支持单列或多列）；
- `join_type`：`inner|left|right|full`；
- 键分布：重复键越集中，分组产出 `K` 越大。

调优建议：

- 已有索引或有序存储时优先复用排序结果；
- 高重复键场景提前估计输出规模；
- 复合键需确保左右字段顺序完全一致。

## R14

常见失效模式：

1. 左右连接键字段对应错位；
2. 键类型混杂且未做规范化（例如 `"101"` 与 `101`）；
3. 外连接漏补 `NULL`，导致结果行数偏小；
4. 重复键分组未完整展开，导致少算匹配对。

排查方法：

- 打印排序后键序列与分组边界；
- 人工手算少量样例并与程序对比；
- 验证 `inner + unmatched_left + unmatched_right == full`。

- 把“键规范化”独立成可复用模块；
- 在 CI 覆盖空表、全不匹配、全匹配、重复键爆发等测试。

## R16

与其他连接算法对比：

- 嵌套循环连接（NLJ）：通用但通常 `O(N*M)`；
- 哈希连接（Hash Join）：无序等值连接常见高性能方案；
- 排序合并连接（本条目）：排序后线性扫描，适合有序数据；
- 索引嵌套循环连接：在内表索引命中良好时表现突出。

本目录聚焦 Sort-Merge Join 的源码级最小实现。

## R18

`demo.py` 源码级算法流程（8 步）：

1. `normalize_key_spec` 把连接键定义规整成元组，确保左右键长度一致。  
2. `extract_key` 从每行提取原始键值，`canonicalize_value/key` 将键转为可稳定比较的规范形式。  
3. `prepare_sorted_rows` 对左右输入构造 `(row, canonical_key)` 并按键排序。  
4. 初始化双指针 `i, j`，进入主循环，逐次比较当前左右键并累计 `key_comparisons`。  
5. 若 `left_key < right_key`，当前左行不可能再匹配；在 `left/full` 下输出左行+右侧 `NULL`，然后 `i += 1`。  
6. 若 `left_key > right_key`，对称处理右行；在 `right/full` 下输出左侧 `NULL`+右行，然后 `j += 1`。  
7. 若键相等，向前扩展得到左右同键分组 `[i, i_end)`、`[j, j_end)`，输出两组笛卡尔积并累加 `matched_pairs`，随后将指针跳到分组末端。  
8. 主循环结束后，按连接类型补齐剩余未匹配尾部，返回 `JoinResult`；`main()` 再用断言验证行数、未匹配数量与比较次数。

该实现把排序、分组、指针推进与外连接补齐全部显式写在源码中，可逐行审计算法行为。
