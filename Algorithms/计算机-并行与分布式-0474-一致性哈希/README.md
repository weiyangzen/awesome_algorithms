## R01

一致性哈希（Consistent Hashing）是一种用于分布式节点映射的哈希策略，核心目标是：
- 节点数量变化时，尽量减少已有键的重映射；
- 在多节点之间保持较均衡的键分布；
- 通过虚拟节点降低物理节点负载倾斜。

它常用于分布式缓存、对象存储分片、分布式 KV 路由等场景。

## R02

问题定义（MVP 范围）：
- 输入：
1. 物理节点集合（初始 `cache-a/cache-b/cache-c`）；
2. 每个物理节点的虚拟节点数（`replicas_per_node=128`）；
3. 键集合（`user-00000` 到 `user-05999` 共 6000 个）。
- 输出：
1. 3 节点时的键分布统计；
2. 新增 `cache-d` 后的键分布与迁移率；
3. 删除 `cache-b` 后的键分布与迁移率；
4. 关键正确性断言结果。

## R03

本实现覆盖的一致性哈希核心语义：
- 哈希环：把键和虚拟节点都映射到同一有序整数环；
- 顺时针命中：键命中其哈希位置顺时针遇到的第一个节点；
- 虚拟节点：单个物理节点映射多个 token，改善负载均衡；
- 动态变更：`add_node` 与 `remove_node` 仅影响局部区间，不会全量洗牌。

## R04

`demo.py` 高层执行流程：
1. 初始化 `ConsistentHashRing(replicas_per_node=128)`；
2. 添加 `cache-a/cache-b/cache-c` 并映射 6000 个键；
3. 统计 3 节点分布；
4. 添加 `cache-d`，重映射并计算新增节点场景迁移率；
5. 删除 `cache-b`，重映射并计算删节点场景迁移率；
6. 对“迁移键集合”做精确断言（新增节点/被删除节点对应键集）；
7. 输出分布表、迁移率、负载均衡指标（CV）；
8. 输出最终通过结论。

## R05

关键数据结构：
- `positions: list[int]`：环上虚拟节点位置（升序）；
- `owners: list[str]`：与 `positions` 对齐的物理节点名；
- `node_positions: dict[str, list[int]]`：每个物理节点拥有的全部虚拟点，便于删除；
- `mapping: dict[str, str]`：键到节点的映射结果；
- `pandas.DataFrame`：按节点统计键数量与比例。

## R06

正确性直觉：
- 顺时针规则保证每个键唯一命中一个节点；
- 新增一个节点时，只会“截获”它前驱区间内的键，未被截获键保持不变；
- 删除一个节点时，只有原本落到该节点的键迁移到后继节点，其他键不变；
- 虚拟节点增加后，节点在环上分布更细，负载倾斜会明显降低。

## R07

复杂度分析：设
- 物理节点数为 `N`；
- 每节点虚拟节点数为 `V`；
- 环总点数 `M = N * V`；
- 键数量为 `K`。

则：
- `add_node`：插入 `V` 个点，每次二分定位 + 列表插入，约 `O(V * M)`（列表插入主导）；
- `remove_node`：删除 `V` 个点，约 `O(V * M)`；
- `get_node`：二分查找 `O(log M)`；
- `assign_keys`：`K` 次查询，总计 `O(K log M)`；
- 统计分布：`O(K)`。

对于本 MVP（`N<=4, V=128, K=6000`）开销很小，可直接本地秒级运行。

## R08

边界与异常处理：
- `replicas_per_node <= 0`：抛 `ValueError`；
- 空环调用 `get_node`：抛 `ValueError`；
- 重复添加同名节点：抛 `ValueError`；
- 删除不存在节点：抛 `ValueError`；
- 删除时若内部位置不一致：抛 `RuntimeError`（防止环结构损坏被静默忽略）；
- 哈希碰撞时采用 `salt` 重试，确保虚拟点唯一可插入。

## R09

MVP 设计取舍：
- 选用“可读源码实现”而非直接依赖现成一致性哈希库，避免黑箱；
- 使用 `md5` 仅为确定性与可复现，不用于安全语义；
- 选择单进程仿真路由，不涉及网络与真实节点通信，聚焦算法机制；
- 用固定键集和断言输出，优先可验证性与稳定复现。

## R10

`demo.py` 函数/类职责：
- `ConsistentHashRing`：维护哈希环及节点增删；
- `_hash_to_int`：把字符串映射到整数环位置；
- `add_node`：为物理节点添加多个虚拟点；
- `remove_node`：删除节点对应全部虚拟点；
- `get_node`：查询单个键应落在哪个节点；
- `assign_keys`：批量键映射；
- `distribution_df`：统计每节点键数量与占比；
- `moved_ratio`：计算两次映射间键迁移比例；
- `main`：组织实验、执行断言、打印结果。

## R13

最小验证清单：
- 新增节点后，“发生迁移的键集合”必须等于“当前落到新节点的键集合”；
- 删除节点后，“发生迁移的键集合”必须等于“删除前落到该节点的键集合”；
- 删除后结果中不允许再出现被删节点；
- 扩缩容迁移率应在合理区间（本示例断言 `15%~35%`）；
- 分布统计输出完整且与总键数一致。

## R15

与相关方案差异：
- 相比取模哈希 `hash(key) % N`：一致性哈希在节点变化时迁移量显著更小；
- 相比无虚拟节点的一致性哈希：本实现加入虚拟节点，负载更平滑；
- 相比 Rendezvous Hashing：两者都支持较低迁移，本条目聚焦“哈希环 + 顺时针命中”机制；
- 相比真实分布式服务路由：本实现不含网络故障、副本一致性、重试等工程细节。

## R16

适用场景：
- 分布式缓存分片（节点动态扩缩容）；
- 对象存储或 KV 存储的路由层；
- 教学或原型阶段验证“一致性哈希迁移最小化”特性。

不适用场景：
- 需要强一致副本协议（如 Raft/Paxos）的写入仲裁；
- 需要跨机房容灾、网络分区恢复等完整系统仿真；
- 需要高性能并发路由基准测试（该脚本是算法演示，不是压测工具）。

## R17

可扩展方向：
- 增加多副本选择（同键映射多个不同节点）；
- 支持节点权重（不同节点虚拟节点数可配置）；

## R18

`demo.py` 源码级算法流（8 步，非黑箱）：
1. `main` 构造 `ConsistentHashRing(128)`，并添加 `cache-a/cache-b/cache-c`。  
2. `add_node` 为每个物理节点生成 `128` 个虚拟点：`node#replica#salt`，经 `_hash_to_int` 得到环位置并有序插入。  
3. `assign_keys` 对每个键调用 `get_node`：先算键哈希，再用 `bisect_right` 在有序环中找顺时针第一个节点。  
4. `distribution_df` 统计三节点阶段每节点键数量和比例。  
5. 扩容时再次调用 `add_node("cache-d")`，重算映射；`moved_ratio` 统计扩容迁移率，并断言“迁移键集合 == 新节点当前键集合”。  
6. 缩容时调用 `remove_node("cache-b")` 删除该节点全部虚拟点，重算映射并统计迁移率。  
7. 断言“缩容迁移键集合 == 缩容前属于 `cache-b` 的键集合”，并断言缩容后映射结果不再包含 `cache-b`。  
8. 计算各阶段负载 CV（`std/mean`），打印分布表、迁移率和最终通过结论。  

该实现将哈希环维护、查找、扩缩容迁移验证都写在源码里，没有把关键步骤外包给不可见第三方黑箱。
