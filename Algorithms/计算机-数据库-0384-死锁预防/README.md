## R01

死锁预防（Deadlock Prevention）是数据库并发控制中的一类策略：

- 目标不是“发生后检测并恢复”，而是“从规则上让死锁根本无法形成”；
- 常见实现包括资源有序分配、`wait-die`、`wound-wait` 等；
- 本条目采用经典时间戳协议 `wait-die` 作为可执行 MVP。

一句话概括：通过强制等待关系满足单调约束，让等待图天然无环。

## R02

为什么会死锁？因为并发事务形成了循环等待，例如：

- `T1` 持有 `A` 等 `B`；
- `T2` 持有 `B` 等 `A`。

`wait-die` 的核心直觉是“只允许老事务等待年轻事务”，反向不允许：

- 若请求者更老：可以等待；
- 若请求者更年轻：立即回滚重启（die）。

这样等待边只能从“老 -> 青年”，时间戳严格单调，不可能闭环。

## R03

本条目要解决的问题：

- 输入：若干事务程序（加锁、工作、解锁、提交）；
- 输出：一个可运行的并发调度器，使用 `wait-die` 预防死锁；
- 验证目标：在典型“交叉加锁”场景中，无死锁，最终全部事务可完成。

`demo.py` 使用纯 Python 内存结构，不依赖数据库引擎，便于追踪算法细节。

## R04

MVP 的核心数据结构：

1. `lock_owner[resource] -> tx_id`：记录每个资源的当前持有者（简化为排它锁）；
2. `waiting_queues[resource] -> deque[tx_id]`：按 FIFO 记录等待该资源的事务；
3. `Transaction`：保存时间戳、程序计数器、持锁集合、等待状态、重启次数；
4. `event_log`：记录授锁、等待、回滚、唤醒、提交等关键事件。

调度采用轮询（round-robin）推进，每一步都可审计。

## R05

复杂度分析（`N` 为总操作数，`R` 为资源数，`W` 为等待事务数）：

- 单次加锁决策：`O(1)`（查询持有者 + 时间戳比较）；
- 单次释放后唤醒：均摊 `O(1)` 到 `O(k)`（`k` 为该资源队列前缀清理长度）；
- 一轮调度：`O(T)`（`T` 为事务数）；
- 等待图构建与无环检查：`O(W)` 到 `O(T + W)`；
- 空间复杂度：`O(R + T + W)`。

这里强调的是“机制正确性与可解释性”，而非吞吐极限。

## R06

示例场景刻意构造“本会死锁”的交叉请求：

- `T1`（老事务）先锁 `A` 再请求 `B`；
- `T2`（年轻事务）先锁 `B` 再请求 `A`；
- `T3` 访问独立资源 `C`，用于展示系统仍能并发推进。

在 `wait-die` 下，`T2` 会在请求 `A` 时被回滚重启，释放 `B`，从而解除 `T1` 的等待。

## R07

优点：

- 从协议层直接消除死锁可能性（无需“检测环 + 挑受害者”）；
- 判定规则简单，可在锁冲突点本地执行；
- 行为可解释，便于日志审计与教学演示。

局限：

- 年轻事务可能反复回滚，造成饥饿风险；
- 在高冲突负载下，重启开销可能显著；
- 单靠预防规则不能替代完整事务恢复与持久化机制。

## R09

适用场景：

- 锁冲突较高且希望避免“停摆式死锁”的 OLTP 系统；
- 可以接受部分事务重启，以换取协议层无死锁保证；
- 需要可证明、可审计的并发控制路径。

不适用场景：

- 事务代价极高、无法承受回滚重放；
- 对公平性/尾延迟要求极高且缺少额外反饥饿机制。

## R10

正确性直觉（无死锁证明骨架）：

1. 等待边仅在“老事务请求被年轻事务持有资源”时建立；
2. 因此每条边满足 `timestamp(waiter) < timestamp(holder)`；
3. 若存在环，则沿环时间戳必须严格递增并最终回到起点；
4. 这与严格递增不可能回到原值矛盾。

故等待图不可能有环，系统不会进入死锁状态。

## R11

实现中的鲁棒性细节：

- 被回滚事务会释放全部锁并从程序起点重启（保留原时间戳）；
- 等待队列中会清理“陈旧等待项”（已不再等待的事务）；
- 每次调度步骤后执行等待图无环断言，做运行时自检；
- 提交时统一释放锁并即时唤醒等待者，避免“锁泄露”。

## R12

关键可调参数：

- `max_rounds`：最大调度轮数，防止异常实现导致无限循环；
- 事务时间戳分配策略：决定谁更可能等待、谁更可能回滚；
- 工作负载冲突模式：直接影响重启次数与吞吐。

工程中通常会再叠加重启退避、优先级提升等策略缓解饥饿。

## R14

常见失效模式：

1. 回滚后未释放全部资源，导致伪死锁；
2. 回滚时错误重置时间戳，破坏协议假设；
3. 等待队列不清理陈旧项，导致虚假阻塞；
4. 提交后未唤醒等待事务，系统吞吐异常下降。

排查建议：

- 打印“每一步锁表 + 等待边 + 事务状态”；
- 对照事件日志检查回滚/唤醒是否成对出现。

## R15

工程化落地建议：

- 与 WAL/恢复模块对接，确保回滚重启可恢复一致状态；
- 监控 `abort_rate`、`restart_count`、`wait_time_p99`；
- 对高价值事务引入优先级继承或年龄提升，缓解长期饥饿；
- 把预防策略与检测策略组合：默认预防，异常时启用诊断检测。

## R16

与其他方案对比：

- 死锁检测（detection）：允许等待，再周期性查环并回滚受害者；
- 死锁避免（avoidance，典型 Banker）：按最大需求做“安全状态”判定；
- 死锁预防（prevention，本条目）：用规则直接限制等待关系，先天无环。

`wait-die` 的优点是实现简单且可证明，代价是可能增加重启次数。

## R17

`demo.py` 的 MVP 功能清单：

- 实现 `wait-die` 加锁冲突决策（等待/回滚）；
- 支持事务回滚后重启（时间戳不变）；
- 支持资源释放后的等待者唤醒；
- 每步构建等待图并断言无环；
- 打印执行日志、提交顺序、最终统计（含重启次数）。

脚本无需输入，执行一次即可复现“交叉加锁但不死锁”的全过程。

## R18

`demo.py` 源码级算法流程（9 步）：

1. `build_demo_transactions()` 构造带时间戳的事务程序，其中 `T1/T2` 形成潜在循环等待模式。  
2. `WaitDieScheduler.__init__` 初始化锁表、等待队列、事件日志与事务映射。  
3. 调度器轮询事务，遇到 `lock(resource)` 时进入 `_handle_lock`。  
4. 若资源空闲，立即授锁并推进程序计数器。  
5. 若资源被占用，按 `wait-die` 判定：请求者更老则入等待队列；更年轻则 `_abort_and_restart`。  
6. `_abort_and_restart` 会释放该事务全部持锁、记录重启次数，并把程序计数器复位到起点。  
7. 资源释放后 `_wake_waiter` 从队列头挑选可继续事务，补授锁并恢复其执行。  
8. 每步后调用 `assert_wait_for_graph_acyclic()`：从等待状态构图并 DFS 检测环，若有环立即抛错。  
9. `main()` 运行调度、打印事件轨迹，并用断言验证“全部提交 + 至少一次回滚 + 锁表清空”。

该流程完整展开了死锁预防的关键机制，不依赖第三方数据库黑盒。
