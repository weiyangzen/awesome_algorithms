## R01

并行 DFS（Parallel Depth-First Search）是在深度优先搜索框架中引入并发执行能力的图遍历方法。  
核心思想是：在保持“深度优先”倾向的前提下，把不同分支子树交给多个执行单元同时探索，以缩短总体 wall-clock 时间。

本条目提供一个可运行 Python MVP，展示共享内存场景下的线程并行 DFS：
- 顺序 DFS 作为基线；
- 并行 DFS 使用共享 `visited` 标记和互斥锁避免重复访问；
- 输出正确性校验与耗时对比。

## R02

为什么需要并行 DFS：
- 在大规模图上，单线程 DFS 可能成为可达性分析与结构扫描的瓶颈；
- 许多图任务天然存在“分支子树可并发探索”的机会；
- 在图挖掘、状态空间搜索、依赖分析中，降低遍历延迟比追求单核吞吐更重要。

并行 DFS 不是简单“开更多线程”，关键在于并发访问去重（`visited` 一致性）和任务粒度设计。

## R03

本实现采用“根邻居分支并发 + 子树内深搜”的最小策略：
1. 起点 `start` 先由主线程访问；
2. 将 `start` 的邻居中尚未访问的节点作为多个并行种子；
3. 每个工作线程对一个或多个种子执行本地栈 DFS；
4. 线程扩展邻居时用锁保护 `visited` 的“检查并标记”过程；
5. 所有线程结束后合并访问结果。

这是一种共享内存并行 DFS 的可追踪简化版，适合教学和最小工程验证。

## R04

与顺序 DFS 的关键差异：
- 顺序 DFS：单栈、单执行流、访问顺序稳定（在邻居顺序固定时）；
- 并行 DFS：多执行流并发扩展，访问顺序通常非确定，但访问集合应一致；
- 并行版的正确性目标从“固定序列”转为“可达节点集合完整且无重复”。

因此，验证并行 DFS 时更应比较集合一致性与覆盖率，而不是逐元素顺序一致性。

## R05

复杂度分析（邻接表）：
- 顺序 DFS：时间 `O(V + E)`，额外空间 `O(V)`；
- 并行 DFS：理论总工作量仍约 `O(V + E)`，但可将时间压缩到近似 `O((V + E) / p + 同步开销)`（`p` 为并行度）。

现实环境中还需叠加：
- 锁竞争开销；
- 任务分配不均衡导致的负载倾斜；
- Python 线程受 GIL 影响的 CPU 并行效率损失。

## R06

`demo.py` 的实验配置：
- 构造可复现的稀疏无向图（固定随机种子）；
- 节点规模 `n=8000`，边数 `m=32000`；
- 先加一条链保证图连通，再补充随机边；
- 并行线程数默认 `workers=min(8, os.cpu_count())`。

脚本会打印：图规模、顺序耗时、并行耗时、速度比、结果校验状态。

## R07

MVP 中的并行一致性保障：
- `visited` 为全局共享布尔数组；
- 每次扩展邻居时执行原子化“检查+标记”（由锁保护）；
- 只有抢到访问权的线程才会将节点压入本地栈；
- 因而每个节点至多被首次声明一次，不会被重复深搜。

这对应并行图遍历中的常见原则：原子 test-and-set visited。

## R09

适用场景：
- 大图可达性分析、图结构巡检、状态空间探索；
- 共享内存机器上的中等规模并行实验；
- 需要快速验证并行 DFS 访问一致性的教学/研发场景。

不适用场景：
- 强依赖确定访问顺序的业务逻辑；
- Python 线程受限严重且要求显著 CPU 加速的纯计算任务；
- 需要跨机器容错与通信优化的分布式生产系统（需更复杂框架）。

## R10

正确性要点：
1. 顺序基线 DFS 的标准不变式：节点首次出栈时被访问，随后扩展全部未访问邻居；
2. 并行版保留相同邻居扩展语义，只改变“谁来扩展”；
3. 由于 `visited` 的受控抢占，同一节点不会被多个线程重复深搜；
4. 因而并行版访问到的节点集合应与顺序版一致（在同一起点与同一图上）。

`demo.py` 用集合相等、长度相等、唯一性检查来验证这些条件。

## R11

边界与鲁棒性处理：
- 空图、非法起点会抛出明确异常；
- `workers<=1` 时自动回退顺序 DFS；
- 起点无邻居时并行函数直接返回仅含起点的结果；
- 图生成器保证无自环、无重复边，便于稳定对比。

此外，脚本包含小图自检，优先验证实现正确再进行基准测试。

## R12

`demo.py` 关键函数职责：
- `build_undirected_graph`：由边集构造邻接表并排序邻居；
- `generate_connected_sparse_graph`：生成可复现连通稀疏图；
- `sequential_dfs`：顺序基线实现；
- `parallel_dfs`：线程并行 DFS；
- `_validate_orders`：验证集合一致性与唯一性；
- `_benchmark`：统计函数耗时；
- `main`：执行自检、基准与结果输出。

## R13

性能解释建议：
- 并行 DFS 理论可并发扩展多个子树，但实际收益受图结构影响很大；
- 若起点邻居很少或图呈长链，分支并行空间有限；
- 若分支丰富且访问竞争低，更容易观察到延迟下降；
- 在 CPython 下，线程模型主要体现算法并发结构，不保证线性加速。

因此本 MVP 的重点是“并行流程与正确性透明”，而非绝对性能纪录。

## R14

常见错误与规避：
1. `visited` 未加锁，导致竞态下重复访问；
2. 将节点出栈后才标记 visited，可能被多线程重复入栈；
3. 只比较访问顺序而忽略集合一致性，误判并行结果；
4. 随机图不可复现，导致基准波动难定位。

本实现在访问权抢占、固定随机种子、显式校验方面做了对应约束。

## R15

可扩展方向：
- 将线程版替换为多进程 + 分区图，减轻 GIL 限制；
- 引入无锁位图或分段锁降低锁竞争；
- 使用工作窃取（work-stealing）队列改善负载均衡；
- 扩展为分布式 DFS（消息传递 + 去重协议 + 终止检测）；
- 对接真实图数据集，输出更完整的吞吐与延迟曲线。

## R16

相关算法与技术：
- 顺序 DFS、迭代 DFS、递归 DFS；
- 并行 BFS（层次并行，更适合 GPU/批处理）；
- 并行图遍历中的原子访问标记与工作队列；

## R18

`demo.py` 源码级并行 DFS 流程（8 步）：
1. `main` 先运行 `run_self_test`，在小图上验证顺序与并行遍历都覆盖同一可达集合。  
2. `generate_connected_sparse_graph` 先构造链保证连通，再用 `numpy` 追加随机边，得到可复现稀疏图。  
3. `build_undirected_graph` 把边集转为邻接表，并对邻居排序以稳定局部遍历行为。  
4. 顺序路径 `sequential_dfs` 使用单栈执行标准 DFS，得到基线访问序列。  
5. 并行路径 `parallel_dfs` 先访问起点，再将其未访问邻居作为并行种子提交到线程池。  
6. 每个线程在 `_worker` 中维护本地栈深搜；扩展邻居前先在锁内执行 `visited` 的检查并标记，抢到访问权才继续深入。  
7. 主线程收集各 worker 的局部访问序列并拼接为并行访问结果，再与顺序结果做集合一致性和唯一性校验。  
8. `_benchmark` 统计顺序与并行耗时，输出时间与速度比，完成可复现实验。
