## R01

本条目将“任务调度问题”精确定义为经典的 **Job Sequencing with Deadlines** 变体，而不是泛指所有调度问题。

问题定义：

- 有一组任务 `job = (id, deadline, profit)`；
- 每个任务执行时间固定为 `1` 个时间槽；
- 单机执行，同一时间槽最多安排一个任务；
- 任务必须在 `deadline` 之前或当时完成；
- 目标是最大化被安排任务的总利润。

因此，这里研究的是“**带截止期与利润的单位时长任务调度**”。

## R02

MVP 输入输出定义：

- 输入：任务列表 `jobs = [(name, deadline, profit), ...]`
- 约束：
  - `deadline >= 1`
  - `profit >= 0`
  - 每个任务时长恒为 `1`
- 输出：
  - 被调度任务及其时间槽
  - 总利润
  - 与精确解对照得到的最优性验证结果

`demo.py` 中使用 `Job` 数据类表达任务。

## R03

原始蓝图中的“最大化完成的任务数量或利润”其实混合了多个常见调度模型。为了让条目具备可验证的 MVP，本条目明确采用教材与竞赛中最常见的一类：

- 每个任务长度相同
- 每个任务有截止期
- 每个任务有利润
- 目标是总利润最大

这个版本最适合展示：

- 贪心排序思想
- “尽量放到最晚可行槽位”的策略
- 并查集/空槽回填的工程实现

## R04

贪心算法流程：

1. 将所有任务按 `profit` 降序排序；若利润相同，再按 `deadline` 升序稳定比较。
2. 令总时间槽数为 `max_deadline`。
3. 维护“每个槽位左侧最近可用槽位”的并查集结构。
4. 依次取出利润最高的任务 `job`。
5. 查询 `find(min(job.deadline, max_deadline))`，得到不晚于截止期的最右可用槽位。
6. 若该槽位为 `0`，说明不存在可用时间槽，跳过该任务。
7. 否则把任务放入该槽位，并把该槽位与左邻可用槽位合并：`parent[slot] = find(slot - 1)`。
8. 全部扫描后得到总利润最大的可行调度。

## R05

复杂度分析：

- 排序：`O(n log n)`
- 每个任务执行一次并查集查询与合并：均摊接近 `O(alpha(T))`
- 总体时间复杂度：`O(n log n + n alpha(T))`
- 当 `T = max_deadline <= n` 时，通常简写为 `O(n log n)`
- 空间复杂度：`O(T)`

其中 `T` 是时间槽总数。

## R09

适用前提与边界条件：

- 仅适用于“每个任务时长都等于 1”的模型；
- 目标函数是总利润最大，不是总完成数最大；
- 假设任务之间没有前驱依赖；
- 假设单机、离散时间槽；
- 如果任务时长不等、存在机器数 > 1、存在释放时间或切换成本，本模型不再直接适用。

## R10

正确性依据要点：

- 若某个高利润任务能被安排，则把它尽量放到最晚可行槽位，不会减少前面槽位给其他任务的空间；
- 因为放得更晚只会保留更多早期空槽，不会损害未来可行性；
- 按利润降序处理时，每次优先保住当前最有价值任务；
- 并查集只是在工程上高效地实现“找到最右可用槽位”，不改变贪心策略本身。

这就是该算法的核心交换论证。

## R11

数值稳定性与误差传播：

本问题属于离散组合优化问题，不涉及浮点数值迭代，因此传统意义上的数值稳定性不是核心风险。

主要风险来自：

- 比较器写错导致排序不稳定
- 并查集 `find/union` 实现错误
- 时间槽索引偏移（`1..T` 与 `0..T-1`）混淆

因此这里更应关注“离散状态一致性”而不是浮点误差传播。

## R12

真实计算成本：

- 当 `deadline` 很大时，时间槽数组会按 `max_deadline` 分配，内存成本可能超过 `n`
- 在很多工程场景里，常会先把 `deadline` 截断到 `n`，因为最多只能安排 `n` 个任务
- DSU 版本在 Python 中常数很小，通常明显优于逐槽倒序扫描
- 若任务数很少，暴力或简单回填法也能接受，但规模变大后 DSU 的优势很明显

## R13

近似比/误差界/概率保证：

这个贪心算法对本条目采用的模型是 **精确最优算法**，不是近似算法。

因此：

- 不存在近似比损失
- 不存在随机成功概率
- 误差界为 `0`（在满足模型前提时）

## R14

鲁棒性与失效模式：

- 若把“任务持续时间可变”误当成该模型的一部分，算法会失效；
- 若忘记把任务放入“最晚可行槽位”，而是放最早可行槽位，会破坏最优性；
- 若并查集初始化错误，可能出现重复占槽；
- 若 `deadline` 包含 `0` 或负值但未过滤，会出现非法调度状态；
- 若同一个任务被重复插入，利润会被错误累计。

## R18

`demo.py` 的源码级算法流（8 步）：

1. 在 `main` 中构造固定样例与随机样例任务集。  
2. 对每组任务调用 `schedule_jobs_greedy_dsu`。  
3. `schedule_jobs_greedy_dsu` 先按利润降序排序任务。  
4. 初始化并查集 `parent`，其中 `find(t)` 表示“不晚于 `t` 的最右可用槽位”。  
5. 对每个任务查询其截止期左侧的最右空槽。  
6. 若找到可用槽位，则把任务放入该槽，并执行 `parent[slot] = find(slot - 1)` 以占用该槽。  
7. 同时用 `schedule_jobs_exact_bruteforce` 对小样本做子集枚举，求得精确最优利润。  
8. 比较贪心结果与精确解利润，并输出调度表、利润与断言结果；全部通过后打印 `All checks passed.`。  

这 8 步已经把“排序、占位、并查集回填、暴力对照”完整展开，不依赖黑盒调度 API。
