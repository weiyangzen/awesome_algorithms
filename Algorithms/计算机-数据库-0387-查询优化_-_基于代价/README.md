## R01

基于代价的查询优化（Cost-Based Query Optimization, CBO）是在同一逻辑查询下，枚举多个候选执行计划，并用代价模型选取估计成本最低的方案。

与仅靠规则的重写不同，CBO 的核心是“可计算比较”：

- 对访问路径做成本估计（如 `SeqScan` vs `IndexScan`）；
- 对连接顺序做成本估计（如 `(A⋈B)⋈C` vs `(A⋈C)⋈B`）；
- 对连接算法做成本估计（如 `NestedLoop`/`HashJoin`/`MergeJoin`）。

## R02

核心直觉是：同一 SQL 的执行代价差异可以非常大，而差异往往来自中间结果规模。

如果先做高选择性过滤、再做合适连接顺序，可能把中间结果从“笛卡尔级”降到“线性级”。  
因此 CBO 通常围绕两个问题展开：

1. 如何估计每个子计划的行数（基数估计）；
2. 如何把行数转成执行成本（CPU、I/O、排序、哈希等）。

## R03

本条目实现一个可运行的最小 CBO MVP，覆盖“访问路径 + 连接顺序 + 连接算法”三层选择。

演示查询由 3 张表构成：

- `orders(o)`：订单；
- `customers(c)`：客户；
- `lineitems(l)`：订单明细。

谓词与连接：

- 单表过滤：`o.status='paid'`、`o.amount>120`、`c.tier='gold'`、`l.qty>=2`；
- 连接条件：`o.customer_id=c.customer_id`、`o.order_id=l.order_id`。

优化器输出最低估计代价的物理计划，并与固定顺序 naive 执行结果做等价断言。

## R04

`demo.py` 的实现结构：

- 统计层：`gather_table_stats`（行数、列 NDV、谓词选择率）；
- 访问路径层：`choose_access_plan`（`SeqScan` vs `IndexScan`）；
- 搜索层：`optimize_cost_based`（Selinger 风格 DP 子集搜索）；
- 连接物理层：`choose_join_plan`（`NestedLoop`/`HashJoin`/`MergeJoin` 三选一）；
- 执行层：`execute_plan`（源码实现三种连接，不依赖数据库黑盒）；
- 验证层：`execute_naive` 对照 + 结果集合规范化比较。

## R05

复杂度（设表数为 `n`）：

- 动态规划枚举复杂度约为 `O(3^n)`（Selinger 经典子集划分规模）；
- 每次候选连接的代价计算是常数级；
- 本 demo 为 `n=3`，候选空间很小，重点是展示算法闭环而非大规模工程优化。

执行复杂度：

- `NestedLoop` 连接近似 `O(L*R)`；
- `HashJoin` 期望 `O(L+R)`；
- `MergeJoin` 近似 `O(L log L + R log R + L + R)`（含排序）。

## R06

样例数据刻意包含选择性差异，以体现 CBO 决策价值：

- `orders` 中既有 `paid` 也有 `cancelled`，且金额分布不均；
- `customers` 中仅部分用户为 `gold`；
- `lineitems` 中仅部分明细满足 `qty>=2`。

另外定义了索引列集合（如 `o.status`、`c.tier`、`l.order_id`），使访问路径选择可观测。

## R07

优势：

- 能统一考虑“先做什么、怎么做”；
- 可通过统计信息适配不同数据分布；
- 与纯规则法相比，通常更接近低成本计划。

局限：

- 估计误差会传导到计划选择；
- 搜索空间随表数增长很快，需要剪枝与启发式；
- 代价模型简化过度会降低计划质量。

## R08

前置知识：

- 关系代数中的连接与过滤；
- 基数估计基础（选择率、NDV）；
- 动态规划子问题最优性思想。

运行环境：

- Python 3.10+
- 依赖：`numpy`、`pandas`（来自仓库统一 `pyproject.toml`）
- 无交互输入。

## R09

适用场景：

- 教学和原型验证 CBO 主流程；
- 小型执行引擎中构建“可解释优化器”；
- 在接入真实数据库统计模块前做端到端演练。

不适用场景：

- 超多表复杂查询（需要更激进剪枝与并行优化）；
- 强依赖真实 I/O 模型和缓存行为的生产场景；
- 需要谓词相关性建模的高精度估计场景。

## R10

正确性不变式：

1. 连接只在存在显式 `JoinCondition` 的子集间构造；
2. 单表谓词仅作用于对应表的访问节点；
3. 物理执行层无论选哪种连接算法，都按相同等值条件合并行；
4. 最终用 `canonical_rows` 比较 naive 与 CBO 结果，保证语义一致。

## R11

鲁棒性设计：

- 代价和基数估计都做下界保护（`max(1.0, est)`），避免零行导致除零或异常决策；
- `find_join_condition` 在左右子树交换时会自动重排连接键方向；
- DP 若找不到合法连接会抛错，防止静默返回无效计划；
- 主流程含断言：结果等价 + 计划中至少出现一次 `HashJoin` 或 `MergeJoin` 决策信号。

## R12

关键参数与调参方向：

- `SeqScan` 成本参数：`row_count + 0.3*est_rows`；
- `IndexScan` 成本参数：`log2(N)*2.5 + 1.6*est_rows`；
- `HashJoin`、`MergeJoin`、`NestedLoop` 的系数设置。

调参建议：

1. 若优化器过度偏向索引，增大随机访问系数；
2. 若优化器过度偏向嵌套循环，增大 `left_rows*right_rows` 权重；
3. 若排序代价低估，调高 `MergeJoin` 中 `N log N` 系数。

## R13

理论性质：

- 全局最优性（在当前候选空间内）：DP 对每个子集保留最小成本计划，满足最优子结构；
- 确定性：给定同一统计信息与参数，输出计划稳定；
- 可终止性：表数有限，子集枚举有限，必然结束。

说明：此处“最优”是相对模型和候选空间的最优，不等于真实系统绝对最优。

## R14

常见失效模式：

1. 谓词独立性假设失真，导致基数系统性偏差；
2. NDV 估计过粗，连接基数低估或高估；
3. 代价系数不贴近真实硬件，物理算法选择偏移；
4. 数据倾斜严重时，哈希连接实际成本可能劣化。

排查建议：

- 打印每步估计行数与真实行数对比；
- 记录候选计划成本排序，观察是否“误判临界点”；
- 单独压测三种连接实现，检查模型系数是否需要重标定。

## R15

工程化落地建议：

- 将统计采集模块与优化器解耦，支持外部注入直方图/采样结果；
- 把代价函数参数化并支持在线重标定；
- 给每个规则点增加可观测日志：访问路径命中、DP 子集成本、最终 plan hash；
- 在生产里保留 fallback（超时或异常时退化到稳健计划）。

## R16

与其他范式对比：

- RBO：解释性强、实现快，但缺少数据分布感知；
- CBO：实现复杂度更高，但在多路径选择问题上更有优势；
- 混合范式：先做规则规范化（下推、去冗余），再做代价驱动决策，是实际系统常见路径。

本条目聚焦“纯 CBO 主循环”最小实现。

## R17

`demo.py` MVP 能力：

- 基于 `pandas` 收集统计信息（行数、NDV、谓词选择率）；
- 自动选择单表访问路径（`SeqScan` / `IndexScan`）；
- 用 Selinger 风格 DP 搜索连接顺序；
- 在每次连接时比较 `NestedLoop`/`HashJoin`/`MergeJoin` 成本；
- 执行优化后计划并与 naive 结果断言等价；
- 打印统计摘要、最终计划树、候选计划数量、查询结果。

## R18

`demo.py` 源码级算法流程（8 步）：

1. `build_demo_data` 与 `build_query_spec` 构造三表数据、单表谓词和连接图。  
2. `gather_table_stats` 用 `pandas` 计算每表行数与列 NDV，并统计各谓词选择率。  
3. `choose_access_plan` 对每张表计算 `SeqScan` 与 `IndexScan` 两种成本，形成 DP 初始状态。  
4. `optimize_cost_based` 对表子集做动态规划：枚举划分 `(left_set, right_set)`，寻找可连接条件。  
5. 对每个候选连接，`estimate_join_cardinality` 用 `left_rows*right_rows/max(NDV)` 估计输出基数。  
6. `choose_join_plan` 计算三种连接算法成本并择最小，更新该子集最优计划。  
7. 完整计划生成后，`execute_plan` 按选中物理算子真正执行（源码实现 NLJ/Hash/Merge）。  
8. `execute_naive` 作为基线执行，`canonical_rows` 比较结果集合一致性，最后输出计划与结果。  

以上流程没有把第三方数据库优化器当黑盒，代价估计、搜索与执行都在源码中可追踪。
