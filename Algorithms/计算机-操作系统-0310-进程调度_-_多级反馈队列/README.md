- 分类: `操作系统`

## R01

问题定义：给定一组进程（到达时间 `arrival`、CPU 需求 `burst`），用多级反馈队列（Multi-Level Feedback Queue, MLFQ）在单核 CPU 上进行调度模拟，输出调度时间线与每个进程的完成/周转/等待/响应指标。

## R02

输入：
- 进程列表 `[(pid, arrival, burst), ...]`
- 队列时间片配置 `quantums`，例如 `[2, 4, 8]`，下标越小优先级越高
- 可选全局优先级提升周期 `boost_interval`（防止饥饿）

输出：
- 时间线：`(start, end, pid, level)`，其中 `level` 是执行时队列层级
- 每进程指标：
  - `completion`
  - `turnaround = completion - arrival`
  - `waiting = turnaround - burst`
  - `response = first_start - arrival`
- 平均指标：平均周转、平均等待、平均响应时间

## R03

MLFQ 核心思想：
- 新到达进程先进入最高优先级队列（`Q0`）。
- 每层队列使用独立时间片；进程若用完整个时间片仍未完成，则降到下一层。
- 调度总是选择“最高优先级的非空队列”。
- 低优先级运行期间，若高优先级出现新任务，可抢占当前任务。
- 周期性优先级提升会把就绪队列中的进程提升回 `Q0`，缓解饥饿。

## R04

关键数据结构：
- `Process`：`pid/arrival/burst/remaining/level/first_start/completion`
- `queues: List[deque[Process]]`：每层一个 FIFO 队列
- `timeline: List[(start, end, pid, level)]`
- `next_idx`：按到达时间排序后的“尚未入队”扫描指针

## R05

算法流程（高层）：
1. 校验输入，按 `(arrival, pid)` 排序并初始化 `remaining=burst`。
2. 维护多层就绪队列；新到达进程进入 `Q0`。
3. 若到达提升周期，执行一次全局优先级提升（`Q1..Qk` 全部搬到 `Q0`）。
4. 若所有队列为空，CPU 记录 `IDLE` 并把时间推进到“下一到达或下一次提升”。
5. 从最高非空队列取进程执行，时间上限由该层时间片决定。
6. 执行中若发生“提升边界”或“高优先级新到达”，提前切段并重新入队。
7. 进程完成则记录 `completion`；时间片耗尽未完成则降级。
8. 所有进程完成后计算指标与平均值。

## R06

正确性要点：
- 优先级规则正确：始终从最高非空队列取任务。
- 反馈规则正确：完整用完时间片才降级，未用完不降级（本 MVP 无 I/O，仅由抢占导致提前返回）。
- 抢占规则正确：低优先级运行遇到高优先级到达可中断并回队。
- 无饥饿机制：`boost_interval` 定期提升低优先级任务。
- 指标可验证：`first_start/completion` 都来自时间线真实时间点。

## R07

复杂度分析：
- 设进程数 `n`，总 CPU 需求 `B = sum(burst)`，队列层数 `L`，时间线段数 `S`。
- 排序成本 `O(n log n)`。
- 调度主循环与时间线事件处理总体近似 `O(S * L)`（选最高非空队列最坏扫描 `L` 层）。
- 空间复杂度 `O(n + S + L)`。

## R08

边界与异常处理：
- `quantums` 为空或存在非正值：抛出异常。
- `boost_interval <= 0`：抛出异常。
- `arrival < 0`、`burst <= 0`、`pid` 重复：抛出异常。
- 全部任务都在未来到达：时间线前缀出现 `IDLE`。
- 单层队列时退化为 RR；`boost_interval=None` 时关闭优先级提升。

## R09

MVP 范围约束：
- 单核、仅 CPU burst、无 I/O 阻塞模型。
- 离散整数时间。
- 使用 Python 标准库实现（无黑盒调度库）。
- 输出文本化时间线与指标表，便于验证和教学。

## R10

`demo.py` 内置示例：
- `quantums = [2, 4, 8]`
- `boost_interval = 12`
- 进程集合：
  - `P1(0, 9)`
  - `P2(1, 4)`
  - `P3(2, 6)`
  - `P4(8, 5)`
  - `P5(10, 2)`

## R11

预期现象：
- 新到达短作业通常在高优队列快速获得首次响应。
- CPU 密集型长作业会逐步下沉到低优先级队列。
- 周期性提升后，低优队列任务会重新获得高优先级机会。
- 与单一 RR 相比，交互响应更好，但调度行为更复杂。

## R12

指标解释：
- `response` 强调“第一次被服务”延迟，最贴近交互体验。
- `waiting` 描述总排队时间，可衡量队列拥塞程度。
- `turnaround` 是端到端时延，反映任务完成效率。
- 平均指标用于比较不同 `quantums/boost_interval` 参数配置。

## R13

与经典策略关系：
- MLFQ 可看作“多层 RR + 动态优先级调整”。
- 当仅 1 层队列时，算法退化为 RR。
- 若关闭反馈与提升，容易退化为静态多级队列并产生饥饿风险。
- 相比 FCFS，MLFQ 对短任务与交互任务更友好。

## R14

参数选择建议：
- 顶层时间片宜小（提升响应），底层可大（降低切换频率）。
- 常见设置是近似指数增长，如 `2, 4, 8`。
- `boost_interval` 过小会削弱分层区分度，过大可能导致低优任务等待过久。
- 需结合工作负载类型（交互/批处理占比）做实验调优。

## R15

常见实现错误：
- 新任务未进入 `Q0` 而错误进入当前层。
- 时间片用尽后忘记降级，导致“伪高优先级垄断”。
- 低优执行时忽略高优到达抢占。
- 只更新时间不记录 `IDLE`，导致时间线与指标不一致。
- 优先级提升只改 `level` 不搬队列，造成状态错乱。

## R16

最小测试清单：
- 正常混合负载（短作业 + 长作业 + 交错到达）。
- 到达时间相同的多任务。
- 含 `IDLE` 区间的场景。
- `boost_interval=None` 与开启提升两组对比。
- 非法输入校验：空 `quantums`、负到达、非正 burst、重复 pid。

## R17

可扩展方向：
- 引入上下文切换开销，评估吞吐与响应折中。
- 支持 I/O 阻塞与“主动让出 CPU”行为。
- 添加 aging（按等待时长渐进提升）替代固定周期提升。
- 扩展到多核（全局队列或每核本地队列 + work stealing）。
- 统计更多指标：吞吐量、99 分位响应时间、队列驻留分布。

## R18

源码级算法拆解（对应 `demo.py`，非黑盒）：
1. `_validate_inputs` 检查 `quantums`、`boost_interval`、`arrival/burst` 和 `pid` 唯一性，防止非法状态进入调度主循环。
2. `mlfq_schedule` 把输入进程转换为 `Process`，并按 `(arrival, pid)` 排序，保证模拟可复现。
3. `enqueue_arrived` 使用指针 `next_idx` 把 `arrival <= time` 的进程统一放入最高优先级队列 `Q0`。
4. `apply_priority_boost_if_due` 在提升边界把低层队列任务搬到 `Q0`，并推进下一次提升时刻。
5. 主循环先处理“到达 + 提升”，再选取最高非空队列任务；若无就绪任务则记录 `IDLE` 并把时间跳到下一个关键事件。
6. 任务运行时，单段执行长度由 `min(remaining, 当前层时间片)` 决定，再被“下一次提升时刻”或“高优先级到达时刻”截断，形成可抢占切段。
7. 执行段结束后，根据状态分流：完成则记录 `completion`；命中提升边界则回到 `Q0`；被高优到达打断则回原层；仅时间片耗尽则降级到下一层。
8. 全部任务完成后，用 `completion/first_start` 计算 `turnaround/waiting/response` 及平均值，最终由 `main` 打印时间线和指标表。
