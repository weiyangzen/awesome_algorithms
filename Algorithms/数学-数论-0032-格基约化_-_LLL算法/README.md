## R01

LLL（Lenstra-Lenstra-Lovasz）算法用于把一组整数格基转换为“较短且近似正交”的新基，是格计算中的基础约化算法。

- 输入:
  - 基矩阵 `B`（常用行向量表示），元素通常为整数
  - 参数 `delta`，满足 `1/4 < delta < 1`（工程上常用 `0.75` 或 `0.99`）
- 输出:
  - 与原格等价的约化基 `B'`
- 核心约束:
  - Size reduction：`|mu_{i,j}| <= 1/2`
  - Lovasz 条件：`||b*_k||^2 >= (delta - mu_{k,k-1}^2) * ||b*_{k-1}||^2`

## R02

LLL 由 Arjen K. Lenstra、Hendrik W. Lenstra Jr. 与 Laszlo Lovasz 在 1982 年提出。其里程碑意义在于：

- 首次给出格基约化的多项式时间算法（相对于输入位长）
- 把格工具系统化引入数论与计算代数
- 成为后续 BKZ、格攻击、格优化算法的重要基础组件

## R03

问题背景是“高维整数格上的计算难题”：

- 同一格可有很多基，某些基向量很长且高度相关，导致后续计算困难
- 许多数论和密码问题（如近似最短向量、子集和、同余结构分析）都需要“更好”的基
- 在 LLL 之前，缺少一个兼顾可证明复杂度与工程可用性的通用方法

LLL 的价值是提供一个可计算、可证明、可复用的“中等强度约化”标准。

## R04

设基维数为 `n`，输入整数的位长上界为 `L`。

- 经典结果（位复杂度层面）通常写作多项式时间，例如 `poly(n, L)`
- 常见教材量级描述：约 `O(n^5 * L^3)` 位操作（取决于实现细节与算术模型）
- 在浮点近似实现中，单轮成本主要来自 Gram-Schmidt 与向量更新，经验上接近 `O(n^3)` 每轮，轮数与数据形态相关

结论：LLL 明显重于初等数论算法，但相比更强格约化方法（如高块大小 BKZ）仍更轻量。

## R05

空间复杂度主要来自三部分：

- 基矩阵 `B`：`O(n^2)`（方阵情形）
- Gram-Schmidt 向量 `B*`：`O(n^2)`
- 系数矩阵 `mu`：`O(n^2)`

因此总体额外空间通常为 `O(n^2)` 级别（不含大整数库的额外内存管理开销）。

## R06

二维直观示例（`delta = 0.75`）：

给定基向量

- `b1 = (4, 1)`
- `b2 = (1, 3)`

步骤摘要：

1. 计算 `mu_21 = <b2, b1*> / ||b1*||^2 = 7/17`，已满足 `|mu_21| <= 1/2`，无需 size reduction。
2. 检查 Lovasz：`||b2*||^2 ? >= (delta - mu_21^2)||b1*||^2`。
3. 若不满足则交换向量并重算；若满足则结束。
4. 最终得到更短、更接近正交的等价基。

在高维情形中，算法会在“size reduction + Lovasz 检查 + 必要交换”之间反复迭代。

## R07

LLL 的意义主要体现在三个层面：

- 理论层：把格基约化引入多项式时间框架
- 算法层：作为大量格算法的预处理器与基线组件
- 应用层：在计算数论、符号计算、编码理论和格密码分析中长期使用

它通常不是终点算法，而是“把输入变成可处理形态”的关键起点。

## R08

先导知识与依赖：

- 线性代数：内积、范数、线性无关、正交化
- 数论/代数：整数格、基变换、行列式与格体积
- 算法基础：循环不变量、复杂度分析
- 实现技术：
  - Gram-Schmidt（或改进的 GSO）
  - 最近整数舍入（用于 size reduction）

## R09

适用前提与边界条件：

- 输入基应满秩（至少在其张成子空间内线性无关）
- `delta` 必须在 `(1/4, 1)`，否则收敛性与性质不成立
- 维度上升时，LLL 的结果质量与成本都显著变化
- LLL 给的是“近似质量保证”，不是严格最短向量解

不适用场景：

- 追求极强约化质量但可接受更高成本时，通常转向 BKZ 等方法

## R10

正确性直觉来自两个不变量：

1. **格不变性**：每次 size reduction（`b_k <- b_k - q*b_j`）与交换都对应可逆整系数变换，不改变格本身。
2. **约化推进性**：
   - size reduction 保证 `mu` 系数被压入 `[-1/2, 1/2]`
   - 若 Lovasz 不满足就交换，交换会推动某个势函数下降
3. 势函数有下界，故“无限交换”不可能持续，算法终止。
4. 终止时同时满足 size reduction 与 Lovasz 条件，即得到 LLL-reduced 基。

## R11

LLL 常见实现使用浮点 Gram-Schmidt，存在数值误差风险：

- 误差来源:
  - 高维下正交化累积舍入误差
  - 接近线性相关时 `||b*_i||^2` 很小，导致系数不稳定
- 工程缓解:
  - 加容差（例如 `1e-12`）判断 Lovasz 条件
  - 必要时重算 GSO
  - 对高精度需求场景使用有理数/多精度实现

本目录 `demo.py` 采用 `numpy.float64`，属于教学与中小规模演示级方案。

## R12

真实工程成本（超越大 O）：

- 最耗时环节通常是反复 Gram-Schmidt 重算
- `delta` 越接近 1，约化更“激进”，交换次数往往更多
- 输入基越病态（向量接近共线、长度悬殊大），迭代更慢
- 纯 Python 循环在高维会成为瓶颈，生产实现通常用 C/C++ 库或专门数值后端

## R13

LLL 的关键近似保证之一：

- 设 `lambda_1(L)` 是格的最短非零向量长度
- 对 `delta = 3/4` 的常见设定，可得
  - `||b1|| <= 2^((n-1)/2) * lambda_1(L)`

这说明 LLL 不保证最优最短向量，但给出可证明上界，是“可计算近似”而非黑箱启发式。

## R14

鲁棒性与失效模式：

- 失效模式 1：输入不满秩，导致 `||b*_i||^2` 接近 0，数值不稳定
- 失效模式 2：维度过高导致运行时间过长
- 失效模式 3：整数过大时，简单 `int64` 可能溢出（若实现未使用大整数）
- 失效模式 4：浮点误差导致边界条件误判

缓解策略：

- 先做秩检查和输入规模检查
- 使用容差与重计算
- 必要时切换高精度或成熟格库
- 为迭代设置安全上限与日志

## R18

源码级流程追踪（本实现无第三方黑箱，直接对应 `demo.py`）：

1. **输入校验**：检查矩阵维度、`delta` 区间与秩条件。  
2. **初始 GSO**：调用 `gram_schmidt` 计算 `mu`、`b*`、`||b*||^2`。  
3. **进入主循环**：从 `k = 1` 开始逐步推进。  
4. **Size reduction**：对 `j = k-1..0`，用最近整数 `q = round(mu[k,j])` 执行 `b_k <- b_k - q*b_j`。  
5. **重算正交信息**：size reduction 后重新计算 GSO，确保后续判定基于当前基。  
6. **Lovasz 检查**：判断 `||b*_k||^2` 是否满足 `(delta - mu_{k,k-1}^2)||b*_{k-1}||^2`。  
7. **交换与回退**：若不满足，交换 `b_k` 与 `b_{k-1}`，并令 `k <- max(1, k-1)`；否则 `k <- k+1`。  
8. **终止与验收**：当 `k == n` 结束；再调用 `check_lll_conditions` 验证 size reduction 与 Lovasz 条件。  

以上 8 步完整对应 LLL 的标准控制流，便于从源码层面逐条对照算法定义。
