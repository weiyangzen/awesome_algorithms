## R01

读者-写者问题（Readers-Writers Problem）是经典并发同步问题：

- 多个读者可同时访问共享资源（读-读可并发）；
- 写者访问时必须独占（写-读、写-写都互斥）；
- 需要避免死锁，并尽量避免某一方长期饥饿。

它本质上是在“吞吐量（允许并发读）”与“公平性（写者不被饿死）”之间做协议设计。

## R02

该问题在操作系统教材中常用于讲解：

- 临界区与互斥；
- 信号量 / 锁 / 条件变量的组合用法；
- 调度策略导致的饥饿与公平。

经典变体有三类：

1. 读者优先：吞吐高，但写者可能长期等待；
2. 写者优先：写延迟低，但读者可能被频繁阻塞；
3. 公平策略：按到达次序或闸门机制，减少双向饥饿风险。

## R03

问题可形式化为以下约束：

- 安全性约束：
  - `active_writers <= 1`
  - `active_writers > 0 => active_readers == 0`
- 活性约束：
  - 只要线程持续执行，等待方最终应进入临界区（无无限期等待）

若只做“读者计数 + 资源锁”而不控制新读者插队，写者在高频读流量下可能始终拿不到写锁，这是典型饥饿失效。

## R04

本目录 MVP 采用**公平读写锁**（turnstile + room_empty + read_count）：

- `turnstile`（闸门）：写者先占住闸门，阻止后续新读者继续插队；
- `room_empty`（房间空锁）：保证写临界区独占；
- `read_count + read_mutex`：首个读者获取 `room_empty`，最后一个读者释放。

这是一种工程上常见、代码规模小且逻辑可追踪的公平方案。

## R05

设读操作总次数为 `R`，写操作总次数为 `W`：

- 每次读/写的锁操作是常数个信号量/互斥量动作；
- 时间复杂度：`O(R + W)`（忽略线程调度开销）；
- 额外空间：`O(1)`（不计日志时），本 demo 为了验证记录了 `O(R + W)` 事件日志。

## R06

简化时序示例：

1. `R1` 到达：通过闸门，`read_count=1`，占住 `room_empty`，开始读；
2. `R2` 到达：`read_count=2`，与 `R1` 并发读；
3. `W1` 到达：占住 `turnstile` 并等待 `room_empty`；
4. 之后新读者到达会卡在 `turnstile`，无法继续插队；
5. `R1/R2` 结束后释放 `room_empty`，`W1` 进入写临界区；
6. `W1` 写完释放 `room_empty` 与 `turnstile`，后续读写继续竞争。

这个流程体现了“读可并发、写独占、写者可最终进入”。

## R07

优点：

- 读共享 + 写互斥的安全性明确；
- 相比读者优先，显著降低写者饥饿概率；
- 实现短小，便于教学和代码审计。

局限：

- Python 线程受 GIL 影响，示例偏同步协议验证，不代表 CPU 密集并行加速；
- 公平性受调度器影响，不是严格实时上界；
- 高冲突下上下文切换开销会放大。

## R09

适用场景：

- 缓存或配置对象“读多写少”；
- 需要比简单互斥锁更高读吞吐；
- 需要显式控制写者长期等待风险。

不适用场景：

- 纯 CPU 密集任务（线程并发收益有限）；
- 需要跨进程同步（应选进程级原语）；
- 需要严格实时调度保证（需 RTOS/优先级协议）。

## R10

本实现的正确性检查点：

1. 写者进入前必须同时拿到 `turnstile` 与 `room_empty`；
2. 首个读者获取 `room_empty`，最后一个读者释放；
3. 读者进入临界区时断言 `active_writers == 0`；
4. 写者进入时断言 `active_readers == 0 and active_writers == 0`；
5. 所有线程结束后，写次数与共享值增量一致。

## R11

死锁与饥饿分析：

- 死锁方面：
  - 锁获取顺序固定，写者路径 `turnstile -> room_empty`；
  - 读者不会持有 `read_mutex` 再等待 `turnstile`，避免环路等待。

- 饥饿方面：
  - 关键是 `turnstile`：写者到达后，新的读者被阻塞，写者不再被“后来的读者洪流”持续插队；
  - demo 记录写者等待时间并做上界断言，作为一次运行级别的活性信号。

## R12

`demo.py` 主要参数：

- `num_readers`、`num_writers`：线程规模；
- `reader_rounds`、`writer_rounds`：每线程操作次数；
- `seed`：固定随机等待，提升可复现性。

调参建议：

- 想更容易观察冲突，增大线程数和轮次；
- 想缩短运行时间，降低轮次；
- 若机器负载高，可放宽写者最大等待阈值。

## R14

常见失效模式与防护：

1. 忘记“最后一个读者释放 `room_empty`”，导致写者永久阻塞；
2. 写者未先占 `turnstile`，导致新读者可持续插队；
3. 更新 `read_count` 未加互斥，导致读者计数竞态；
4. 用无限 `join()` 且线程异常退出未处理，脚本可能卡住。

本实现通过：固定锁序、计数锁保护、断言、`join(timeout)` 来降低以上风险。

## R18

`demo.py` 源码级算法流程（8 步）：

1. 初始化 `FairRWLock`：创建 `turnstile`、`room_empty`、`read_mutex` 与 `read_count`。  
2. `reader_acquire` 先经过 `turnstile`（获取后立即释放），再在 `read_mutex` 保护下递增 `read_count`，首读者占用 `room_empty`。  
3. `reader_release` 在 `read_mutex` 下递减 `read_count`，末读者释放 `room_empty`。  
4. `writer_acquire` 依次获取 `turnstile` 和 `room_empty`，确保写阶段独占且阻止新读者插队。  
5. `run_simulation` 创建读/写线程：读者只读取共享值，写者每次将共享值 `+1`。  
6. 在线程临界区入口处记录事件并执行断言：读者进入时必须无写者，写者进入时必须无读者无写者。  
7. 所有线程结束后统计 `max_active_writers`、冲突事件数、最终共享值，并用 `numpy` 计算读写等待时间的均值/p95/最大值。  
8. `main` 输出指标并执行最终断言（写入次数一致、无读写重叠、写者等待有上界），确认本次并发协议运行有效。  

该实现没有依赖黑盒并发库：关键同步路径（闸门、公平性、互斥断言）都在源码中显式展开并可逐步追踪。
