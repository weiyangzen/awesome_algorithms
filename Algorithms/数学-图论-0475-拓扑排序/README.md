## R01

拓扑排序（Topological Sorting）是对有向图顶点的一种线性排列，使得每条有向边 `u -> v` 中，`u` 都出现在 `v` 之前。  
它只对有向无环图（DAG, Directed Acyclic Graph）有完整解；若图中存在环，则不存在覆盖全部节点的合法拓扑序。

## R02

拓扑排序是经典图算法基础组件，广泛用于“先后依赖”问题。常见两条主线：
- Kahn 算法（1962）：基于入度与队列/堆逐步剥离入度为 0 的点；
- DFS 逆后序法：深度优先遍历后按完成时间逆序输出，并通过回边检测环。

## R03

它解决的核心问题是“约束调度”：
- 输入：若干对象（课程、任务、编译单元）及依赖边；
- 输出：满足全部依赖约束的执行顺序。  
若依赖关系形成闭环（例如 `A->B->C->A`），算法应返回“有环/无解”，而不是给出伪顺序。

## R04

Kahn（入度法）流程：
1. 统计每个节点入度；
2. 将所有入度为 0 的节点放入容器（本实现用最小堆做稳定输出）；
3. 反复取出一个节点加入结果，并“删除”其出边（即邻居入度减 1）；
4. 邻居入度降为 0 时入堆；
5. 最终若处理节点数小于总节点数，则图中存在环。

## R05

设节点数为 `V`、边数为 `E`：
- 时间复杂度：
  - 用队列时为 `O(V+E)`；
  - 本实现用最小堆做确定性并列打破，复杂度为 `O((V+E) log V)`。
- 空间复杂度：`O(V+E)`（邻接表、入度表、结果序列和容器）。

## R06

示例（DAG）：
- 边：`需求分析->接口设计`、`接口设计->后端实现`、`后端实现->联调测试`、`联调测试->上线发布` 等。
- 一个合法拓扑序：`需求分析 -> 数据库建模 -> 接口设计 -> 后端实现 -> 前端实现 -> 联调测试 -> 上线发布`。  
注意拓扑序通常不唯一，只要满足所有边方向约束就是正确解。

## R07

优点：
- 能直接把“偏序关系”转成“可执行序列”；
- 可同时完成环检测（无解识别）；
- 实现简单，工程可解释性强。  
局限：
- 只能用于有向图依赖问题；
- 结果受并列节点选择策略影响（不唯一）。

## R08

前置知识与数据结构：
- 邻接表表示有向图；
- 入度数组/哈希表；
- 队列或堆；
- 基础图遍历与复杂度分析。  
实现上需明确边语义：`u -> v` 表示 `u` 必须先于 `v`。

## R09

适用边界：
- 适用：课程安排、任务编排、构建系统依赖解析、DAG 工作流。
- 不适用：
  - 含有向环且必须全量排序的场景（只能报告冲突）；
  - 无向图“排序”问题（语义不同）；
  - 需要最短路径/最小生成树等其他图目标问题。

## R10

正确性要点（Kahn）：
1. 任意 DAG 至少存在一个入度为 0 的节点；
2. 每次取出的入度为 0 节点放入结果，不会违反先后约束；
3. 删除其出边后，新的入度为 0 节点正是“已满足前驱依赖”的节点；
4. 若最终无法继续且仍有未处理节点，则这些节点互为前驱链，必含环。  
因此：处理完全部节点当且仅当图无环，且输出序列合法。

## R11

工程稳定性注意点：
- 同一批入度为 0 的节点若使用哈希迭代，输出顺序可能随运行环境变化；
- 本实现采用最小堆进行并列打破，保证可复现；
- 输入出现孤立点时也应进入输出序列；
- 输入边引用了未显式声明的节点时，应自动补入节点集合（本实现已处理）。

## R12

真实成本与实现取舍：
- 当 `E` 远大于 `V` 时，主要开销在遍历边；
- 若只要任意解，用 `deque` 更快；
- 若要稳定、可测试的固定输出，用 `heapq` 更合适；
- 生产中常同时输出“已排序前缀 + 剩余节点数”，便于快速定位循环依赖。

- 有环时报告无完整序。

## R14

常见失效模式与防护：
- 边方向写反，得到看似“错位”的顺序。
  - 防护：统一语义为 `u -> v` 表示 `u` 在前。
- 未统计全部节点（仅统计出现在边中的节点）。
  - 防护：显式节点列表与边节点并集。
- 将“处理节点数 < 总节点数”误判为算法 bug。
  - 防护：这是标准环检测条件，应作为业务错误返回。
- 单测只测 DAG，不测含环图。
  - 防护：至少准备一组正例和一组环例。

## R15

工程实践建议：
- 输出结果后做二次校验：对每条边检查 `pos[u] < pos[v]`；
- 为可复现测试固定 tie-break 规则（如字典序）；
- 在日志中打印 `processed_count / total_nodes`；
- 大图场景优先邻接表而非邻接矩阵，降低空间占用。

相关算法关系：

## R18

源码级流程拆解（对应 `demo.py`，8 步）：
1. `main` 定义 DAG 和含环图两组测试数据，作为最小可验证场景。
2. `kahn_topological_sort` 调用 `build_graph` 构建邻接表与入度表，并补齐边中隐含节点。
3. 将所有入度为 0 的节点压入最小堆，建立“当前可执行节点集”。
4. 循环弹出堆顶节点加入结果序列，同时计数 `processed_count`。
5. 遍历该节点所有出边，对邻居执行入度减 1；入度降到 0 时再次入堆。
6. 循环结束后比较 `processed_count` 与总节点数，判定 `has_cycle` 并封装 `TopoResult`。
7. 对 DAG 结果调用 `validate_topological_order`：建立位置映射，逐边检查 `pos[u] < pos[v]`。
8. 对含环图检查 `has_cycle=True`，并输出可排序前缀，完成“可排序 + 可报错”双路径验证。
