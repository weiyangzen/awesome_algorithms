# Diffie-Hellman密钥交换

- UID: `CS-0132`
- 学科: `计算机`
- 分类: `密码学`
- 源序号: `274`
- 目标目录: `Algorithms/计算机-密码学-0274-Diffie-Hellman密钥交换`

## R01

Diffie-Hellman（DH）密钥交换用于让通信双方在不安全信道上协商出同一个共享秘密。

给定公开参数 `p`（大素数）和 `g`（生成元）：

- Alice 选私钥 `a`，发送公钥 `A = g^a mod p`
- Bob 选私钥 `b`，发送公钥 `B = g^b mod p`
- Alice 计算 `K_A = B^a mod p`
- Bob 计算 `K_B = A^b mod p`

由于 `K_A = K_B = g^(ab) mod p`，双方得到同一共享秘密 `K`。

## R02

- 提出者: Whitfield Diffie 与 Martin Hellman（1976）。
- 历史意义: 首次系统性给出“公开信道协商密钥”的可行方案，开启现代公钥密码学范式。
- 工程地位: TLS、SSH、IPsec 等协议都采用其变体（如 DHE/ECDHE）进行会话密钥协商。

## R03

问题动机是“如何在窃听环境里建立会话密钥”。

- 传统对称加密要求预共享密钥，不适合开放网络。
- DH 使双方只交换公钥值，不直接传输秘密本身。
- 安全性依赖离散对数困难性：攻击者即使拿到 `g, p, A, B`，通常也无法高效恢复 `a` 或 `b`。

## R04

朴素不可行基线（攻击视角）:

1. 窃听者截获 `A=g^a mod p`。
2. 试图枚举或求解 `a=log_g(A) mod p`。
3. 对 `b` 同理，再计算 `g^(ab) mod p`。

该过程本质是离散对数问题。对足够大、结构正确的参数群，当前没有多项式时间通用解法。

## R05

DH 核心算法结构:

1. 选择参数群（本 MVP 用 RFC 3526 2048-bit MODP group 14）。
2. 双方分别生成随机私钥 `a,b`。
3. 计算并交换公钥 `A,B`。
4. 分别计算共享秘密 `K_A=B^a mod p`、`K_B=A^b mod p`。
5. 对 `K` 做 KDF（本 MVP 用 SHA-256）得到会话密钥。

时间复杂度主要由模幂运算决定，可视为 `O(log e)` 次大整数乘模（`e` 为指数比特长度）。

## R06

小规模示例（仅用于演示，不具备真实安全性）:

- 取 `p=23, g=5`
- Alice 私钥 `a=6`，公钥 `A=5^6 mod 23=8`
- Bob 私钥 `b=15`，公钥 `B=5^15 mod 23=19`
- Alice 算 `K_A=19^6 mod 23=2`
- Bob 算 `K_B=8^15 mod 23=2`

两者一致，得到共享秘密 `K=2`。

## R07

算法价值:

- 协议层: 解决会话密钥“首次建立”难题。
- 安全层: 配合前向安全（ephemeral key）可降低长期密钥泄露影响。
- 设计层: 是“数学难题支撑协议安全”最经典的工程样板之一。

## R08

前置知识:

- 模运算与快速幂。
- 素数群、生成元与子群基础。
- 离散对数困难性假设。
- KDF 基本概念（把共享秘密转成固定长度会话密钥）。

## R09

适用条件与边界:

- 必须使用足够大的安全参数（本 MVP 使用 2048-bit MODP）。
- 必须验证对端公钥合法性，防止明显无效输入和部分子群攻击。
- 纯 DH 只解决“保密协商”，不提供身份认证。
- 若缺少认证，协议可被中间人攻击（MITM）。

## R10

正确性要点（简述）:

1. 定义 `A=g^a mod p`，`B=g^b mod p`。
2. Alice 计算 `K_A=B^a mod p=(g^b)^a mod p=g^(ab) mod p`。
3. Bob 计算 `K_B=A^b mod p=(g^a)^b mod p=g^(ab) mod p`。
4. 因模幂乘法结合律成立，故 `K_A=K_B`。

因此双方在不暴露私钥的前提下协商同一秘密。

## R11

数值与实现稳定性:

- Python 大整数可精确处理模幂，不存在浮点误差。
- 内置 `pow(base, exp, mod)` 使用高效模幂算法，避免手写乘方溢出风险。
- KDF 前把整数按大端字节序编码，避免跨实现字节表示不一致。

## R12

工程成本（超越大 O）:

- 2048-bit 模幂 CPU 开销明显高于对称加密。
- 随机源质量直接影响私钥安全性。
- 参数选择、合法性校验、协议拼装（认证、重放防护）往往比“核心公式”更容易出错。

## R13

- 近似比/随机保证: N/A。
- 原因: DH 不是近似优化算法，而是确定性代数协议；随机性用于私钥采样而非近似求解。

## R14

常见失效模式与防护:

- 使用过小或不安全参数群: 容易被离线攻击。
- 不校验对端公钥: 可能触发无效曲线/小子群类问题。
- 私钥复用过久: 降低前向安全性。
- 仅做 DH 不做认证: 可被 MITM 完整劫持。

防护措施:

- 采用标准参数（或安全曲线）。
- 强制公钥合法性校验。
- 使用临时密钥（DHE/ECDHE）。
- 在协议层引入签名/证书或预共享认证。

## R15

实现建议:

- 直接使用 `pow(g, x, p)` 与 `pow(Y, x, p)`，避免手写慢速或错误模幂。
- 把“共享秘密”和“会话密钥”分离：先 DH，再 KDF。
- 对外部输入（对端公钥）做显式校验并抛异常。
- 在演示代码中加入负例（无效公钥、MITM）而不只演示 happy path。

## R16

前驱与后继关系:

- 前驱: 模运算、快速幂、群论基础。
- 同代/变体: ElGamal、DSA、MQV、ECDH。
- 后继工程形态:
  - TLS 1.2/1.3 中的 DHE/ECDHE 握手
  - Noise/Signal 等协议中的 DH 组合模式

## R17

本目录 `demo.py` 的 MVP 设计:

- 依赖: `numpy`（随机数生成与回归控制），`hashlib`（SHA-256 KDF）。
- 参数: RFC 3526 MODP group 14（`p` 为 2048-bit 素数，`g=2`）。
- 核心函数:
  - `sample_private_key()` 生成私钥。
  - `compute_public_key()` 生成公钥。
  - `validate_public_key()` 校验对端公钥。
  - `compute_shared_secret()` 计算共享秘密。
  - `kdf_sha256()` 派生会话密钥。
- 运行内容:
  - 一致性回归（多轮 DH，检查双方共享秘密一致）。
  - MITM 演示（说明未认证 DH 的风险）。
  - 无效公钥拦截测试。
- 运行命令:
  - `uv run python demo.py`

## R18

`demo.py` 源码级算法流（9 步，非黑箱）:

1. `main()` 依次调用 `run_regression()`、`run_mitm_demo()`、`run_invalid_key_test()`。
2. `load_default_params()` 载入 RFC 3526 group 14 参数，计算 `q=(p-1)//2`。
3. `sample_private_key()` 用 `numpy` RNG 采样随机字节，映射到区间 `[2, q-2]` 形成私钥。
4. `compute_public_key()` 用 `pow(g, x, p)` 计算公钥 `Y=g^x mod p`。
5. `validate_public_key()` 先做区间校验 `2 <= Y <= p-2`，再用 `pow(Y, q, p)==1` 校验子群归属。
6. `compute_shared_secret()` 对通过校验的对端公钥执行 `pow(Y_peer, x_self, p)` 得到共享秘密。
7. `run_single_exchange()` 创建 Alice/Bob，双方各自计算共享秘密并断言相等。
8. `kdf_sha256()` 将共享秘密整数做大端编码并加上下文标签，计算 SHA-256 输出 32-byte 会话密钥。
9. `run_mitm_demo()` 构造 Mallory 分别与 Alice/Bob 建立两把不同密钥，证明“未认证 DH”下 Alice 与 Bob 的密钥不再一致。
