## R01

外部归并排序（External Merge Sort）是一种面向“数据量远超内存容量”场景的排序算法。  
核心思想是把大文件切分为多个可放入内存的小块分别排序（生成有序段 run），再把这些有序段做多路归并，得到最终全局有序文件。

## R02

在数据库与数据工程中，它常用于：
- `ORDER BY` 结果集过大，内存无法一次装下；
- 大规模日志/明细文件离线排序；
- 构建或重建需要有序输入的数据结构（如批量索引构建前的数据预处理）；
- 任何需要“顺序磁盘 I/O + 可控内存占用”的排序任务。

## R03

问题抽象：
- 输入：一个包含 `N` 条记录的大文件（本示例记录为整数，每行一个）。
- 约束：可用内存只允许每次处理 `M` 条（`M << N`）。
- 输出：与输入记录同构、按键值非降序排列的新文件。

## R04

算法分两阶段：
1. 分块排序（Run Generation）  
   顺序读取输入文件，每 `M` 条形成一个块，在内存里排序后写出为独立有序 run 文件。
2. 多路归并（K-way Merge）  
   同时打开所有 run 文件，用最小堆维护每个 run 当前最小值，反复弹出最小项并补充下一项，直到堆空。

## R05

与“内部归并排序”区别：
- 内部归并：数据主要在内存中递归分治，I/O 不是主瓶颈；
- 外部归并：核心目标是减少随机 I/O，把排序过程转为顺序读写磁盘；
- 关注点从“比较次数”转向“磁盘读写轮次、归并路数、缓冲区管理”。

## R06

正确性要点：
- 每个 run 内部有序（来自内存排序的正确性）；
- 多路归并每次输出当前全局最小候选（堆顶），保持输出前缀始终有序；
- 当所有 run 耗尽时，输出文件包含全部记录且整体有序。

## R07

时间复杂度（比较视角）：
- 设总记录数 `N`，run 数量 `R`，归并路数约为 `R`；
- 分块排序：`sum O(m_i log m_i)`，整体近似 `O(N log M)`；
- 归并阶段：每条记录一次堆操作，`O(N log R)`；
- 总体可写为 `O(N log M + N log R)`，工程上通常近似 `O(N log N)`。

## R08

空间复杂度：
- 内存：`O(M + R)`（块缓存 + 堆/文件指针元信息）；
- 外存：除输入外，需中间 run 文件，总量与输入同阶 `O(N)`（可能略有格式开销）。

## R09

稳定性：
- 理论上外部归并可稳定；
- 本实现中：
  - 块内使用 Python `sort`（稳定）；
  - 归并时堆键采用 `(value, run_index)`，相等键优先来自更早 run；
  - 对“顺序切块”输入可保持全局稳定语义。

## R10

I/O 模型特点：
- 输入阶段：顺序读原文件 + 顺序写 run；
- 归并阶段：顺序读各 run + 顺序写输出；
- 避免大规模随机寻址，符合机械盘与大文件处理的性能习惯，也适用于对象存储分片落盘场景。

## R11

数据库系统中的典型对应：
- `External Sort` 是查询执行器中实现大规模 `ORDER BY`、`DISTINCT`、`GROUP BY`（排序聚合路径）的基础算子；
- 查询优化器会基于 `work_mem` / 可用内存估计是否走外部排序；
- 中间 run 可视作临时文件（temp files）。

## R12

本目录 MVP 设计选择：
- 语言：Python 3（仅标准库，便于直接运行）；
- 数据模型：整数文本文件（每行一个）；
- 参数：`chunk_size` 控制单块内存上限；
- 核心函数：`split_into_sorted_runs`、`merge_sorted_runs`、`external_merge_sort`。

## R13

边界与鲁棒性处理：
- 空文件：直接产生空输出文件；
- 仅一条记录：仍按统一流程处理；
- 重复值：允许且正确排序；
- 负数/大整数：按 Python `int` 处理；
- 运行后清理临时 run，避免工作目录污染。

## R14

可调优参数与权衡：
- `chunk_size` 越大，run 数越少，归并堆更小，但内存占用更高；
- `chunk_size` 越小，run 数增多，归并阶段 `log R` 增大；
- 实际工程中还可引入“分层归并”（多轮归并）控制同时打开文件数。

## R15

运行方式（在仓库根目录）：

```bash

```

脚本会自动：

- 执行外部归并排序；
- 验证输出有序性并打印统计信息。

## R16

预期输出解读：
- `records`: 总记录数；
- `chunk_size`: 每个 run 的最大记录数；
- `run_count`: 生成的有序段数量；
- `is_sorted`: 输出文件是否通过线性校验；
- `head/tail`: 排序结果前后若干项（快速人工检查）。

## R17

工程扩展方向：
- 记录从纯整数扩展为 `(key, payload)` 并保持稳定排序；
- 二进制块格式与批量缓冲，降低文本解析开销；
- 限制最大归并路数，加入多轮归并计划；
- 对接数据库临时文件管理与监控指标（spill bytes、merge pass 次数）。

## R18

`demo.py` 源码级算法流程（8 步）：
1. `generate_input_file` 生成测试数据文件，作为外部排序输入。  
2. `external_merge_sort` 创建临时目录并启动两阶段流程。  
3. `split_into_sorted_runs` 顺序读取输入，每累计 `chunk_size` 条记录就触发一次块处理。  
4. `_flush_chunk_to_run` 对块内记录执行稳定排序并落盘为 `run_*.txt`，返回 run 路径。  
5. `merge_sorted_runs` 打开所有 run 文件，读取每个 run 首元素，构建最小堆 `(value, run_index)`。  
6. 归并循环中反复 `heappop` 最小值写入输出，再从对应 run 读取下一条并 `heappush`。  
7. 堆清空后关闭文件句柄，得到全局有序输出文件；`external_merge_sort` 清理临时 run。  
8. `is_file_sorted` 与 `sample_head_tail` 对输出做线性校验与样本展示，主程序打印最终统计。
