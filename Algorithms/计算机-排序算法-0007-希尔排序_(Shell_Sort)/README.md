## R01

希尔排序（Shell Sort）是对插入排序的分组增量优化版本，核心目标是减少“元素一次只移动一格”带来的低效。

基本思想：

- 先用较大间隔 `gap` 把数组分成多个子序列；
- 对每个子序列做“间隔版插入排序”；
- 再逐步缩小 `gap`，直到 `gap=1` 时完成最终细排。

它是原地排序算法，平均性能通常优于简单 `O(n^2)` 排序。

## R02

问题定义（升序）：

- 输入：长度为 `n` 的可比较序列 `A`；
- 输出：`A` 的升序重排 `B`，满足 `B[0] <= B[1] <= ... <= B[n-1]`。

本条目 MVP 交付：

- 纯源码实现 Shell Sort（不调用第三方排序黑箱）；
- 支持两种常见 gap 序列（`halving` 与 `knuth`）；
- 输出比较次数、移动次数和每轮 gap 信息；
- 自动验证与 Python `sorted()` 一致。

## R03

为什么需要 Shell Sort：

- 插入排序在逆序或乱序数据中移动次数高，常见为二次复杂度；
- Shell Sort 先“粗排”再“细排”，让元素提前接近最终位置；
- 在中小规模数组和工程轻量场景中，往往能取得不错的时间-实现复杂度平衡；
- 算法结构简单，适合作为“从 `O(n^2)` 向更优排序过渡”的教学条目。

## R04

核心机制（以某个 `gap` 为例）：

1. 固定 `gap` 后，索引集合会形成若干组：`i, i+gap, i+2gap, ...`。  
2. 每组内部执行插入排序，但比较/移动步长为 `gap`。  
3. 当 `gap` 缩小，前一轮粗排结果会显著降低后一轮插入成本。  
4. 最后一轮 `gap=1` 时等价于普通插入排序，但数组通常已“接近有序”。

## R05

本实现流程：

1. 根据 `strategy` 生成 gap 列表（递减且最终含 `1`）；  
2. 对每个 `gap` 执行一次 `gapped_insertion_pass`；  
3. 在 pass 内做“按 gap 右移”直到找到插入位置；  
4. 记录每轮比较数、移动数、写入数与对应 gap；  
5. 返回排序结果、总体统计和每轮轨迹。

## R06

正确性要点（循环不变式）：

- 对固定 `gap` 的一轮 pass 结束后，数组满足“`gap`-有序”：
  - 对任意合法 `i`，有 `a[i] <= a[i+gap]`。
- 随着 `gap` 递减，`gap`-有序条件越来越强；
- 当最后一轮 `gap=1` 完成后，`1`-有序即全局非降序；
- 因此算法最终输出正确排序结果。

## R07

复杂度（取决于 gap 序列）：

- 最坏时间复杂度通常仍可记为 `O(n^2)`；
- 常见 gap 序列下平均性能通常优于插入/冒泡/选择等简单二次排序；
- 最好情况可接近线性级别，但受序列与数据分布影响；
- 额外空间复杂度为 `O(1)`（原地，除少量临时变量外无额外数组）。

## R08

稳定性与原地性：

- 原地：是，仅在原数组内移动元素；
- 稳定：否。  

原因：跨 gap 的远距离交换/移动可能让“相同键”的先后顺序反转。`demo.py` 中包含一个记录型反例并自动检测该性质。

## R09

边界处理策略：

- 空数组：直接返回空；
- 单元素：直接返回；
- 重复值、负数、浮点数：按统一比较规则处理；
- 混合不可比较类型：Python 会抛出类型错误，MVP 保持显式失败，不做静默吞错。

## R10

`demo.py` 的核心结构：

- `GapPassStat` / `ShellSortStats`：记录各轮与总体统计；
- `build_gaps`：生成 `halving` 或 `knuth` 序列；
- `gapped_insertion_pass`：单轮 gap 插入排序；
- `shell_sort_in_place` / `shell_sort`：排序主入口；
- `run_case`：单样例执行并对照 `sorted()`；
- `demonstrate_not_stable`：构造记录型样例验证“不稳定”；
- `main`：组织固定+随机测试并输出汇总。

## R14

关键参数与影响：

- `strategy=halving`：实现最简、教学友好；
- `strategy=knuth`：常见经验上表现更好（视数据分布而定）；
- 输入规模 `n` 越大，gap 策略差异越明显；
- 数据“接近有序”时，后期 pass 的移动与比较通常会更少。

## R15

与其他排序对比：

- 对比插入排序：同样原地，但 Shell Sort 通过增量分组减少了长距离逆序成本；
- 对比归并/堆/快排：后者平均/上界通常更强（`O(n log n)` 级），但实现复杂度也更高；
- 对比稳定排序（归并/Timsort）：Shell Sort 不稳定，不适用于必须保序的业务场景。

## R16

适用场景与不适用场景：

- 适用：
  - 中小规模数组的轻量排序；
  - 教学演示“增量思想 + 插入排序改进”；
  - 受限环境下需要简洁原地实现的场景。
- 不适用：
  - 需要稳定性的排序任务；
  - 大规模高性能生产排序（通常选 `O(n log n)` 算法或语言内建排序）。

## R18

`demo.py` 源码级算法流拆解（7 步）：

1. `main` 构造固定样例与 `numpy` 随机样例，分别调用 `run_case`。  
2. `run_case` 调用 `shell_sort`，后者复制输入并进入 `shell_sort_in_place`。  
3. `shell_sort_in_place` 先用 `build_gaps(n, strategy)` 生成递减 gap 序列（保证最终包含 `1`）。  
4. 对每个 `gap` 调用 `gapped_insertion_pass`：从 `i=gap` 到 `n-1`，取 `current=a[i]`。  
5. 在该 pass 的内层循环里，以步长 `gap` 反向比较：若 `a[j-gap] > current` 则右移到 `a[j]`，直到找到插入点。  
6. 将 `current` 写回 `a[j]`，并累计该轮 `comparisons/shifts/writes` 与总统计；全部 gap 处理完即得到排序数组。  
7. `run_case` 用 `sorted()` 验证正确性；`demonstrate_not_stable` 再用记录型数据验证同键相对次序可能改变，证明算法不稳定。  

说明：排序核心完全在本地源码中展开实现，第三方库仅用于随机样例生成与可复现测试。
