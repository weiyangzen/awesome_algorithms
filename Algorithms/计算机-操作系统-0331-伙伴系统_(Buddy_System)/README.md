## R01

伙伴系统（Buddy System）是操作系统中经典的动态内存分配算法。

核心思想是把一段总内存按 2 的幂次大小进行管理：
- 分配时，找到足够大的最小块；若块太大就不断二分；
- 释放时，尝试与“伙伴块（buddy）”合并，逐级恢复成更大块。

它在“分配速度快”和“碎片可控”之间提供了工程上可接受的折中。

## R02

问题输入：
- 总内存大小 `total_size`（2 的幂）
- 最小块大小 `min_block_size`（2 的幂，且 `<= total_size`）
- 一组内存请求序列：
  - `alloc(name, request_size)`
  - `free(name)`

问题输出：
- 每次 `alloc` 的返回地址（失败则返回 `None`）
- 每次操作后的自由链表状态（各阶空闲块）
- 统计指标：
  - 当前空闲总量、最大连续空闲块
  - 外部碎片率（`1 - largest_free / total_free`）
  - 内部碎片字节（已分配块总大小 - 实际请求总大小）

## R03

算法核心机制：
- 内存块大小限制为 `2^k`；
- 请求 `s` 会向上取整到 `max(min_block_size, ceil_pow2(s))`；
- 若目标阶无空闲块，则在更高阶找块并逐级拆分；
- 释放时用 `buddy = addr XOR block_size` 计算伙伴地址；
- 如果伙伴也空闲且同阶，就合并并继续向上尝试。

## R04

MVP 关键数据结构：
- `free_lists: dict[int, set[int]]`
  - key 为阶 `order`（`block_size = 2^order`）
  - value 为该阶所有空闲块起始地址
- `allocations: dict[int, AllocationRecord]`
  - 记录已分配地址对应的请求大小、块阶、块大小
- `name_to_addr: dict[str, int]`
  - demo 用于按名字释放

## R05

高层流程：
1. 初始化时仅有一个最大块（地址 0，阶 `max_order`）。
2. `alloc(size)`：计算目标阶 `need_order`。
3. 从 `need_order` 向上查找首个非空阶 `k`。
4. 取出一个块并从 `k` 连续拆分到 `need_order`，每次把右半块放回自由链表。
5. 返回左半块地址并登记到 `allocations`。
6. `free(addr)`：从 `allocations` 取出记录。
7. 计算伙伴地址；若伙伴同阶且空闲，则删除伙伴并向上合并。
8. 无法继续合并时，把结果块加入对应自由链表。

## R06

正确性要点（不变量）：
- 空闲块和已分配块两两不重叠；
- 任一块地址都按其块大小对齐（`addr % block_size == 0`）；
- 每次拆分/合并都保持“总内存守恒”；
- 释放后若伙伴可用，必定优先合并，不会漏掉可合并路径。

`demo.py` 在每步操作后做区间覆盖断言，直接检查上述不变量。

## R07

复杂度分析（设阶数为 `L = log2(total_size/min_block_size) + 1`）：
- 分配：最坏需要向上查找并向下拆分，时间 `O(L)`；
- 释放：最坏逐级合并到顶层，时间 `O(L)`；
- 空间：`O(number_of_blocks)`，MVP 中由自由链表与分配表主导。

## R08

边界与异常处理：
- `total_size`、`min_block_size` 非 2 的幂：抛异常；
- `min_block_size > total_size`：抛异常；
- `alloc(size<=0)`：抛异常；
- 请求大于总内存或当前无法满足：返回 `None`；
- `free` 非法地址或重复释放：抛 `KeyError`。

  - 多次分配（不同尺寸）

## R11

预期现象：
- 小请求会被提升到最近的 2 次幂块，形成内部碎片；
- 释放后如果伙伴仍在用，无法立即合并，外部碎片会短时升高；
- 当成对伙伴都释放后，块会级联合并，最大连续空闲块迅速恢复；
- 最终全部释放时，应恢复到单一最大块（无碎片）。

## R12

指标解释：
- `internal_frag_bytes`：已分配块浪费空间（向上取整成本）；
- `external_frag_ratio`：空闲内存被切碎程度（越接近 0 越好）；
- `largest_free`：当前一次可满足的大请求上限；
- `alloc_failures`：请求超过可用结构时的失败次数。

## R13

与其他分配策略对比：
- 相比“任意大小链表”，伙伴系统更易快速合并，结构简单；
- 相比固定大小池（slab/对象池），伙伴系统更通用，但内部碎片更明显；
- 真实操作系统常把伙伴系统用于页框级管理，再叠加 slab/arena 管理小对象。

## R14

参数选择经验：
- `min_block_size` 过小：链表层数增多，管理开销上升；
- `min_block_size` 过大：小请求内部碎片明显；
- `total_size` 通常按页大小与物理内存分区规划，便于与页分配器对齐。

## R15

常见实现错误：
1. buddy 计算错误（应使用 `addr XOR block_size`）；
2. 拆分时把错误半块放回空闲链表；
3. 释放时未循环向上合并，只合并一层；
4. 忘记校验地址对齐与重复释放，导致隐式内存破坏。

## R18

`demo.py` 源码级流程拆解（8 步，非黑盒）：
1. 初始化 `BuddyAllocator`：计算 `min_order/max_order`，并在 `free_lists[max_order]` 放入地址 `0`。  
2. `allocate(size)` 先把请求向上取整到目标块阶 `need_order`，再从该阶向上搜索可用空闲块。  
3. 找到高阶块后循环拆分：每次把右半块加入下一阶自由链表，左半块继续用于下一轮拆分。  
4. 到达目标阶后，将左半块登记到 `allocations` 并返回地址。  
5. `free(addr)` 取出分配记录，计算同阶伙伴地址 `buddy = addr ^ block_size`。  
6. 若伙伴在同阶空闲链表中，则移除伙伴并合并为更高阶块，继续循环尝试向上合并。  
7. 每次操作后执行 `assert_integrity()`：检查所有空闲/已分配区间无重叠、全覆盖且总长度等于 `total_size`。  
8. 演示序列结束后汇总 `internal/external` 碎片与失败次数，并断言“最终恢复单一最大空闲块”。
