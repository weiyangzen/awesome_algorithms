## R01

Lamport 时间戳（Lamport Logical Clock）是分布式系统中的逻辑时钟机制，用来在没有全局物理时钟的情况下给事件赋予“可比较的时间标签”。

它解决的核心问题是：

- 分布式节点物理时钟不同步；
- 网络延迟不确定；
- 仍然需要表达事件之间的先后关系（尤其是因果相关的先后）。

## R02

典型动机来自事件排序与一致性推断：

- 分布式日志重放需要稳定排序；
- 多副本冲突处理需要“谁先发生”的依据；
- 调试与追踪需要把跨机器事件对齐到同一条逻辑时间线。

Lamport 时间戳不依赖 NTP 精度，也不要求共享硬件时钟，工程可落地性很高。

## R03

基础模型：

- 每个进程内部事件有局部顺序；
- 消息发送与接收构成跨进程因果边；
- “happens-before”（记作 `a -> b`）由以下规则定义：
  - 同一进程中，先执行事件指向后执行事件；
  - 一次消息的发送事件指向该消息的接收事件；
  - 关系满足传递性。

Lamport 时间戳的目标是满足：若 `a -> b`，则 `L(a) < L(b)`。

## R04

算法规则（每个进程维护一个整数时钟 `C`）：

1. 本地事件发生前：`C = C + 1`，该事件时间戳记为 `C`；
2. 发送消息前：`C = C + 1`，把 `C` 附在消息里；
3. 接收消息时（消息携带时间戳 `Tmsg`）：
   - `C = max(C, Tmsg) + 1`；
   - 接收事件时间戳记为更新后的 `C`。

这三条规则保证因果边在时间戳上单调递增。

## R05

复杂度非常低：

- 每个事件更新时钟是 `O(1)`；
- 每条消息额外携带一个整数；
- 若系统总事件数为 `E`，总维护开销是 `O(E)`。

因此它适合高吞吐、低开销的分布式基础组件。

## R06

本条目 `demo.py` 的实验设计：

- 3 个进程（`P1/P2/P3`）；
- 固定脚本生成 13 个事件（本地、发送、接收）；
- 4 条消息链（`m1..m4`）构成跨进程因果路径；
- 自动校验 Lamport 时钟条件；
- 使用 `numpy` 计算 happens-before 的传递闭包并找并发事件示例。

运行时无交互输入，便于批处理验证。

## R07

优点：

- 实现简单，工程成本低；
- 能正确保持“因果先后必然对应时间戳递增”；
- 可直接用于全局日志排序的第一层键。

局限：

- `L(a) < L(b)` 不能反推 `a -> b`（可能只是并发事件）；
- 不能识别并发关系本身；
- 若业务严格依赖因果可判定，通常要用向量时钟。

## R08

前置知识与环境：

- 分布式系统中的消息传递模型；
- 偏序关系（happens-before）；
- Python 3.10+；
- `numpy`（用于闭包计算与并发检测示例）。

## R09

适用场景：

- 分布式日志排序与回放；
- 跨节点事件追踪（Tracing）中的逻辑时间标注；
- 多副本事件流的稳定归并。

不适用场景：

- 需要精确识别并发冲突集合；
- 需要度量真实物理时延；
- 需要与外部时间（如交易所毫秒级时间）强绑定。

## R10

正确性要点（简版）：

- 同一进程内，每次事件前时钟至少加 1，所以进程序列边满足严格递增；
- 对发送 `s` 和接收 `r`，接收端执行 `max(C, Ts) + 1`，必有 `L(r) > Ts = L(s)`；
- 对传递边 `a -> b -> c`，严格不等式可传递。

因此任意 `a -> b`，都有 `L(a) < L(b)`，满足 Lamport clock condition。

## R11

`demo.py` 的鲁棒性处理：

- 拒绝重复进程 ID；
- 拒绝重复在途消息 ID；
- 拒绝接收不存在或已消费的消息；
- 拒绝“消息目的进程与接收方不一致”；
- 结束时断言没有未投递消息残留。

这些检查能避免示例“跑通但语义错误”。

## R12

实现细节选择：

- 事件同时记录：全局 `event_id`、进程内 `local_index`、Lamport 时间戳；
- happens-before 边拆成两类：
  - 进程内相邻事件边；
  - 发送到接收的消息边；
- 使用确定性脚本而非随机调度，保证每次运行输出一致，便于验证。

## R13

理论边界：

- Lamport 时间戳给出的是“与因果兼容的标号”，不是完整因果编码；
- 通过 `(timestamp, process_id, local_index)` 可以构造确定性总序；
- 这个总序可用于实现层面的“可复现排序”，但总序中的相邻事件不一定有真实因果关系。

## R14

常见错误：

1. 接收事件忘记 `+1`，写成 `max(C, Tmsg)`；
2. 发送事件不递增时钟，导致跨进程边无法保证严格不等；
3. 把 `L(a) < L(b)` 误当成 `a -> b` 的充要条件；
4. 日志排序只按 Lamport 值，不加稳定 tie-break 键。

## R15

可扩展方向：

- 升级为向量时钟（可判定并发）；
- 把消息通道改成异步队列并模拟乱序到达；
- 与分布式 KV 冲突解决策略结合（Last-Writer-Wins / CRDT）；
- 增加故障注入（丢包、重复包、延迟）做鲁棒性实验。

## R16

相关算法与机制：

- Vector Clock（向量时钟）：更强的因果表达能力；
- Hybrid Logical Clock（HLC）：把物理时间与逻辑时间结合；
- Causal Broadcast：在广播系统中维持因果交付；
- Paxos/Raft 中日志索引与任期并非 Lamport 时钟，但同样服务于跨节点有序性约束。

## R17

## R18

`demo.py` 的源码级算法流程（8 步）：

1. `build_demo_scenario` 初始化 `P1/P2/P3` 三个进程，每个进程时钟初值为 0。  
2. 依次执行本地事件、发送、接收；本地和发送调用 `tick_local_or_send`，使本地时钟加 1。  
3. 发送时把当前 Lamport 值写入消息对象，并记录发送事件 ID，消息放入 `in_flight`。  
4. 接收时读取消息中的发送时间戳 `Tmsg`，执行 `C = max(C, Tmsg) + 1`，再记录接收事件。  
5. 每记录一个事件都添加进程内顺序边；每次接收再添加“发送 -> 接收”消息边，形成 happens-before 图。  
6. `assert_clock_condition` 遍历全部因果边，逐条断言源事件时间戳严格小于目标事件时间戳。  
7. `compute_reachability` 使用布尔 Warshall 算法计算传递闭包，并据此检测一组并发事件。  
8. 程序输出执行日志、基于 `(LamportTs, ProcessId, LocalIndex)` 的总序和最终时钟，展示“保因果但不反推因果”的性质。

本 MVP 没有调用第三方分布式框架黑盒实现 Lamport 时钟；核心规则在源码中逐行实现并可直接追踪。
