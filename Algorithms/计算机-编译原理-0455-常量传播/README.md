## R01

常量传播（Constant Propagation）是经典编译优化：在编译期通过数据流分析推断变量是否为常量，并把后续表达式中的变量替换为字面量，从而减少运行期计算。

本条目实现的是“前向、流敏感、基于 CFG 的稠密常量传播”MVP，不依赖编译器框架，直接在 Python 中用最小 IR 复现算法核心。

## R02

目标问题：给定控制流图（CFG）与三地址风格指令，求每个基本块入口/出口处每个变量的抽象值：

- `UNDEF`：当前尚无信息；
- `CONST(c)`：可证明恒为常量 `c`；
- `NAC`（Not A Constant）：无法视为单一常量（例如不同路径给出不同值）。

分析结果可用于常量替换、常量折叠触发、死代码清理前置判断等。

## R03

本实现使用三点格（three-point lattice）：

- 底：`UNDEF`
- 中间：`CONST(c)`（不同 `c` 属于不同格元素）
- 顶：`NAC`

合流（meet）规则：

- `meet(UNDEF, x) = x`
- `meet(CONST(a), CONST(a)) = CONST(a)`
- 其他冲突情况统一提升到 `NAC`

这保证了跨路径信息合并时的保守正确性。

## R04

数据流方程（前向分析）：

- `IN[B] = meet(OUT[P])`，其中 `P` 为 `B` 的所有前驱；
- `OUT[B] = F_B(IN[B])`，`F_B` 为基本块传递函数。

求解策略采用工作队列迭代，直到所有 `IN/OUT` 状态收敛（不再变化）。

## R05

`demo.py` 的指令级传递函数支持三类操作：

- `("const", x, k)`：`x <- CONST(k)`
- `("copy", x, y)`：`x <- value(y)`
- `("binop", x, op, a, b)`：根据 `a,b` 的格值推断 `x`

对二元运算：

- 若任一操作数为 `NAC`，结果 `NAC`；
- 若任一操作数为 `UNDEF`，结果 `UNDEF`；
- 若两者均是常量，则编译期直接计算得到 `CONST`。

## R06

MVP 的 CFG 用一个小型案例展示两种典型行为：

- 菱形分支中两条路径都给 `p=24`，在汇合点仍保持常量；
- 第二次分支中路径分别给 `mix=1` 与 `mix=2`，汇合后 `mix` 变为 `NAC`，下游表达式也会退化为 `NAC`。

因此同一示例里同时验证“可传播常量”和“冲突失常量”两类结果。

## R07

复杂度（设变量数 `V`、边数 `E`、指令数 `I`）：

- 单次传递成本约 `O(I)`；
- 每次合流涉及变量集合合并，量级约 `O(V)`；
- 工作队列总迭代次数与图结构和格高度有关，MVP 场景下可视作有限收敛。

整体上对教学/中小 CFG 足够轻量。

## R08

优点：

- 没有黑盒依赖，所有推断步骤可逐行追踪；

## R10

正确性直觉（非形式化证明）：

- 合流函数对冲突信息上推到 `NAC`，保证不会把“非常量”误判为常量；
- 传递函数仅在可证明时产生 `CONST`，否则保持 `UNDEF/NAC`；
- 工作队列持续传播变化直到不动点，得到全图一致解。

因此结果满足“保守但安全”的静态分析要求。

## R11

边界处理：

- 使用未知变量时默认为 `UNDEF`（状态稀疏存储）；
- 整数除法 `//` 遇到除数常量为 0 时返回 `NAC`，避免分析阶段异常中断；
- 若 CFG 后继引用不存在的块，立即抛出 `ValueError`。

这些处理让 MVP 在演示场景下更稳健可复现。

## R12

与相关优化的关系：

- 常量传播：推断变量是否恒定；
- 常量折叠：在表达式层面把常量运算直接计算；
- 复制传播：把 `x=y` 的引用链继续展开。

本条目聚焦第一项，但在打印阶段已经演示了“把已知常量替换到指令文本”的传播效果。

## R13

实现细节选择：

- 用 `LatticeValue(kind, value)` 显式建模三点格，便于断言与调试；
- 状态采用“稀疏字典 + 缺失即 UNDEF”减少冗余；
- 使用确定性工作队列，保证多次运行输出顺序稳定；
- 把解析、传递、合流、打印拆成独立函数，便于单测化扩展。

## R14

常见实现错误：

1. 把路径冲突常量错误保留为某一路值，而非上推到 `NAC`；
2. 没有把 `UNDEF` 作为格底处理，导致初始状态污染；
3. 只更新 `OUT` 不回灌后继，工作队列无法收敛到真实不动点；
4. 直接依赖第三方“优化库”而不解释传递/合流细节，难以验证正确性。

## R16

适用与不适用场景：

- 适用：教学演示、编译优化原型、静态分析入门工具。
- 不适用：需要精确内存模型、跨过程分析、异常流和别名分析的工业编译器全量优化。

对大型真实程序，常量传播通常只是优化管线中的一个子阶段。

## R17

`demo.py` 功能清单：

- `constant_propagation`：工作队列求不动点；
- `meet_value/meet_states`：合流逻辑；
- `transfer_instruction/transfer_block`：传递函数；
- `print_analysis`：打印每个基本块 IN/OUT 与替换后的指令；
- `summarize_state_counts`：用 `numpy` 汇总每块 `CONST/NAC/UNDEF` 数量；
- `run_demo`：构造固定 CFG、执行断言并输出结果。

脚本无需任何交互输入，可直接运行完成验证。

## R18

`demo.py` 的源码级算法流程（8 步）：

1. 在 `run_demo` 中手工定义 CFG（块内指令 + 后继关系），并设置入口块 `entry`。  
2. `constant_propagation` 调用 `build_predecessors` 生成前驱表，初始化每个块的 `IN/OUT` 状态为空字典（即全 `UNDEF`）。  
3. 使用工作队列迭代：取出一个块后，先把所有前驱 `OUT` 通过 `meet_states` 合并成该块新的 `IN`。  
4. `transfer_block` 逐条执行 `transfer_instruction`，得到新的 `OUT`；若 `IN/OUT` 发生变化，就把后继块重新入队。  
5. `transfer_instruction` 对三种指令分派：常量赋值、复制赋值、二元运算；二元运算通过 `eval_binop` 根据格值规则得出 `CONST/UNDEF/NAC`。  
6. 队列收敛后，`print_analysis` 逐块打印 `IN/OUT`，并用 `render_instruction` 把当前可证明常量直接替换进文本，展示传播效果。  
7. `run_demo` 中的断言验证关键结论：`t`、`s` 为常量，而 `merge2` 的 `mix/n/final` 为 `NAC`。  
8. `summarize_state_counts` 使用 `numpy` 统计每个块输出状态里 `CONST/NAC/UNDEF` 的数量，最后打印通过信息。

本实现没有调用任何“几行就完成优化”的黑盒库；第三方依赖仅用于统计展示，常量传播核心流程全部在本地源码中可追踪。
