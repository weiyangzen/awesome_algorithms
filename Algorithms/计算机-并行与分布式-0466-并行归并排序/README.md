## R01

并行归并排序（Parallel Merge Sort）是在经典归并排序基础上引入并行执行的一类排序算法。  
核心目标是把“可独立处理”的子问题同时交给多个计算单元处理，再把局部有序结果按归并规则合并为全局有序结果。

本条目给出一个可直接运行的 Python MVP：
- 顺序归并排序作为基线；
- 使用多进程实现并行分块排序 + 分层并行归并；
- 自动做正确性校验与耗时对比。

## R02

为什么要并行化归并排序：
- 归并排序天然分治，左右子问题彼此独立，结构上非常适合并行；
- 在多核 CPU 上，排序常是数据预处理瓶颈（如日志、特征工程、离线 ETL）；
- 对大数组，单核递归排序容易受到总时长限制，并行可降低 wall-clock 时间。

归并排序还具有稳定性和 `O(n log n)` 的可预期复杂度，工程上比部分“平均快但最坏退化”的算法更稳健。

## R03

算法思路（本实现）：
1. 将原始数组切分成多个块（chunk）；
2. 每个块独立执行顺序归并排序（并行阶段 1）；
3. 将多个“局部有序块”按层进行二路归并（并行阶段 2）；
4. 重复归并层，直到只剩一个有序块。

这里并不是调用黑盒排序库完成“并行归并排序”，而是把“分块、排序、分层归并”全过程显式写在源码里。

## R04

与传统单线程版本的差异：
- 单线程：递归地先排左半、再排右半、最后归并；
- 并行版：将“多个子块排序”和“同层多对归并”同时投递到进程池；
- 通过 `min_chunk_size` 控制并行粒度，避免任务过细导致进程调度开销反噬；
- 当数据规模小或 `workers=1` 时自动回退顺序实现，保证行为稳定。

## R05

复杂度分析：
- 顺序归并排序：时间 `O(n log n)`，额外空间 `O(n)`；
- 并行版本（理想负载均衡）：
  - 总工作量仍约 `O(n log n)`；
  - 临界路径近似可降到 `O((n log n)/p + log p * n/p)`（`p` 为并行度，简化估计）；
  - 现实中还需叠加进程通信/序列化开销与任务调度开销。

因此并行并不保证总是更快，关键取决于数据规模与并行粒度配置。

## R06

`demo.py` 的实验设置：
- 随机数据规模：`n=150000`；
- 数据生成：`numpy.random.default_rng(seed=20260407)`，保证可复现；
- 并行工作进程：`workers=min(4, os.cpu_count())`；
- 并行阈值：`min_chunk_size=20000`；
- 输出：顺序版耗时、并行版耗时、两者结果一致性检查。

脚本无交互输入，直接运行即可复现实验。

## R07

本 MVP 的并行策略优势：
- 仅用标准库 `concurrent.futures.ProcessPoolExecutor`，依赖简单；
- 分块排序与分层归并均支持并行，不是只“并行排块、串行合并”；
- 结果确定性强：同一输入必得相同有序输出；
- 易扩展到外部排序或分布式场景（把 chunk 从内存块换成分区文件/分区节点即可）。

## R09

适用场景：
- 多核机器上的大规模整数/可比较对象排序；
- 数据预处理流水线中“先排序再聚合/去重”的步骤；
- 教学与实验：演示分治算法如何迁移到并行执行模型。

不太适用：
- 极小数据集（并行调度开销可能大于收益）；
- 需要就地排序且内存极端受限的场景（归并需要额外缓存）；
- 单核或进程启动成本很高的运行环境。

## R10

正确性要点（简版）：
1. `merge_two_sorted_lists(left, right)` 保证若 `left/right` 有序，则输出整体有序；
2. `sequential_merge_sort` 对长度 0/1 的数组显然正确，长度 `n` 的正确性可由归纳法得到；
3. 并行版只改变“子任务执行时机与并发方式”，不改变每个子任务的排序/归并语义；
4. 各层归并始终只合并有序块，因此最终结果与顺序归并排序一致。

`demo.py` 通过与 `sorted(case)` 对比及顺序版/并行版一致性断言来做实证校验。

## R11

实现中的边界与鲁棒性处理：
- 空数组、单元素数组直接返回副本；
- `max_workers` 自动规整到 `>=1`；
- 小数据自动回退顺序版本，防止无意义并行；
- `_assert_sorted` 对输出做单调性检查；
- `_self_test` 覆盖空输入、重复值、负数、逆序等典型边界。

## R12

关键函数职责：
- `merge_two_sorted_lists`：二路线性归并；
- `sequential_merge_sort`：递归顺序归并排序；
- `_split_into_chunks`：按块切分输入；
- `parallel_merge_sort`：并行分块排序 + 分层并行归并；
- `_self_test`：快速正确性回归；
- `_benchmark`：测量不同实现耗时；
- `main`：构造数据、执行实验、打印结论。

## R13

性能观察建议：
- 调大 `n`（如 3e5、5e5）通常更容易看到并行收益；
- `workers` 不宜盲目拉满，常见经验是 `CPU 核心数` 或略低；
- `min_chunk_size` 太小会造成大量短任务，IPC 成本偏高；
- 对纯 Python 归并逻辑，进程间序列化成本是主要瓶颈之一。

因此本条目更强调“算法与并行流程可追踪”，而非追求极限性能。

## R14

常见实现错误：
1. 只并行了切分排序，但最后串行逐个 merge，导致并行收益有限；
2. 在并行函数中直接调用不可 pickle 的闭包/局部函数，引发进程池报错；
3. 分块逻辑遗漏尾块，导致元素丢失；
4. 归并时比较符使用不当，破坏稳定性或顺序性；
5. 忽略小数据回退策略，结果是并行版反而更慢。

## R15

可扩展方向：
- 用共享内存或 `numpy` 结构减少大列表序列化开销；
- 将进程池并行替换为多机分布式任务（Spark/Dask 风格）；
- 增加外排序流程：块排序后落盘，再多路归并；
- 对对象排序增加 `key` 提取器与稳定性验证；
- 加入多轮基准测试（均值/中位数/方差）形成更可靠的性能评估。

## R18

`demo.py` 源码级算法流程（8 步）：
1. `main` 先执行 `_self_test`，用多组边界样例验证顺序版与并行版都等于 `sorted(case)`。  
2. `main` 用 `numpy` 生成固定随机种子的数据集，并设定 `workers` 与 `min_chunk_size`。  
3. 顺序路径调用 `sequential_merge_sort`：递归拆分数组到长度 1，再由 `merge_two_sorted_lists` 自底向上归并。  
4. 并行路径调用 `parallel_merge_sort`：先根据 `min_chunk_size` 与 `workers` 计算分块数，并通过 `_split_into_chunks` 切片。  
5. 进程池第一阶段执行 `pool.map(sequential_merge_sort, chunks)`，得到多个局部有序块。  
6. 进入分层归并循环：把相邻两块打包成任务，`pool.map(_merge_pair, pair_tasks)` 并行归并；若层内块数为奇数则保留尾块进入下一层。  
7. 当仅剩一个块时返回全局有序结果，并与顺序版结果逐元素比较，同时调用 `_assert_sorted` 检查单调性。  
8. `main` 打印顺序耗时、并行耗时和 speedup，形成可复现实验闭环。  

本实现没有把第三方库当作“并行排序黑箱”直接调用，核心分治与归并逻辑都在源码中可逐步追踪。
