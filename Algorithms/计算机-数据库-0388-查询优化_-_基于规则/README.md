## R01

基于规则的查询优化（Rule-Based Query Optimization, RBO）是在**不依赖代价模型**的前提下，利用一组语义等价重写规则改写逻辑查询计划。

典型目标是：

- 更早过滤数据（减少中间结果）；
- 更早裁剪列（减少 I/O 与内存传输）；
- 消除冗余算子（减少执行路径长度）。

## R02

核心直觉是“固定启发式优先级”。

例如在大多数场景，`Selection Pushdown`（选择下推）几乎总是有利：

- `Filter` 放在 `Join` 上方会先产生更大的中间结果；
- 下推到 `Join` 两侧可以让参与连接的数据量更小；
- 即使没有精确代价估算，通常也会得到更好的执行形态。

## R03

本条目解决的问题：实现一个可运行的最小 RBO MVP，输入为逻辑计划树，输出为规则重写后的等价计划。

`demo.py` 中的演示查询对应如下结构：

- `Project(o.order_id, c.name)`
- `Filter(o.amount>100 AND o.status='paid' AND c.tier='gold')`
- `Join(o.customer_id=c.customer_id)`
- `Scan(orders)` 与 `Scan(customers)`

优化后应把过滤与列裁剪推到更靠近数据源的位置。

## R04

本目录实现采用显式计划树与显式规则函数，不把数据库优化器当黑盒。

计划节点：

- `Scan`
- `Filter`
- `Project`
- `Join`

主要规则：

- 规则组 A（规范化）：合并相邻 `Filter`、去重谓词、去重/折叠 `Project`、移除恒等 `Project`；
- 规则组 B（下推）：`Filter` 穿越 `Project`，`Filter` 按引用列拆分后下推到 `Join` 左右子树；
- 规则组 C（列裁剪）：根据输出列与谓词引用列做 top-down `Project` 插入。

## R05

复杂度分析分两部分：

- 优化阶段：每轮重写遍历计划树，设节点数为 `N`，迭代轮数为 `I`，则约 `O(I*N)`；
- 执行阶段（本演示 Join 实现为嵌套循环）：若左右输入行数分别为 `L`、`R`，连接成本约 `O(L*R)`。

本 MVP 关注“规则重写正确性与可解释性”，不是执行引擎性能极限。

## R06

`demo.py` 使用固定数据集，刻意包含无关列：

- `orders` 含 `o.extra`；
- `customers` 含 `c.region`。

这让列裁剪规则有可观测效果：优化后计划中不再保留与输出和谓词无关的列。

## R07

优势：

- 可解释：每条规则的触发与结果都可追踪；
- 成本低：不需要统计信息、直方图或复杂代价模型；
- 工程可控：规则可按白名单逐条上线。

局限：

- 对“是否更优”的判断粗粒度，可能错过更佳执行顺序；
- 缺少数据分布感知；
- 对复杂查询（多表连接重排序）通常不如 CBO（Cost-Based Optimization）。

## R08

前置知识：

- 关系代数等价变换（Selection/Projection Pushdown）；
- 逻辑计划树表示；
- 谓词引用列集合（`predicate refs`）概念。

运行环境：

- Python 3.10+
- 仅标准库（`dataclasses`, `typing`），无交互输入。

## R09

适用场景：

- 教学、原型系统或轻量引擎；
- 缺乏稳定统计信息时的保守优化；
- 作为 CBO 前置规范化阶段。

不适用场景：

- 超大规模复杂查询仅靠规则难以达到最佳执行计划；
- 强依赖基数估计与 join order 搜索的场景。

## R10

正确性依据是“语义等价重写”不变式：

1. `Filter` 下推仅在谓词引用列被子计划完整提供时进行；
2. `Filter` 在 `Join` 上方拆分为左/右/残留三类，跨表谓词不会被错误下推；
3. 列裁剪始终保留输出列、Join 键列、Filter 引用列；
4. 最终通过 `execute_plan` 对比优化前后结果行，保证结果一致。

## R11

鲁棒性实现细节：

- 谓词使用 `Predicate(name, refs)` 显式声明依赖列，避免字符串猜测；
- `dedup_predicates` 与规则统计 `RuleStats` 防止重复重写导致计划膨胀；
- 优化采用 fixpoint 循环并设 `max_iters=10`，防止异常规则循环；
- `assert original_rows == optimized_rows` 做语义回归校验。

## R12

关键参数：

- `max_iters`：规则迭代上限；
- `output_columns`：最终查询输出列，驱动列裁剪强度；
- `refs` 精度：谓词引用列越准确，规则越安全。

调参建议：

- 规则开发初期保持较小 `max_iters` 并增加日志；
- 对复杂谓词先保守放入 `residual`，确认安全后再增强下推；
- 先保证等价性，再追求更多激进重写。

## R13

理论性质：

- 近似比：N/A（该算法是语义等价重写，不是近似算法）；
- 随机化成功率：N/A（全流程确定性）；
- 终止性：在有限节点与有限规则集下，且有迭代上限，优化过程必终止。

## R14

常见失效模式：

1. 谓词 `refs` 写错，导致错误下推；
2. 列裁剪遗漏 Join 键或 Filter 列，导致执行时报字段缺失；
3. 规则次序不当，出现“推下去又拉回来”的震荡；
4. 仅做计划形态断言，未做结果等价断言。

排查建议：

- 打印优化前后计划树；
- 打印规则触发计数；
- 做结果行级对比（本 demo 已内置）。

## R15

工程化建议：

- 将规则定义为可插拔列表，支持灰度开关；
- 给每条规则增加前置条件和单元测试；
- 在真实系统中把 RBO 作为“逻辑规范化层”，下游再接 CBO；
- 输出可观测信息（规则命中、节点数变化、列数变化）。

## R16

与其他优化范式对比：

- RBO：可解释、轻量、无需统计，但全局最优能力有限；
- CBO：依赖统计信息和代价估算，通常更接近最优但实现复杂；
- Heuristic + CBO 混合：先用 RBO 做安全缩减，再用 CBO 做高价值决策，是工程上常见路径。

本条目专注 RBO 的“源码可追踪最小闭环”。

## R17

`demo.py` 的 MVP 能力：

- 构建逻辑计划树（`Scan/Filter/Project/Join`）；
- 执行规则优化到不再变化（fixpoint）；
- 统计每条规则触发次数；
- 执行优化前后计划并断言结果完全一致；
- 输出优化前计划、优化后计划、规则命中统计与最终结果行。

运行方式：

```bash
cd Algorithms/计算机-数据库-0388-查询优化_-_基于规则
uv run python demo.py
```

## R18

`demo.py` 源码级算法流程（8 步）：

1. `build_demo_plan` 先构建一个刻意“未优化”的计划：`Project -> Filter -> Join -> Scan`。  
2. `optimize_rule_based` 进入 fixpoint 循环，先调用 `canonicalize` 做规范化（合并 Filter、清理冗余 Project）。  
3. 同一轮里调用 `pushdown_filters`：把 `Filter` 从 `Join` 上方拆成左侧谓词、右侧谓词和残留谓词，并下推到子树。  
4. 再次 `canonicalize` 清理重写后可能出现的重复结构；若本轮无变化则停止迭代。  
5. 退出循环后执行 `prune_columns`：从输出列集合出发向下传播 required columns，在 `Scan` 上方插入必要 `Project`。  
6. `execute_plan` 分别执行优化前后计划（本 MVP 的 Join 执行为显式双重循环），得到两份结果集。  
7. 用 `assert original_rows == optimized_rows` 验证语义等价，同时断言过滤下推与列裁剪确实发生。  
8. `format_plan` 与 `summarize_stats` 输出计划树和规则命中统计，形成可审计的优化闭环。

以上流程完全在源码中展开，没有把第三方优化器或数据库引擎当黑盒。
