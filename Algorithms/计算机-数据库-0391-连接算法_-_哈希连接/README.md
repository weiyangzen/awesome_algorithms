## R01

哈希连接（Hash Join）是数据库中处理等值连接（equi-join）的核心算法之一。

它把连接拆成两个阶段：

- Build：对一侧表按连接键建哈希表；
- Probe：扫描另一侧表，用连接键探测哈希表并输出匹配行。

相对嵌套循环连接，哈希连接通常能显著减少无谓比较，尤其适合大表等值连接。

## R02

算法直觉：

- 先把“查找对象”组织成键到行列表的映射（哈希桶）；
- 再把“查询对象”逐行拿来做 O(1) 期望时间的键查找；
- 找到桶后直接产出匹配行对（支持一对多、多对多）。

这相当于用空间换时间，把双重扫描中的大量“失败比较”变成哈希定位。

## R03

本条目解决的问题：

- 输入：左表、右表（`list[dict]`）、左右连接键（支持单列或复合键）；
- 输出：连接结果行（列名前缀 `l_` / `r_`）与执行统计（build/probe/桶数量/匹配数）。

MVP 支持以下连接类型：

- `inner`
- `left`
- `right`
- `full`

## R04

本目录的实现策略：

1. 规范化键定义（`normalize_key_spec`）；
2. 按 `build_side` 决定构建侧（默认 `auto` 选更小侧，等长时固定右侧，保证输出可复现）；
3. 用 Python `dict[key] -> [row_index,...]` 构建哈希表；
4. Probe 阶段按键查桶，输出所有匹配组合；
5. 用匹配位图回填外连接未匹配行；
6. 返回 `JoinResult(rows, stats)`，并在 `main` 中断言关键语义。

## R05

复杂度分析（设 Build 侧 `B` 行，Probe 侧 `P` 行）：

- Build 时间：`O(B)`
- Probe 时间：`O(P + K)`，`K` 为匹配对数（输出规模）
- 总时间：`O(B + P + K)`（平均哈希查找）
- 额外空间：`O(B)`（哈希表与匹配标记）

当连接键分布极端倾斜（热点键）时，某些桶会很大，单次 probe 的展开成本会上升。

## R07

优点：

- 等值连接场景下性能通常优于纯嵌套循环；
- 实现结构清晰，Build/Probe 两段可单独统计与调优；
- 对未排序输入友好，不要求预先排序。

局限：

- 主要适用于等值连接，不直接覆盖范围连接；
- 需要额外内存存放哈希表；
- 键倾斜严重时会退化到“大桶展开”瓶颈。

## R09

适用场景：

- 大量等值连接；
- 输入未排序，且希望避免排序开销；
- 需要在教学或轻量代码中复现数据库连接核心思想。

不适用场景：

- 非等值连接（如 `<`, `BETWEEN`）为主；
- 可用内存受限，Build 侧无法放入内存；
- 键分布极度倾斜且无法分区缓解。

## R10

正确性直觉：

1. Build 阶段将每条 Build 行按键完整记录到哈希桶，不丢行；
2. Probe 阶段对每条 Probe 行都执行一次查桶，不漏检；
3. 命中桶时输出“Probe 行 × 桶内 Build 行”的全部组合，不漏匹配；
4. 外连接通过匹配位图补齐未命中侧，保证语义完整。

因此该实现满足等值连接的完备枚举与外连接补齐规则。

## R11

鲁棒性与实现细节：

- 对连接键规格做长度一致性校验；
- 键提取统一使用 `build_key`，复合键转元组比较；
- 输出列统一前缀化，避免同名列冲突；
- `build_side`、`join_type` 都有显式参数校验，防止静默降级。

## R12

关键参数：

- `left_on` / `right_on`：连接键
- `join_type`：`inner|left|right|full`
- `build_side`：`auto|left|right`

调参建议：

- 一般使用 `build_side="auto"`，优先构建更小侧；
- 如需稳定对齐某执行计划，可显式固定 build 侧；
- 若出现热点键，可先做分区或预聚合再连接。

## R14

常见失效模式：

1. 左右键类型不一致（如字符串 `"101"` 对整数 `101`）导致匹配丢失；
2. 复合键列顺序不一致导致“逻辑同键、实现不同键”；
3. 把高基数大表误设为 Build 侧，造成内存压力；
4. 外连接回填模板列缺失，导致下游字段不完整。

排查方式：

- 打印 Build 桶数与样例桶内容；
- 统计 `matched_pairs` 与人工预期对比；
- 对未匹配行数量做断言（本 MVP 已示例）。

## R15

工程化建议：

- 为连接算子统一接口（NLJ/Hash/Merge）方便优化器切换；
- 记录执行统计（桶数、probe 次数、匹配对数）用于成本模型；
- 对热点键加入倾斜检测与告警；
- 大规模场景扩展为分区哈希连接（Grace Hash Join）以控制内存峰值。

## R16

与其他连接算法对比：

- 嵌套循环连接（NLJ）：实现最简单，但大表代价高；
- 排序归并连接（Sort-Merge Join）：适合已排序或可排序数据；
- 哈希连接（本条目）：等值连接性能与工程实用性平衡最好之一。

数据库优化器通常会基于数据规模、是否有序、可用内存来选择这三类主流连接策略。

## R18

`demo.py` 源码级算法流程（8 步）：

1. `normalize_key_spec` 将单列或复合键统一成元组格式，并在 `hash_join` 里校验左右键长度一致。  
2. `choose_build_side` 根据 `build_side` 参数决定 Build 侧；`auto` 下选择更小侧（等长时固定右侧）。  
3. Build 阶段遍历 Build 表：`build_key` 提取键，写入 `hash_table[key] -> [row_index,...]`。  
4. 初始化 `build_matched` 与 `probe_matched` 位图，用于后续外连接回填。  
5. Probe 阶段逐行取键查桶：`candidate_ids = hash_table.get(key)`，并累计 `probe_lookups`。  
6. 命中桶时展开所有候选 `build_idx`，产出匹配行（`l_`、`r_` 前缀合并），并更新匹配位图与 `matched_pairs`。  
7. 按 `join_type` 回填未匹配侧：`left/full` 补右侧 `NULL`，`right/full` 补左侧 `NULL`。  
8. 封装 `JoinStats` 与 `JoinResult` 返回；`main` 通过断言校验匹配数、未匹配数、桶数与全连接行数守恒。

该 MVP 未把第三方库当黑盒，关键流程（选 Build、建桶、探测、外连接补齐）都可在源码逐行追踪。
