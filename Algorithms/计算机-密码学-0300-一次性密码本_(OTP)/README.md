## R01

`一次性密码本`（One-Time Pad, OTP）是一种基于按位异或（XOR）的对称加密方案。  
在满足“密钥真随机、长度等于明文、且绝不复用”三个前提时，OTP 具有信息论意义上的`完美保密性`。

## R02

本条目目标是给出一个最小可运行 MVP，用代码直接展示：
- OTP 的加密与解密流程；
- 在合规前提下可正确恢复明文；
- 一旦复用同一密钥，会泄漏明文关系并可被利用。

## R03

核心思想：
- 记明文为 `P`，密钥为 `K`，密文为 `C`；
- 加密：`C = P XOR K`；
- 解密：`P = C XOR K`；
- 因为 `XOR` 自反，且随机密钥对任意明文“掩码均匀”，攻击者仅观察 `C` 时无法偏好任何特定明文。

## R04

数学要点（字节级）：
- 对每个位置 `i`：`c_i = p_i ⊕ k_i`；
- 逆运算：`p_i = c_i ⊕ k_i`；
- 若 `K` 在同长度空间均匀分布，则对任意候选明文 `p` 都存在唯一 `k = c ⊕ p`，且概率相同；
- 这对应 Shannon 完美保密：`Pr(P=p | C=c) = Pr(P=p)`。

## R05

输入与输出定义（对应 `demo.py`）：
- 输入：
  - `plaintext: bytes`
  - `key: bytes`（长度必须与 `plaintext` 或 `ciphertext` 一致）
- 输出：
  - `otp_encrypt` 返回 `ciphertext: bytes`
  - `otp_decrypt` 返回恢复明文 `bytes`

## R06

数据结构与约束：
- 全部采用 Python 原生 `bytes`；
- `xor_bytes(a, b)` 负责逐字节异或；
- 长度不一致直接抛 `ValueError`，避免静默截断造成错误安全假设。

## R07

实现模块划分：
- `xor_bytes`：底层逐字节 XOR；
- `generate_key`：生成给定长度的随机密钥；
- `otp_encrypt / otp_decrypt`：对称加解密主流程；
- `to_hex`：输出辅助，方便观察密文；
- `main`：执行正确性测试和“密钥复用攻击面”演示。

## R08

复杂度分析：
- 设消息长度为 `n` 字节；
- `xor_bytes` 为单次线性扫描，时间复杂度 `O(n)`，空间复杂度 `O(n)`（输出缓冲）；
- 加密与解密均由一次 XOR 主导，总体同为 `O(n)`。

## R09

正确性直观证明：
- 加密：`C = P ⊕ K`；
- 解密：`C ⊕ K = (P ⊕ K) ⊕ K = P ⊕ (K ⊕ K) = P`；
- 因此只要解密方持有同一把一次性密钥，就可无损恢复明文。

## R11

预期输出特征：
- 打印原始明文、密钥/密文十六进制截断信息；
- `decrypt with correct key: True`；
- `decrypt with wrong key equals original: False`；
- 在复用密钥场景，`C1 XOR C2 == P1 XOR P2` 为 `True`；
- 已知 `P1, C1, C2` 时可恢复 `P2`，演示结果为 `True`。

## R12

边界与异常处理：
- 空消息 `b""` 允许处理（长度为 0 时 XOR 结果仍为空）；
- 任意长度不匹配均报错；
- 错误密钥解密不会抛异常，但结果通常是无意义字节串（演示里验证“不等于原文”）。

## R13

安全注意事项：
- OTP 安全性高度依赖密钥管理，而不是算法复杂度；
- 密钥必须：
  - 真随机；
  - 长度不短于消息；
  - 仅使用一次；
  - 安全分发并保密存储；
- 实际工程中，OTP 更适合极小规模高保密场景；大多数系统会选择流密码或分组密码+工作模式。

## R14

与常见方案对比：
- 相比 AES/ChaCha20：OTP 在理论安全性更强，但密钥分发和存储成本极高；
- 相比“伪随机流 + XOR”的流密码：OTP 不依赖计算困难假设，但不具备实用伸缩性；
- 相比重复密钥异或（错误做法）：OTP 明确要求“一次性”，复用会立即破坏安全性。

## R18

源码级算法流程拆解（对应 `demo.py`）：
1. `main` 定义演示明文 `message`，并调用 `generate_key(len(message))` 生成同长度随机密钥。  
2. `otp_encrypt` 内部调用 `xor_bytes(plaintext, key)`，逐字节计算 `c_i = p_i ⊕ k_i` 得到密文。  
3. `otp_decrypt` 对密文和同一把密钥再次执行 XOR，利用 `XOR` 自反性恢复明文。  
4. 断言 `recovered == message`，验证加解密闭环成立。  
5. 构造一个“错误密钥”（仅翻转 1 bit）并解密，验证结果不再等于原文。  
6. 构造两个等长明文 `p1/p2`，故意复用同一密钥分别得到 `c1/c2`。  
7. 计算 `c1 ⊕ c2` 与 `p1 ⊕ p2`，验证两者恒等，显示复用导致明文关系泄漏。  
8. 进一步用 `p1, c1, c2` 直接恢复 `p2`，演示复用密钥下的可利用攻击面。  

以上 8 步完全基于源码可见操作，不依赖第三方密码学黑盒函数。
