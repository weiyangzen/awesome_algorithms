## R01

哲学家就餐问题（Dining Philosophers Problem）是操作系统并发控制中的经典资源分配模型：

- `N` 个哲学家围坐圆桌；
- 相邻两人之间有一把叉子，共 `N` 把；
- 每位哲学家循环执行“思考 -> 进餐”；
- 进餐必须同时拿到左右两把叉子。

核心难点是：在保证互斥安全的同时，避免死锁和长期饥饿。

## R02

该问题最早由 Dijkstra 提出，常用于说明：

- 多资源请求导致的死锁风险（环路等待）；
- 条件变量 / 信号量 / 监视器的协调设计；
- 活性（liveness）与公平性（fairness）之间的工程权衡。

如果所有哲学家都先拿左叉再等右叉，会形成循环占有并等待，从而出现系统级僵死。

## R03

可形式化为如下约束：

- 安全性：任意时刻一把叉子最多被一个哲学家占有；
- 邻接互斥：相邻哲学家不能同时进餐；
- 活性：系统不能卡死（至少有人能持续推进）；
- 公平性目标：等待进餐者不应无限期被延后。

本题是“互斥 + 资源分配 + 调度策略”三者结合的最小教学模型。

## R04

本目录 MVP 使用**监视器方案（monitor + condition variable）**：

- 每个哲学家有三态：`THINKING/HUNGRY/EATING`；
- 全局互斥锁保护状态数组；
- 每个哲学家对应一个条件变量；
- 当哲学家放下叉子时，主动尝试唤醒左右邻居。

关键规则：只有当某哲学家左右邻居都不在 `EATING` 且自身 `HUNGRY` 时，才允许转为 `EATING`。

## R05

设哲学家人数 `N`，每人轮次 `R`：

- 总状态切换次数为 `O(NR)`；
- 每次 `pickup/putdown` 仅执行常数次邻居检查（2 个邻居）；
- 时间复杂度（抽象模型）为 `O(NR)`；
- 额外空间为 `O(N)`（状态数组、条件变量、统计数组）。

`demo.py` 额外记录运行日志用于验证，因此日志空间为 `O(NR)`。

## R06

简化时序（`N=5`）如下：

1. `P0` 饥饿，左右邻居未进餐，进入 `EATING`；
2. `P1` 饥饿，但其左邻居 `P0` 在进餐，只能等待；
3. `P2` 饥饿，左右邻居（`P1`,`P3`）未进餐，可进入 `EATING`；
4. `P0` 放下叉子后触发邻居检查，若 `P1` 条件满足会被唤醒；
5. 系统持续推进，不会出现“所有人都拿一把叉子互相等待”的死锁形态。

## R10

本实现在运行时做以下正确性检查：

1. 哲学家进入 `EATING` 前必须满足“左右邻居均非 `EATING`”；
2. 统计 `active_eaters` 时，若发现邻居同时进餐立刻计为冲突；
3. 所有线程结束后，每个哲学家的进餐次数必须等于 `rounds`；
4. 线程 `join(timeout)` 后不得残留活跃线程；
5. 最终断言 `adjacent_conflicts == 0`。

## R11

死锁与饥饿分析：

- 死锁避免：
  - 不是“先锁左叉再锁右叉”的阻塞式资源链；
  - 通过监视器统一裁决“谁可以吃”，不会形成环路等待。

- 饥饿风险：
  - 理论上仍受调度器影响，无法数学上保证绝对无饥饿；
  - 本 demo 用随机思考/进餐间隔和固定轮次，验证在实际运行中每位哲学家都能完成同等次数进餐。

## R12

`demo.py` 关键参数：

- `num_philosophers`：哲学家数量（默认 5）；
- `rounds`：每位哲学家循环次数（默认 30）；
- `seed`：随机种子，保证结果可复现；
- `think_range` / `eat_range`：模拟思考和进餐耗时区间。

调参建议：

- 想快速 smoke test：降低 `rounds`；
- 若机器负载高：可放宽线程 `join` 超时。

## R14

常见失效模式与防护：

1. 忘记在 `putdown` 后检查并唤醒邻居，导致可运行线程被遗漏；
2. 在非互斥区域读写状态数组，造成竞态；
3. 用 `if` 替代 `while` 等待条件，可能被伪唤醒破坏逻辑；
4. 不设置 `join(timeout)`，异常时可能长时间卡住。

本实现分别通过：单锁保护状态、`while` 条件等待、邻居唤醒、超时断言来降低风险。

## R18

`demo.py` 源码级算法流程（8 步）：

1. 初始化 `DiningMonitor`：创建全局互斥锁、每位哲学家的条件变量、状态数组（初始全 `THINKING`）。
2. 哲学家调用 `pickup(i)` 时将自身状态置为 `HUNGRY`，并立即执行 `_test(i)` 判断是否可吃。
3. `_test(i)` 只检查 3 个局部条件：自己 `HUNGRY`、左邻居非 `EATING`、右邻居非 `EATING`；满足则置 `EATING` 并 `notify`。
4. 若 `pickup(i)` 未获准，哲学家在 `while state[i] != EATING` 中等待条件变量，防止伪唤醒造成错误进入。
5. 哲学家进餐结束调用 `putdown(i)`，先把自己置回 `THINKING`，再依次 `_test(left(i))` 与 `_test(right(i))` 触发邻居可进餐检查。
6. `run_simulation` 创建 `N` 个线程并行执行“思考 -> pickup -> 进餐 -> putdown”，固定随机种子保证复现。
7. 运行中记录等待时间、并发进餐集合、冲突计数；若发现相邻哲学家同时进餐即记为冲突。
8. 全线程结束后做断言：每人进餐次数等于 `rounds`、冲突为 0、无线程残留，并输出 `numpy` 统计（mean/p95/max）。

该实现没有把并发控制交给黑盒库：状态机、可进餐判定、阻塞唤醒路径都在源码中可逐行追踪。
