# 分数背包问题

- UID: `CS-0062`
- 学科: `计算机`
- 分类: `贪心算法`
- 源序号: `82`
- 目标目录: `Algorithms/计算机-贪心算法-0082-分数背包问题`

## R01

分数背包问题（Fractional Knapsack）描述如下：

给定若干物品，每个物品有重量 `w_i` 与价值 `v_i`，背包容量为 `C`。与 0/1 背包不同，本题允许拿取物品的任意比例 `x_i ∈ [0,1]`，目标是最大化总价值：

`max Σ(v_i * x_i)`，约束为 `Σ(w_i * x_i) <= C`。

## R02

输入输出可形式化为：

- 输入：物品集合 `items = [(name, weight, value), ...]` 与容量 `capacity`；
- 约束：`weight >= 0`、`value >= 0`、`capacity >= 0`；
- 决策：每个物品选择比例 `fraction`，范围 `[0, 1]`；
- 输出：
  - 最大总价值；
  - 每个被选物品的取用比例与贡献价值（便于审计）。

`demo.py` 中使用 `Item` 和 `SelectedPortion` 两个数据结构承载这两类信息。

## R03

核心贪心结论：

- 对任意两个物品 `a`、`b`，若单位价值（价值密度）`v_a / w_a >= v_b / w_b`，则优先拿 `a` 不会比优先拿 `b` 更差；
- 因此只要按价值密度从高到低排序，依次尽可能多拿每个物品，最终最多只有一个物品是“部分拿取”；
- 该策略即为全局最优，不是启发式近似。

这也是分数背包和 0/1 背包在算法上最关键的分界点。

## R04

算法流程（贪心主解）：

1. 校验输入并构造物品列表；
2. 计算每个物品密度 `density = value / weight`（对 `weight=0` 进行特殊处理）；
3. 按密度降序排序；
4. 从高密度到低密度遍历：
   - 若可完整放入，则 `fraction=1`；
   - 若容量不足，则取 `fraction = remaining / weight`；
   - 更新总价值与剩余容量；
5. 容量耗尽后停止，返回结果。

## R05

复杂度分析：

- 时间复杂度：`O(n log n)`（排序主导）；
- 空间复杂度：`O(n)`（保存排序结果与选取明细）；
- 单次扫描阶段为 `O(n)`。

当物品数增大时，性能瓶颈在排序而非装填循环。

## R06

MVP 验证策略：

- 固定样例：覆盖经典教材案例、零容量、全拿、部分拿、含零重量物品；
- 随机样例：使用 `numpy` 生成小规模随机数据；
- 精确对照：实现 `exact_fractional_by_enumeration`，用指数枚举在小规模上给出真值；
- 一致性断言：比较贪心结果与精确结果（允许浮点微小误差）。

运行命令：`uv run python demo.py`。

## R07

优点：

- 实现短小，逻辑清晰；
- 证明成熟，结果全局最优；
- 对工程场景（资源按比例分配）很实用。

局限：

- 前提是“允许按比例拆分”；
- 若物品必须整件选择，应改用 0/1 背包（DP/分支限界）；
- 浮点输入会引入数值误差，需要设置容差。

## R08

前置知识与运行环境：

- 贪心算法基础；
- 排序与比较器；
- 基本线性规划直觉（单约束下的极点解特性）。

运行环境：

- Python 3.10+；
- `numpy`（用于随机样例生成与数组桥接）；
- 无需交互输入。

## R09

适用场景：

- 资源可分割（带宽、预算、可切分原材料）；
- 目标是单位收益最大化；
- 需要快速给出可解释分配方案。

不适用场景：

- 物品不可拆分（采购整件、任务不可切）；
- 存在多维容量约束（多背包/多资源）；
- 存在复杂依赖关系（先后约束、互斥约束）。

## R10

正确性直觉（交换论证）：

- 假设某最优解中先拿了低密度物品 `b` 的一部分，同时还有高密度物品 `a` 未拿满；
- 将同等重量从 `b` 换到 `a`，总价值不减且通常增加；
- 反复交换后，可得到“密度从高到低拿满，最后最多一个分数项”的结构；
- 这正是贪心算法的输出结构，因此贪心最优。

## R11

边界与鲁棒性处理：

- `capacity = 0` 时直接返回 0；
- 空物品列表返回 0；
- `weight < 0`、`value < 0`、`capacity < 0` 直接抛错；
- `weight = 0` 且 `value > 0` 的物品应无条件全取（不消耗容量）；
- 浮点比较使用 `EPS` 容差，避免误判。

## R12

实现细节选择：

- `Item`：原始输入实体；
- `SelectedPortion`：输出解释实体（比例、取重、增益、密度）；
- `_normalize_items` / `_normalize_capacity`：统一做数据清洗；
- `_density`：集中处理零重量逻辑；
- `fractional_knapsack_greedy`：主算法；
- `exact_fractional_by_enumeration`：小规模精确基线，仅用于校验。

这种结构让“求解”和“验证”职责分离，便于定位问题。

## R13

理论保证与结果性质：

- 在本题假设下，贪心解与线性规划最优解一致；
- 最优值可唯一，但达到最优值的取法可能不唯一（密度相同物品可互换）；
- 算法输出自带可解释性：每个物品为何被全取、部分取或不取都能追溯。

## R14

常见错误：

1. 误用 0/1 背包思路，拒绝分数取用；
2. 按总价值排序而不是按单位价值排序；
3. 忽略 `weight=0` 情况导致除零错误；
4. 对浮点数使用严格相等比较；
5. 把验证代码当生产解法，导致大规模输入超时。

## R15

扩展方向：

- 加入多维容量约束（多资源分配）；
- 加入最低采购量/最小切分粒度；
- 目标函数从线性扩展到分段线性或凸优化；
- 输出可视化报表（累计重量-价值曲线）。

这些扩展通常不再能由单一排序贪心直接解决。

## R16

相关算法对照：

- 0/1 背包：DP，时间常见 `O(nC)`；
- 完全背包：DP，允许无限件整取；
- 分数背包：贪心排序，`O(n log n)`；
- 线性规划：可统一表达分数背包，但工程上贪心更轻量。

本题是“贪心可证明最优”的教科书级代表问题。

## R17

`demo.py` 功能清单：

- `fractional_knapsack_greedy`：主算法，输出最优价值和取用明细；
- `exact_fractional_by_enumeration`：指数枚举小规模精确值；
- `assert_consistency`：比对主算法与精确值；
- `run_fixed_cases`：运行确定性样例并校验期望；
- `run_random_verification`：随机回归验证；
- `run_numpy_case`：展示 numpy 输入桥接。

## R18

`demo.py` 源码级流程（8 步）：

1. `main` 依次调用固定样例、随机验证、`numpy` 样例，形成端到端闭环。  
2. `_normalize_capacity` 与 `_normalize_items` 先做输入清洗，确保容量、重量、价值合法。  
3. `_density` 对每个物品计算价值密度，并对 `weight=0` 做可解释特殊分支。  
4. `fractional_knapsack_greedy` 按密度降序遍历物品，逐个决定“全取 / 部分取 / 跳过”。  
5. 每次决策都记录为 `SelectedPortion`，同时维护 `remaining` 与 `total_value`。  
6. `exact_fractional_by_enumeration` 在小规模下枚举“其余物品 0/1 + 一个候选分数物品”，计算精确最优值。  
7. `assert_consistency` 用容差比较贪心结果与精确值，确保主算法实现无偏差。  
8. 所有断言通过后打印总结信息，说明该 MVP 在固定与随机样例上均通过验证。  

说明：本实现没有调用“背包黑盒求解器”；第三方库 `numpy` 仅用于数据生成与数组桥接，核心算法路径完全在源码中可追踪。
