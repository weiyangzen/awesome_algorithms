# 活动选择问题

## R01

活动选择问题（Activity Selection）是贪心算法中的经典模型：

给定若干活动，每个活动有开始时间 `start` 与结束时间 `finish`，要求从中选出尽可能多且互不冲突的活动。

本条目采用半开区间语义 `[start, finish)`，两个活动 `a`、`b` 兼容当且仅当：

`a.finish <= b.start` 或 `b.finish <= a.start`。

## R02

MVP 输入与输出定义：

- 输入：活动列表 `activities = [(name, start, finish), ...]`；
- 约束：`start <= finish`（允许零时长活动）；
- 目标：最大化可兼容活动数量（基数最大化，不考虑权重）；
- 输出：
  - 一组兼容活动（贪心解）；
  - 对照用精确解（用于验证最优性）；
  - 每组测试的活动数量统计。

`demo.py` 中使用 `Activity` 数据类承载活动定义。

## R03

核心贪心结论（最早结束优先）：

- 在所有尚可选择的活动中，优先选择结束时间最早的活动，不会降低后续可选择空间；
- 该选择能为后续活动留下最大的“剩余时间区间”；
- 重复执行后得到的活动数是全局最优。

这是本题的标准最优策略，不是启发式近似。

## R04

算法流程（贪心主解）：

1. 对活动按 `(finish, start, name)` 升序排序；
2. 初始化空集合 `selected`；
3. 线性扫描排序后的活动：
   - 若 `selected` 为空，直接选入；
   - 否则检查与上一个已选活动是否兼容：`last.finish <= current.start`；
   - 兼容则加入，不兼容则跳过；
4. 扫描结束即得到最大兼容活动集合。

## R05

复杂度分析：

- 时间复杂度：`O(n log n)`（排序主导）；
- 扫描部分：`O(n)`；
- 空间复杂度：`O(n)`（存储排序结果与选择结果）。

因此该算法在大规模输入上仍非常高效。

## R06

MVP 验证策略：

- 固定样例：
  - CLRS 经典活动集合；
  - 全兼容链式样例；
  - 高重叠样例；
  - 边界相接与零时长活动混合样例。
- 随机样例：`numpy` 生成小规模区间数据；
- 精确对照：`exact_maximum_schedule_bruteforce` 枚举子集获得最优基数；
- 一致性断言：比较贪心解与精确解的活动数量是否一致。

运行命令：`uv run python demo.py`。

## R07

优点：

- 实现非常短小，工程可维护性高；
- 有严格最优性理论支持；
- 适用于会议排程、任务窗口筛选等高频场景。

局限：

- 只优化“活动数量”，不优化总收益；
- 对带权版本（每个活动收益不同）不再成立；
- 默认离线输入，不处理在线动态插入删除。

前置知识与运行环境：

- 贪心算法基础；
- 排序与区间冲突判定；
- 子集枚举（用于验证而非主求解）。

运行环境：

- Python 3.10+；
- `numpy`（用于随机样例构造与数组桥接）；
- 无交互输入。

## R09

典型应用场景：

- 会议室时间段安排（尽量安排更多会议）；
- 设备占用窗口选择（最大化批次数）；
- 课程/实验时段冲突消解；
- 批处理任务在单资源上的时隙筛选。

不适用场景：

- 活动有收益权重，目标是最大收益；
- 多资源并行调度（多机、多房间）；
- 带前驱依赖、优先级、惩罚项的复杂约束调度。

## R10

正确性直觉（交换论证）：

- 设某最优解第一项不是“最早结束”的活动 `g`，而是活动 `x`；
- 因为 `g.finish <= x.finish`，把 `x` 替换成 `g` 后，不会减少后续可接活动空间；
- 所以存在一个同样最优的解以 `g` 开头；
- 去掉 `g` 后，剩余问题与原问题同构，继续递归得到整体最优。

因此“每步选最早结束”活动是安全且全局最优的。

## R11

边界与鲁棒性处理：

- 输入为空：返回空解；
- 非法活动（`start > finish`）：抛出 `ValueError`；
- 零时长活动（`start == finish`）：允许参与选择；
- 兼容判定统一使用 `[start, finish)` 语义，避免边界歧义。

## R12

实现细节设计：

- `Activity`：活动实体（名称、开始、结束）；
- `_normalize_activities`：输入规范化与校验；
- `activity_selection_greedy`：核心贪心实现；
- `_is_valid_schedule`：检查结果是否无冲突；
- `exact_maximum_schedule_bruteforce`：小规模精确对照；
- `activities_from_numpy`：`numpy` 输入桥接。

这种分层便于把“求解逻辑”和“验证逻辑”解耦。

## R13

理论保证与结果性质：

- 本问题（无权区间调度）存在贪心最优算法；
- 最优值是“最大兼容活动数量”；
- 最优解可能不唯一（不同活动集合可达到相同最大数量）；
- `demo.py` 通过暴力真值对照，验证实现层面未偏离理论。

## R14

常见错误：

1. 按开始时间排序而不是结束时间排序；
2. 冲突判定写成 `last.finish < current.start`，导致边界相接被误判冲突；
3. 把本题贪心直接套到带权版本；
4. 忽略非法输入校验，导致结果不可解释；
5. 只测固定样例，不做随机回归与真值对照。

## R15

可扩展方向：

## R16

相关问题对照：

- 无权活动选择：贪心，`O(n log n)`；
- 带权活动选择：DP，常见 `O(n log n)`；
- 区间合并：同属区间类贪心，但目标是并集压缩；
- 最少删除重叠区间：可转化为“保留最多不重叠区间”的等价问题。

本题是理解“贪心选择性质”的代表入门问题。

## R18

`demo.py` 的源码级算法流程（8 步）：

1. `main` 依次执行固定样例、随机验证和 `numpy` 样例，构成完整回归闭环。  
2. `_normalize_activities` 先校验每个活动满足 `start <= finish`，并做类型规范化。  
3. `activity_selection_greedy` 按 `(finish, start, name)` 排序活动，建立“最早结束优先”处理序。  
4. 线性扫描排序结果：若当前活动与已选最后活动兼容（`last.finish <= current.start`）则加入。  
5. 扫描结束后输出一个无冲突活动集合，作为贪心候选最优解。  
6. `exact_maximum_schedule_bruteforce` 对小规模输入枚举所有子集，并用 `_is_valid_schedule` 过滤冲突子集，得到精确最优基数。  
7. `assert_optimality` 比较贪心解与精确解的基数，确保实现与理论一致。  
8. `run_random_verification` 使用 `numpy` 生成随机样例重复校验，输出“通过”结论。  

说明：本实现未调用任何第三方调度黑盒；`numpy` 仅用于样例生成和数组输入桥接，核心贪心与精确对照都可在源码中逐行追踪。
