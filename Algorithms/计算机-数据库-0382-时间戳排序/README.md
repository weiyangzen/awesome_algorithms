# 时间戳排序

## R01

时间戳排序（Timestamp Ordering, TO）是经典的无锁并发控制协议。

核心思想：

- 每个事务在开始时获得一个全局唯一且单调递增的时间戳 `TS(T)`；
- 系统强制所有冲突操作按时间戳顺序生效；
- 若某操作违反该顺序，则事务被回滚（或在 Thomas 规则下忽略过时写）。

它的目标是保证冲突可串行化，同时避免锁等待链。

## R02

TO 要解决的问题是：

- 在并发读写同一数据项时，如何不用锁也保证结果等价于某个串行执行；
- 如何在无锁条件下阻断“老事务覆盖新事务结果”这类错误；
- 如何用轻量元数据（时间戳）替代复杂锁管理。

典型冲突包括读写冲突、写读冲突、写写冲突。

## R03

基础模型（单版本 TO）为每个数据项 `X` 维护两类时间戳：

- `RTS(X)`：最近一次成功读取 `X` 的事务时间戳最大值；
- `WTS(X)`：最近一次成功写入 `X` 的事务时间戳最大值。

每个事务 `T` 固定持有 `TS(T)`，整个执行期不变（若重启则分配新时间戳）。

## R04

核心规则：

- 读 `R_T(X)`：
  - 若 `TS(T) < WTS(X)`，说明 `T` 太旧，读会破坏顺序，`T` 回滚；
  - 否则读成功，并更新 `RTS(X) = max(RTS(X), TS(T))`。
- 写 `W_T(X)`：
  - 若 `TS(T) < RTS(X)`，说明已有更“新”事务读过旧值，`T` 写入会逆序，回滚；
  - 若 `TS(T) < WTS(X)`，该写已过时：
    - Basic TO：回滚事务；
    - Thomas Write Rule：忽略这次写并继续。
  - 其余情况写成功，更新 `WTS(X) = TS(T)`。

## R05

复杂度（设总操作数 `N`，数据项数 `M`）：

- 单次读/写判定：`O(1)`（常数次哈希查询与比较）；
- 总执行代价：`O(N)` + 回滚重启额外成本；
- 元数据空间：`O(M)`（每个数据项维护 `RTS/WTS`）。

TO 的吞吐通常受“冲突导致的回滚率”主导，而非单步判定开销。

## R06

`demo.py` 包含 3 个内置场景：

- 场景 A（Basic TO + 自动重启）：
  - 老事务 `T1` 在新事务 `T2` 写后再写同项，触发 `TS(T1) < WTS(X)` 回滚；
  - `T1` 用新时间戳重启后成功提交。
- 场景 B（Thomas 规则）：
  - 对同样的“过时写”不回滚，只忽略该写；
  - 展示与 Basic TO 的差异。
- 场景 C（写被读时间戳拒绝）：
  - 构造 `TS(T_old) < RTS(Y)`，触发“写太晚”回滚。

## R07

优点：

- 无锁，无死锁；
- 判定逻辑局部且简单，单步 `O(1)`；
- 对高并发短事务场景可提供较好并发度。

局限：

- 冲突高时回滚率可能显著上升；
- 长事务更容易“变老”并被年轻事务挤掉；
- 基础单版本 TO 需要谨慎处理恢复与可见性细节。

## R08

前置知识：

- 事务隔离与冲突可串行化；
- `RTS/WTS` 含义；
- 回滚与重启机制。

运行环境：

- Python 3.10+
- 无第三方依赖（标准库）

## R09

适用场景：

- 内存数据库/教学系统中需要无锁并发控制原型；
- 短事务为主、冲突可控的业务；
- 希望用时间戳机制替代锁等待链的系统。

不适用场景：

- 高热点写冲突且事务较长；
- 对回滚重试非常敏感的实时路径；
- 需要严格可恢复调度但未配套恢复机制的实现。

## R10

正确性直觉（冲突可串行化）：

1. 每个事务有固定 `TS`，可看作其串行顺序位置；
2. 每次读写都检查是否与已确认的 `RTS/WTS` 冲突；
3. 一旦发现逆序访问，立即拒绝该操作（回滚或忽略过时写）；
4. 因而留下的有效冲突边不会违反时间戳顺序，执行等价于按 `TS` 排列的串行历史。

## R11

本实现的鲁棒性细节：

- 时间戳分配器全局单调递增，重启事务必须拿新时间戳；
- 统一事件日志，记录 `READ/WRITE/ABORT/RESTART/IGNORE/COMMIT`；
- 每个场景带断言，验证最终值与状态；
- 设置 `max_restarts` 防止病态场景无限重启。

## R12

关键参数：

- `mode`：`"basic"` 或 `"thomas"`；
- `max_restarts`：单事务允许的最大重启次数；
- `schedule`：确定性调度序列（用于复现实验）。

调参建议：

- 研究回滚行为时使用 `basic`；
- 研究“过时写优化”时使用 `thomas`；
- 通过改变 `schedule` 控制冲突强度与触发路径。

## R13

理论保证：

- Basic TO：通过拒绝所有逆序冲突访问来保证冲突可串行化；
- Thomas 规则：在 `TS < WTS` 时忽略过时写，不改变最终可见版本的正确性；
- 近似比/随机成功率：N/A（协议型算法，非近似优化问题）。

## R14

常见失效模式：

1. 重启事务复用旧时间戳，导致协议失效；
2. 忘记更新 `RTS/WTS`，使检查失真；
3. 把 `TS < WTS` 错写为 `<=`，引入错误拒绝；
4. 仅回滚状态不处理调度，造成事务永久饿死。

排查建议：

- 打印每次操作前后的 `RTS/WTS`；
- 固定调度复现实验；
- 对比 Basic 与 Thomas 结果差异定位分支错误。

## R15

工程化建议：

- 将协议判定层（TO 引擎）与执行层（事务调度）解耦；
- 日志结构化（JSON）便于离线分析回滚率；
- 对热点键增加冲突监控，必要时切换到 MVCC/锁方案；
- 结合退避重试策略降低饥饿概率。

## R16

与其他并发控制方案对比：

- Strict 2PL：会阻塞且可能死锁，但恢复路径清晰；
- TO：不阻塞、无死锁，但可能高回滚；
- OCC：提交阶段集中验证，低冲突时优秀；
- MVCC：读性能更强，但版本管理更复杂。

TO 的典型定位是“用回滚换等待”。

## R18

`demo.py` 源码级算法流程（8 步）：

1. `TimestampOrderingEngine.begin_transaction` 为事务分配单调递增时间戳 `TS`。  
2. 调度器按给定 `schedule` 逐步推进事务，每次仅执行一条操作。  
3. 执行 `READ` 时调用 `read(tx, item)`：若 `TS < WTS(item)` 则返回 `ABORT`，否则读成功并更新 `RTS(item)`。  
4. 执行 `WRITE` 时调用 `write(tx, item, value)`：先检查 `TS < RTS(item)`；若成立直接 `ABORT`。  
5. 继续检查 `TS < WTS(item)`：`basic` 模式下 `ABORT`，`thomas` 模式下返回 `IGNORE`（跳过过时写）。  
6. 写通过时立即更新 `data[item]` 与 `WTS(item)`，并记录事件日志。  
7. 事务被回滚后，调度器若未超 `max_restarts`，调用 `restart_transaction` 重置 `pc` 并分配新 `TS`。  
8. 所有事务到达 `COMMIT/ABORTED_FINAL` 后输出最终数据、`RTS/WTS`、重启次数和事件轨迹，用断言校验关键性质。  

上述流程完全在源码里显式实现 `TS/RTS/WTS` 规则，没有依赖第三方数据库黑盒。
