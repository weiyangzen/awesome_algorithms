## R01

向量时钟（Vector Clock）是分布式系统中用于刻画因果关系的逻辑时间机制。

核心思想是：每个进程维护一个长度为 `N` 的向量（`N` 为进程数），向量第 `i` 位记录“该进程已知的 `Pi` 事件计数”。

它相比单值逻辑时钟（如 Lamport）更强，因为它不仅保证“因果先后不被颠倒”，还能判定两个事件是否并发。

## R02

典型动机：

- 多副本系统中的写冲突需要判断“是否存在因果覆盖关系”；
- 分布式调试希望明确区分“先后事件”与“并发事件”；
- 因果一致性（Causal Consistency）协议需要可验证的因果元数据。

物理时间（wall clock）受时钟漂移和网络延迟影响，不能可靠代表因果；向量时钟是与物理时间解耦的工程化替代。

## R03

理论模型基于 `happens-before`（记作 `a -> b`）：

- 同一进程内，先发生事件 `->` 后发生事件；
- 消息发送事件 `->` 对应接收事件；
- 关系满足传递性。

向量时钟判定规则：

- 设事件 `a,b` 对应向量 `Va,Vb`；
- 若 `Va <= Vb`（逐分量不大于）且至少一个分量严格小于，则 `a -> b`；
- 若 `Va` 与 `Vb` 互不满足上述偏序（既非 `Va < Vb` 也非 `Vb < Va`），则二者并发。

## R04

算法更新规则（进程 `Pi` 维护向量 `VCi`）：

1. 本地事件发生前：`VCi[i] += 1`；
2. 发送消息前：`VCi[i] += 1`，并把 `VCi` 副本附在消息中；
3. 接收消息（携带 `VCm`）时：
   - 先做逐分量最大值合并：`VCi = max(VCi, VCm)`；
   - 再执行本地分量自增：`VCi[i] += 1`。

该规则保证了向量时钟与因果关系的一致性。

## R05

复杂度：

- 每个本地/发送事件开销 `O(1)` 到 `O(N)`（取决于是否复制向量）；
- 接收事件需要逐分量 `max`，开销 `O(N)`；
- 每条消息携带 `N` 维元数据，通信元开销 `O(N)`。

因此向量时钟更强调因果表达能力，代价是比 Lamport 额外的向量维度开销。

## R06

本条目 `demo.py` 的 MVP 设计：

- 固定 3 个进程：`P1/P2/P3`；
- 执行一段确定性事件脚本（本地、发送、接收共 14 个事件）；
- 维护两类边构成 happens-before 图：
  - 进程内相邻事件边；
  - 发送事件到接收事件的消息边；
- 使用 `numpy` 实现布尔传递闭包；
- 自动校验“向量偏序 <=> 因果可达”的等价性；
- 输出一个并发事件对作为直观示例。

## R07

优点：

- 可以识别并发（这是 Lamport 时间戳做不到的）；
- 因果关系判定更精确，可直接用于因果冲突检测；
- 规则简单，易于在消息元数据层实现。

局限：

- 进程数变大时，向量维度和消息体膨胀明显；
- 节点动态加入/离开时，维度管理更复杂；
- 在超大规模系统中通常需要压缩或近似方案（版本向量、区间树时钟等）。

## R08

前置知识与运行环境：

- 分布式系统中的事件模型与消息传递；
- 偏序关系、传递闭包概念；
- Python 3.10+；
- `numpy`（本示例用于向量运算与布尔闭包）。

## R09

适用场景：

- 多副本数据库中的因果冲突判定；
- 因果一致性存储、聊天系统的消息依赖追踪；
- 分布式调试工具中跨进程事件的因果还原。

不适用场景：

- 极端追求网络报文最小化（`N` 维元数据不可接受）；
- 只需要“近似排序”而不需要并发判定；
- 节点规模非常动态且缺少额外维度管理机制。

## R10

正确性直觉（简版）：

- 本地单调：同一进程每次事件都会使本地分量递增；
- 消息因果：接收方先 `max` 合并再 `+1`，确保发送向量被接收向量覆盖并严格推进；
- 传递保持：向量逐分量偏序与 `happens-before` 传递性兼容。

因此在标准消息传递模型下，向量时钟满足：

- `a -> b` 当且仅当 `VC(a) < VC(b)`。

## R11

`demo.py` 的健壮性检查：

- 禁止重复进程 ID；
- 禁止重复在途消息 ID；
- 禁止接收不存在/已消费消息；
- 禁止“消息目的进程与接收方不一致”；
- 结束时断言不存在未投递消息；
- 逐条断言所有因果边都满足向量严格偏序。

## R12

实现细节：

- 事件记录包含：`event_id / process_id / local_index / kind / vector`；
- 消息记录包含：发送时向量副本和发送事件 ID；
- 用 `tuple[int, ...]` 存储事件向量，保证事件快照不可变；
- 向量比较函数明确拆分：
  - `vector_leq`：逐分量小于等于；
  - `vector_lt`：偏序严格小于；
  - `vector_concurrent`：互不可比。

## R13

理论边界与工程注意：

- 向量时钟描述的是因果偏序，不是全序；
- 两个并发事件可以映射到任意稳定输出顺序，但该顺序不代表因果先后；
- 当进程集合变化时，需引入进程成员管理策略（例如 epoch + 扩展维度映射）；
- 生产系统常用“版本向量”只跟踪副本写版本，属于向量时钟的特化。

## R14

常见错误：

1. 接收时只做 `max` 忘记对本地分量再 `+1`；
2. 发送消息时传引用而不是快照，导致后续改写污染历史事件；
3. 用“任意一个分量小于”误判偏序，忽略“所有分量不大于”前提；
4. 把不可比关系当成错误，而不是正确的并发判定结果。

## R15

可扩展方向：

- 动态成员下的可伸缩版本向量；
- 与 CRDT 合并策略结合，实现因果冲突最小化；
- 引入故障注入（乱序、延迟、重复消息）做压力实验；
- 与 Hybrid Logical Clock（HLC）结合，兼顾物理时间可读性与因果性。

## R16

相关机制：

- Lamport 时间戳：维护因果必要条件，但不能判并发；
- Version Vector：面向副本版本管理的实用变体；
- Matrix Clock：记录“对他人知识的知识”，信息更丰富但开销更高；
- Causal Broadcast / Causal Delivery：常依赖向量时钟进行依赖检查。

`demo.py` 主要模块：

## R18

`demo.py` 的源码级算法流程（8 步）：

1. `build_demo_scenario` 初始化 `P1/P2/P3`，每个进程向量初值均为全 0。  
2. 本地事件与发送事件调用 `tick_local_or_send`，对自身分量执行 `+1` 并记录事件快照。  
3. 发送时把当前向量复制到 `Message.clock`，防止后续本地推进篡改已发送元数据。  
4. 接收时调用 `tick_receive`：先 `np.maximum(local, remote)` 逐分量合并，再对本地分量 `+1`。  
5. 记录事件时维护两类 `happens-before` 边：进程内前驱边、发送到接收边。  
6. `assert_vector_clock_condition` 逐边验证 `VC(src) < VC(dst)`，确保因果方向不被破坏。  
7. `compute_reachability` 用布尔 Warshall 算法得到传递闭包，`assert_hb_equivalence` 校验闭包与向量偏序等价。  
8. 程序输出事件表、并发事件示例和最终时钟状态，展示向量时钟可判并发的关键性质。

本实现没有把向量时钟交给第三方分布式框架黑盒处理；核心更新和判定规则均在源码中逐行展开，可直接追踪。
