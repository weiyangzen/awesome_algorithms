## R01

AC 自动机（Aho-Corasick Automaton）是用于**多模式串匹配**的经典算法。  
它把多个模式串构造成一棵 Trie，并在 Trie 上增加失败指针（fail link），从而在一次线性扫描中同时找到所有模式串出现位置。

本目录 MVP 的目标：
- 输入：模式串集合 `patterns` 与文本串 `text`。
- 输出：所有命中结果（模式串、起止位置），并给出每个模式串出现次数。
- 特点：实现完全展开（Trie 构建 + fail 构建 + 扫描），不依赖第三方黑箱实现。

## R02

问题形式化：

- 模式串集合：`P = {p1, p2, ..., pk}`，长度总和记为 `L = sum |pi|`。
- 文本串：`T`，长度记为 `N`。
- 任务：找出所有 `(pid, start, end)`，满足 `T[start:end+1] == p_pid`。

朴素做法通常是“每个模式串在文本上单独匹配”，复杂度约 `O(N * k)`（或更精确为 `O(sum N*|pi|)`）；  
AC 自动机把它降为构建后单次扫描 `O(N + 匹配输出数)`。

## R03

算法核心思想分三层：

1. Trie 层：把所有模式串共享前缀压缩到同一棵字典树。
2. 失败指针层：当当前状态无法按字符转移时，沿 `fail` 跳转到“最长可用后缀状态”继续尝试。
3. 输出层：每个状态保存 `out`（在该状态结束的模式串 id），并继承 `fail` 链上的输出，实现后缀模式自动命中。

因此文本扫描时无需回退文本指针，只回退自动机状态。

## R04

复杂度分析：

- 构建 Trie：`O(L)`。
- 构建 fail 指针（BFS）：`O(L * alpha_check)`，在字典实现下可视作 `O(L)` 平均复杂度。
- 扫描文本：`O(N + Z)`，其中 `Z` 为匹配结果数量。

总体：`O(L + N + Z)`。  
这也是 AC 自动机相比逐模式匹配的关键优势。

## R05

空间复杂度：

- Trie/自动机节点总数：不超过 `L + 1`。
- 每个节点存转移表 `next`、失败指针 `fail`、输出表 `out`。
- 总空间约 `O(L + Z_out_struct)`，通常记为 `O(L)`（不计最终输出结果存储）。

在字符集较大时，稀疏映射（`dict`）比定长数组更节省空间。

## R06

小例子：

- 模式串：`["he", "she", "his", "hers"]`
- 文本：`"ahishers"`

扫描过程会命中：
- `"his"`（位置 `1..3`）
- `"she"`（位置 `3..5`）
- `"he"`（位置 `4..5`，由 `she` 的后缀触发）
- `"hers"`（位置 `4..7`）

这里 `"he"` 的命中体现了 fail 链输出继承的作用。

## R07

AC 自动机的工程价值：

- 一次构建，多次查询：模式串集合固定时，可复用自动机匹配多段文本。
- 可解释：节点、fail、out 都是显式结构，便于调试。
- 输出完整：天然支持“全部命中”而非仅首个命中。
- 可拓展：支持大小写归一化、过滤、关键词告警等文本流水线任务。

## R08

理论前置：

- Trie（前缀树）基本操作；
- BFS 层序遍历（用于 fail 构建）；
- KMP 中“失配后沿最长可用前后缀跳转”的思想；
- 字符串下标与区间管理（起止位置计算）。

可以把 AC 理解为“多模式版 KMP + Trie”。

## R09

适用边界：

- 适用：
  - 模式串数量较多；
  - 文本较长；
  - 需要找全命中（包括重叠命中）；
  - 模式集可提前已知（离线或准静态）。
- 不太适用：
  - 模式集频繁增删且要实时生效（需动态 AC 或重建）；
  - 仅少量短模式，简单方法已足够快。

## R10

正确性要点：

1. Trie 路径与模式串字符顺序一一对应，终止节点记录该模式串 id。
2. 对任意节点 `u` 和字符 `c`，失配时沿 `fail[u]` 继续找可转移状态，等价于查找最长可匹配后缀前缀。
3. `out[u]` 合并 `out[fail[u]]`，保证所有后缀模式命中被枚举。
4. 扫描文本时每个字符仅推进一次文本指针，状态通过转移与 fail 跳转保持一致性。
5. 每次命中位置 `end=i`，起点计算为 `start = i - len(pattern) + 1`，故输出位置正确。

## R11

数值稳定性：

- 本算法是离散状态机与整数下标运算，不涉及浮点数值稳定性问题。
- 主要风险来自实现细节：
  - fail 构建顺序错误；
  - 未合并 fail 输出导致漏报；
  - 命中起点计算偏移一位；
  - 字符集预处理不一致（大小写、空白、Unicode 归一化）。

MVP 通过朴素匹配对照测试规避这些风险。

## R12

真实成本（常数项）关注：

- 节点转移结构：
  - 稀疏字符集用 `dict` 常数更优；
  - 稠密小字符集用数组更快但更占内存。
- 模式串分布：
  - 公共前缀越多，Trie 压缩收益越大；
  - 输出密度高时，`Z`（命中数）主导总时间。
- Python 层面：
  - 对象数量较大时内存分配成本明显；
  - 适当减少临时对象可提升吞吐。

## R14

常见失效模式与防护：

- 失效模式：空模式串 `""` 导致语义混乱（理论上会命中每个位置）。
  - 防护：MVP 明确禁止空模式串并抛出异常。
- 失效模式：重复模式串导致统计重复。
  - 防护：允许重复并按 `pattern id` 分别统计，行为显式。
- 失效模式：只记录当前状态输出，未合并 fail 输出，导致漏掉后缀命中。
  - 防护：fail BFS 时执行 `out.extend(out[fail])`。
- 失效模式：下标边界错误（负起点或越界）。
  - 防护：统一用 `start = i - len(p) + 1` 并在测试中与朴素结果对拍。

## R18

源码级流程拆解（`demo.py`，非黑箱）：

1. 读取模式串列表，逐个插入 Trie；每个字符沿 `next` 建边，终点节点记录模式串 id 到 `out`。
2. 将根节点的直接子节点入队，初始化它们的 `fail=0`。
3. 进行 BFS：弹出节点 `u`，对每条字符边 `(u --c--> v)` 计算 `v` 的失败指针。  
   规则：从 `fail[u]` 反复跳转，直到找到可走 `c` 的状态或回到根。
4. 设定 `fail[v]` 后，把 `out[fail[v]]` 合并进 `out[v]`，确保后缀模式也能在 `v` 命中。
5. 扫描文本时维护当前状态 `state`；读入字符 `ch` 后，若无转移则沿 fail 回退，直到可转移或回根。
6. 若存在 `ch` 转移则进入新状态；该状态的 `out` 中每个模式 id 都代表一次命中。
7. 对每个命中 id，按 `start = i - len(pattern) + 1` 计算起点，记录 `(pattern, start, end)`。
8. 扫描结束后返回全部命中列表，并与朴素算法结果排序对比，作为正确性验证。
