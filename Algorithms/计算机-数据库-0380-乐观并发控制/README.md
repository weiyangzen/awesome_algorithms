## R01

乐观并发控制（Optimistic Concurrency Control, OCC）是一种“先执行、后校验”的并发控制策略。

核心思想：

- 事务在读阶段不加锁，先在本地缓冲区记录读集与写集；
- 提交时进入验证阶段，检查与已提交事务是否冲突；
- 验证通过才写回数据库，否则回滚并可重试。

它假设冲突概率整体较低，尽量避免长时间锁等待。

## R02

OCC 主要解决的问题是：

- 在读多写少、冲突相对稀疏的负载下，减少锁管理与阻塞开销；
- 让事务执行阶段尽量并行，把串行化约束推迟到提交瞬间；
- 在保持可串行化正确性的前提下，提高吞吐与响应性。

与 2PL 不同，OCC 的代价从“等待”转移为“可能的提交失败与重试”。

## R03

本条目 MVP 的输入与输出模型：

- 输入：
  - 初始数据库键值（内存字典）；
  - 多个事务程序（`R/W/C` 操作序列）；
  - 调度参数（最大轮次、最大重试次数）。
- 输出：
  - 调度事件日志；
  - 每个事务最终状态（`committed/aborted`）；
  - 最终数据库状态与提交历史。

`demo.py` 用纯 Python 实现，不依赖外部数据库内核。

## R04

OCC 的三阶段在本实现中对应为：

1. 读阶段（Read Phase）：
   - `R(x)` 读取当前已提交值或本事务写缓冲值；
   - `W(x,v)` 只写入本地 `write_buffer`，不改全局数据库。
2. 验证阶段（Validation Phase）：
   - 提交时检查“开始时间之后已提交事务”是否写了本事务读/写过的键。
3. 写阶段（Write Phase）：
   - 验证通过后，原子写回 `write_buffer` 到全局数据库并记录提交时间戳。

## R05

本 MVP 采用“向后验证（backward validation）”的简化规则。

设事务 `Ti` 开始时间戳为 `start_ts(i)`，提交时遍历所有已提交事务 `Tj`：

- 若 `commit_ts(j) <= start_ts(i)`：`Tj` 在 `Ti` 之前完成，不构成并发重叠冲突；
- 若 `commit_ts(j) > start_ts(i)` 且
  `write_set(j) ∩ (read_set(i) ∪ write_set(i)) != ∅`：
  `Ti` 验证失败，必须回滚。

这对应 OCC 常见的“读写/写写冲突提交时检查”思想。

## R06

`demo.py` 内置两个场景：

- 场景 1（低冲突）：
  - `T1` 仅改 `A`，`T2` 仅改 `B`；
  - 两者均一次提交成功。
- 场景 2（高冲突 + 重试）：
  - `T1` 与 `T2` 都读写 `X`；
  - `T1` 先提交后，`T2` 首次验证失败并回滚；
  - `T2` 自动重试一次并成功提交。

用例能直观看到 OCC 在冲突场景下的“回滚重试”特征。

## R07

复杂度分析（设事务数 `T`，总操作数 `N`，已提交记录数 `H`）：

- 单次 `R/W` 操作：平均 `O(1)`（哈希字典访问）；
- 单次验证：`O(H * K)`，`K` 为集合交集检查成本（近似键数规模）；
- 调度主循环：`O(N + retries * N_retry)` 级别。

空间复杂度主要来自：

- 每事务的读集、写缓冲；
- 全局提交历史。

总体约 `O(total_read_keys + total_write_keys + H)`。

## R09

适用场景：

- 读多写少、冲突稀疏；
- 事务较短、重试成本可接受；
- 业务更在意低等待而非极低回滚率。

不适用场景：

- 热点写入严重、冲突密集；
- 长事务且重复执行代价高；
- 对尾延迟波动极度敏感（频繁重试会放大抖动）。

## R10

正确性直觉（关键不变式）：

1. 事务写入在验证前不会污染全局已提交状态；
2. 任意事务提交时，都会与“可能并发且已提交”的事务做写集冲突检查；
3. 验证失败事务不会进入写回阶段；
4. 因此最终提交历史可映射为按提交顺序串行执行。

## R11

鲁棒性实现细节：

- 事务状态显式区分 `active/committed/aborted`；
- 逻辑时钟统一分配 `start_ts` 与 `commit_ts`，保证验证判定可复现；
- 事件日志记录每个读、写缓冲、提交、回滚与重试；
- 设置 `max_rounds` 防止异常情况下调度死循环；
- 使用断言验证最终状态和关键行为顺序。

## R12

关键参数与调优建议：

- `max_retries`：单事务最大重试次数。
  - 值大：成功率更高，但可能增加尾延迟；
  - 值小：失败更快暴露，适合上层补偿策略。
- `max_rounds`：调度轮次上限。
  - 防止程序逻辑错误导致无限循环。
- 调度顺序（本实现按事务编号轮询）：
  - 固定顺序便于教学复现；
  - 工程上可替换成更公平或优先级策略。

## R14

常见失效模式：

1. 验证范围不足（漏检并发提交事务）导致错误提交；
2. 只检查读写冲突而漏掉写写冲突，导致丢失更新；
3. 回滚后未重置读写集合，导致重试污染；
4. 无重试上限，极端热点下可能活锁式反复失败。

排查建议：

- 打开详细事件日志核对时间戳与集合交集；
- 用最小两事务冲突样例做回归；
- 检查每次重试是否创建了干净事务上下文。

## R15

工程化落地建议：

- 将事务执行、验证器、时间戳分配器拆分为独立模块；
- 结合指标监控：提交率、平均重试次数、验证失败原因分布；
- 对热点键引入混合策略（热点键走锁/队列，冷键走 OCC）；
- 在分布式场景结合全局版本号或 TrueTime 类时序边界。

## R16

与其他并发控制方案对比：

- 对比 2PL：
  - OCC 减少等待，但会回滚；
  - 2PL 等待更多，但提交失败率通常更低。
- 对比严格 2PL：
  - OCC 无锁读阶段吞吐高；
  - 严格 2PL 恢复路径更直接、语义更保守。
- 对比 MVCC：
  - OCC 实现更轻量；
  - MVCC 在高并发读上通常更稳定，但版本管理更复杂。

## R18

`demo.py` 源码级算法流程（8 步）：

1. `OCCEngine.begin` 为每个事务分配 `start_ts`，初始化读集、写缓冲和程序计数器。  
2. 调度器 `run_workload` 按轮询策略推进事务，每轮每个事务至多执行一条操作。  
3. 遇到 `R(item)` 时，优先读本事务 `write_buffer`，否则读 `committed_db`，并登记到 `read_set`。  
4. 遇到 `W(item, value)` 时，仅写入 `write_buffer`，不立即修改全局数据库。  
5. 遇到 `C` 时执行 `validate`：遍历 `start_ts` 之后提交的事务，检查其 `write_set` 与当前事务 `read_set ∪ write_set` 是否相交。  
6. 若验证失败，事务标记为 `aborted` 并记录冲突原因，不进入写回阶段。  
7. 若验证通过，`commit` 将 `write_buffer` 原子应用到 `committed_db`，写入提交历史并分配 `commit_ts`。  
8. 调度器对 `aborted` 事务按 `max_retries` 决定是否 `begin` 新尝试；所有事务终态确定后输出结果与断言。  

这个流程把 OCC 的关键机制全部拆解为显式源码步骤，没有依赖数据库黑盒接口。
