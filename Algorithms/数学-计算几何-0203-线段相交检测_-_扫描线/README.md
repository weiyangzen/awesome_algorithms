# 线段相交检测 - 扫描线

## R01

本条目讨论二维线段相交检测的扫描线（Sweep Line）方法。

问题定义：给定若干线段 `S = {s1, s2, ..., sn}`，判断哪些线段对存在交点（包括真交、端点相交、共线重叠）。

核心思想：
- 用一条从左到右移动的“垂直扫描线”访问平面。
- 仅维护当前与扫描线相交的“活动线段集合”。
- 只检查活动集合中相邻线段是否可能产生新的交点事件。

这比朴素 `O(n^2)` 两两检测更接近 Bentley-Ottmann 的事件驱动思路。

## R02

历史背景：
- 扫描线思想在 20 世纪计算几何发展中成为基础范式。
- 1970s 以后，Shamos-Hoey（判定是否存在交）与 Bentley-Ottmann（枚举交点）成为线段相交经典框架。
- 经典理论复杂度通常写作 `O((n + k) log n)`，其中 `k` 为交点数。

本目录 MVP 采用其核心机制（事件队列 + 活动顺序 + 邻居检查），并做工程化简化实现。

## R03

要解决的工程问题：
- 输入一组 2D 线段。
- 输出所有相交线段对 `(i, j)`。
- 结果需覆盖：真交、端点接触、重叠。

为什么不用纯暴力：
- 暴力法直接比较 `n(n-1)/2` 对，规模稍大时成本迅速上升。
- 扫描线通过“局部邻接关系”避免大量不必要比较。

## R04

算法流程（抽象层）：

1. 为每条线段生成左右端点事件。
2. 按 `(x, y, 类型)` 排序事件并逐个处理。
3. `LEFT`：把线段插入活动结构，检查其上下邻居。
4. `RIGHT`：把线段移出活动结构，检查其原上下邻居彼此是否相交。
5. `INTERSECTION`：记录相交对，并在活动结构中交换两线段顺序，再检查新邻居。
6. 对每次邻居检测得到的交点，若在扫描线右侧则入队为未来事件。

本 MVP 还保留了“交点在当前或左侧则直接记结果”的处理，用于应对端点接触这类退化情形。

## R05

复杂度说明（区分理论与本实现）：

- 经典 Bentley-Ottmann（平衡树实现状态结构）：
  - 时间：`O((n + k) log n)`
  - 空间：`O(n + k)`
- 本 MVP（Python `list` 维护活动结构，插入/删除/查找为线性）：
  - 时间：最坏可达 `O((n + k) * n)`
  - 空间：`O(n + k)`

因此这是“算法思想正确、工程上可运行但非最优数据结构”的最小实现。

## R06

手算示例（5 条线段）：

- `s0: (0,0)-(5,5)`
- `s1: (0,5)-(5,0)`
- `s2: (0.5,4.5)-(2.0,4.5)`
- `s3: (6,0)-(7,1)`
- `s4: (2,-1)-(3,6)`

可见交点关系：
- `s0` 与 `s1` 相交。
- `s0` 与 `s4` 相交。
- `s1` 与 `s2` 在端点相交。
- `s1` 与 `s4` 相交。
- `s2` 与 `s0`、`s4` 不相交。
- `s3` 与其他不相交。

`demo.py` 会打印扫描线结果与暴力基线结果，并断言一致。

## R07

算法价值：
- 对“稀疏相交”数据，通常比暴力快很多。
- 事件驱动结构适合扩展到多种几何任务（可见性、布局冲突检测等）。
- 在工程上可逐步升级：先 list MVP，再替换为平衡树或跳表结构。

## R09

适用前提与边界：
- 适用于二维欧氏平面线段集合。
- 支持浮点输入，但需容忍阈值 `EPS`。
- 共线重叠在本 MVP 中按“存在相交关系”处理，记为一对相交线段。

边界提醒：
- 多重共点、大量垂直线、极端近共线数据会放大数值与事件并列处理难度。
- 若要生产级稳定性，应引入更严格鲁棒几何谓词或有理数内核。

## R10

正确性依据（核心不变式）：

1. 扫描线推进时，只有活动结构中“相邻”线段可能成为下一次新交点候选。
2. 当线段插入或删除时，只会改变局部邻接关系，因此只需检查局部邻居。
3. 当处理交点事件时，交叉线段在活动顺序中发生相对交换；交换后再次局部检查即可保持完整性。
4. 几何判定层用 orientation + on-segment 覆盖了真交、端点相交、共线重叠。

因此输出的相交对可与暴力法对齐（在数值容差下）。

## R11

数值稳定性与误差控制：
- 统一使用 `EPS = 1e-9` 做“近似零”判定。
- `on_segment` 同时检查共线与包围盒，避免仅凭叉积造成误报。
- 事件去重时对交点坐标做 `round(..., 12)`，降低浮点噪声导致的重复事件。
- 活动结构比较使用 `y_at(current_x + EPS)`，避免恰在事件点出现次序抖动。

## R14

常见失效模式：

1. 忽略共线重叠：导致“明明相交却漏报”。
2. 只判真交，不判端点接触：导致拓扑连接关系错误。
3. 活动结构顺序在事件点未正确更新：可能错过后续交点。
4. 同一交点重复入队：事件爆炸、性能恶化。
5. 过于严格或过于宽松的 `EPS`：分别导致漏报或误报。

MVP 对应防护：
- 增加 `Intersection(kind)` 分类。
- 入队前去重。
- 每个测试用例与暴力基线对比。

## R18

源码级算法流程拆解（8 步）：

1. `build_segments` 将输入端点归一化为“左端点到右端点”，建立线段 ID。  
2. `segment_intersection` 用 orientation / on-segment 计算两线段关系，输出 `point`、`overlap` 或 `None`。  
3. `sweep_line_intersections` 初始化事件堆：每条线段生成 `LEFT` 与 `RIGHT` 事件。  
4. 处理 `LEFT` 事件：按 `y_at(current_x + EPS)` 插入活动结构，只检查新线段与上下邻居。  
5. 邻居检查由 `maybe_register_or_schedule` 执行：若交点在未来则压入 `INTERSECTION` 事件，否则直接登记相交对。  
6. 处理 `RIGHT` 事件：移除线段，并检查其原上下邻居是否因“变成相邻”而产生新交点。  
7. 处理 `INTERSECTION` 事件：记录相交对、交换活动结构中的两线段顺序，再做局部邻居检查。  
8. 全部事件处理完后返回相交对集合，并在 `run_case` 中与 `brute_force_intersections` 结果逐案断言一致。

说明：`numpy` 在本实现中仅用于向量和线交点坐标计算（`line_intersection_point`），算法控制流并未交给黑盒几何库。
