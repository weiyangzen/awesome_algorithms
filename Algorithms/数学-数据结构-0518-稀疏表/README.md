# 稀疏表

## R01

稀疏表（Sparse Table）是用于**静态区间查询**的数据结构，典型问题是 RMQ（Range Minimum Query，区间最小值查询）。

它的核心特征是：
- 预处理较重：`O(n log n)`；
- 查询极快：对满足幂等性质的运算（如 `min/max/gcd`）可做到 `O(1)`；
- 不支持原地动态更新（若数组变动，通常需要重建）。

本目录实现一个最小可运行 MVP：静态数组上的区间最小值查询。

## R02

问题定义（本实现）：
- 输入：长度为 `n` 的整数数组 `a[0..n-1]`，`n >= 1`；
- 查询：`range_min(l, r)`，其中 `0 <= l <= r < n`；
- 输出：闭区间 `[l, r]` 的最小值。

目标是在一次预处理后，让每次查询尽可能快，并通过固定样例与随机回归验证正确性。

## R03

数学基础与递推关系：

1. 记 `st[k][i]` 表示区间 `[i, i + 2^k - 1]` 的最小值。  
2. 边界层：`st[0][i] = a[i]`。  
3. 状态转移：
   `st[k][i] = min(st[k-1][i], st[k-1][i + 2^(k-1)])`。  
4. 查询 `[l, r]` 时，令 `len = r-l+1`，`k = floor(log2(len))`，则：
   `min(l,r) = min(st[k][l], st[k][r - 2^k + 1])`。

最后一步成立的关键是 `min` 的幂等性：两段可重叠覆盖不会影响答案。

## R04

算法流程（高层）：
1. 预计算 `log2[x] = floor(log2(x))`（`x=1..n`）。
2. 构建第 0 层稀疏表，直接拷贝原数组。
3. 逐层构建第 `k` 层，每格由第 `k-1` 层两个半区间合并得到。
4. 执行查询时，依据区间长度找到 `k`，取两个长度为 `2^k` 的块做一次 `min`。
5. 用暴力 `min(a[l:r+1])` 对拍验证，确保实现无误。

## R05

核心数据结构：
- `data: list[int]`：原始静态数组；
- `log2: list[int]`：长度到 `floor(log2)` 的查表；
- `st: list[list[int]]`：二维表，第 `k` 层存所有长度 `2^k` 区间的最小值；
- `n`：数组长度。

空间规模约为 `n + n/2 + n/4 + ... < 2n` 每层展开总和，整体 `O(n log n)`。

## R06

正确性要点：
- 构建正确：每个 `st[k][i]` 都对应精确区间 `[i, i+2^k-1]` 的最小值（数学归纳法可证）。
- 查询正确：
  - 两个长度 `2^k` 的区间分别覆盖查询区间左端与右端；
  - 这两个区间并集覆盖 `[l, r]`；
  - 对 `min` 运算，重叠元素不会破坏结果，因此一次 `min` 合并即得答案。

## R07

复杂度分析：
- 预处理：`O(n log n)`；
- 单次查询：`O(1)`；
- 额外空间：`O(n log n)`；
- 重建代价：若数据更新，通常需要重新构建整张表。

这使稀疏表非常适合“查询远多于更新”的静态场景。

## R08

边界与异常处理：
- 空数组：构造时抛 `ValueError`；
- 非法区间（越界或 `l > r`）：查询时抛 `IndexError`；
- 单元素区间（`l == r`）：应直接返回该元素；
- 全等值、负数、混合正负数：都应正常工作。

本实现选择显式抛错，不做静默修正，避免隐藏输入问题。

## R09

MVP 取舍说明：
- 仅实现 `min` 查询，不泛化到自定义聚合函数；
- 不支持动态更新（不做分块重建或树结构增量维护）；
- 仅依赖 Python 标准库，算法流程完全可读，不依赖黑盒数据结构库。

该取舍让实现短小、行为明确，便于教学和审计。

## R10

`demo.py` 代码结构：
- `SparseTableMin.__post_init__`：参数校验与预处理入口；
- `_build_log_table`：构建 `log2` 查表；
- `_build_sparse_table`：按层构建 `st`；
- `_check_query`：统一查询边界校验；
- `range_min`：`O(1)` 查询接口；
- `run_deterministic_case`：固定样例演示；
- `run_randomized_regression`：随机对拍验证；
- `main`：执行全部流程并输出成功信息。

## R12

关键实现参数（与 R04 对应）：
- `max_k = floor(log2(n)) + 1`：稀疏表层数；
- 每层有效长度 `width = n - 2^k + 1`：第 `k` 层列数；
- 查询选层 `k = log2[r-l+1]`：决定使用哪一层做 `O(1)` 合并；
- 回归参数：`num_cases` 与 `queries_per_case` 控制验证强度。

调参建议：
- 快速冒烟可减小回归规模；
- 提高置信度可增大随机案例数与查询数。

## R13

保证与限制：
- 在输入合法且数组静态不变时，查询结果与暴力最小值一致；
- 预处理后每次查询是常数时间；
- 不提供动态更新保证（修改数据后旧表失效，必须重建）。

因此它是“静态 RMQ 最优查询时间”方案，而非通用动态区间结构。

## R14

常见失效模式与排查：
1. 把开区间当闭区间，导致边界偏移；
2. 误传 `l > r` 或越界索引；
3. 数据更新后忘记重建稀疏表；
4. 将此查询公式误用于非幂等运算（如区间和），得到错误结果。

诊断策略：
- 对异常输入直接抛错；
- 在开发阶段用暴力对拍做回归；
- 对可疑查询打印 `(l, r, k, left_block, right_block)` 进行定位。

## R15

与相关结构对比：
- 前缀和：区间和 `O(1)`，但不适用于最小值 RMQ；
- 线段树：支持动态更新，查询 `O(log n)`，更通用；
- 树状数组：擅长前缀可加型问题，不直接解决 RMQ；
- 稀疏表：静态场景下 RMQ 查询 `O(1)`，实现也较简洁。

选型经验：
- 静态大量 RMQ 查询，优先稀疏表；
- 有频繁更新，优先线段树等动态结构。

## R16

适用与不适用场景（与 R08 前提一致）：
- 适用：
  - 数组构建后不再修改；
  - 查询量大，且关注 `min/max/gcd` 等幂等区间查询；
  - 需要稳定、可预测的 `O(1)` 查询耗时。
- 不适用：
  - 高频更新场景；
  - 需要非幂等查询并仍想 `O(1)` 的场景；
  - 内存极端受限且无法接受 `O(n log n)` 空间。

## R17

`demo.py` 的 MVP 交付清单：
- 实现 `SparseTableMin` 并支持 `range_min(l, r)`；
- 固定案例输出多组查询并与暴力结果逐条对比；
- 随机回归（多组随机数组 + 多次随机查询）自动断言；
- 无交互输入，可直接 `uv run python demo.py` 运行。

## R18

`demo.py` 源码级算法流程（8 步）：
1. `main()` 调用 `run_deterministic_case()` 与 `run_randomized_regression()`。  
2. 在构造 `SparseTableMin(data)` 时，`__post_init__` 先校验数组非空并记录 `n`。  
3. `_build_log_table()` 线性预计算 `log2[1..n]`，供查询 O(1) 选层。  
4. `_build_sparse_table()` 先把原数组作为第 0 层，再按 `k=1..max_k-1` 用递推式构建各层。  
5. 查询 `range_min(l,r)` 时先 `_check_query()` 做边界检查，避免非法索引污染结果。  
6. 计算 `k = log2[r-l+1]`，从第 `k` 层取两个块：左块起点 `l`，右块起点 `r-2^k+1`。  
7. 对两个块值做一次 `min` 合并，即得到区间 `[l,r]` 的最小值。  
8. 两个测试函数把每次查询结果与暴力 `min(data[l:r+1])` 对照；若全部断言通过即输出成功信息。
