## R01

Karatsuba 算法是一种用于大整数乘法的分治算法。它把一次 `n` 位乘法拆成 3 次约 `n/2` 位乘法，从而优于朴素乘法的二次复杂度。

给定两个非负整数 `x, y`，取一个分割位宽 `m`，写成:

- `x = x1 * B^m + x0`
- `y = y1 * B^m + y0`

其中 `B` 是进制（实现中常用 `2` 的幂或 `10` 的幂）。则:

- `z2 = x1 * y1`
- `z0 = x0 * y0`
- `z1 = (x1 + x0) * (y1 + y0) - z2 - z0`
- `x*y = z2 * B^(2m) + z1 * B^m + z0`

核心收益是把传统需要的 4 次子乘法（`x1y1, x1y0, x0y1, x0y0`）压缩到 3 次。

## R02

- 提出者: Anatoly Karatsuba（与 Yuri Ofman 共同发表）。
- 时间: 1962 年。
- 历史意义: 这是第一个打破“整数乘法必为 `O(n^2)`”直觉的重要结果，开启了后续 Toom-Cook、FFT/NTT 等更快乘法算法路线。

## R03

问题背景是“大整数乘法在密码学、任意精度计算中的成本过高”。

- 朴素竖式乘法在位数增长时成本是 `O(n^2)`。
- 当 `n` 很大时，乘法成为许多数值流程（模幂、RSA、ECC、符号计算）的瓶颈。
- Karatsuba 通过分治减少乘法次数，属于“用加减和位移换乘法”的经典思想。

## R04

设输入规模为 `n`（按位数计）。Karatsuba 满足递推:

- `T(n) = 3T(n/2) + O(n)`

由 Master 定理可得:

- `T(n) = O(n^(log2 3)) ≈ O(n^1.585)`

对比:

- 朴素乘法: `O(n^2)`
- Karatsuba: `O(n^1.585)`

当 `n` 足够大时，Karatsuba 渐近更优；但在较小输入上，递归与常数开销可能让朴素法更快，所以工程上需要阈值切换。

## R05

空间复杂度主要来自递归栈与中间结果:

- 递归深度约 `O(log n)`
- 每层会持有若干子问题结果与加减临时量

若按实现细节优化，中间内存可控制在近线性量级，常见描述为 `O(n)` 级额外空间（不含输入输出本身）。

## R06

一个十进制拆分示例:

- `x = 1234`, `y = 5678`
- 取 `m=2`, `B=10`
- `x1=12, x0=34; y1=56, y0=78`

计算:

- `z2 = 12*56 = 672`
- `z0 = 34*78 = 2652`
- `(x1+x0)(y1+y0) = 46*134 = 6164`
- `z1 = 6164 - 672 - 2652 = 2840`

合并:

- `x*y = 672*10^4 + 2840*10^2 + 2652`
- `= 6,720,000 + 284,000 + 2,652 = 7,006,652`

与直接计算 `1234*5678=7006652` 一致。

## R07

算法价值:

- 理论层面: 证明整数乘法可严格快于二次复杂度。
- 工程层面: 是高性能大整数库中的关键中层算法（通常位于学校乘法与 Toom/FFT 之间）。
- 教学层面: 分治思想、复杂度分析、代数变形三者结合的经典案例。

## R08

先验知识与组件:

- 分治法与递归。
- 多项式/整数展开与代数化简。
- 基础位运算（移位）或进制拆分。
- 渐近复杂度（Master 定理）。

## R09

适用场景:

- 输入是大整数，且规模达到 Karatsuba 优势区间。
- 需要大量重复乘法（如大整数模幂、任意精度库内部）。

边界与不适用:

- 小整数场景通常直接用朴素乘法或语言内建乘法更快。
- 递归实现若无阈值和尾部优化，常数开销会偏大。
- 在极大规模上，通常还会继续切换到 Toom-Cook/FFT，而非只靠 Karatsuba。

## R10

正确性要点（简述）:

设

- `x = x1*B^m + x0`
- `y = y1*B^m + y0`

则

- `x*y = x1y1*B^(2m) + (x1y0 + x0y1)*B^m + x0y0`

并且

- `(x1+x0)(y1+y0) = x1y1 + x1y0 + x0y1 + x0y0`

所以

- `x1y0 + x0y1 = (x1+x0)(y1+y0) - x1y1 - x0y0 = z1`

代回可得合并公式:

- `x*y = z2*B^(2m) + z1*B^m + z0`

递归基（小规模直接乘）显然正确，递归步基于恒等式成立，因此整体正确。

## R11

该算法是整数精确运算，不涉及浮点误差放大问题。

实现时主要关注:

- 进制拆分与合并的边界处理是否正确。
- 符号（正负号）是否统一在外层处理。
- 递归基阈值选择是否导致极端性能退化。

`demo.py` 中采用按二进制位宽拆分，并用移位合并，可避免十进制字符串换算带来的额外误差与复杂度。

## R12

真实工程成本（常数项）:

- 递归函数调用与对象分配成本不可忽略。
- 加减法数量显著增加，若底层大整数实现弱，会拉低收益。
- 阈值 `cutoff` 对性能影响很大，需要实测调优。
- Python 内置 `int` 乘法本身已是优化实现，纯 Python Karatsuba 常用于教学与可解释验证，而非追求极致性能。

## R13

Karatsuba 是确定性算法，不依赖随机化，因此不存在“概率正确性”问题。

- 只要实现正确，输出必然等于数学定义的乘积。
- 不同输入只影响运行时间，不影响结果正确性。
- 与随机分解算法（如 Pollard's Rho）不同，Karatsuba 没有“成功概率”概念。

## R14

常见失效模式:

- 递归终止条件设置不当导致栈过深或性能异常。
- 负数处理遗漏导致符号错误。
- 拆分位宽 `m` 选择或移位拼接写错导致结果错位。

鲁棒性策略:

- 统一在入口处理符号，内部只做非负数递归。
- 用大量随机回归测试对比 Python 内建乘法。
- 对 `0`、`1`、不同位宽、奇偶位宽等边界做专项测试。

## R15

工程实现建议:

- 设置阈值: 当位宽小于阈值时直接 `x*y`。
- 使用位运算拆分: `high = x >> m`, `low = x - (high << m)`。
- 递归函数只接收非负整数，减少分支复杂度。
- 保持函数纯净与可测试，便于后续替换阈值或扩展到 Toom-Cook。

## R16

前驱与后继关系:

- 前驱: 学校乘法（竖式乘法）`O(n^2)`。
- 当前: Karatsuba `O(n^1.585)`。
- 后继: Toom-3/Toom-Cook、Schönhage-Strassen（FFT）、Fürer、Harvey-van der Hoeven 等更快算法。

实践中常见“分段切换”:

1. 小规模用学校乘法。
2. 中规模用 Karatsuba。
3. 更大规模切到 Toom/FFT。

## R17

Python MVP 设计（见 `demo.py`）:

- 依赖: Python 3 标准库（`random`, `time`, `typing`），无第三方依赖。
- 核心函数:
  - `karatsuba_mul(x, y, cutoff_bits)`：对外接口，含符号处理。
  - `_karatsuba_nonneg(x, y, cutoff_bits)`：非负整数递归主体。
  - `schoolbook_mul_decimal(x, y)`：十进制竖式乘法实现，用于对照。
  - `run_correctness_checks()`：固定样例 + 随机样例校验。
  - `run_benchmarks()`：比较学校乘法、Karatsuba、内建乘法。
- 运行: `python3 demo.py`，无交互输入。

## R18

源码级流程拆解（对应 `demo.py`，非黑箱）:

1. `karatsuba_mul` 先处理 `0` 与符号，转为非负乘法问题。
2. `_karatsuba_nonneg` 对小位宽直接走基线 `x*y`（阈值剪枝）。
3. 取 `n = max(bit_length(x), bit_length(y))`，`m = n // 2` 作为分割位。
4. 按位拆分: `x = x1*2^m + x0`, `y = y1*2^m + y0`。
5. 递归计算 `z2 = x1*y1` 与 `z0 = x0*y0`。
6. 递归计算交叉项基础值 `(x1+x0)*(y1+y0)`，再减去 `z2` 与 `z0` 得 `z1`。
7. 用移位合并: `(z2 << (2*m)) + (z1 << m) + z0` 得到乘积。
8. `run_correctness_checks` 用随机与固定样例对比内建乘法，保证实现正确。
9. `run_benchmarks` 输出不同位宽下的耗时对比，展示 Karatsuba 与学校法的规模差异。

以上 9 步就是本实现的完整可执行算法链路。
