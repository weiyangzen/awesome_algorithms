# 递归下降分析

## R01

递归下降分析（Recursive-Descent Parsing）是一类自顶向下语法分析方法：为每个非终结符编写一个递归函数，按文法规则逐步消费 token，最终得到语法树或报错。

本条目实现一个“算术表达式 LL(1) 文法”的最小可运行 MVP，覆盖词法分析、递归下降语法分析、AST 构建、表达式求值与错误处理。

## R02

目标问题：把输入字符串（如 `10 / (2 + 3) + 7 * 2`）解析为抽象语法树（AST），并验证语法结构是否符合文法约束。

该问题在编译器前端中对应“语法分析阶段”，核心输出不是数值本身，而是结构化语法表示（AST）；求值只是演示 AST 可执行性的附加步骤。

## R03

MVP 使用的文法（已消除左递归）如下：

- `Expr -> Term ((PLUS | MINUS) Term)*`
- `Term -> Factor ((STAR | SLASH) Factor)*`
- `Factor -> NUMBER | LPAREN Expr RPAREN | MINUS Factor`

其中 `Expr` 处理加减优先级层，`Term` 处理乘除优先级层，`Factor` 处理原子与括号，保证运算符优先级和结合性正确。

## R04

为什么该文法适合递归下降：

- 选择分支时只需看当前 token（例如 `Factor` 中看是 `NUMBER`、`LPAREN` 还是 `MINUS`）；
- 不需要回溯和复杂冲突消解；
- 文法结构与代码函数一一对应，便于调试与教学。

这就是典型的 LL(1) 风格实现路径。

## R05

`demo.py` 先执行手写词法分析 `tokenize`：

- 支持 token：`NUMBER + - * / ( ) EOF`；
- 忽略空白字符；
- 支持整数与小数（如 `3.5`）；
- 对非法字符或非法数字字面量直接抛出 `SyntaxError` 并带位置。

词法阶段把原始字符串转成线性 token 流，为语法阶段提供确定输入。

## R06

语法阶段由 `Parser` 类完成，核心函数：

- `parse_expr`：处理 `+/-` 链；
- `parse_term`：处理 `*//` 链；
- `parse_factor`：处理数字、括号、一元负号；
- `expect`：断言当前 token 类型并前进。

解析过程中同步构建 AST 节点：`Number`、`UnaryOp`、`BinaryOp`。

## R07

复杂度分析（设 token 数量为 `n`）：

- 词法分析：`O(n)`；
- 语法分析：`O(n)`（每个 token 被消费常数次）；
- AST 求值：`O(n)`；
- 额外空间：`O(n)`（token 流 + AST + 递归栈）。

因此该 MVP 在表达式场景下是线性复杂度。

## R08

优点：

- 实现小而透明，算法路径可逐行追踪；
- 文法与代码同构，维护和扩展成本低；
- 错误定位粒度清晰（包含 token 类型和值、字符位置）。

局限：

- 只覆盖表达式子语言，不含变量、函数调用、语句块；
- 递归深度受 Python 栈限制，不适合超深嵌套输入；
- 未实现 panic-mode 等恢复策略，首次错误即中止。

## R09

环境与运行方式：

- Python 3.10+；
- 依赖 `numpy`（仅用于统计输出，解析核心是纯 Python）；
- 无交互输入，直接运行内置样例。

执行命令：

```bash
cd Algorithms/计算机-编译原理-0445-递归下降分析
uv run python demo.py
```

## R10

正确性直觉（非形式证明）：

- `Expr/Term/Factor` 三层拆分保证优先级：`Factor` 最紧，`Term` 次之，`Expr` 最外层；
- 每个分支都由 `expect`/`current` 严格检查 token 类型，避免“跳读”；
- `parse()` 最后强制 `EOF`，确保输入被完整消费，不接受尾随垃圾字符。

因此得到的 AST 与文法推导一致，且不会静默吞掉错误输入。

## R11

边界与异常处理策略：

- 非法字符：`tokenize` 直接报 `SyntaxError`；
- 非法数字（如 `.` 或 `1..2`）：在词法阶段拦截；
- 括号不匹配、缺失操作数：在 `expect` 或 `parse_factor` 中报错；
- 除零：在 `evaluate` 中抛 `ZeroDivisionError`。

`run_demo` 额外包含三条非法样例，确保错误路径可复现。

## R12

与其他语法分析技术对比：

- 相比表驱动 LL/LR：递归下降代码更直观、教学友好；
- 相比 PEG/packrat：本实现不做回溯缓存，内存更轻但文法表达能力较受限；
- 相比自动生成器（Yacc/Bison/ANTLR）：手写版可控性强，但需要人工保证文法无冲突。

本条目聚焦“可读、可验证”的手写解析器 MVP。

## R13

实现细节选择：

- 用 `Token(kind, value, pos)` 保存位置信息，提升报错质量；
- 用 dataclass 定义 AST，结构清晰且便于递归处理；
- 解析 trace 保存在 `Parser.trace`，用于展示推导路径；
- 将 `ast_size/ast_depth/to_infix` 独立成工具函数，便于测试与扩展。

## R14

递归下降常见实现错误：

1. 文法仍含左递归（如 `Expr -> Expr + Term`），导致无限递归；
2. 漏掉 `EOF` 校验，出现“前半段合法就通过”的假阳性；
3. 把一元负号和二元减号混淆，优先级处理错误；
4. 错误信息不带位置，调试成本高；
5. 直接调用第三方黑盒解析器，无法解释语法函数级流程。

## R15

可扩展方向：

- 增加标识符、赋值语句、函数调用等语法；
- 增加一元加号、幂运算与右结合规则；
- 输出 AST 到 JSON，用于后续语义分析/中间代码生成；
- 增加错误恢复（例如同步到分号或右括号）。

## R16

适用场景：

- 编译原理教学和课程实验；
- DSL 原型验证；
- 需要快速构建、可解释性高的解析器场景。

不适用场景：

- 高度复杂、冲突较多的大型工业语法；
- 需要强恢复能力与高吞吐批量解析的生产编译器前端。

## R17

`demo.py` 功能清单：

- `tokenize`：把文本转成 token 流；
- `Parser.parse_expr/parse_term/parse_factor`：执行递归下降；
- `evaluate`：递归解释 AST；
- `to_infix`：把 AST 还原成带括号中缀表达式；
- `ast_size/ast_depth`：统计 AST 结构；
- `run_demo`：执行 4 条合法样例 + 3 条非法样例 + 自动断言；
- `numpy` 统计：汇总 token 数、节点数、树深均值。

脚本可直接 `uv run python demo.py` 运行，不需要任何交互输入。

## R18

`demo.py` 的源码级算法流程（8 步）：

1. `run_demo` 准备固定表达式样例，分为合法用例和错误用例两组。  
2. 每个合法表达式先进入 `tokenize`：按字符线性扫描，产出 `NUMBER/+/-/*///(/)/EOF` token 流。  
3. `parse_expression` 创建 `Parser` 并调用 `parse()`；`parse` 从起始符 `Expr` 开始递归下降。  
4. `parse_expr` 先解析一个 `Term`，再循环吸收后续 `+/- Term`，构建左结合二元节点。  
5. `parse_term` 同理吸收 `*// Factor`，从而把乘除绑定得比加减更紧。  
6. `parse_factor` 处理三类原子：数字、括号子表达式、一元负号；遇到不匹配 token 即抛 `SyntaxError`。  
7. 解析成功后，`evaluate` 递归遍历 AST 计算结果，`run_demo` 用断言校验与期望值一致；同时输出前 8 条推导 trace。  
8. 最后把每条合法样例的 token 数、AST 节点数、树深送入 `numpy` 计算均值/最大值，打印汇总统计。

实现中未调用“几行封装完成语法分析”的黑盒库；词法、递归下降和 AST 处理均在源码内可追踪。
