## R01

资源分配图算法（Resource Allocation Graph, RAG）用于描述并发系统中的“进程-资源”依赖关系，常用于死锁分析：

- 进程节点 `P_i` 与资源节点 `R_j` 共同构成有向图；
- 请求边：`P_i -> R_j`（进程正在等待该资源）；
- 分配边：`R_j -> P_i`（资源已分配给该进程）。

核心问题：图中是否存在导致系统无法继续推进的等待闭环。

## R02

在操作系统课程中，RAG 常用于解释：

- 死锁四要素中的“循环等待”如何在图结构里显式呈现；
- 资源分配策略与进程阻塞状态之间的因果关系；
- 为什么“检测”与“避免”（如银行家算法）是两类不同思路。

对单实例资源类型（每种资源仅 1 个实例）而言，RAG 中出现环路与死锁有等价关系，适合做最小可运行演示。

## R03

形式化建模（本目录采用单实例版本）：

- 进程集合：`P = {P1, P2, ..., Pn}`
- 资源集合：`R = {R1, R2, ..., Rm}`，且每个 `Rj` 至多分配给一个进程
- 请求关系：`Req ⊆ P x R`
- 分配关系：`Alloc ⊆ R x P`

若存在闭环 `P_a -> R_b -> P_c -> ... -> P_a`，则该环上的进程彼此等待对方持有的资源，系统进入死锁。

说明：多实例资源时，“有环”仅是死锁必要条件，不是充分条件。

## R04

本目录 MVP 算法路径：

1. 输入 RAG（进程、资源、分配边、请求边）；
2. 将 RAG 折叠为等待图（Wait-For Graph, WFG）：
   - 若 `P_i` 请求 `R_k` 且 `R_k` 正被 `P_j` 占有，则加入边 `P_i -> P_j`；
3. 在 WFG 上用 DFS 三色标记法找回边（gray -> gray）；
4. 找到回边即提取一个进程环，判定死锁。

该流程代码短、可解释性强，适合作为教学级最小实现。

## R05

设进程数 `n`，请求边数 `e_r`，等待图边数 `e_w`：

- RAG -> WFG 构建复杂度：`O(e_r)`（查询资源所有者为常数）；
- DFS 环检测复杂度：`O(n + e_w)`；
- 总体复杂度：`O(n + e_r + e_w)`，在单实例模型中通常近似线性；
- 额外空间：`O(n + e_w)`（图存储 + DFS 状态）。

`demo.py` 额外输出邻接矩阵（`numpy`）用于可视化检查。

## R06

示例（死锁场景）：

- 分配：`R1->P1, R2->P2, R3->P3`
- 请求：`P1->R2, P2->R3, P3->R1`

折叠到 WFG 后：

- `P1` 等待 `R2` 的持有者 `P2`，得 `P1->P2`
- `P2` 等待 `R3` 的持有者 `P3`，得 `P2->P3`
- `P3` 等待 `R1` 的持有者 `P1`，得 `P3->P1`

形成进程环 `P1->P2->P3->P1`，因此死锁成立。

## R07

优点：

- 图模型直观，便于把阻塞关系可视化；
- 单实例条件下判定清晰（环 <=> 死锁）；
- 能输出具体环路，利于定位问题进程。

局限：

- 多实例资源下仅凭环检测会出现“有环但可继续推进”的情形；
- 仅做检测，不直接给出避免策略；
- 静态快照分析不等于在线调度最优决策。

## R10

本实现中的正确性检查点：

1. 输入校验：请求与分配关系中的进程/资源必须属于声明集合；
2. 资源单实例约束：一个资源只能有一个所有者；
3. WFG 构造只在“请求资源已被占用”时连边；
4. DFS 使用三色标记，回边出现即说明存在环；
5. 返回的环必须首尾同一节点，且长度不少于 2 条边。

## R11

死锁判定逻辑（单实例）：

- 充分性：若发现进程环，每个进程都在等待环上下一进程持有的唯一资源，无法推进；
- 必要性：若系统死锁，阻塞进程必形成循环等待，映射到图上即环。

因此在本模型里，环检测可作为完整判定器。

扩展说明：

- 多实例资源下应使用更一般的死锁检测（如 Work/Finish 迭代），RAG 环检测只能作为快速预警信号。

## R12

`demo.py` 主要参数由两个内置测试用例给出：

- `deadlock_case`：构造 3 进程闭环，期望检测为死锁；
- `safe_case`：构造等待链但无回环，期望检测为非死锁。

每个用例包含：

- `requests`（等待请求）

脚本固定数据、无随机输入，便于稳定回归。

## R14

常见失效模式与防护：

1. 把“请求空闲资源”错误连成等待边，导致假阳性；
2. 输入中资源重复分配，破坏单实例前提；
3. DFS 只返回布尔值、不保留路径，难以定位问题；
4. 忽略节点合法性检查，导致脏数据误判。

本实现通过输入验证、单实例断言、显式提取环路径来降低以上风险。

## R18

`demo.py` 源码级算法流程（8 步）：

1. 定义 `RAGCase`：封装进程集合、资源集合、分配关系、请求关系与期望死锁标签。  
2. `build_wait_for_graph` 先做输入校验，再遍历每条请求 `P_i->R_k`：若 `R_k` 被 `P_j` 持有，则添加等待边 `P_i->P_j`。  
3. 构建完成后得到仅含进程节点的 WFG 邻接表（`dict[str, set[str]]`）。  
4. `find_cycle_dfs` 对 WFG 执行三色 DFS：`WHITE(未访问) -> GRAY(在栈中) -> BLACK(完成)`。  
5. 当 DFS 遇到一条指向 `GRAY` 节点的边时，说明发现回边；利用当前递归栈切片提取一个显式环路。  
6. `adjacency_matrix` 使用 `numpy` 将 WFG 转成 0/1 邻接矩阵，便于打印和人工核对。  
7. `analyze_case` 汇总案例结果：等待边数量、是否有环、环路节点序列，并与 `expected_deadlock` 对比。  
8. `main` 依次运行死锁/非死锁案例并做断言，若全部符合预期则输出 `All assertions passed.`。  

该实现没有依赖黑盒死锁检测库：RAG 折叠、DFS 回边判定、环提取都在源码中逐步展开，可直接追踪每个判定步骤。
