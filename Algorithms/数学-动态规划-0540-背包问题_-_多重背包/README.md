# 背包问题 - 多重背包

- UID: `MATH-0540`
- 学科: `数学`
- 分类: `动态规划`
- 源序号: `540`
- 目标目录: `Algorithms/数学-动态规划-0540-背包问题_-_多重背包`

## R01

本条目实现“多重背包（Bounded Knapsack）”的最小可运行版本（MVP），目标是：
- 给定若干物品类型，每种物品有 `weight / value / count` 三元约束；
- 在总容量 `capacity` 下最大化总价值；
- 输出最优价值与每种物品的选择数量；
- 用“优化算法 vs 朴素算法”交叉校验，保证可解释与可验证。

## R02

问题定义（MVP 范围）：
- 输入：
  - 背包容量 `W >= 0`；
  - `n` 种物品，第 `i` 种有重量 `w_i > 0`、价值 `v_i >= 0`、可选件数上限 `c_i >= 0`。
- 约束：
  - 每种物品选择件数 `x_i` 为整数，且 `0 <= x_i <= c_i`；
  - 总重量满足 `sum(x_i * w_i) <= W`。
- 目标：
  - 最大化 `sum(x_i * v_i)`。
- 输出：
  - `max_value`：最优总价值；
  - `total_weight`：对应解的总重量；
  - `picks`：每种物品的选择件数。

## R03

动态规划建模：

1. 朴素多重背包 DP（基线）  
   令 `dp[i][cap]` 表示只考虑前 `i` 类物品、容量为 `cap` 时的最大价值。  
   转移为：
   `dp[i][cap] = max_{k=0..min(c_i, cap/w_i)}(dp[i-1][cap-k*w_i] + k*v_i)`。
2. 二进制拆分优化（主算法）  
   将每类物品数量 `c_i` 拆成若干组 `1,2,4,...,rest`，每组变成一个 0/1 物品包：
   - 组重量：`group_units * w_i`
   - 组价值：`group_units * v_i`
3. 拆分后问题转为 0/1 背包，可用标准二维 DP：
   `dp[p][cap] = max(dp[p-1][cap], dp[p-1][cap-pack_w] + pack_v)`。
4. 通过 `take` 选择表逆推可恢复每类物品实际选取件数。

## R04

算法流程（MVP）：
1. 校验输入合法性（容量非负、重量正、价值非负、数量非负）。
2. 对每种物品做二进制拆分，构造拆分包列表 `SplitPack`。
3. 建立二维数组 `dp`（价值）和 `take`（是否选择当前拆分包）。
4. 按拆分包逐层推进 0/1 背包转移。
5. 从 `dp[m][W]` 读取最优价值（`m` 为拆分包总数）。
6. 通过 `take` 从后向前回溯，聚合出每个原始物品类型的选取件数。
7. 计算回溯解的总重量与总价值，形成结果对象。
8. 再运行朴素多重背包 DP，做最优值一致性校验。
9. 对两种解做可行性与数值复核，打印并断言通过。

## R05

核心数据结构：
- `ItemType(weight, value, count)`：原始物品类型；
- `SplitPack(type_index, units, weight, value)`：二进制拆分后的 0/1 包；
- `MultiKnapsackResult(max_value, total_weight, picks)`：统一输出结果；
- `numpy.ndarray dp`：DP 值表；
- `numpy.ndarray take / choose`：回溯选择表（优化算法与基线算法分别使用）。

## R06

正确性要点（简述）：
- 二进制拆分等价性：任意 `0..c_i` 的整数都可由拆分组和表示，且每组最多取一次，故与原件数上限约束等价。
- 0/1 背包最优子结构：当前包“取/不取”的最优解由上一层最优值决定，二维 DP 转移成立。
- 回溯一致性：`take` 记录了每层状态是否来自“取当前包”，逆推可恢复一个与最优值一致的可行解。
- 交叉验证：朴素多重背包 DP 与拆分优化 DP 对同一输入应得到相同最优价值，作为实现正确性的外部证据。

## R07

复杂度分析：
- 朴素多重背包 DP：
  - 时间复杂度：`O(n * W * avg_count)`（更精确是 `O(sum_i W * c_i)` 上界）；
  - 空间复杂度：`O(n * W)`（本实现为便于回溯使用二维表）。
- 二进制拆分 + 0/1 背包：
  - 拆分包数 `m = sum_i O(log2(c_i + 1))`；
  - 时间复杂度：`O(m * W)`；
  - 空间复杂度：`O(m * W)`（同样为回溯保留二维信息）。

## R08

边界与异常处理：
- `capacity = 0`：最优价值必为 `0`；
- `items = []`：输出全零解；
- `count = 0` 的物品：拆分阶段自然忽略；
- 非法输入（负容量、负数量、非正重量、负价值）：抛出 `ValueError`；
- 脚本采用固定样例运行，不需要交互输入。

## R09

MVP 取舍：
- 选择“二进制拆分”而非“单调队列优化”，因为实现更短、可读性更高；
- 保留朴素 DP 基线，牺牲部分性能换取自校验透明度；
- 不引入复杂框架与黑盒求解器，核心逻辑全部在 `demo.py` 内可逐行跟踪；
- 当前仅关注非负价值场景，未扩展到负价值或多目标版本。

## R10

`demo.py` 职责划分：
- `validate_items`：输入约束检查；
- `split_items_binary`：把多重物品拆成若干 0/1 包；
- `solve_multi_knapsack_binary_split`：主算法（优化版 DP + 回溯）；
- `solve_multi_knapsack_naive`：基线算法（朴素多重 DP + 回溯）；
- `evaluate_solution`：复核解的重量、价值、可行性；
- `run_case`：执行单个测试样例并打印与断言；
- `main`：组织固定测试集并批量运行。

## R15

方法对比：
- 朴素多重背包 DP  
  - 优点：定义直接、便于教学与验证；  
  - 缺点：当 `count` 大时计算量高。
- 二进制拆分 + 0/1 背包（本 MVP 主算法）  
  - 优点：实现简洁、复杂度显著下降；  
  - 缺点：仍依赖 `W`，超大容量时内存与时间压力上升。
- 单调队列优化多重背包  
  - 优点：可做到 `O(nW)`；  
  - 缺点：实现与调试复杂度更高，本条目暂不展开。

## R18

源码级算法流（对应 `demo.py`，9 步）：
1. `main` 构建固定案例列表并逐个调用 `run_case`。  
2. `run_case` 先调用 `solve_multi_knapsack_binary_split` 计算优化解。  
3. `solve_multi_knapsack_binary_split` 调用 `validate_items` 检查容量与物品合法性。  
4. `split_items_binary` 将每类 `(w, v, c)` 拆成若干 `SplitPack`（`1,2,4,...,rest`）。  
5. 在拆分包维度上执行二维 0/1 背包转移，填充 `dp` 与 `take`。  
6. 从 `take` 表逆向回溯，累计得到每类物品的件数 `picks`，并计算 `max_value` 与 `total_weight`。  
7. `run_case` 再调用 `solve_multi_knapsack_naive`，执行朴素多重背包 DP 作为基线。  
8. `evaluate_solution` 对两套解做可行性、重量和值复算，形成一致性检查。  
9. 若最优值不一致或任一可行性/复算失败，立即抛出异常；否则打印结果并继续下一案例。
