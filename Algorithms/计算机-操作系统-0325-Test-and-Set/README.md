## R01

Test-and-Set（TAS）是经典硬件原子原语，用于实现互斥同步。其语义是：

- 原子读取一个布尔锁变量旧值；
- 同一原子操作内把该变量置为 `true`（表示“已被占用”）；
- 返回旧值供调用方判断是否拿到锁。

因此，若返回 `false`，说明调用线程成功获取锁；若返回 `true`，表示锁已被其他线程占用，需要等待或重试。

## R02

在操作系统与并发运行时中，TAS 常用于：

- 构造自旋锁（spin lock）；
- 在内核短临界区做低开销互斥；
- 作为更复杂同步原语（票据锁、队列锁等）的历史基础。

本题不是调用黑盒锁库，而是从 TAS 语义出发，手写“尝试获取-失败自旋-成功进入-释放清零”的完整路径。

## R03

可把 TAS 自旋锁形式化为以下状态机：

- 共享状态：`L in {0, 1}`，`0` 表示未加锁，`1` 表示已加锁；
- 原子转移 `old = TAS(L)`：返回旧值并设置 `L := 1`；
- 获取锁判定：当 `old == 0` 时成功进入临界区；
- 释放锁操作：`L := 0`。

关键不变量：

- 任意时刻最多一个线程处于临界区；
- 每次成功获取锁对应一次成功临界区执行；
- 统计关系：`tas_calls = failed_spins + acquires`。

## R04

本目录 MVP 采用“多线程争抢 TAS 自旋锁并累加共享计数器”的方式验证算法语义：

1. `TestAndSetFlag` 实现原子 `test_and_set` 与 `clear`；
2. `TASSpinLock.acquire` 持续调用 `test_and_set`，失败则指数退避后重试；
3. 成功线程进入临界区修改共享计数器；
4. 记录每次获取锁的失败自旋次数、等待时间和线程级统计；
5. 结束后做一致性断言并输出统计表。

说明：Python 没有直接暴露通用硬件 TAS 指令，本实现用极小互斥区模拟“单条原子指令”的不可分割语义，重点是验证同步逻辑正确性。

## R05

设线程数 `T`、每线程进入临界区次数 `K`、总成功获取次数 `N = T*K`，总失败自旋次数 `F`：

- 总 TAS 调用次数：`N + F`；
- 时间复杂度：`O(N + F)`；
- 空间复杂度：
  - 锁与共享状态为 `O(1)`；
  - 若记录每次获取锁的失败次数用于统计，则为 `O(N)`。

当竞争加剧时，`F` 会增大，意味着吞吐下降与等待增加。

## R06

双线程竞争示意：

1. 锁初值 `L=0`；
2. `T1` 调用 `TAS(L)`，读到 `0`，并把 `L` 置为 `1`，成功进入临界区；
3. `T2` 调用 `TAS(L)`，读到 `1`，获取失败，只能继续自旋；
4. `T1` 完成后执行 `clear`，使 `L=0`；
5. `T2` 下一次 `TAS(L)` 读到 `0`，成功进入临界区。

这体现了 TAS 的核心：通过单原子操作保证互斥，不会出现两线程同时拿到锁。

## R07

优点：

- 原语简单，硬件支持广；
- 实现路径短，适合极短临界区；
- 失败可立即得知，便于做退避策略。

局限：

- 高竞争时会产生大量无效自旋（总线与 CPU 周期浪费）；
- 公平性弱，可能出现线程饥饿；
- 不适合长临界区与用户态高延迟场景。

## R10

`demo.py` 的核心正确性检查：

1. `shared_counter == num_threads * entries_per_thread`；
2. `total_acquires == expected_entries`；
3. `total_tas_calls == total_failed_spins + total_acquires`；
4. `max_threads_in_cs == 1`（互斥不被破坏）；
5. 所有线程都在超时内结束，且最终锁状态为未持有。

这几项断言覆盖“互斥正确性 + 统计守恒 + 活性基本检查”。

## R11

活性与工程风险：

- 活锁风险：大量线程同时高速重试会相互干扰；
- 饥饿风险：TAS 不提供天然公平排队；
- 可扩展性问题：竞争越高，自旋失败越多，吞吐越差。

本实现通过指数退避（exponential backoff）降低冲突峰值，但它不是严格公平锁，也不保证饥饿完全避免。

## R12

主要参数：

- `num_threads`：并发线程数（默认 `8`）；
- `entries_per_thread`：每线程进入临界区次数（默认 `240`）；
- `seed`：随机种子，保证可复现；
- `critical_section_max_s`：临界区内随机停留上限，用于调节竞争强度。

调参建议：

- 想观察更明显竞争：增大线程数或临界区停留上限；
- 想更快 smoke test：减小 `entries_per_thread`；
- 想比较不同配置：固定 `seed` 后逐项改参。

## R13

理论保证与本题边界：

- 这是同步原语问题，不是近似优化问题，故“近似比”不适用；
- 正确性来自原子语义与不变量，而非随机概率保证；
- 该 MVP 聚焦单锁互斥，不覆盖 NUMA、抢占、优先级反转等系统级复杂因素。

## R14

常见错误与防护：

1. 误把“读锁变量 + 写锁变量”拆成两步非原子操作，导致双重进入临界区；
2. 获取失败后不重试或重试无退避，导致逻辑错误或活锁恶化；
3. 释放路径缺失，造成永久占锁；
4. 只看最终计数，不校验 `tas_calls` 与 `failed_spins` 账本。

本实现分别用 `test_and_set` 原子接口、退避策略、`release`、断言与统计表进行防护。

## R15

工程实践建议：

- 先测竞争分布（失败率、p95 自旋次数、平均等待）再决定是否继续用 TAS；
- 高竞争场景优先考虑更公平或可阻塞方案（票据锁、MCS、互斥量）；
- 对实时性敏感场景，控制临界区长度比盲目优化自旋更关键；
- 在 CI 中保留“正确性断言 + 指标输出”，避免仅凭吞吐做判断。

## R18

`demo.py` 源码级算法流程（8 步）：

1. 初始化 `TestAndSetFlag`，在 `test_and_set` 中用单个短临界区实现“返回旧值并置位为 true”的原子语义。  
2. 基于该原语封装 `TASSpinLock.acquire`：循环执行 `test_and_set`，若返回旧值为 `false` 则成功获取。  
3. 若获取失败（旧值为 `true`），累计一次失败自旋并做指数退避，再次重试。  
4. `run_simulation` 启动多个线程，每线程重复 `entries_per_thread` 次“获取锁 -> 临界区 -> 释放锁”。  
5. 进入临界区后更新 `threads_in_cs` 并断言其恒为 `1`，随后递增共享计数器，保证互斥可验证。  
6. 线程释放锁后记录本次获取对应的 `tas_calls`、`failed_spins` 与等待时间。  
7. 所有线程结束后，用 `numpy` 汇总总调用次数、失败率、均值/p95/最大失败自旋；用 `pandas` 生成线程级统计表。  
8. 执行一致性断言（计数守恒、互斥成立、线程全部退出、锁最终清零）并打印结果报告。  

该实现没有把核心同步行为交给黑盒：TAS 原语、加锁循环、退避策略、临界区检查与统计账本都在源码中逐步展开。
