## R01

生产者-消费者问题描述了多个生产者线程向共享缓冲区写入数据、多个消费者线程从缓冲区读取数据的并发协作场景。

- 输入: 生产速率、消费速率、缓冲区容量 `N`、线程数量。
- 输出: 在“无丢失、无重复、无越界”的前提下完成数据传递。
- 约束: 缓冲区满时生产者必须等待；缓冲区空时消费者必须等待。

## R02

该问题是操作系统并发控制的经典模型，常用于说明“互斥 + 同步”的区别。

- 互斥: 对共享缓冲区的读写是临界区，必须串行进入。
- 同步: 生产与消费存在先后依赖，需要条件等待（满/空）。
- 典型教学路径: 信号量（`empty/full/mutex`）或条件变量（`not_full/not_empty`）。

## R03

核心动机是避免并发下的数据破坏与资源浪费。

- 若无同步，可能出现缓冲区溢出、读取空槽、覆盖未消费数据。
- 若用忙等轮询，会浪费 CPU。
- 正确做法是让线程在条件不满足时阻塞，条件满足时被唤醒。

## R04

错误基线（反例）:

1. 共享数组 + 头尾指针，不加锁。
2. 生产者直接写入，消费者直接读取。
3. 通过 `if` 判断满/空但不等待。

问题:

- 存在竞态条件（race condition）。
- 可见性和顺序无法保证。
- 在高并发下会出现重复消费、丢数据或越界。

## R05

标准解法（本目录实现为条件变量版本）:

1. 用互斥锁保护缓冲区队列及计数器。
2. `put` 时若队列已满，生产者在 `not_full` 上等待。
3. `get` 时若队列为空，消费者在 `not_empty` 上等待。
4. `put` 成功后通知 `not_empty`；`get` 成功后通知 `not_full`。
5. 队列语义为 FIFO，保证按入队顺序消费。

对应的信号量等价结构是 `empty/full/mutex` 三元组。

## R06

示例时间线（容量 `N=2`）:

1. 初始缓冲区空，`C`（消费者）先到达，因空而阻塞。
2. `P1`（生产者）写入 item-a，通知 `C` 可读。
3. `P2` 写入 item-b，缓冲区满。
4. `P1` 再次尝试写入时阻塞（等待 `not_full`）。
5. `C` 读取 item-a，唤醒一个等待写入的生产者。
6. 系统继续推进，始终满足“空不读、满不写”。

## R09

适用条件与边界:

- 适用: 生产和消费速率不匹配，需要缓冲解耦时。
- 不适用: 强实时硬约束下仅靠通用线程调度无法保证截止期。
- 边界输入:
  - 容量 `<=0` 应直接报错。
  - 线程终止要有明确协议（如哨兵）。

## R10

正确性要点（简述）:

1. 互斥性: 所有队列修改都在同一把锁内完成，无并发写破坏。
2. 安全性: `put` 前检查“未满”，`get` 前检查“非空”，避免越界。
3. FIFO: 使用队尾入队、队首出队，消费顺序与入队顺序一致。
4. 活性: 条件满足后 `notify` 唤醒对端线程，系统可持续推进。
5. 终止性: 主线程在生产者结束后投递哨兵，消费者收到后退出。

## R11

并发细节与稳定性:

- `while` 等待而不是 `if`，防止伪唤醒导致错误执行。
- 计数与队列状态在同一临界区维护，避免“检查-使用”竞争。
- Python 层虽然有 GIL，但 GIL 不等于并发逻辑正确性，锁与条件变量仍必需。

## R12

工程成本（超越大 O）:

- 线程上下文切换和唤醒存在常数开销。
- 容量过小会增加阻塞次数，容量过大增加内存占用与延迟抖动。
- 生产/消费速率波动会导致背压，需要监控（如满缓冲等待次数）。

## R14

常见失效模式与防护:

- 漏掉互斥锁: 导致竞态与数据损坏。
- `if` 代替 `while` 等待: 伪唤醒后可能读空/写满。
- 只通知不对端（漏 `notify`）: 线程永久阻塞。
- 终止协议不完整: 消费者无法退出或提前退出丢数据。

## R15

实现建议:

- 优先封装 `BoundedBuffer`，把同步细节局部化。
- 记录监控指标: 峰值占用、等待次数、吞吐量。
- 保留自动断言: 总量一致、无重复、FIFO 顺序、容量不越界。
- 用小随机延迟制造并发交错，提升对竞态问题的检出能力。

## R17

本目录 `demo.py` 的 MVP 设计:

- 依赖: Python 标准库并发原语；`numpy` 仅用于轻量统计与可重复随机延迟（缺失时自动降级）。
- 核心组件:
  - `BoundedBuffer.put/get`: 条件等待 + FIFO 队列。
  - `producer_worker`: 生成 `(producer_id, local_index)` 数据。
  - `consumer_worker`: 消费并记录入队序号。
- 自动校验:
  - 生产总数 = 消费总数
  - `Counter(produced)==Counter(consumed)`（无丢失/重复）
  - 出队 `enqueue_seq` 严格有序（FIFO）
  - 峰值占用不超过容量
- 运行命令:
  - `uv run python demo.py`

## R18

`demo.py` 源码级算法流（8 步）:

1. `main()` 配置线程数、容量和每个生产者任务量，初始化 `BoundedBuffer` 与统计容器。
2. 启动消费者线程后启动生产者线程，形成“先等待后唤醒”的真实并发场景。
3. 生产者在 `producer_worker()` 中构造 `Payload`，调用 `buffer.put()` 入队。
4. `put()` 先在锁内检查容量；若满则在 `not_full` 上 `wait()`，否则写入队尾并 `notify(not_empty)`。
5. 消费者在 `consumer_worker()` 中循环调用 `buffer.get()`。
6. `get()` 在锁内检查是否为空；若空则在 `not_empty` 上 `wait()`，否则队首出队并 `notify(not_full)`。
7. 生产者全部结束后，主线程投递与消费者数量相同的哨兵 `None`，每个消费者收到一个哨兵后退出。
8. 主线程汇总并断言四类性质（总量、无重复丢失、FIFO、容量上界），最后打印吞吐与等待统计。
