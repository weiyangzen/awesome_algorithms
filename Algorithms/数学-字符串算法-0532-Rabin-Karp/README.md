# Rabin-Karp

- UID: `MATH-0532`
- 学科: `数学`
- 分类: `字符串算法`
- 源序号: `532`
- 目标目录: `Algorithms/数学-字符串算法-0532-Rabin-Karp`

## R01

Rabin-Karp 是一种基于滚动哈希（rolling hash）的字符串匹配算法，用于在主串 `text` 中查找模式串 `pattern` 的所有出现位置。
它的核心目标是把“逐字符比较”变成“先比较哈希，必要时再精确比对”，以减少平均比较成本。
本条目聚焦单模式匹配的最小可运行实现，并给出可验证的输出与复杂度分析。

## R02

核心思想：
1. 先计算 `pattern` 的哈希值。
2. 对 `text` 中每个长度为 `m=len(pattern)` 的窗口维护哈希值。
3. 若窗口哈希与模式哈希不同，则窗口必不匹配。
4. 若哈希相同，再做一次子串精确比较，规避哈希碰撞误判。

因此，算法把大量“明显不匹配”的窗口过滤掉，只在少量候选上做字符串比较。

## R03

设字符编码映射为 `v(c)`，基数为 `B`，模数为 `M`，长度为 `m` 的串 `s` 的哈希定义为：

`H(s) = (s0*B^(m-1) + s1*B^(m-2) + ... + s(m-1)) mod M`

滚动更新（窗口从 `i` 滑到 `i+1`）：
1. 去掉旧首字符贡献：`h = (h - v(text[i]) * B^(m-1)) mod M`
2. 左移一位并加新字符：`h = (h * B + v(text[i+m])) mod M`

本 MVP 采用 `B=257`、`M=1_000_000_007`，并使用 `v(c)=ord(c)+1` 支持 Unicode 字符。

## R04

单模式 Rabin-Karp 过程（与 `demo.py` 一致）：
1. 处理边界：`pattern == ""` 时返回 `0..n`；`m > n` 时返回空结果。
2. 预计算 `high_base = B^(m-1) mod M`，用于快速移除窗口首字符。
3. 计算 `pattern` 哈希与首窗口哈希。
4. 遍历所有窗口：若哈希相等，执行 `text[i:i+m] == pattern` 二次校验。
5. 记录通过校验的位置为匹配下标。
6. 若未到最后窗口，使用滚动公式更新窗口哈希。

## R05

碰撞处理策略：
1. 哈希相等不直接判定匹配，必须再做一次子串精确比较。
2. 因此本实现不会因哈希碰撞产生错误结果，只会在极端情况下增加比较次数。
3. 若业务对性能更敏感，可扩展为双模哈希（两个不同 `M`）进一步降低碰撞触发率。

## R06

正确性要点：
1. 对每个窗口都计算了与模式长度一致的哈希，保证候选窗口枚举完整。
2. 滚动更新公式与多项式哈希定义等价，保证每步窗口哈希正确。
3. 仅当“哈希相等 + 子串精确相等”同时成立才记为匹配，保证无假阳性。
4. 所有真实匹配窗口必然满足精确相等，因此不会漏检。

## R07

Rabin-Karp 的典型优势是“同长度多模式”扩展：
1. 先把多个模式串哈希放入集合（或哈希桶）。
2. 扫描文本窗口时先查哈希集合，再对候选做精确比较。
3. 在多模式过滤场景（如敏感词初筛）常优于逐模式暴力扫描。

本目录 MVP 先实现单模式版本，保持代码最小且可验证。

## R08

复杂度（`n=len(text)`, `m=len(pattern)`）：
1. 预处理：`O(m)`（计算模式哈希与首窗口哈希）。
2. 扫描窗口：`O(n-m+1)` 次滚动更新，均摊 `O(n)`。
3. 总体期望时间：`O(n+m)`。
4. 最坏时间：若大量哈希碰撞且触发精确比较，可退化到 `O(n*m)`。
5. 额外空间：`O(1)`（不计输出匹配下标）。

## R09

与其他单模式匹配算法对比：
1. KMP：最坏严格 `O(n+m)`，无哈希碰撞问题，但实现包含前缀函数逻辑。
2. Boyer-Moore：在自然语言文本上常数很优，但最坏分析更复杂。
3. Rabin-Karp：实现简洁，且容易扩展到多模式同长度匹配。

当需求包含“可快速扩展为哈希过滤器”时，Rabin-Karp 通常是更实用选择。

## R10

边界与约定（已在 `demo.py` 中覆盖）：
1. 空模式串：返回所有可插入位置 `[0, 1, ..., n]`。
2. 模式长于文本：返回空列表。
3. 重复字符文本（如 `aaaaaa`）可验证重叠匹配逻辑。
4. Unicode 文本可直接处理（`ord` 编码）。

## R11

`demo.py` 提供以下核心接口：
1. `RollingHashParams`：封装 `base` 与 `mod` 参数。
2. `rabin_karp_search(text, pattern, params)`：返回 `(匹配下标列表, 统计信息)`。
3. `brute_force_search(text, pattern)`：暴力真值函数（对拍用）。
4. `validate_case(text, pattern, params)`：单样例校验并返回结构化结果。
5. `run_demo_samples()`：运行内置样例并打印报告。

## R12

运行方式（无交互输入）：

```bash
uv run python demo.py
```

程序执行流程：
1. 逐个加载内置 `(text, pattern)` 样例。
2. 调用 `rabin_karp_search` 扫描窗口（对应 R04 的步骤）。
3. 用 `brute_force_search` 做真值对拍。
4. 输出匹配结果与哈希统计，最后打印整体通过结论。

## R13

输出字段说明：
1. `text` / `pattern`：当前测试样例。
2. `matches`：Rabin-Karp 返回的匹配起始下标。
3. `expected`：暴力法结果（用于核对）。
4. `windows_scanned`：扫描窗口数。
5. `hash_equal_checks`：哈希相等后触发精确比较的次数。
6. `verified_matches`：精确比较后确认匹配的次数。

## R14

工程化建议（与 R11/R13 一致）：
1. 生产环境可把 `rabin_karp_search` 封装为库函数，返回“匹配 + 指标”便于监控。
2. 若 `hash_equal_checks` 异常偏高，可切换参数或启用双哈希。
3. 大规模多模式场景可按模式长度分桶，复用同一窗口扫描逻辑。

## R15

常见错误：
1. 滚动更新时漏取模或减法后未归一化，导致负值哈希。
2. `high_base` 计算错误（应为 `B^(m-1)`）。
3. 忽略空模式与 `m>n` 边界，导致越界或语义不一致。
4. 把“哈希相等”直接当作“字符串相等”，产生碰撞误报。

## R16

本目录 MVP 的验证策略：
1. 使用多组内置样例覆盖普通文本、重叠匹配、Unicode、空模式、`m>n`。
2. 每组样例都与 `brute_force_search` 对拍，不一致即抛异常。
3. 打印 `windows_scanned/hash_equal_checks/verified_matches`，用于观察 R08 复杂度行为是否符合预期。

## R17

可扩展方向：
1. 双哈希或 64 位哈希，降低碰撞触发率。
2. 多模式同长度批量匹配（哈希集合过滤）。
3. 流式文本匹配：在日志/消息流中按块维护滚动窗口。
4. 与 KMP/Boyer-Moore 组合，按文本特征动态选择策略。

## R18

`demo.py` 源码级算法流（8 步）：
1. 在 `rabin_karp_search` 中读取 `n,m`，先处理空模式和 `m>n` 的边界返回。
2. 计算 `high_base = pow(base, m-1, mod)`，用于窗口左端字符出窗。
3. 通过 `_poly_hash` 计算 `pattern_hash` 与首窗口 `window_hash`。
4. 用循环遍历每个窗口起点 `start`，窗口总数为 `n-m+1`。
5. 若 `window_hash == pattern_hash`，计入 `hash_equal_checks`，并做 `text[start:start+m] == pattern` 精确校验。
6. 校验通过则把 `start` 加入 `matches`，并增加 `verified_matches`。
7. 若存在下一个窗口，按“减旧首字符贡献 -> 左移乘基数 -> 加入新字符”更新 `window_hash`。
8. 在 `validate_case` 中把 Rabin-Karp 结果与 `brute_force_search` 对拍，全部样例通过后由 `main` 输出汇总成功信息。
