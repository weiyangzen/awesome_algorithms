## R01

并行矩阵乘法（Parallel Matrix Multiplication）是在计算 `C = A × B` 时，把原本单线程的工作拆分到多个执行单元并发完成。  
矩阵乘法是科学计算、机器学习、图计算、物理仿真里的核心算子之一，因此它也是并行计算课程中的经典基准问题。

本条目提供一个最小可运行 Python MVP，覆盖：
- 顺序分块矩阵乘法基线；
- 行切分 + 线程池并行实现；
- 与 NumPy 参考结果的数值一致性校验；
- 基准耗时与速度比输出。

## R02

并行化这个问题的重要性在于：
- 算法总工作量大，`O(mkn)` 级别在中大规模矩阵下很快成为瓶颈；
- 矩阵乘法通常是更复杂流程（如神经网络、最小二乘、特征分解）的底层热点；
- 对该算子的并行优化，往往能直接拉动整个系统吞吐。

但并行矩阵乘法不等于“把 `A @ B` 扔给库函数”，还需要明确：
- 如何划分任务（行切分、块切分、二维网格切分）；
- 如何减少内存带宽与缓存失配；
- 如何保证并发写回时没有数据竞争。

## R03

本实现选择“分块 + 行切分并行”的策略：
1. 先把输出矩阵 `C` 按行切成若干块，分别交给不同 worker；
2. 每个 worker 只负责自己行块的结果，不与其他 worker 写同一区域；
3. 在 worker 内部按 `k` 维进行分块累加，减少大矩阵一次性乘法造成的缓存压力。

这种设计有两个优势：
- 并行边界清晰，不需要锁（每个线程写回互不重叠的行块）；
- 代码量小，易验证正确性，适合作为并行矩阵乘法教学级 MVP。

## R04

数学上矩阵乘法定义为：

\[
C_{ij}=\sum_{t=0}^{k-1} A_{it} B_{tj}
\]

分块视角下，可以按 `k` 轴把累加拆分成多段：

\[
C_{i,:} = \sum_{p} A_{i,\,k_p:k_{p+1}} \cdot B_{k_p:k_{p+1},\,:}
\]

`demo.py` 的核心就是在每个 worker 中执行这条“分块累加”公式。

## R05

复杂度分析：
- 时间复杂度：顺序与并行总工作量都为 `O(mkn)`；
- 空间复杂度：额外结果矩阵 `C` 为 `O(mn)`，再加少量临时块；
- 并行理论耗时：`O(mkn / p + 调度与同步开销)`，其中 `p` 是 worker 数量。

实际速度上限会受到以下因素限制：
- Python 层调度与线程管理成本；
- 底层 BLAS 与外层线程是否发生过度并行（oversubscription）；
- 内存带宽与缓存命中率。

## R06

`demo.py` 默认实验配置：
- 矩阵规模：`A(768, 768)`、`B(768, 768)`；
- 数据类型：`float64`；
- 分块大小：`block_size=128`；
- 并行度：`workers=min(8, os.cpu_count())`；
- 随机种子：`20260407`（可复现实验）。

脚本输出会展示：
- 顺序分块耗时；
- 并行分块耗时；
- NumPy 参考耗时；
- 误差指标与速度比。

## R07

并发正确性的关键约束：
- 每个 worker 仅写 `C[start:end, :]`，写入区间互斥；
- worker 内部对局部结果先累加再一次性回传；
- 主线程按行块位置写回，不存在交叉覆盖。

因此该并行方案不依赖锁来保护写回结果，避免了锁竞争成本和死锁风险。

## R10

正确性检查分两层：
1. 小矩阵精确测试：用手工可验证样例检查输出是否与期望一致；
2. 大矩阵数值测试：与 `numpy` 参考结果比较，检查 `allclose`、最大绝对误差、相对 Frobenius 误差。

`demo.py` 在 `run_self_test` 中先做第 1 层验证，通过后才进入基准测试，避免“先跑性能再发现算法错”的低效流程。

## R11

边界与鲁棒性处理：
- `rows/cols <= 0` 会抛出明确异常；
- 输入矩阵若不是 2D 或形状不匹配会直接报错；
- `workers` 自动限制在 `[1, m]` 合理区间；
- `block_size` 小于 1 时自动归一为 1。

这样可保证脚本在非法输入时快速失败，错误定位明确。

## R12

`demo.py` 关键函数职责：
- `generate_matrix`：生成可复现随机矩阵；
- `_validate_shapes`：检查矩阵维度和乘法可行性；
- `_iter_row_ranges`：把输出行区间分配给 worker；
- `sequential_blocked_matmul`：顺序分块基线；
- `_worker_row_block`：worker 内分块累加；
- `parallel_blocked_matmul`：线程池并行调度与结果拼装；
- `max_abs_error` / `relative_fro_error`：误差统计；
- `_benchmark` / `run_self_test` / `main`：实验闭环入口。

## R13

性能理解建议：
- `block_size` 过小会导致调用次数过多，调度开销上升；
- `block_size` 过大又可能降低缓存友好性；
- `workers` 增加并不总是线性提速，受限于内存带宽和 BLAS 行为；
- 行切分适合“矩阵较方、行块工作量均衡”的场景。

所以这个 MVP 的核心价值是“可解释的并行结构 + 可重复实验”，不是跑分竞赛。

## R14

常见错误与规避：
1. 让多个线程写同一块 `C`，导致数据竞争和结果漂移；
2. 只做性能对比，不做数值一致性验证；
3. 直接把第三方库当黑盒，不说明并行任务如何切分；
4. 忽略随机种子，导致结果难复现。

本实现通过“行块独占写入 + 误差断言 + 源码分步 + 固定种子”规避这些问题。

## R18

`demo.py` 的源码级并行矩阵乘法流程（8 步）：
1. `main` 先调用 `run_self_test`，在已知答案的小矩阵上验证顺序、并行、NumPy 三者结果完全一致。  
2. `generate_matrix` 按固定种子生成 `A`、`B`，保证每次实验输入一致可复现。  
3. `parallel_blocked_matmul` 先做维度检查，并计算行切分区间 `[(start,end), ...]`。  
4. 主线程为每个行区间提交一个 worker；每个 worker 只负责输出 `C` 的对应行块。  
5. `_worker_row_block` 在本地创建 `c_rows`，随后沿 `k` 轴按 `block_size` 循环分块。  
6. 对每个 `k` 分块执行 `c_rows += a_rows[:, k0:k1] @ b[k0:k1, :]`，把子乘积逐段累加到同一结果块。  
7. 主线程按区间顺序收集 worker 返回的 `c_rows`，写回全局 `C[start:end, :]`，完成并行结果拼装。  
8. 结果阶段用 `numpy` 参考值做 `allclose`、最大绝对误差和相对 Frobenius 误差校验，并输出耗时与速度比。
