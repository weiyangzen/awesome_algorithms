## R01

Compare-and-Swap（CAS）是并发编程中的基础原子原语，用于无锁/低锁同步：

- 输入三元组：`(address, expected, new_value)`；
- 若 `*address == expected`，则原子写入 `new_value` 并返回成功；
- 否则不修改内存并返回失败。

CAS 的核心价值在于把“比较 + 条件写入”合成一个不可分割操作，从而避免典型竞态更新丢失。

## R02

在操作系统与并发运行时中，CAS 常用于：

- 构建自旋锁、无锁队列、引用计数等基础组件；
- 替代“先读再写”的非原子更新路径；
- 作为多核并发下实现线性化更新的重要指令语义。

典型模式是“CAS 循环”：失败则重读当前值并重试，直到成功。

## R03

可形式化为单变量状态迁移：

- 共享状态：`x`；
- 原子转移：`CAS(x, e, n)`
  - 若 `x == e`，转移后 `x := n`，返回 `true`；
  - 否则 `x` 不变，返回 `false`。

并发场景的关键不变量：

- 成功 CAS 的线性化点可全序化；
- 对单调递增计数器，每次成功 CAS 使值 `+1`；
- 最终值应等于总成功次数。

## R04

本目录 MVP 采用“线程竞争 + CAS 增量循环”演示：

1. `AtomicCASInteger` 提供 `load` 和 `compare_and_swap`；
2. 多线程执行 `increment`：先读 `expected`，再 CAS 成 `expected + 1`；
3. 失败时指数退避（backoff）并重试；
4. 结束后校验最终计数、尝试次数守恒和重试统计。

说明：Python 无通用硬件 CAS 指令接口，示例使用短临界区锁模拟“单条 CAS 原子语义”，用于教学验证而非性能极限实现。

## R05

设线程数为 `T`，每线程操作数为 `K`，总成功操作 `N = T*K`：

- 成功次数固定为 `N`；
- 若总失败重试为 `F`，总尝试次数为 `N + F`；
- 时间复杂度可写为 `O(N + F)`；
- 空间复杂度：
  - 核心计数器为 `O(1)`；
  - demo 为统计重试分布，额外记录 `O(N)` 数据。

## R06

简化时序（两线程竞争同一计数器）：

1. `T1` 读到 `x=7`；`T2` 也读到 `x=7`；
2. `T1` 先执行 `CAS(7, 8)` 成功，`x` 变为 `8`；
3. `T2` 执行 `CAS(7, 8)` 失败（观测到当前值已是 `8`）；
4. `T2` 重读 `x=8`，再执行 `CAS(8, 9)` 成功。

该过程展示了 CAS 如何在竞争下避免“两个线程都把 7 写成 8”的丢失更新。

## R07

优点：

- 单原语即可表达常见并发更新；
- 在冲突不高时可减少长临界区阻塞；
- 失败可观测（返回 false），便于实现自适应退避。

局限：

- 高冲突下会出现频繁失败重试，吞吐下降；
- 可能发生 ABA 问题（值看起来没变但中间被改过）；
- 需要配合内存序语义与回收策略，工程复杂度高。

## R10

`demo.py` 的正确性检查点：

1. 所有线程必须在超时内完成（无卡死）；
2. `final_value == num_threads * ops_per_thread`；
3. `successes == expected_value`；
4. `attempts == successes + failures`；
5. 每个成功操作都记录一次重试次数（记录条数等于成功数）。

这些断言共同保证“语义正确 + 统计一致”。

## R11

并发语义与风险说明：

- ABA 问题：
  - 若值从 `A -> B -> A`，仅比较值可能误判“未变化”；
  - 工程中常用版本号（tagged pointer）或 hazard pointer/epoch 机制缓解。

- 内存序问题：
  - 真实系统需关注 acquire/release/seq_cst 语义；
  - 本 demo 重点在 CAS 成功/失败控制流，不展开 CPU 内存模型细节。

- 活锁风险：
  - 全体线程都在失败重试时可能长期无有效推进；
  - demo 中加入指数退避减少该风险。

## R12

`demo.py` 主要参数：

- `num_threads`：并发线程数（默认 8）；
- `ops_per_thread`：每线程成功增量次数（默认 300）；
- `seed`：随机种子，保证结果可复现；
- `read_pause_prob/read_pause_max_s`：制造竞争窗口的暂停参数。

调参建议：

- 想观察更多 CAS 失败：增大线程数或 `read_pause_prob`；
- 想快速 smoke test：降低 `ops_per_thread`；
- 想降低运行波动：固定 seed 并减少外部负载干扰。

## R14

常见失效模式与防护：

1. 把“读取 + 写入”拆开但无 CAS，导致丢失更新；
2. CAS 失败后不重试，导致操作静默丢弃；
3. 高冲突下无退避，导致活锁或极差吞吐；
4. 只看最终值，不校验 attempts/success/failure 账本一致性。

本实现通过：CAS 循环、指数退避、完整断言与统计输出来降低以上风险。

## R18

`demo.py` 源码级算法流程（8 步）：

1. 初始化 `AtomicCASInteger`：维护共享整数和内部锁，`compare_and_swap` 在单临界区内完成“读旧值 -> 比较 -> 条件写入”。  
2. `run_simulation` 创建共享计数器、统计容器（每线程 attempts/failures/successes、每次成功的 retry 次数）。  
3. 每个工作线程进入循环：先 `load()` 得到 `expected`，按概率短暂停顿制造并发竞争窗口。  
4. 线程调用 `compare_and_swap(expected, expected + 1)`：成功则一次增量完成；失败表示其他线程已抢先更新。  
5. 失败路径执行 `retry_backoff_sleep` 指数退避后重试，直到该次操作成功，期间累计失败次数与尝试次数。  
6. 所有线程结束后汇总：`final_value`、`total_attempts`、`total_failures`、`total_successes`，并用 `numpy` 计算重试均值/p95/最大值。  
7. 执行一致性断言：最终值等于总成功数、尝试账本守恒、线程全部退出、重试记录条数与成功操作数一致。  
8. 打印指标报告（冲突率、重试分布、线程分布）并返回 `SimulationResult`。  

本实现没有把关键逻辑交给黑盒库：CAS 比较与写入、失败重试、退避策略、统计与断言都在源码中逐步展开并可直接追踪。
