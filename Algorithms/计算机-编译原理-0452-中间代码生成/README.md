# 中间代码生成

- UID: `CS-0292`
- 学科: `计算机`
- 分类: `编译原理`
- 源序号: `452`
- 目标目录: `Algorithms/计算机-编译原理-0452-中间代码生成`

## R01

中间代码生成（Intermediate Code Generation）的目标是把语法树（AST）翻译成更易优化与目标代码生成的中间表示（IR）。
本条目实现一个最小可运行 MVP：
- 定义一个小型 AST（赋值、分支、循环、返回、二元表达式）；
- 生成三地址码（Three-Address Code, TAC）；
- 通过内置 TAC 执行器验证语义正确性。

## R02

为什么需要中间代码：
- 解耦前端语法分析与后端目标机细节；
- 为数据流分析、优化、寄存器分配提供统一输入；
- 让“语义正确性”和“机器相关性”分层处理。

在教学与工程中，TAC 都是非常常见的桥接层。

## R03

本实现采用的 IR 形式是显式三地址码指令序列，支持以下 opcode：
- `label`：定义标签；
- `assign`：简单赋值；
- `binop`：二元运算（算术/关系）；
- `cjump`：条件跳转；
- `jump`：无条件跳转；
- `return`：返回值。

每条指令都保留结构化字段（如 `dest/arg1/arg2/target`），而不是仅保存字符串，便于后续分析与执行。

## R04

示例源程序（由 AST 手工构造）语义如下：
- `a = 2`
- `b = 4`
- `if a < b:`
: `c = (a + b) * (b - 1)`
- `else:`
: `c = a - b`
- `while c < 20:`
: `c = c + 3`
- `return c`

该样例同时覆盖表达式翻译、分支控制流与循环回边。

## R05

表达式翻译策略：后序递归 + 临时变量。
- 常量直接返回字面量；
- 变量直接返回变量名；
- 二元表达式先翻译左右子表达式，再生成一条 `binop`，结果放入新临时变量 `t_k`。

例如 `(a + b) * (b - 1)` 会生成两条子表达式 `binop`，再生成乘法 `binop`。

## R06

语句翻译策略：
- `Assign(x, e)`：先翻译 `e` 得到 place，再发射 `assign x = place`；
- `IfElse(cond, then, else)`：
: 生成 `cjump` 到 `IF_TRUE_i/IF_FALSE_i`，then 末尾插入 `jump IF_END_i`；
- `While(cond, body)`：
: 生成 `WHILE_HEAD_i`、`WHILE_BODY_i`、`WHILE_END_i` 三标签，构建回边；
- `Return(e)`：翻译 `e` 后发射 `return`。

## R07

标签与临时变量分配规则：
- `new_temp()` 产生 `t1, t2, ...`；
- `new_label(prefix)` 产生带语义前缀的唯一标签，如 `IF_TRUE_1`、`WHILE_HEAD_4`。

这保证 TAC 无命名冲突，并可从名字直观看出控制流结构。

## R08

TAC 执行器（`TACExecutor`）的作用不是替代真实后端，而是做语义自校验：
- 先建立 `label -> pc` 映射；
- 按程序计数器解释执行 `assign/binop/cjump/jump/return`；
- 用环境字典维护变量与临时变量值；
- 关系运算输出 `0/1`，`cjump` 按 `!= 0` 判真。

这样可以直接验证“生成的 IR 是否保持原语义”。

## R09

时间复杂度（设 AST 节点数为 `N`，生成 TAC 条数为 `M`）：
- 代码生成：每个 AST 节点至多被访问常数次，`O(N)`；
- 执行器运行：顺序与跳转执行总步数为 `K`，复杂度 `O(K)`；
- 在本示例中循环次数固定很小，`K` 与 `M` 同数量级。

总体为线性规模，适合教学和回归验证。

## R10

空间复杂度：
- 指令存储 `O(M)`；
- 运行时环境（变量 + 临时）`O(V + T)`；
- 标签映射 `O(L)`。

其中 `V/T/L` 分别表示变量数、临时变量数、标签数，均远小于大型编译器真实规模。

## R11

技术栈与依赖：
- Python 3.10+；
- 核心算法只依赖标准库；
- `numpy` / `pandas` 为可选依赖，仅用于操作码统计输出增强，缺失时自动回退。

运行命令：

```bash
uv run python Algorithms/计算机-编译原理-0452-中间代码生成/demo.py
```

## R12

脚本输出结构：
- `Generated TAC`：带索引的三地址码序列；
- `Opcode Summary`：各 opcode 的频次统计；
- `Execution Result`：返回值与关键变量值；
- `Validation: PASS`：断言全部通过。

输出是确定性的，便于自动化检查。

## R13

正确性保障包含三层：
- 结构层：标签唯一性检查；
- 覆盖层：要求 `assign/binop/cjump/jump/label/return` 六类 opcode 全出现；
- 语义层：执行结果断言 `return == 21` 且 `c == 21`。

只要其中一层失败，脚本会抛异常并终止。

## R14

边界与异常处理：
- 未定义变量/临时变量会报错；
- 跳转到不存在标签会触发查表异常；
- 除零操作显式抛 `ZeroDivisionError`；
- 超过执行步数上限会报潜在死循环。

这些防护保证 MVP 在错误输入下“失败可解释”。

## R15

当前限制：
- AST 是手工构造，未包含词法/语法分析器；
- 仅支持少量语句与表达式类型；
- 未实现类型系统、函数调用、数组/指针、短路求值细化；
- 未接入真实优化管线（SSA、数据流框架等）。

本条目目标是“透明地展示中间代码生成核心过程”。

## R16

可扩展方向：
- 接入 parser，把源码字符串自动转 AST；
- 扩展 IR 指令集（函数调用、参数传递、phi 节点）；
- 增加 CFG 构建与基本块级优化（常量传播、DCE）；
- 引入属性测试/随机程序生成，做翻译与执行一致性检验。

## R17

与黑箱库方案对比：
- 黑箱可几行完成“生成 IR”，但无法看到临时变量与跳转如何组织；
- 本 MVP 将命名、发射、分支翻译、循环翻译、解释执行全部源码展开；
- 因此更适合教学、审计和二次改造。

换言之，这里强调的是可追踪性而非框架规模。

## R18

`demo.py` 源码级算法流程（8 步）：
1. `build_sample_program()` 构造固定 AST，包含赋值、`if-else`、`while`、`return`。
2. `TACGenerator.generate()` 按语句顺序调用 `gen_stmt()`，进入递归翻译。
3. `gen_expr()` 对 `Binary` 后序遍历，先生成子表达式，再发射 `binop` 到新临时变量。
4. `gen_stmt()` 对 `IfElse` 生成 `IF_TRUE/IF_FALSE/IF_END` 标签与 `cjump + jump` 控制流骨架。
5. `gen_stmt()` 对 `While` 生成 `HEAD/BODY/END` 三标签，建立条件分支和回边。
6. `format_instruction()` 把结构化指令转成可读 TAC 文本，输出完整指令表。
7. `TACExecutor.run()` 建立标签地址映射并解释执行 TAC，得到最终 `return` 与变量环境。
8. `assert_expected()` 校验标签唯一、opcode 覆盖与数值结果，全部通过后输出 `Validation: PASS`。
