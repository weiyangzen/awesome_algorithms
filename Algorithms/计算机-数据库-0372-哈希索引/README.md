## R01

哈希索引（Hash Index）是数据库中用于加速等值查询（`=`）的一类索引结构。

核心思想：

- 把索引键通过哈希函数映射到桶（bucket）；
- 每个桶维护该键对应的一组记录位置（RID 或主键）；
- 查询时直接定位桶并在桶内比对键，避免全表扫描。

它最适合高频点查场景，例如按 `user_id`、`order_id` 精确命中记录。

## R02

算法直觉可以理解为“先把候选空间切小，再做精确比对”。

- 全表扫描需要检查 `N` 行；
- 哈希索引先把数据分散到 `B` 个桶；
- 查询键只需访问一个桶，平均比较次数接近桶内链长度。

所以哈希索引在负载因子可控时，等值查询的平均复杂度可接近 `O(1)`。

## R03

本条目解决的问题：

- 输入：一组行记录（`list[dict]`）与索引列名（如 `customer_id`）；
- 输出：一个可执行的哈希索引 MVP，支持构建、等值查询、插入、删除；
- 目标：展示数据库哈希索引的源码级机制，而不是调用数据库或第三方索引黑盒。

`demo.py` 同时输出统计信息，帮助观察桶分布、冲突与查询探测成本。

## R04

本目录实现策略：

1. 使用“桶数组 + 链地址法（separate chaining）”保存键到 RID 列表的映射；
2. 每个桶内保存若干 `IndexEntry(key, row_ids)`；
3. 构建索引时按行提取键并插入对应桶；
4. 当负载因子超过阈值触发扩容并重哈希；
5. 查询时只扫描目标桶，返回 RID 列表；
6. 通过 RID 回表取出完整行，并用线性扫描结果做断言校验。

## R05

复杂度分析（`N` 为记录数，`B` 为桶数）：

- 构建索引：平均 `O(N)`，最坏 `O(N^2)`（极端冲突）；
- 等值查询：平均 `O(1 + alpha)`，`alpha = N/B` 为负载因子；
- 插入：平均 `O(1)`，偶发扩容时需要 `O(N)` 重哈希；
- 删除单个 RID：平均 `O(1 + alpha)`；
- 额外空间：`O(N + B)`。

在工程上，性能关键是哈希函数质量与负载因子控制。

## R06

`demo.py` 的样例数据是订单表（含重复 `customer_id`）：

- 适合演示一对多映射（一个键对应多个 RID）；
- 包含不存在键查询（验证空结果）；
- 包含插入与删除（验证增量维护逻辑）；
- 覆盖扩容路径（初始桶数较小，构建后会触发 rehash）。

## R07

优点：

- 等值查询速度快，平均接近常数时间；
- 结构简单，增删改查维护成本可控；
- 在高并发点查负载中常有稳定收益。

局限：

- 不擅长范围查询（`<`, `>`, `BETWEEN`）；
- 哈希冲突会拉长桶链，极端时退化到线性复杂度；
- 桶扩容需要重哈希，存在停顿成本（可通过渐进 rehash 缓解）。

## R09

适用场景：

- OLTP 场景中的主键/唯一键点查；
- 高 QPS 的等值过滤条件；
- 写入频繁但键分布相对均匀的数据。

不适用场景：

- 需要大量范围检索或有序遍历；
- 必须支持前缀扫描与排序输出；
- 键分布极端倾斜且无法优化哈希策略。

## R10

正确性直觉：

1. 构建阶段每条记录都会调用 `insert(key, rid)`，不会漏插；
2. `lookup(key)` 只在哈希命中的桶内比对真实键，避免误命中；
3. 同键 RID 存在同一 `IndexEntry.row_ids` 中，保证完整返回；
4. 删除逻辑在命中键后只移除目标 RID，不影响同键其他 RID；
5. 扩容后通过重插入重建桶结构，语义与扩容前保持等价。

因此，只要哈希函数与键比较是确定性的，查询结果与线性扫描应一致。

## R11

鲁棒性与实现细节：

- 对空键（`None`）直接报错，防止静默污染索引；
- 通过 `row.get(column)` 防止字段缺失导致崩溃；
- 删除后若 `row_ids` 为空，自动清理该键条目；
- 提供 `bucket_histogram()` 观察桶负载，便于诊断热点桶；
- 使用断言对照“索引查找 vs 线性扫描”保证行为可回归。

## R12

关键工程参数：

- `bucket_count`：初始桶数，越大冲突越少、内存越高；
- `load_factor_threshold`：扩容阈值，常见取值 `0.7~0.85`；
- 键类型与哈希稳定性：影响冲突率与可比性。

调参建议：

- 若冲突偏高，优先增大桶数或降低阈值；
- 若内存敏感，可略升阈值并接受少量查询退化；
- 对热点键明显的数据，可增加分片或二级结构缓解倾斜。

## R14

常见失效模式：

1. 忘记在写路径维护索引（数据变了，索引没变）；
2. 键类型不一致（`"101"` 与 `101`）导致查不到；
3. 负载因子过高造成桶链过长；
4. 扩容重哈希实现错误导致丢失条目；
5. 删除时误删整个键而不是单个 RID。

排查方法：

- 打印桶直方图与冲突统计；
- 抽样对比索引查询结果和全表扫描结果；
- 对扩容前后做“键集合 + RID 集合”一致性校验。

## R15

工程化建议：

- 将索引维护挂入事务日志或 WAL 流程，避免崩溃后不一致；
- 提供后台重建能力（bulk rebuild）以修复长期碎片；
- 将“索引命中率、平均桶长度、rehash 次数”纳入监控；
- 与查询优化器联动：等值谓词优先走哈希索引，范围谓词走 B+ 树。

## R16

与其他索引结构对比：

- B+ 树索引：范围查询与有序扫描更强，但点查常数项更高；
- 位图索引：低基数列聚合分析很强，不适合高频单点写入；
- 哈希索引（本条目）：等值查询快、实现直接，但范围能力弱。

实际数据库通常会组合多种索引，按查询模式选择。

## R18

`demo.py` 源码级算法流程（8 步）：

1. `HashIndex.__init__` 创建固定数量桶，每个桶是一个 `IndexEntry` 列表，并设置负载因子阈值。  
2. `build(records, key_column)` 顺序遍历数据行，提取键并调用 `insert(key, row_id)`。  
3. `insert` 先算 `bucket_id = hash(key) % bucket_count`，再扫描该桶：若命中同键就追加 RID，否则新增条目。  
4. 每次新增新键后更新 `entry_count`，当 `entry_count / bucket_count` 超过阈值时触发 `_rehash`。  
5. `_rehash` 分配更大桶数组，并把旧桶中每个 `(key, row_ids)` 重新按新桶数分发，确保扩容后语义不变。  
6. `lookup(key)` 同样先定位桶，只在该桶内做键比较，返回匹配键的 RID 列表，并记录探测次数。  
7. `delete(key, row_id)` 在目标桶内定位键并删除对应 RID；若该键 RID 为空则移除整个键条目。  
8. `main()` 用线性扫描结果校验索引查询、插入后查询、删除后查询，最后打印桶直方图和统计信息。

该流程把哈希分桶、冲突处理、扩容重哈希和查询判定都展开在源码中，可直接追踪每一步。
