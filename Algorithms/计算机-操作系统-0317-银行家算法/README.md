## R01

银行家算法（Banker's Algorithm）是操作系统里的经典**死锁避免**算法。  
它不等死锁发生后再检测，而是在每次分配前先“试分配 + 预演”，只允许系统保持在**安全状态**时继续分配。

直观上，系统像银行放贷：只有确认未来仍存在“所有客户都能结清”的还款顺序，才批准当前贷款。

## R02

该算法由 Dijkstra 提出，核心用于多进程、多资源类型环境下的资源分配决策。  
它与死锁检测/恢复的差异在于：

- 死锁检测：先运行，死锁后再处理；
- 银行家算法：在分配时预防进入不安全状态；
- 代价：需要提前知道每个进程的最大需求 `Max`。

## R03

形式化数据结构（`n` 个进程，`m` 类资源）：

- `Available[m]`：系统当前可用资源向量；
- `Max[n][m]`：每个进程对每类资源的最大声明；
- `Allocation[n][m]`：当前已分配资源；
- `Need[n][m] = Max - Allocation`：仍需资源。

安全状态定义：存在一个进程序列 `P_k`，使得每一步都满足 `Need[P_k] <= Work`，并在执行后 `Work += Allocation[P_k]`，最终所有进程都可完成。

## R04

核心策略分两层：

1. **安全性检测**：在当前状态上寻找安全序列；
2. **请求处理**：对请求向量 `Request_i` 先做约束检查，再做试分配并调用安全性检测。

仅当“试分配后仍安全”时才真正批准；否则回滚到原状态并拒绝请求。

## R05

设进程数 `n`，资源种类数 `m`：

- 一次安全性检测最坏复杂度约 `O(n^2 * m)`；
- 一次请求处理 = 常数向量比较 + 一次安全性检测，总体仍为 `O(n^2 * m)`；
- 额外空间主要是 `Need`、`Work`、`Finish`，为 `O(n*m)`。

## R06

本目录 `demo.py` 使用教材常见 5 进程、3 资源实例：

- 初始 `Available = [3, 3, 2]`；
- 初始状态可找到安全序列（示例：`[1, 3, 4, 0, 2]`）；
- 请求 `P1: [1,0,2]` 被批准；
- 请求 `P4: [3,0,0]` 因 `request > available` 拒绝；
- 请求 `P0: [0,2,0]` 虽满足 `request <= available`，但试分配后不安全，回滚拒绝。

## R11

银行家算法处理的是“避免进入可能死锁的状态”，因此关注点是**安全性**而非“当前是否已死锁”。  
关系可总结为：

- 安全状态：一定不会死锁；
- 不安全状态：不代表已死锁，但存在走向死锁的可能；
- 算法目标：禁止系统从安全状态过渡到不安全状态。

## R14

常见失效模式与规避：

1. 忘记检查 `request <= need`，导致进程越界申请；
2. 试分配失败后未回滚，破坏状态一致性；
3. `Need` 未实时同步更新；
4. 安全性检测循环终止条件写错，误判安全。

本实现通过统一 `request()` 流程和快照断言规避这些问题。

## R18

`demo.py` 的源码级算法流程（8 步）：

1. 初始化 `Banker`：读取 `Available`、`Max`、`Allocation`，校验维度与非负约束，计算 `Need = Max - Allocation`。
2. 调用 `check_safety()` 时，创建 `Work = Available` 与 `Finish[n]=False`，并准备 `safe_sequence`、`work_trace`。
3. 在未完成进程中扫描：若某进程满足 `Need[i] <= Work`，视为可执行，执行后释放其分配资源（`Work += Allocation[i]`），并标记完成。
4. 若一轮扫描无任何进展则停止；根据 `Finish` 是否全真判断 `is_safe`，并返回未完成进程列表 `pending_processes`。
5. 处理请求 `request(pid, vector)` 时，先做两道门槛检查：`request <= Need[pid]` 与 `request <= Available`。
6. 门槛通过后执行试分配：`Available -= request`、`Allocation[pid] += request`、`Need[pid] -= request`，随后立刻跑一次 `check_safety()`。
7. 若试分配后安全，则正式批准；若不安全，则执行精确回滚（把三处状态逐项加减还原）并拒绝。
8. `run_demo()` 用三组请求覆盖批准、可用量不足拒绝、以及“不安全回滚拒绝”三条路径，并用快照断言验证状态一致性。

该实现未把核心逻辑交给黑盒库：安全序列搜索、试分配和回滚都能在源码中逐行追踪。
