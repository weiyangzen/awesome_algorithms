## R01

两阶段锁（Two-Phase Locking, 2PL）是一类并发控制协议：

- **增长阶段（growing phase）**：事务可以不断加锁，但不能解锁。
- **收缩阶段（shrinking phase）**：事务一旦开始解锁，就不能再加任何新锁。

经典结论：遵循 2PL 的调度是**冲突可串行化**的。实践中常用更强的 **严格两阶段锁（Strict 2PL）**：所有写锁（常见实现中包括读锁）直到提交/回滚才释放，避免级联回滚。

## R02

2PL 解决的核心问题是：在多事务并发读写同一批数据时，既要保证正确性（等价于某个串行执行），又要保留一定并发度。

典型冲突包含：

- 读-写冲突（`R(X)` 与 `W(X)`）
- 写-读冲突（`W(X)` 与 `R(X)`）
- 写-写冲突（`W(X)` 与 `W(X)`）

没有并发控制时，上述冲突会导致丢失更新、脏读、不可重复读等异常。

## R03

2PL 的核心做法是把“访问权限”显式建模为锁：

- 读操作先申请共享锁 `S`。
- 写操作先申请排它锁 `X`。
- 只有锁兼容时才允许继续执行，否则事务进入等待队列。

兼容矩阵（请求锁 vs 已持有锁）：

- `S` 与 `S` 兼容。
- `X` 与任何锁都不兼容（除同一事务已持有的场景）。

通过“先加锁后访问、到达收缩阶段后不再加锁”的约束，形成可证明的串行化顺序。

## R04

本目录 MVP 使用的关键数据结构：

- `LockManager.table[item] -> ItemLockState`
: 每个数据项的锁状态（已授予锁列表 + 等待队列）。
- `held_by_tx[tx][item] -> mode`
: 每个事务当前持有的锁集合。
- `pending_by_tx[tx] -> LockRequest`
: 每个阻塞事务当前等待的锁请求。
- `phase_by_tx[tx] -> growing/shrinking`
: 事务所处的 2PL 阶段。
- `Txn`
: 事务程序计数器、状态、操作序列（`R/W/C`）。

## R05

MVP 调度流程（严格 2PL）：

1. 轮询事务当前操作。
2. 若是 `R(item)`，请求 `S(item)`；若是 `W(item)`，请求 `X(item)`。
3. 锁可授予则执行操作并前进 PC；否则进入等待。
4. `C`（提交）时进入收缩阶段，释放全部锁。
5. 释放锁后尝试从等待队列按 FIFO 继续授予。
6. 若全体事务都无进展，构建 wait-for graph 做死锁检测。
7. 发现环则回滚一个受害者事务，释放其锁并继续。

## R06

正确性直觉：

- 任意冲突操作对（同一数据项且至少一个写）在 2PL 下会被锁顺序强制排序。
- 这种冲突顺序可映射为一个前驱图（precedence graph）。
- 2PL 保证该图可无环化为某个串行顺序（冲突可串行化）。
- 严格 2PL 进一步保证提交前不释放写锁，避免其他事务读取未提交写入。

## R07

设：

- `N` = 事务总操作数
- `L` = 当前系统总锁请求数
- `E` = wait-for graph 边数

则典型复杂度为：

- 单次加锁判定：`O(k)`，`k` 为目标数据项当前持锁数（通常远小于 `N`）。
- 一轮调度：`O(N + lock_checks)`。
- 死锁检测（DFS）：`O(V + E)`。

空间复杂度主要来自锁表、等待队列、事务状态，约 `O(L + N)`。

## R08

适用前提与边界：

- 事务必须在读写前显式经过锁管理器。
- 需要定义锁粒度（行级/页级/表级）与兼容矩阵。
- 2PL 不自动消除死锁，需配合检测或超时策略。
- 锁粒度越粗并发度越低，越细则管理成本更高。

## R09

简化伪代码：

```text
for tx in round_robin:
    if tx.blocked: continue
    op = tx.next()

    if op is READ(x):
        if acquire(tx, x, S): execute_read(tx, x)
        else block(tx)

    if op is WRITE(x):
        if acquire(tx, x, X): execute_write(tx, x)
        else block(tx)

    if op is COMMIT:
        phase[tx] = SHRINKING
        release_all(tx)
        wake_waiters()

if no_progress:
    if detect_deadlock(wait_for_graph): abort(victim)
```

## R12

预期现象（示例场景）：

- `T1` 和 `T2` 都先读 `A`，可并发持有 `S(A)`。
- `T1` 之后要写 `A`，需升级到 `X(A)`，被 `T2` 的共享锁阻塞。
- `T2` 完成并提交后释放锁，`T1` 的升级请求被授予并继续执行。
- 最终日志显示阻塞与唤醒过程，冲突图给出一个可串行化顺序。

## R13

与其他并发控制方案对比：

- 对比时间戳排序（TO）：2PL 可能阻塞，但不必频繁回滚旧事务。
- 对比乐观并发控制（OCC）：2PL 在高冲突写负载下通常更稳定。
- 对比 MVCC：2PL 实现更直接，但读写并发能力通常弱于多版本读。

## R14

2PL 常见变体：

- **Basic 2PL**：满足“两阶段”即可，可能出现级联回滚。
- **Strict 2PL**：写锁持有到提交/回滚，工程上最常见。
- **Rigorous 2PL**：读写锁都持有到提交，隔离更强但并发更低。
- **Conservative 2PL**：事务开始前一次性申请全部锁，避免死锁但可用性较差。

本 MVP 接近 Strict/Rigorous 风格（提交时统一释放）。

## R15

工程实现中的常见坑：

- 忘记处理锁升级，导致读后写事务异常阻塞或错误放行。
- 等待队列无公平策略，可能出现饥饿。
- 死锁检测只看直接依赖，遗漏环。
- 回滚后未彻底清理等待请求，导致“幽灵阻塞”。
- 锁释放时未及时唤醒后继请求，吞吐显著下降。

## R16

建议最小测试集：

- `S/S` 并发读应同时通过。
- `X` 与 `S/X` 冲突应阻塞。
- `S -> X` 升级在“仅自己持有”时应立即成功。
- 两事务互等形成死锁时应检测到环并回滚一方。
- 提交后锁表应清空，等待队列应正确推进。

## R17

可扩展方向：

- 增加多粒度锁（意向锁 `IS/IX/SIX`）与锁升级策略。
- 加入伤害-等待（wound-wait）或等待-死亡（wait-die）预防死锁。
- 引入超时与重试策略，控制长尾事务。
- 对接真实存储引擎的日志/恢复模块，实现完整 ACID 路径。

## R18

`demo.py` 的源码级算法流（非黑盒）可拆成 8 步：

1. **事务建模**：`Txn` 用 `pc` 顺序执行 `R/W/C` 操作序列。
2. **锁表建模**：`LockManager.table` 维护每个数据项的已授予锁与等待队列。
3. **请求入场**：`request_lock()` 根据请求模式与已持有锁判断“立即授予 / 入队等待”。
4. **升级处理**：若事务已持有 `S(item)` 且请求 `X(item)`，走升级分支而不是新建独立锁。
5. **提交释放**：`begin_shrinking()` 后 `release_all()` 释放事务全部锁，进入收缩阶段。
6. **队列推进**：`_try_grant_waiting()` 按 FIFO 检查兼容性并批量唤醒可继续的请求。
7. **无进展判定**：调度循环若一轮无事务前进，则构建 `wait_for_graph()`。
8. **死锁恢复与验证**：`find_cycle()` 发现环后回滚受害者；执行结束后由 `precedence_graph()` + `topo_sort()` 给出冲突可串行化顺序。
