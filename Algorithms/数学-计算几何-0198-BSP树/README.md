## R01

BSP 树（Binary Space Partitioning Tree）是一种通过“超平面递归切分空间”来组织几何对象的结构。  
在二维里，切分超平面退化为直线；在三维里通常是平面。

本条目目标:
- 实现一个可运行、可验证的二维 BSP 树 MVP；
- 支持对线段集合建树；
- 支持给定视点后输出 back-to-front 绘制顺序（Painter 思路）。

## R02

典型应用背景:
- 早期实时图形渲染中的可见性排序；
- 碰撞检测和空间查询预处理；
- CSG（Constructive Solid Geometry）布尔运算中的空间分类。

为什么是 BSP:
- 能把全局排序问题转为局部递归决策；
- 能利用“相机在分割面哪一侧”快速确定遍历顺序；
- 对静态场景预处理后，多次查询有明显收益。

## R03

问题形式化（二维线段版本）:
- 输入: 有限线段集 `S = {s_1, ..., s_n}`，每条线段有两个端点。
- 输出:
  1. 一个 BSP 树 `T`；
  2. 对任意视点 `eye`，返回线段（或线段片段）的绘制顺序 `order(eye)`。
- 约束:
  - 若线段跨越某节点分割线，需要被切分后分别下放；
  - 不允许把第三方库当黑盒直接“调用 BSP”。

## R04

核心思想:
1. 在当前集合中选一条线段作为分割线（其支撑线作为切分超平面）。
2. 将其余线段按相对该分割线的位置分成 `front / back / coplanar / spanning`。
3. `spanning` 线段要做几何切分，分别放入前后子问题。
4. 递归构建前子树和后子树。
5. 渲染排序时根据视点相对分割线的位置，决定“先遍历哪侧子树”。

## R05

本 MVP 的数据结构:
- `Segment2D(p0, p1, label)`: 线段端点与标签。
- `BSPNode(splitter, coplanar, front, back)`:
  - `splitter`: 当前分割线段；
  - `coplanar`: 与分割线共线的线段列表；
  - `front/back`: 前后子树。

这里“前后”用有向面积符号定义:
- `side(p) = cross(splitter_dir, p - splitter_p0)`；
- `side > 0` 视作 front，`side < 0` 视作 back。

## R06

建树流程（`build_bsp_tree`）:
1. 若当前线段集为空，返回空节点。
2. 选取“最长线段”作为分割线（简单启发式，减轻退化）。
3. 对其余线段分类:
   - `coplanar`: 归入当前节点；
   - `front/back`: 放入对应桶；
   - `spanning`: 计算交点后切分成两段，分别入 front/back。
4. 递归构建前子树与后子树并挂接。

输出是一棵对当前场景几何关系可复用的分割树。

## R07

遍历流程（Painter 顺序）:
- 若视点在当前分割线 front 侧:
  1. 先遍历 `back` 子树；
  2. 再输出当前节点 `coplanar`；
  3. 最后遍历 `front` 子树。
- 若视点在 back 侧，上述顺序对调。

这对应“从远到近”的绘制策略，能在不使用 Z-buffer 的语境下得到合理覆盖顺序（二维演示版）。

## R08

复杂度分析（设当前对象数为 `n`）:
- 建树平均复杂度难给严格闭式，受分割策略与切分次数影响。
- 直观上:
  - 分类成本: 每层线性扫描；
  - 若平衡且切分适中，常见行为接近 `O(n log n)`；
  - 在极端退化（分割极差、大量切分）下可能接近 `O(n^2)` 甚至更差常数。
- 查询（一次 painter 遍历）:
  - 访问所有节点与片段，时间 `O(m)`，`m` 为树中片段总数。
- 空间复杂度:
  - `O(m)`，其中 `m >= n`（因为跨线段会被切分）。

## R09

本例构造了 8 条互相交叉的线段:
- 横线 `H0/H1/H2`
- 竖线 `V0/V1/V2`
- 对角线 `D0/D1`

这些线段会在建树时产生切分，树内片段数会大于原始线段数。  
用两个不同视点 `eye_a=(6,6)` 与 `eye_b=(-6,-6)` 遍历，可观察到输出顺序显著不同。

## R10

正确性检查点:
1. 线段分类必须覆盖 `front/back/coplanar/spanning` 四类且互斥。
2. `spanning` 线段交点参数 `t = s0/(s0-s1)` 要落在 `[0,1]`。
3. 每条（子）线段在树中应且仅应出现一次。
4. painter 遍历结果长度应等于树内片段总数。
5. 不同视点通常应产生不同顺序（本 demo 中用断言验证）。

## R11

数值稳定性与几何鲁棒:
- 使用 `EPS=1e-9` 判断“在线上”与侧别，降低浮点抖动影响。
- 对 `t` 做 `[0,1]` 截断，避免极小误差导致交点越界。
- 对“端点恰在分割线”的 spanning 边界情况做专门分支，避免生成零长度片段。

这不是工业级鲁棒几何内核，但足够支撑最小演示与逻辑验证。

## R12

参数与实现决策:
- 分割线选择: 当前实现为“最长线段优先”，简单且确定性好。
- 维度: 固定二维，便于直接看懂 cross-product 判侧逻辑。
- 输出: 仅输出顺序与统计，不做图形界面渲染。

若追求更稳定性能，可引入更复杂的 splitter 评分（如最小化切分数与树不平衡度）。

## R13

理论保证说明:
- 本条目不提供近似比意义下的保证（该问题不属于典型近似算法框架）。
- 在给定 BSP 树结构的前提下，painter 遍历遵循严格递归规则，顺序可复现。
- 真实渲染管线中的“完全正确可见性”还依赖对象类型、遮挡模型与数值鲁棒处理。

因此本 MVP 的“保证”是算法流程正确与可执行，而非图形学完备可见性证明。

## R14

常见失败模式:
- 分割线选择过差:
  - 现象: 树严重不平衡、片段爆炸；
  - 应对: 改 splitter 策略或加入深度/片段上限。
- 浮点误差导致错分:
  - 现象: 片段遗漏或顺序异常；
  - 应对: 统一 EPS 策略、增加鲁棒谓词。
- 过多共线对象:
  - 现象: 节点 `coplanar` 过大，遍历退化；
  - 应对: 在 coplanar 内再做局部排序或次级结构。

## R15

工程实践建议:
- 保持标签可追踪（demo 用 `|F`、`|B` 后缀标记切分来源）。
- 输出树统计（节点数、深度、片段数）用于评估分割质量。
- 在 CI 里保留无交互断言:
  - 片段计数一致；
  - 视点顺序差异；
  - 标签集合一致性。

这些检查比“只打印结果”更容易发现回归错误。

## R16

相关算法对比:
- `k-d tree`: 轴对齐切分，常用于最近邻和范围查询，通常不切几何对象本体。
- `BVH/R-tree`: 基于包围体层次，更新/查询模式与 BSP 不同。
- `Quadtree/Octree`: 规则网格递归切分，构造简单但对斜向结构适应较弱。

BSP 的特点是“分割超平面可任意方向”，灵活但构造与鲁棒处理更复杂。

## R18

`demo.py` 源码级算法流程拆解（8 步）:
1. `make_demo_segments` 构造 8 条相交线段，形成可切分的测试场景。
2. `build_bsp_tree` 选择最长线段作为当前 `splitter`，其余线段逐条分类。
3. `classify_segment` 通过端点相对支撑线符号，把线段判为 `front/back/coplanar/spanning`。
4. 对 `spanning` 线段，`split_spanning_segment` 用 `t=s0/(s0-s1)` 求交点并切成前后两段。
5. 前后桶递归建树，得到完整 `BSPNode` 结构（含 `coplanar/front/back`）。
6. `painter_order` 用视点与分割线侧别决定访问顺序: `back -> coplanar -> front` 或反向。
7. `tree_stats` 与 `collect_labels` 统计节点/深度/片段总数并收集标签用于一致性验证。
8. `main` 对两个视点分别求顺序，断言长度与标签集合一致且顺序不同，最后打印结果。

实现中未调用第三方 BSP 黑盒接口，关键几何判侧、切分、递归与遍历都在源码内可逐行追踪。
