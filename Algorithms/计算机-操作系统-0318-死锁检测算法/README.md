## R01

死锁检测算法用于在系统某个时刻的资源分配快照上，判断是否已有进程集合进入“彼此等待、无法继续推进”的状态。

本条目采用操作系统教材中的多实例资源检测模型：

- 输入 `Allocation / Request / Available`；
- 通过 `Work / Finish` 迭代判断可推进进程；
- 迭代结束后 `Finish=False` 的进程即为死锁进程。

## R02

与常见相关概念的区别：

- 死锁预防/避免：在分配前规避风险（如银行家算法）；
- 死锁检测：允许系统先运行，再周期性检测是否已经卡死；
- 死锁恢复：检测后采取回滚、抢占、终止等措施解除死锁。

工程上这三者经常组合：`检测` 负责发现，`恢复` 负责处置。

## R03

形式化定义（`n` 个进程，`m` 类资源）：

- `Allocation[n, m]`：当前已分配给各进程的资源数量；
- `Request[n, m]`：各进程仍在等待的资源数量（当前未满足请求）；
- `Available[m]`：系统当前可用资源数量。

检测目标：找出所有无法在现有资源约束下最终完成的进程集合。

## R04

Work/Finish 检测流程：

1. `Work <- Available`；
2. 若进程 `i` 当前不占有任何资源（`Allocation[i] == 0`），令 `Finish[i]=True`，否则为 `False`；
3. 寻找某个 `Finish[i]=False` 且 `Request[i] <= Work` 的进程；
4. 若找到，则视为其可完成并释放资源：`Work += Allocation[i]`，`Finish[i]=True`，回到步骤 3；
5. 若找不到可推进进程，算法停止；
6. 所有 `Finish[i]=False` 的进程判为死锁进程。

该算法是确定性的，不依赖随机化。

## R05

复杂度分析：

- 每轮最多扫描 `n` 个进程，每次比较 `m` 类资源；
- 最多推进 `n` 轮；
- 时间复杂度 `O(n^2 * m)`；
- 空间复杂度 `O(n + m)`（不计输入矩阵存储）。

对教学或中小规模快照检测足够直接且可解释。

## R06

示例（完整死锁）：

- 进程：`P0, P1, P2`
- 资源类型：`A, B, C`
- `Allocation`：
  - `P0` 持有 `A`
  - `P1` 持有 `B`
  - `P2` 持有 `C`
- `Request`：
  - `P0` 请求 `B`
  - `P1` 请求 `C`
  - `P2` 请求 `A`
- `Available = (0, 0, 0)`

此时没有任何进程满足 `Request <= Work`，三者全部被判定为死锁。

## R07

优点：

- 直接适配多实例资源，不依赖“有环即死锁”的单实例假设；
- 输出死锁进程集合，便于后续恢复策略执行；
- 逻辑与银行家安全性判定近似，易于教学迁移。

局限：

- 属于离线快照检测，不直接给出最优恢复动作；
- 若系统规模很大且需高频检测，需增量化或分区化优化；
- 检测结果依赖状态采样一致性，脏快照会误导结论。

## R10

本实现中的关键正确性检查：

1. 进程名、资源名不可重复；
2. `Allocation/Request/Available` 维度必须匹配；
3. 所有输入数量必须是非负整数；
4. `expected_deadlocked` 必须是声明进程子集；
5. 每次推进都必须满足 `Request[i] <= Work`，然后才允许释放 `Allocation[i]`。

## R11

正确性直觉：

- 若某进程满足 `Request <= Work`，说明当前资源足以让它完成；完成后释放资源不会使系统更差；
- 反复“可完成 -> 释放”直到无法继续，是对“系统仍可推进部分”的最大化提取；
- 最终仍不能推进的进程集合，意味着它们之间对资源形成僵局，即死锁集合。

因此，算法终态中的 `Finish=False` 是死锁判定结果。

## R12

`demo.py` 内置 3 个固定用例：

- `Case-Full-Deadlock`：三进程环形等待，预期全部死锁；
- `Case-No-Deadlock`：存在可释放链，预期无死锁；
- `Case-Partial-Deadlock`：一个无持有进程可忽略，剩余两个互等，预期部分死锁。

每个用例都包含进程名、资源名、`Allocation`、`Request`、`Available` 与期望死锁进程列表。

## R14

常见失效模式与对策：

1. 将 `Request` 误当作“最大需求”而非“当前未满足请求”；
2. 维度错位（进程数、资源数不一致）导致误判；
3. 把“无资源占有进程”也当作死锁成员；
4. 只输出布尔值不输出具体死锁进程，难以恢复。

本实现通过严格校验、`Allocation==0` 的初始处理和显式死锁列表输出规避以上问题。

## R18

`demo.py` 源码级算法拆解（8 步）：

1. 定义 `DeadlockCase`：封装进程名、资源名、`Allocation`、`Request`、`Available` 与期望死锁集合。  
2. `materialize_case` 将元组输入转为 `numpy` 矩阵/向量，并做维度与非负校验。  
3. `validate_case` 额外校验命名唯一性、期望死锁集合合法性。  
4. `detect_deadlock` 初始化 `Work = Available`，并按 `Allocation[i] == 0` 初始化 `Finish`。  
5. 主循环扫描进程：若 `Finish[i]=False` 且 `Request[i] <= Work`，则认定可完成并执行 `Work += Allocation[i]`、`Finish[i]=True`。  
6. 当一轮扫描无任何推进时停止；读取全部 `Finish=False` 进程作为死锁进程集合。  
7. `analyze_case` 汇总检测结果，记录推进顺序、`Work` 轨迹和最终死锁列表。  
8. `main` 顺序运行 3 个内置案例并断言结果与期望一致，全部通过后输出 `All assertions passed.`。  

实现未调用黑盒死锁检测库，核心判定步骤全部在源码中可跟踪。
