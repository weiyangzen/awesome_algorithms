# 进程调度 - SJF

- UID: `CS-0162`
- 学科: `计算机`
- 分类: `操作系统`
- 源序号: `307`
- 目标目录: `Algorithms/计算机-操作系统-0307-进程调度_-_SJF`

## R01

问题定义：给定一组进程（包含到达时间 `arrival` 与 CPU 执行时间 `burst`），在单核环境下使用最短作业优先（SJF, Shortest Job First）策略进行调度，输出执行时间线与每个进程的完成时间、周转时间、等待时间、响应时间。

## R02

输入：
- 进程集合 `P = {p1, p2, ...}`，每个进程包含：
  - `pid`：进程标识（唯一）
  - `arrival`：到达时间（非负整数）
  - `burst`：CPU 执行时间（正整数）

输出：
- 甘特图式时间线 `[(start, end, pid)]`，其中 `pid="IDLE"` 表示 CPU 空闲
- 每个进程指标：
  - `completion`：完成时刻
  - `turnaround = completion - arrival`
  - `waiting = turnaround - burst`
  - `response = first_start - arrival`
- 平均指标：平均周转、平均等待、平均响应时间

## R03

核心思想（非抢占式 SJF）：
- 在每次“CPU 可重新选择任务”时，从就绪队列中选择 `burst` 最短的进程执行。
- 一旦某进程被选中，本次会连续运行到完成（不被新到达进程打断）。
- 若当前没有可运行进程，CPU 进入空闲并将时间推进到下一到达时刻。
- 因为每轮都优先短作业，SJF 在很多负载下可以降低平均等待时间。

## R04

关键数据结构：
- `Process`：保存 `pid/arrival/burst/first_start/completion`
- 已排序进程表：按 `(arrival, pid)` 排序，保证到达顺序可复现
- 最小堆 `ready_heap`：按 `(burst, arrival, pid)` 作为优先键，快速选择最短作业
- `timeline`：记录执行片段 `(start, end, pid)`
- `metrics`：按 `pid` 汇总统计指标

## R05

算法流程（高层）：
1. 校验输入并构造进程对象。
2. 按 `(arrival, pid)` 排序，初始化 `time=0`，`next_idx=0`。
3. 将所有 `arrival <= time` 的进程压入最小堆。
4. 若最小堆为空：
   - 若仍有未来进程，记录 `IDLE` 并把 `time` 跳到下一到达时刻；
   - 否则调度结束。
5. 弹出堆顶（最短作业）进程，若首次执行则记录 `first_start`。
6. 该进程一次运行到完成：`end = time + burst`，记录时间线与 `completion`。
7. 更新时间并吸纳这段时间内新到达进程进入堆。
8. 重复直到所有进程完成，最后计算每进程和平均指标。

## R06

正确性要点：
- 可执行性：每轮只从“已到达”任务中选择，满足调度可行性约束。
- 最短优先性：堆顶由 `(burst, arrival, pid)` 决定，保证当前可选集合中 burst 最短者先执行。
- 非抢占性：选中后直接运行到完成，符合“非抢占式 SJF”定义。
- 指标一致性：`completion` 和 `first_start` 都直接来自时间线，派生指标公式可校验。

## R07

复杂度分析：
- 设进程数为 `n`。
- 排序开销：`O(n log n)`。
- 每个进程最多入堆/出堆各一次，总堆操作 `O(n log n)`。
- 总时间复杂度：`O(n log n)`。
- 额外空间复杂度：`O(n)`（进程状态、堆、时间线）。

## R08

边界与异常处理：
- 空输入：抛出异常（MVP 约束至少一个进程）。
- `arrival < 0`：非法。
- `burst <= 0`：非法。
- `pid` 重复：非法（避免统计覆盖）。
- 首个进程晚于 `t=0`：时间线前缀出现 `IDLE`。
- 多进程同到达且同 burst：按 `(arrival, pid)` 稳定打破平局，结果可复现。

## R09

MVP 实现范围：
- 仅实现单核、非抢占式 SJF。
- 不考虑 I/O 阻塞、多段 CPU burst、上下文切换开销。
- 输入为离散整数时间，输出为文本化时间线与指标表。
- 不依赖第三方调度库，核心流程完全手写，便于学习与审计。

## R10

`demo.py` 内置示例：
- `P1(arrival=1, burst=6)`
- `P2(arrival=2, burst=8)`
- `P3(arrival=3, burst=2)`
- `P4(arrival=5, burst=4)`

对应时间线（预期）：
- `[0,1] IDLE -> [1,7] P1 -> [7,9] P3 -> [9,13] P4 -> [13,21] P2`

## R11

预期现象：
- 短作业（如 `P3`）在进入就绪队列后会被优先调度。
- 长作业（如 `P2`）可能等待更久，出现“短任务友好、长任务延迟”的特征。
- 当系统中持续有短作业到达时，长作业存在饥饿风险（理论层面）。

## R12

指标解释：
- `Turnaround`：任务从提交到完成的总时长。
- `Waiting`：任务在就绪队列中排队等待 CPU 的累计时长。
- `Response`：任务首次得到 CPU 的延迟。
- 在非抢占式 SJF 中，每个进程只被调度一次，因此 `response == waiting`。

## R13

与其他调度策略关系：
- 与 FCFS 对比：SJF 会打破纯到达顺序，通常降低平均等待时间。
- 与抢占式 SJF（SRTF）对比：本实现不抢占，避免频繁切换但对突发短任务响应慢于 SRTF。
- 与 RR 对比：SJF 偏向优化平均性能，RR 更偏向交互公平与响应均衡。

## R14

局限与工程注意事项：
- 饥饿问题：长作业可能长期得不到执行。
- 需要已知或可估计 burst，真实系统常只能估算。
- 本 MVP 未建模上下文切换成本；真实内核需权衡切换开销与调度收益。
- 工程中常配合 Aging（老化）或多级反馈队列缓解饥饿。

## R15

常见实现错误：
- 把“非抢占式 SJF”误写成“每个时刻都可抢占”的 SRTF。
- 就绪队列为空时未推进时钟，导致死循环。
- 忘记处理 `IDLE` 时间片，造成时间线断裂。
- 指标计算中把 `response` 错写成 `completion - arrival`。
- 同 burst 平局时未定义规则，导致结果不稳定、不可复现。

## R16

最小测试清单：
- 正常交错到达 + 不同 burst。
- 首个进程非零到达（验证 `IDLE`）。
- 同时到达的多个进程（验证 tie-break）。
- 全部 burst 相同（应退化为按到达/标识顺序）。
- 非法输入：空数组、重复 pid、负到达、非正 burst。

## R17

可扩展方向：
- 实现抢占式 SJF（SRTF）并与本实现做指标对比。
- 引入 burst 预测（如指数平滑）模拟真实系统估计。
- 加入上下文切换开销、I/O 阻塞与多级队列模型。
- 结合可视化输出（甘特图图形化）用于教学演示。

## R18

源码级算法拆解（对应 `demo.py`，非黑盒）：
1. `_validate_inputs` 检查输入合法性：非空、`pid` 唯一、`arrival>=0`、`burst>0`。
2. 将输入转为 `Process` 对象，并按 `(arrival, pid)` 排序，建立可复现的到达序列。
3. 使用 `next_idx` 线性扫描未入队进程，把 `arrival <= current_time` 的任务压入最小堆。
4. 最小堆键为 `(burst, arrival, pid)`，每次 `heappop` 都得到“当前就绪集合中的最短作业”。
5. 若堆为空但仍有未来任务，向 `timeline` 追加 `IDLE` 片段，并把时钟跳到下一到达时刻。
6. 取出进程后记录 `first_start`（仅首次），然后一次执行到完成：更新时间区间并写入 `completion`。
7. 执行结束后再次吸纳新到达进程，循环直到 `completed == n`。
8. 根据 `completion/arrival/burst/first_start` 逐个计算 `turnaround/waiting/response`，再汇总平均值并打印。
