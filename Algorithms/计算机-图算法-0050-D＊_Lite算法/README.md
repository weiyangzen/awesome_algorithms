# D* Lite算法

- UID: `CS-0038`

## R01

D* Lite 是一种面向动态环境的增量启发式搜索算法，常用于移动机器人导航。  
它可以在环境局部变化（如新增障碍）时复用之前的搜索结果，而不是像 A* 一样从头计算整条路径。  
本条目提供二维栅格上的最小可运行实现，演示“初始规划 -> 机器人移动 -> 新障碍出现后重规划”。

## R02

本条目解决的问题：
- 输入：二维栅格地图 `grid`（`0` 可通行，`1` 障碍）、起点 `start`、终点 `goal`。
- 输出：
  - 当前 `start` 到 `goal` 的路径序列；
  - 路径长度（单位步长代价）；
  - 增量重规划过程中的迭代与扩展统计。

`demo.py` 使用固定样例，不需要交互输入。

## R03

核心状态定义：
- `g(s)`：当前已知的从状态 `s` 到目标 `goal` 的路径代价估计。
- `rhs(s)`：一步前瞻值，`rhs(s)=min_{s' in Succ(s)}(c(s,s') + g(s'))`。
- 一致条件：`g(s) == rhs(s)` 表示该节点局部一致；否则不一致，需要放入优先队列维护。
- 键值函数：
  - `key(s) = [min(g, rhs) + h(start, s) + km, min(g, rhs)]`
  - 采用字典序比较。

其中 `km` 记录起点移动带来的启发偏移量，用于保持键值的相对一致性。

## R04

MVP 算法主流程：
1. 初始化：所有节点 `g=rhs=inf`，仅 `rhs(goal)=0`，并把 `goal` 入队。
2. 调用 `compute_shortest_path`，持续处理队列直到 `start` 满足终止条件。
3. 依据当前 `g` 值从 `start` 贪心提取一条可行最短策略路径。
4. 当机器人移动到新位置时更新 `km` 并把 `start` 改为新状态。
5. 当地图局部变化（障碍增删）时，仅更新受影响节点及其邻居。
6. 再次执行 `compute_shortest_path`，得到新路径。

## R05

核心数据结构：
- `grid: np.ndarray[int]`：栅格地图。
- `g: np.ndarray[float]`：`g` 值表。
- `rhs: np.ndarray[float]`：`rhs` 值表。
- `PriorityQueue`：支持 lazy deletion 的最小堆。
  - `heap`：实际堆条目（允许陈旧键）；
  - `active_keys`：节点到“当前有效键”的映射。
- 标量：`start`、`goal`、`km`、`expansions`。

## R06

正确性直觉：
- `update_vertex` 使每个节点的 `rhs` 反映“经由后继节点的最优一步代价”。
- 不一致节点（`g != rhs`）被置入优先队列并按键值优先修复。
- `compute_shortest_path` 循环结束后，`start` 满足 D* Lite 终止条件，因此可从 `start` 依据 `c + g` 提取最优策略。
- 地图变化后只需更新边变化相关的局部节点，再次收敛即可完成重规划。

## R07

复杂度（网格可通行节点数记为 `V`，边数 `E`）：
- 单次从零到收敛与 A*/LPA* 同量级，近似 `O((V+E) log V)`。
- 增量更新时，仅与受影响区域和波及范围相关，通常远小于全图重算。
- 空间复杂度 `O(V)`：`g/rhs`、优先队列映射及少量辅助状态。

## R08

边界与异常处理：
- 非二维或空地图：抛 `ValueError`。
- 地图元素非 `0/1`：抛 `ValueError`。
- 起点/终点越界或落在障碍上：抛 `ValueError`。
- 试图把当前 `start` 或 `goal` 设为障碍：抛 `ValueError`。
- 若当前起点不可达，`current_path()` 返回 `None`，并以 `inf` 代价表示无解。

## R09

MVP 取舍说明：
- 未使用 `networkx` 或现成路径规划黑箱库；D* Lite 核心逻辑完全手写。
- 场景限定为 4 邻接、单位步长代价，优先保证可读性与可验证性。
- 优先队列采用“惰性删除”而非可减键堆，代码更短且足够稳定。

## R10

`demo.py` 函数/类职责：
- `PriorityQueue`：管理 D* Lite 的键值队列并清理陈旧项。
- `DStarLite.__init__`：初始化 `g/rhs`、`km` 和开放列表。
- `calculate_key` / `update_vertex` / `compute_shortest_path`：核心增量搜索流程。
- `move_start`：模拟机器人实际移动并更新 `km`。
- `set_obstacle`：应用动态障碍变化并触发局部节点更新。
- `current_path`：按 `c+g` 提取当前策略路径。
- `render_grid`：文本可视化地图与路径。
- `main`：构造固定案例并打印三阶段结果。

## R12

输出字段说明：
- `compute iterations`：本次 `compute_shortest_path` 迭代次数。
- `expansions` / `total expansions`：累计完成一致性更新的节点次数。
- `path cost`：当前路径长度（步数），无解时为 `inf`。
- 地图渲染：
  - `S`：当前起点；
  - `G`：目标；
  - `#`：障碍；
  - `*`：当前路径。
- `path sequence`：路径坐标序列。

## R13

本条目内置三阶段演示：
- 阶段 1：初始地图规划，路径通过中间墙体的左侧缺口。
- 阶段 2：机器人沿路径前进两步，验证 `km` 更新后无需大规模重算。
- 阶段 3：把左侧缺口设为新障碍，触发增量重规划，路径改走右侧缺口。

这个过程展示了 D* Lite 相比“每次全量 A*”的核心价值：复用历史搜索信息。

## R14

可调参数与扩展接口：
- 邻接方式：可扩展为 8 邻接（需同步调整启发函数）。
- 代价模型：可将 `cost` 从常数 1 扩展到地形权重。
- 动态事件：`set_obstacle` 可扩展为批量边权增减。
- 终止策略：可增加最大重规划预算、实时系统超时策略。

## R15

相关算法对比：
- A*：静态图高效；环境变化频繁时通常需要反复全量重算。
- LPA*：增量最短路框架；D* Lite 可视作其面向移动起点问题的变体。
- D* Lite：适合“边走边发现环境变化”的在线导航场景，重规划成本更低。

## R16

典型应用：
- 移动机器人在未知或半已知环境中的在线导航。
- 仓储 AGV 遇到临时障碍（托盘、人员）时的局部重规划。
- 游戏中 NPC 在动态地图（门开关、临时封锁）中的路径更新。
- 无人系统仿真教学中的“增量搜索”示例。

## R17

## R18

`demo.py` 源码级流程拆解（9 步）：
1. `main` 构造 7x10 栅格，放置中间墙和两个缺口，设置 `start=(6,1)`、`goal=(0,8)`。  
2. 创建 `DStarLite`：初始化 `g/rhs` 全 `inf`，并令 `rhs(goal)=0`，把 `goal` 按键值放入开放队列。  
3. 首次调用 `compute_shortest_path`：循环弹出最小键节点，依据 `g` 与 `rhs` 的关系做“一致化”或“失配传播”。  
4. 在每次一致化后，对该节点邻居执行 `update_vertex`，仅局部刷新 `rhs` 并维护队列。  
5. 调用 `current_path`：从当前 `start` 开始反复选择 `argmin(c + g)` 邻居，得到初始路径。  
6. 机器人前进两步后执行 `move_start`，更新 `start` 与 `km`，再调用 `compute_shortest_path` 做轻量修正。  
7. 新障碍事件通过 `set_obstacle((3,3), True)` 注入，只更新该格与其邻居的 `rhs/队列状态`。  
8. 再次 `compute_shortest_path`，算法仅在受影响区域传播不一致性并恢复终止条件。  
9. 最终 `current_path` 输出改道后的新路径，控制台打印三阶段路径、代价和扩展统计用于验证。
