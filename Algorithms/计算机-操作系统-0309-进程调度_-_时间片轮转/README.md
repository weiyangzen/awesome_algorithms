# 进程调度 - 时间片轮转

- UID: `CS-0164`
- 学科: `计算机`
- 分类: `操作系统`
- 源序号: `309`
- 目标目录: `Algorithms/计算机-操作系统-0309-进程调度_-_时间片轮转`

## R01

问题定义：给定若干进程的到达时间与 CPU 执行时间（burst），使用时间片轮转（Round Robin, RR）策略模拟单核 CPU 调度，输出执行时间线与每个进程的完成时间、周转时间、等待时间、响应时间。

## R02

输入：
- 进程集合 `P = {p1, p2, ...}`，每个进程包含：
  - `pid`：进程标识
  - `arrival`：到达时间（非负整数）
  - `burst`：总 CPU 需求（正整数）
- 时间片 `quantum`（正整数）

输出：
- 甘特图式时间线（每次被调度的区间）
- 每个进程指标：
  - `completion`：完成时刻
  - `turnaround = completion - arrival`
  - `waiting = turnaround - burst`
  - `response = first_start - arrival`
- 全局平均指标（平均周转/等待/响应时间）

## R03

核心思想：
- 维护一个就绪队列（FIFO）。
- 每次取队首进程执行 `min(quantum, remaining)` 时间。
- 若该进程未完成则重新入队尾，已完成则记录完成时刻。
- 在执行过程中，所有新到达进程按到达顺序进入队列。
- 队列为空且仍有未来到达进程时，CPU 进入空闲并将时钟跳到下一到达时刻。

## R04

关键数据结构：
- `Process` 结构体（`pid/arrival/burst/remaining/first_start/completion`）
- `deque`：就绪队列
- `timeline`：记录片段 `(start, end, pid)`，`pid="IDLE"` 表示空闲
- `dict`：按 `pid` 汇总指标

## R05

算法流程（高层）：
1. 按 `(arrival, pid)` 对进程排序，初始化 `time=0`。
2. 将 `arrival <= time` 的进程全部入就绪队列。
3. 若队列为空：
   - 若仍有未到达进程，记录 `IDLE`，并将 `time` 跳到下一到达时刻；
   - 否则结束。
4. 队列非空时弹出队首进程，首次运行则设置 `first_start`。
5. 执行一个时间片（或剩余时间不足一个时间片）。
6. 更新全局时钟；将执行期间新到达进程入队。
7. 若当前进程未完成，入队尾；完成则记录 `completion`。
8. 重复直到所有进程完成。

## R06

正确性要点：
- 公平性：就绪队列 FIFO + 固定时间片，确保每个就绪进程都能轮流获得 CPU。
- 无饥饿：只要进程进入就绪队列，在有限轮转后必被调度。
- 指标可追溯：`completion/first_start` 均来自时间线真实时刻，派生指标公式标准且可验证。

## R07

复杂度分析：
- 设进程数为 `n`，总执行时间为 `B = sum(burst_i)`，时间片为 `q`。
- 调度轮次约为 `K = sum(ceil(burst_i / q))`。
- 时间复杂度：`O(n log n + K)`（排序 + 轮转）
- 空间复杂度：`O(n + K)`（进程状态、队列、时间线）

## R08

边界与异常处理：
- `quantum <= 0`：应抛出异常。
- `burst <= 0` 或 `arrival < 0`：应抛出异常。
- 全部进程在未来才到达：前缀 `IDLE` 正常出现。
- 单进程：退化为按时间片切分的串行执行。
- 时间片很大：行为接近 FCFS（先来先服务）。

## R09

MVP 实现范围：
- 只模拟单核 CPU、纯 CPU burst（不含 I/O 阻塞）。
- 到达时间为整数离散时刻。
- 输出文本化甘特图与指标表，便于验证与教学。

## R10

示例数据（`demo.py` 内置）：
- 时间片：`q = 2`
- 进程：
  - `P1(arrival=0, burst=5)`
  - `P2(arrival=1, burst=3)`
  - `P3(arrival=2, burst=8)`
  - `P4(arrival=4, burst=6)`

## R11

预期现象：
- 早到达进程更早得到首次响应；
- 长作业不会长期独占 CPU，而是被切分到多个轮次；
- 随着 `q` 减小，切换更频繁、响应更快但管理开销更高（理论上）。

## R12

指标解释：
- `Turnaround` 衡量“从到达到完成”总耗时；
- `Waiting` 衡量“在就绪队列等待 CPU”的累计时长；
- `Response` 反映交互体验（首次响应延迟）；
- 平均指标用于比较不同 `quantum` 配置的调度效果。

## R13

与 FCFS 的关系：
- 当 `quantum` 足够大（大于所有剩余执行时间）时，RR 近似 FCFS。
- RR 通常能降低交互任务响应时间，但可能增加上下文切换次数。

## R14

时间片选择经验：
- 过小：响应快，但切换开销大（真实系统中会损失吞吐）。
- 过大：切换少，但交互性退化，趋近 FCFS。
- 经验上选“明显小于典型 CPU burst、但不至于过度切换”的区间。

## R15

常见实现错误：
- 忘记在“执行后”再吸纳新到达进程，导致到达顺序错误。
- 队列为空时不推进时钟，造成死循环。
- 响应时间误写为等待时间。
- 完成时间记录在“入队时”而不是“真正执行结束时”。

## R16

最小测试清单：
- 正常多进程场景（交错到达）
- 同一时刻多进程到达
- 只有一个进程
- 存在 CPU 空闲区间
- 非法输入（负到达、非正 burst、非正 quantum）

## R17

可扩展方向：
- 加入上下文切换开销（每次切换增加固定成本）
- 支持 I/O 阻塞与多段 CPU burst
- 多核调度（每核一个就绪队列或全局队列）
- 与 SJF/优先级调度做对比实验

## R18

源码级算法拆解（对应 `demo.py`，非黑盒）：
1. 读取输入后构造 `Process` 对象，并初始化 `remaining=burst`。
2. 将进程按 `(arrival, pid)` 排序，保证同到达时刻的稳定、可复现实验顺序。
3. 使用指针 `next_idx` 扫描“尚未入队”的进程，凡 `arrival <= current_time` 即入 `deque`。
4. 若就绪队列为空且还有未来进程，则在 `timeline` 追加一段 `IDLE` 并把 `current_time` 跳到下一到达时刻。
5. 弹出队首进程，若 `first_start` 为空则记录当前时刻，确保响应时间只记录首次调度。
6. 计算 `run_for = min(quantum, remaining)`，更新 `timeline`、`current_time` 与 `remaining`。
7. 再次吸纳执行区间内新到达进程；若当前进程仍有剩余则入队尾，否则记录 `completion`。
8. 全部完成后，按标准公式计算每个进程与平均指标，并打印时间线与统计表。
