## R01

哈夫曼编码（Huffman Coding）是一种最优前缀码构造算法。给定符号集及其出现频率，算法通过贪心策略构造一棵二叉树，使得高频符号路径更短、低频符号路径更长，从而最小化平均编码长度。

核心目标：
- 输入：符号频率表 `f(s)`。
- 输出：前缀码 `code(s)`（任一码字都不是另一码字前缀）。
- 优化目标：最小化 `E[L] = Σ p(s) * |code(s)|`。

## R02

该算法由 David A. Huffman 于 1952 年提出，最初用于通信中的无损压缩。它是信息论中经典的贪心算法之一，常见于：
- 文件压缩（例如 DEFLATE 家族中会使用哈夫曼编码组件）；
- 数据传输中的熵编码环节；
- 教学中用于展示“贪心选择 + 最优子结构”。

## R03

它要解决的问题是“如何为已知频率分布设计最省比特的无损码”。

如果直接用定长编码，`k` 个符号每个都需 `ceil(log2 k)` 位；当符号分布明显不均匀时，这会浪费大量比特。哈夫曼编码通过变长前缀码，让常见符号用更短码字，显著降低总比特数。

## R04

标准流程：
1. 统计每个符号的频率。
2. 把每个符号作为叶节点，按频率放入最小堆。
3. 重复执行：弹出两个最小频率节点，合并为新父节点（频率为两者之和），再压回堆。
4. 堆中只剩一个根节点时结束，得到哈夫曼树。
5. 从根遍历到叶：左边记 `0`，右边记 `1`，路径即符号码字。
6. 用码表编码；解码时按比特在树上行走，走到叶就输出符号并回根。

## R05

记原串长度为 `n`，不同符号数为 `k`：
- 统计频率：`O(n)`。
- 建堆：`O(k)`（或 `O(k log k)` 视实现）。
- 合并构树：`k-1` 次弹出/压入，每次 `O(log k)`，总计 `O(k log k)`。
- 生成码表：`O(k)`。
- 编码：`O(n)`（按字符查表拼接）。
- 解码：`O(比特流长度)`。

总体主项常写为 `O(n + k log k)`，空间复杂度 `O(k)`（树、堆、码表）。

## R06

简例：频率 `A:5, B:2, C:1, D:1`。

可能构造出的码表之一：
- `A -> 1`
- `B -> 01`
- `C -> 000`
- `D -> 001`

平均码长：
- 总频率 `9`
- `E[L] = (5*1 + 2*2 + 1*3 + 1*3)/9 = 15/9 = 1.667`

对比定长编码：4 个符号需 2 比特/符号；哈夫曼编码将平均码长降到约 1.667。

## R07

优点：
- 在“已知频率”的前缀码集合中最优。
- 算法实现简单，工程可落地性强。
- 对偏态分布数据压缩收益明显。

局限：
- 需要先知道频率（或分块统计）。
- 只按单符号频率建模，难以捕捉长程上下文。
- 小文件可能被码表存储开销抵消收益。

## R08

前置知识：
- 二叉树与树遍历；
- 最小堆（优先队列）；
- 前缀码与唯一可译码概念；
- 概率分布、期望值、熵 `H(X)` 基本概念。

## R09

适用场景：
- 符号概率分布稳定、可统计；
- 无损压缩；
- 需要低复杂度、可解释编码器。

不太适用：
- 频率剧烈漂移且无法动态更新码表；
- 需要更接近熵极限的高级熵编码（如算术编码）；
- 短文本且头部开销敏感。

## R10

正确性要点（思路）：
1. 贪心选择性质：最优前缀码中，频率最小的两个符号可作为最深层兄弟节点。
2. 最优子结构：把这两个最小频率符号合并为“伪符号”后，原问题化为规模更小的同类问题。
3. 递归/归纳：不断合并最小频率对，得到的整棵树保持全局最优。

因此，哈夫曼算法输出的是最优前缀码（针对给定频率分布）。

## R11

数值与实现稳定性注意点：
- 哈夫曼算法主要是离散操作，不涉及浮点迭代稳定性问题。
- 频率相同时应设置确定性 tie-break（本实现用 `order`），保证可复现。
- 比特流用字符串仅用于教学；大规模工程需位级缓冲区减少内存开销。
- 单符号输入要特殊处理（通常分配码字 `0`），避免空码字导致解码歧义。

## R12

真实工程成本：
- 统计频率是线性扫描，通常不是瓶颈。
- 主要成本来自 I/O 与位流打包解包，而非构树本身。
- 码表序列化（头部）会引入额外比特，需要与正文压缩收益一起评估。
- 在大规模批处理里，通常按块独立编码，平衡局部最优与并行吞吐。

## R13

理论保证：
- 最优性保证：在所有二元前缀码中最小化期望码长。
- 与熵关系：`H(X) <= E[L] < H(X) + 1`（二元哈夫曼码）。
- 近似比/随机化：N/A（这是确定性最优构造，不是近似算法或随机算法）。

## R14

常见失效模式与防护：
- 失效：忘记处理“仅 1 个唯一符号”。
  - 防护：叶节点空前缀时强制给 `0`。
- 失效：解码比特串在内部节点结束，导致截断。
  - 防护：解码结束时检查游标必须回到根。
- 失效：码表不是前缀码，解码歧义。
  - 防护：构树生成天然前缀码，并增加前缀冲突断言。
- 失效：频率相同导致非确定输出，不利测试。
  - 防护：堆键加入稳定次序字段。

## R15

工程实践建议：
- 先做 round-trip 测试：`decode(encode(text)) == text`。
- 对比定长编码基线，验证压缩收益是否为正。
- 记录 `H(X)` 与 `E[L]`，检查是否满足理论边界。
- 长文本场景用 bitarray/bytearray 等位容器替代字符串拼接。
- 若做在线场景，可扩展为自适应哈夫曼或分块重建码表。

## R16

相关算法与前后关系：
- 前驱/同类：香农-范诺编码（通常次优）、定长编码。
- 后续/替代：算术编码、区间编码（常可更逼近熵极限）。
- 组合关系：哈夫曼经常作为压缩流水线末端熵编码器，与 LZ 系列前端配合。

## R18

`demo.py` 的源码级算法流程（8 步）：
1. `build_frequency_table` 统计输入文本每个字符的出现次数，得到 `freq_table`。
2. `build_huffman_tree` 把每个字符封装为叶节点并放入最小堆，循环弹出两个最小频率节点合并，直到生成唯一根节点。
3. `build_codebook` 深度优先遍历树，左边追加 `0`、右边追加 `1`，得到符号到比特串的映射；单符号场景回退到码字 `0`。
4. `assert_prefix_free` 对生成码表做前缀冲突检查，保证可唯一解码。
5. `encode_text` 按原文字符顺序查码表并拼接比特串，得到编码结果。
6. `decode_bits` 从根节点按比特行走，遇到叶节点输出符号并回根，重建原文并校验 round-trip 一致性。
7. `expected_bits_per_symbol` 与 `shannon_entropy_bits` 计算 `E[L]`、`H(X)`，并在 `main` 中验证 `H(X) <= E[L] < H(X)+1`。
8. `main` 同时计算定长编码基线比特数，输出压缩率与高频符号码字，最终给出“运行成功”信号。

本实现未把第三方包当黑盒压缩器；贪心合并、树构建、编码与解码步骤都在源码中可追踪。
