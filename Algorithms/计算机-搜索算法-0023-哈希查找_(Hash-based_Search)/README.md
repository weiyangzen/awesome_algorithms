# 哈希查找 (Hash-based Search)

- UID: `CS-0023`
- 学科: `计算机`
- 分类: `搜索算法`
- 源序号: `23`
- 目标目录: `Algorithms/计算机-搜索算法-0023-哈希查找_(Hash-based_Search)`

## R01

哈希查找（Hash-based Search）通过哈希函数把键映射到桶（bucket）位置，从而把“按值定位”从线性扫描的 `O(n)` 平均降到接近 `O(1)`。它特别适合**无序数据上的高频查询**。

## R02

问题形式化：
- 输入：序列 `arr`（可无序），以及待查询键 `target`
- 输出：
  - 是否存在 `target`
  - 或返回首个命中下标
  - 或返回所有命中下标（当存在重复值时）

本条目在 `demo.py` 中返回“首个下标”和“全部下标”两种结果。

## R03

适用前提：
- 查询键必须可哈希（hashable）
- 若要多次查询，同一批数据最好先建哈希索引再反复查找
- 若数据持续高频更新，需要考虑重哈希/扩容策略（本 MVP 为静态建表）

## R04

核心思想：
- 使用哈希函数 `h(key)` 计算桶号 `slot = h(key) mod m`
- 将同桶元素组织成链（链地址法）解决冲突
- 查找时只扫描目标桶，而非全表
- 对重复值可在桶内维护索引列表

## R05

简化伪代码：

```text
build_index(arr):
    table = [empty_bucket] * m
    for i, value in enumerate(arr):
        slot = hash(value) mod m
        if value already in table[slot]:
            append i to value.indices
        else:
            insert (value, [i]) into table[slot]

find_first(target):
    slot = hash(target) mod m
    scan table[slot]
    if found key == target: return first index
    return -1
```

## R06

正确性要点：
- 同一键总会被映射到同一桶号（在哈希函数和表长不变时）
- 建表阶段把每个元素下标准确附着到其键对应的桶节点
- 查询阶段完整扫描目标桶；若键存在，必在该桶中被命中
- 若目标桶扫描结束仍未命中，则键不存在于当前索引

## R07

复杂度：
- 建表时间：`O(n)`
- 单次查询平均：`O(1)`
- 单次查询最坏：`O(n)`（大量冲突退化到同一桶）
- 空间复杂度：`O(n)`

影响平均性能的关键因素：
- 负载因子 `alpha = n / m`
- 哈希函数分布均匀性
- 冲突处理策略（链地址法/开放定址等）

## R08

边界与特殊情况：
- 空数组：任何查询返回未命中
- 重复值：可返回首个下标或全部下标（本实现支持两者）
- 不可哈希元素（如 `list`）：应明确抛错，避免静默失败
- 键存在但冲突严重：性能会下降，但正确性不受影响

## R09

示例：
- `arr = [42, 7, 13, 7, 99]`
- 建表后，键 `7` 对应索引列表为 `[1, 3]`
- 查询结果：
  - `find_first(7) -> 1`
  - `find_all(7) -> [1, 3]`
  - `find_first(100) -> -1`

## R10

本目录 MVP 实现策略：
- `demo.py` 手写一个最小链地址法哈希表 `HashTableIndex`
- 每个桶存储若干 `Entry(key, indices)`
- 对外提供：
  - `find_first(key)`：返回首个命中下标或 `-1`
  - `find_all(key)`：返回全部命中下标列表
- 使用线性扫描结果做交叉校验，避免“自证正确”

## R12

典型应用：
- 字典/符号表查找
- 数据去重与成员判断（set/map）
- 关系型连接中的哈希连接（Hash Join）
- 编译器词法符号表、缓存键定位、ID 索引

## R13

常见错误：
- 忘记处理冲突，导致同桶后插入覆盖前插入
- 使用可变不可哈希对象作为键
- 把“平均 `O(1)`”误解为“严格最坏 `O(1)`”
- 只返回布尔存在性，却忽略重复值位置需求

## R14

常见变体：
- 冲突处理：链地址法、线性探测、二次探测、双重哈希
- 动态扩容：当负载因子超阈值时重建更大哈希表
- 最小完美哈希：针对静态键集追求无冲突查询
- 一致性哈希：用于分布式节点路由而非单机表查找

## R15

与其他搜索方法对比：
- 线性查找：无需预处理，单次 `O(n)`
- 二分查找：需有序数组，单次 `O(log n)`
- 哈希查找：允许无序，建表后平均单次 `O(1)`

当查询次数较多时，哈希查找通常更划算；当只查询一次且数据很小，线性查找更简单。

## R16

工程实践建议：
- 监控负载因子与桶分布，避免长期高冲突
- 明确键类型与相等语义（`__hash__` 与 `__eq__` 一致）
- 对外 API 说明重复值处理策略（首个/全部/任意）
- 在关键路径保留退化监控与回退策略

## R17

最小测试清单：
- 空数组、单元素命中/未命中
- 多元素命中与未命中
- 重复值（验证首个与全部下标）
- 负数、零、大整数键
- 不可哈希键（应触发 `TypeError`）
- 批量重复查询（验证可复用索引）

## R18

源码级算法流程拆解（以本目录 `demo.py` 为准）：
1. `build_hash_index(arr)` 根据数据长度选择桶数量，初始化 `HashTableIndex`。
2. 逐个读取 `arr` 元素，调用 `insert(value, index)` 进入建表流程。
3. `insert` 内通过 `_slot(key)` 计算桶号，并定位到对应桶链表。
4. 扫描该桶：若键已存在，则把新下标追加到该键的 `indices` 列表。
5. 若桶内不存在该键，则新建 `Entry(key, [index])` 追加到桶末。
6. 查询 `find_first(target)` 时，再次计算同一桶号，只扫描该桶并返回首个下标。
7. 查询 `find_all(target)` 时，命中后返回完整下标列表副本；未命中返回空列表。
8. `main()` 用线性扫描结果做对照断言，覆盖空数组、重复值、未命中与批量查询场景。
9. 所有断言通过后输出 `All checks passed.`，证明 MVP 在给定测试集上行为正确。
