## R01

死锁检测（Deadlock Detection）是数据库并发控制中“先允许等待、再检查是否卡死”的策略。

核心思路：

- 事务冲突时先进入等待，而不是立即回滚；
- 用等待图（wait-for graph）建模“谁在等谁”；
- 发现有向环后，选择受害者回滚/重启，释放资源以打破环。

一句话概括：允许局部阻塞，但通过周期检测和恢复机制避免系统永久停摆。

## R02

与相关概念的区别：

- 死锁预防（prevention）：用规则让等待关系天然无环（如 `wait-die`）；
- 死锁避免（avoidance）：在分配前做安全性判定（如 Banker）；
- 死锁检测（detection，本条目）：允许形成等待，再检测环并恢复。

检测策略的工程价值在于：并发度通常更高，但需要承担“检测与恢复”的额外复杂度。

## R03

本条目要解决的问题：

- 输入：多个事务的操作序列（`lock/work/unlock/commit`）；
- 输出：调度日志、检测到的死锁环、受害者重启信息与最终提交结果；
- 目标：在经典交叉加锁场景中，能够检测环并自动恢复，最终全部事务完成。

`demo.py` 用纯 Python 实现锁管理、等待队列、等待图构建与环检测，不依赖数据库内核黑盒。

## R04

MVP 的核心数据结构：

1. `lock_owner[resource] -> tx_id`：记录排它锁当前持有者；
2. `waiting_queues[resource] -> deque[tx_id]`：资源级等待队列；
3. `Transaction`：保存事务时间戳、程序计数器、等待状态、持锁集、重启次数；
4. `wait_for_graph`：由“等待事务 -> 持锁事务”构成的有向图；
5. `event_log`：记录授锁、等待、检测、回滚重启、唤醒、提交等事件。

调度器按轮询推进，每轮结束后执行一次死锁检测与恢复。

## R05

复杂度分析（设事务数 `T`，资源数 `R`，等待边数 `E`）：

- 单次加锁判定：`O(1)`（哈希查表）；
- 一轮事务推进：`O(T)`；
- 构建等待图：`O(T)` 到 `O(E)`；
- 单次环检测（DFS）：`O(T + E)`；
- 一次受害者恢复：`O(R + Q)`（`Q` 为队列总长度，用于清理等待项）。

空间复杂度约 `O(T + R + E)`。

## R06

内置演示场景：

- `T1`（老事务）先锁 `A`，再请求 `B`；
- `T2`（年轻事务）先锁 `B`，再请求 `A`；
- `T3` 访问独立资源 `C`，用于证明系统仍能并行前进。

当 `T1` 等 `T2`、`T2` 等 `T1` 时，等待图形成环。检测器选年轻事务 `T2` 为受害者重启，释放 `B` 后 `T1` 得以继续。

## R07

优点：

- 冲突时先等待，通常比保守预防策略并发度更高；
- 等待图模型直观，检测结果（环路）可审计；
- 可按策略选择受害者（年龄、代价、优先级等）。

局限：

- 检测是额外开销，不是“零成本”；
- 受害者回滚会带来重放成本；
- 若策略不当，某些事务可能反复被选中（饥饿风险）。

## R09

适用场景：

- 允许短时等待，希望提升并发度的 OLTP 负载；
- 需要“可观测死锁事件 + 可解释恢复路径”的系统；
- 教学与实验环境下演示“检测-恢复”闭环。

不适用场景：

- 事务重放代价极高，几乎不能接受回滚；
- 超低延迟路径中，任何检测开销都不可接受；
- 缺少恢复组件（日志、回滚语义）时不宜单独采用。

## R10

正确性直觉（为何检测有效）：

1. 若等待图无环，则不存在循环等待死锁；
2. 若等待图有环，则环上事务彼此等待，形成死锁；
3. 回滚环中任一事务并释放其锁，至少移除环上一条关键等待边；
4. 环被破坏后，等待队列可继续推进，系统恢复前进能力。

因此，流程关键是“准确构图 + 正确找环 + 正确释放资源”。

## R11

实现中的鲁棒性细节：

- 等待请求去重，避免同事务重复入队；
- 资源释放时清理等待队列中的陈旧项（已不再等待的事务）；
- 受害者恢复时会全局移除其等待项并释放全部持锁；
- 唤醒时仅在资源空闲时授锁，并推进事务程序计数器；
- 采用确定性受害者选择规则，保证演示可复现。

## R12

关键可调参数：

- `max_rounds`：最大调度轮数，防止异常逻辑无限循环；
- 检测频率：当前是“每轮检测一次”，工程中可改为周期/阈值触发；
- 受害者策略：当前按“环内最年轻事务”重启，可替换为成本导向；
- 工作负载冲突模式：决定死锁出现概率与恢复次数。

工程实践中常叠加重启退避、优先级提升来缓解饥饿。

## R14

常见失效模式：

1. 等待图漏边（例如忽略部分等待状态），导致漏检；
2. 受害者回滚未释放全部锁，导致伪死锁持续；
3. 等待队列残留陈旧请求，造成虚假阻塞；
4. 唤醒逻辑错误（授锁后未推进 PC），导致事务卡住。

排查建议：

- 打印每轮锁表、等待图、事务状态；
- 对照事件日志核对“检测 -> 受害者 -> 释放 -> 唤醒”链路。

## R15

工程化落地建议：

- 把检测、恢复、调度、日志拆成独立模块便于替换；
- 记录指标：`deadlock_rate`、`victim_restart_count`、`wait_time_p99`；
- 受害者选择引入业务权重（高价值事务降低被回滚概率）；
- 恢复要和 WAL/回滚日志联动，确保语义正确。

## R16

与其他方案对比：

- 对比死锁预防：检测策略并发度更高，但恢复复杂度更高；
- 对比死锁避免：检测不需要提前声明最大资源需求；
- 对比超时回滚：图检测对“真死锁”更精确，误杀更少；
- 对比无锁/MVCC：锁模型更直接，但等待与回滚成本更显著。

本 MVP 展示的是“数据库锁等待图检测”这一经典路径。

## R17

`demo.py` 的 MVP 功能清单：

- 排它锁授予、冲突等待、释放后唤醒；
- 每轮构建等待图并用 DFS 检测环；
- 发现死锁时确定性选择受害者并重启；
- 输出完整事件轨迹、提交顺序与每事务重启统计；
- 内置断言验证：至少一次死锁恢复，最终全部提交，锁表清空。

脚本无输入参数，执行一次即可复现检测与恢复全过程。

## R18

`demo.py` 源码级算法流程（9 步）：

1. `build_demo_transactions()` 构造事务程序，其中 `T1/T2` 反向请求 `A/B`，可触发循环等待。  
2. `DeadlockDetectingScheduler.__init__` 初始化锁表、等待队列、事件日志与统计字段。  
3. `run()` 按轮询调度事务，每个事务每轮最多执行一步（`_step`）。  
4. `_handle_lock` 在资源冲突时将事务置为 `waiting` 并入资源等待队列，不立即回滚。  
5. 每轮结束 `_detect_and_resolve_deadlock` 调用 `wait_for_graph` 构图，再用 `find_cycle`（DFS）查找环。  
6. 若检测到环，`choose_victim` 按时间戳选最年轻事务为受害者。  
7. `_abort_and_restart` 执行恢复：移除受害者等待项、释放其全部锁、程序计数器归零并重启。  
8. `_release_one/_wake_waiter` 在资源释放时唤醒队首合法等待者并直接授锁，恢复推进链路。  
9. `main()` 打印事件日志与统计，并用断言校验“死锁确实被检测并恢复，最终全部提交且无残留锁/等待”。  

以上流程把检测与恢复细节全部展开在源码中，没有依赖外部数据库黑盒实现。
