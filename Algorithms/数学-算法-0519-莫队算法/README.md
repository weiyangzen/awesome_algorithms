## R01

莫队算法（Mo's Algorithm）是一种用于**静态数组离线区间查询**的重排技巧。它不改变查询语义，只重排查询执行顺序，让相邻查询的区间边界移动尽可能小，从而降低总维护成本。

- 输入: 长度为 `N` 的静态数组，`Q` 个闭区间查询 `[L, R]`。
- 输出: 每个查询对应的统计值（本目录 MVP 为“区间不同元素个数”）。
- 核心机制:
  1. 按块划分左端点（块大小通常取 `sqrt(N)`）。
  2. 在块内按右端点排序（常用奇偶块蛇形排序减少抖动）。
  3. 用双指针维护当前区间 `[curL, curR]`，通过 `add/remove` 增量更新答案。

## R02

莫队算法主要来源于竞赛算法社区中的经验总结，可视作“分块思想 + 离线排序 + 滑动窗口维护”的组合范式。它在 Codeforces、AtCoder 等平台广泛传播，成为静态离线区间查询的经典手段之一。

## R03

很多区间统计问题单次查询并不难，但 `Q` 很大时，逐条独立计算会退化成 `O(NQ)`。例如区间不同值个数、区间频次相关统计等，如果每次都重扫子数组会非常慢。

莫队算法把“单查询重算”改为“查询之间复用状态”，尤其适合以下场景：

- 数组静态（无在线修改，或修改次数较少且可扩展到带修改莫队）。
- 查询可离线（不要求按输入顺序即时返回）。
- 目标统计可通过 `add/remove` 在近似 `O(1)` 时间维护。

## R04

设单次 `add/remove` 成本为 `F`。

- 排序成本: `O(Q log Q)`。
- 指针移动总成本: 典型为 `O((N + Q) * sqrt(N))` 次量级。
- 总体复杂度: `O(Q log Q + (N + Q) * sqrt(N) * F)`。

当 `F = O(1)`（如本例的 distinct count 频次数组维护）时，常写为 `O((N + Q) * sqrt(N))`（忽略排序项与常数）。

## R05

空间复杂度由三部分组成：

- 查询副本与排序索引: `O(Q)`。
- 频次结构: `O(U)`，`U` 为值域大小（本实现通过坐标压缩使 `U <= N`）。
- 输出数组: `O(Q)`。

总体额外空间为 `O(Q + U)`，在压缩后可近似为 `O(Q + N)`。

## R06

以数组 `a = [1,1,2,1,3,4,5,2,8]`、查询集合
`[0,4], [1,3], [2,4], [0,8], [3,7]` 为例：

1. 设块长 `B = floor(sqrt(9)) = 3`，按 `(L//B, R)`（奇偶块蛇形）排序查询。
2. 初始窗口为空 `curL=0, curR=-1`，`distinct=0`。
3. 处理第一个排序后查询时，移动边界并执行 `add/remove`，维护频次数组与 `distinct`。
4. 切换到下一个查询时，不重算整个区间，只做边界差量移动。
5. 最终按原查询顺序回填答案，得到 `4,2,3,6,5`。

## R07

莫队算法的价值主要在工程可实现性与性能折中：

- 相比复杂树结构（如某些可持久化结构），实现更直接。
- 相比暴力重算，能显著降低总开销。
- 提供一个通用框架：只要能写出可逆增量 `add/remove`，就能替换统计目标。

## R08

理论与技术前置主要包括：

- 分块思想（平方根分治直觉）。
- 离线查询重排（结果按原索引回填）。
- 双指针窗口维护。
- 计数结构（数组/哈希表）与坐标压缩。

## R09

适用条件与边界：

- 适用:
  - 静态数组场景。
  - 离线可接受。
  - 统计函数支持“加入一个元素/移除一个元素”的增量维护。
- 不适用或需扩展:
  - 在线必须即时回答的场景。
  - 高频在线修改（需带修改莫队或改用线段树/树状数组等结构）。
  - 统计函数无法高效维护（`F` 很大时莫队收益下降）。

## R10

正确性要点（针对本目录 distinct query 版本）：

1. 查询排序仅改变执行顺序，不改变每个查询区间定义。
2. 任一时刻窗口状态与频次数组严格对应当前 `[curL, curR]`。
3. `add` 逻辑保证“元素频次从 0 到 1 时 distinct +1”。
4. `remove` 逻辑保证“元素频次从 1 到 0 时 distinct -1”。
5. 当窗口被移动到目标查询 `[L, R]` 后，`distinct` 即该区间不同值个数。
6. 将结果写入原索引位置，确保输出顺序与输入查询顺序一致。

## R11

数值稳定性方面，本算法属于离散计数流程：

- 不涉及浮点迭代，因此不存在浮点舍入误差传播。
- 主要风险来自实现细节而非数值分析:
  - 频次数组越界。
  - 边界移动顺序错误。
  - 坐标压缩映射不一致。

在 Python 中整数为任意精度，频次不会因整型溢出导致错误。

## R12

真实成本（超越渐近符号）主要受以下因素影响：

- 查询排序质量（奇偶块排序通常优于单调 `R` 排序）。
- `add/remove` 的常数因子（数组频次通常快于哈希频次）。
- 值域大小与缓存局部性（压缩后更易命中缓存）。
- 数据分布（高度重复值通常让 distinct 维护更“平滑”）。

## R14

典型失效模式与防护：

- 失效模式: 查询下标非法（`L<0`、`R>=N`、`L>R`）。
  - 防护: 入参校验并抛出异常。
- 失效模式: 忘记离线回填原索引，导致输出顺序错乱。
  - 防护: 查询结构保留 `idx` 字段。
- 失效模式: 值域太大直接开频次数组导致内存浪费。
  - 防护: 先做坐标压缩。
- 失效模式: 窗口移动与更新顺序写反。
  - 防护: 固定四个 while 顺序并用朴素解对拍回归。

## R15

工程实践建议：

- 保持 `add/remove` 纯函数式副作用边界（只改频次与统计量）。
- 对空查询集、空数组、越界查询给出明确行为。
- 使用随机回归测试与朴素解对拍，避免“看似快但偶发错”。
- 当 `Q` 很大时优先开启蛇形排序（奇偶块反向 `R`）。
- 若扩展到“带修改莫队”，需加入时间维和回滚/应用操作逻辑。

## R16

前驱、并行路线与后继：

- 前驱思想:
  - 朴素离线重算（每次 `O(R-L+1)`）。
  - 分块统计（但难处理复杂区间统计）。
- 后继与变体:
  - Hilbert order 排序版本（进一步降低指针抖动）。
  - 带修改莫队（加入时间维度）。
  - 树上莫队（把树上路径查询转化为序列区间）。
- 典型应用:
  - 区间不同值个数。
  - 区间频次相关统计（如某值出现次数、可维护的函数统计）。

## R18

源码级流程拆解（`demo.py`，未使用第三方黑箱算法）。

`mo_distinct_count` 的关键执行链路可分为 8 步：

1. 读取 `arr` 与 `queries`，完成空输入与区间合法性校验。
2. 计算块大小 `block_size = floor(sqrt(N))`，为每个查询附加 `(idx, block)` 元数据。
3. 按 `(block, R)` 进行奇偶块蛇形排序，减少连续查询之间的右端点震荡。
4. 对数组执行坐标压缩，把原值映射为紧凑整数编号。
5. 初始化频次数组 `freq`、当前窗口 `curL/curR` 与统计量 `distinct`。
6. 依次处理排序后的查询，通过四个 while 循环移动窗口边界。
7. 每次边界移动调用 `add/remove`：
   - `add`: 频次 `0->1` 时 `distinct += 1`。
   - `remove`: 频次 `1->0` 时 `distinct -= 1`。
8. 当窗口对齐目标 `[L, R]`，把 `distinct` 写回 `ans[idx]`，最终返回原顺序答案。
