## R01

本条目实现的是“一元多项式最大公因式（Polynomial GCD）”算法。  
给定两个多项式 `p(x), q(x)`，求一个次数最高的多项式 `g(x)`，使得：

- `g | p` 且 `g | q`；
- 任意公共因子 `h` 都满足 `h | g`（在单位因子意义下）。

在计算代数里，多项式 gcd 是因式分解、约分、消元和符号化简的基础组件。

## R02

MVP 输入输出约定：

- 输入：两个“降幂系数”数组，例如 `[1, -3, 2]` 表示 `x^2 - 3x + 2`。
- 输出：`gcd(p, q)` 的 **monic** 形式（最高次项系数归一化为 `1`）。

额外边界约定：

- `gcd(0, p) = monic(p)`；
- `gcd(0, 0) = 0`（工程约定，便于程序闭合）；
- 自动去掉首部多余零系数。

## R03

核心数学方法是欧几里得算法在 `Q[x]` 上的推广：

- 若 `q != 0`，则存在 `p = q*s + r`，且 `deg(r) < deg(q)`；
- `gcd(p, q) = gcd(q, r)`；
- 反复取余直到余式为零，最后一个非零余式即 gcd（差一个可逆常数因子）。

本实现最后做 monic 归一化，得到唯一代表。

## R04

为什么在 `Q[x]`（有理系数域）上做欧几里得算法：

- 多项式长除在域上总是可行，步骤简单直接；
- 用 `Fraction` 可保持精确算术，避免浮点误差；
- 结果可再转回 `Z[x]` 的 primitive 代表，便于人类阅读。

因此该实现兼顾“数学正确性”和“可追踪代码实现”。

## R05

`demo.py` 的核心函数：

- `_as_fraction_list`：把输入统一为 `Fraction`。
- `_trim_leading_zeros`：清理首部零系数。
- `poly_divmod`：显式多项式长除，返回 `(quotient, remainder)`。
- `poly_gcd`：欧几里得主循环，输出 monic gcd。
- `to_integer_primitive`：把有理系数 gcd 映射到整系数 primitive 代表（展示用途）。

所有核心步骤均在源码中可见，无黑盒调用。

## R06

长除实现要点（降幂数组）：

- 每轮取当前余式与除式的首项比值 `lead_factor`；
- 计算对应次数偏移 `shift`；
- 用 `lead_factor * x^shift * divisor` 消去余式最高次项；
- 迭代直到 `deg(remainder) < deg(divisor)`。

这与纸笔长除完全一致，只是改为数组运算。

## R07

复杂度分析（设 `deg(p)=m`, `deg(q)=n`, `m>=n`）：

- 一次长除主开销约为 `O((m-n+1) * n)`；
- 欧几里得算法做若干次长除，最坏可近似看作 `O(d^3)` 量级（`d=max(m,n)`，粗略上界）。

空间复杂度主要由多项式系数存储决定，约 `O(d)` 到 `O(d^2)`（取决于中间复制策略）。

## R08

与整数系数 PRS / pseudo-remainder 路线对比：

- 本实现（`Fraction`）优点：逻辑直观、代码短、正确性清晰；
- 缺点：高次数时分子分母可能膨胀；
- PRS 路线优点：能更好控制中间系数增长，通常更高效。

本条目作为 MVP，优先“透明与可验证”，因此选择 `Fraction + Euclid`。

## R09

内置验证案例覆盖：

- 共享二次因子：应返回对应二次多项式；
- 非首一输入：即使有内容因子（常数倍）也应得到同一 monic gcd；
- 互素输入：应返回 `1`；
- 含零多项式：`gcd(0,p)` 与 `gcd(0,0)` 边界。

运行时每个案例打印 `OK/MISMATCH` 与总结果 `PASS/FAIL`。

## R10

正确性直觉：

- 每次 `a = b*q + r` 不改变公共因子集合；
- 迭代保持 `gcd(a,b)=gcd(b,r)` 不变；
- 当 `r=0` 时，`b` 能整除前一轮的两个多项式，是最大公共因子（差单位因子）。

最后 monic 化去掉单位因子不唯一性。

## R11

数值与鲁棒性策略：

- 全程使用 `Fraction`，避免浮点舍入误差；
- 每次更新后做 `_trim_leading_zeros`，防止伪高次项污染循环条件；
- 对除零多项式显式抛错，避免静默错误。

这使得输出可用于符号验证，而非仅数值近似。

## R12

输出解释：

- `gcd_monic`：标准化后的唯一代表，便于程序比较；
- `gcd_primitive_Z[x]`：同一因子在整系数下的 primitive 代表，便于人工查看；
- 若只关心“是否有非平凡公共因子”，判定 `deg(gcd_monic) >= 1` 即可。

## R13

工程扩展方向：

- 更高性能：替换为 subresultant PRS；
- 多变量：先对某变量视作一元做主元化，再配合消元策略；
- 与因式分解联动：先做 gcd 去重，再做 square-free 分解。

当前代码结构已经把“长除/主循环/归一化”拆开，便于替换底层算子。

## R14

常见失败模式与排查：

- 输入不是降幂顺序：会得到错误结果；
- 把 `[0,0,...,0]` 当一般多项式传入：需先规范为 `[0]`；
- 手工期望值未做 monic 归一化：可能与程序输出差常数因子。

若出现 `MISMATCH`，优先检查以上三点。

## R18

`demo.py` 的源码级算法流程（8 步）：

1. 读取输入系数并通过 `_as_fraction_list` 转为 `Fraction`，保证精确算术。  
2. 用 `_trim_leading_zeros` 规范化多项式，统一次数定义与零多项式判定。  
3. 在 `poly_gcd` 中初始化 `a=p, b=q`，处理 `gcd(0,0)` 特例。  
4. 进入欧几里得循环：调用 `poly_divmod(a, b)` 做一次显式长除，得到余式 `r`。  
5. 长除内部用“首项相除 + 次数偏移”构造消元项，逐步降低余式次数直到 `< deg(b)`。  
6. 回到主循环，执行状态推进 `a <- b, b <- r`，直到 `b` 成为零多项式。  
7. 循环结束后将 `a` 除以其首项系数做 monic 归一化，得到唯一 gcd 代表。  
8. `main()` 运行多组案例，打印 `gcd_monic` 与 `gcd_primitive_Z[x]` 并给出 `OK/MISMATCH`。  

该流程完全由源码中的可见函数组成，不依赖第三方 CAS 的黑盒 gcd。
