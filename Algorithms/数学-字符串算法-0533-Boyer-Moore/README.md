## R01

Boyer-Moore（BM）是经典的**精确字符串匹配**算法：在文本 `text` 中查找模式串 `pattern` 的全部出现位置。

与从左到右逐字符比较不同，BM 在每次对齐时从模式串右端开始比较，并在失配后通过启发式规则一次性跳过多个字符。

## R02

BM 的核心价值在于“跳步搜索”：

- 朴素匹配最坏会在每个位置都做大量重复比较；
- BM 结合模式串预处理信息，通常能显著减少比较次数；
- 在自然语言文本、日志、DNA 片段等长文本检索场景中常表现良好。

它是字符串算法中从“逐位扫描”向“信息驱动跳转”演进的代表。

## R03

本条目要解决的问题：

- 输入：文本串 `text` 与模式串 `pattern`；
- 输出：`pattern` 在 `text` 中所有匹配的起始下标；
- 要求：结果与朴素算法一致，支持零次或多次出现。

MVP 采用确定性实现，不依赖黑盒库。

## R04

BM 的两条主启发式：

1. **坏字符规则（Bad Character）**  
   若在模式串位置 `j` 失配，查看文本失配字符 `c` 在模式串中的最右出现位置 `last[c]`，可将模式右移 `j - last[c]`（至少 1）。
2. **好后缀规则（Good Suffix）**  
   失配前若已匹配了一个后缀，可将模式移动到“让该后缀在模式内再次对齐”的位置；若不存在则对齐到可匹配前缀。

搜索时取两种位移的最大值，以获得更大跳步。

## R05

设 `n = len(text)`，`m = len(pattern)`：

- 预处理复杂度：`O(m + |Sigma|)` 或在字典实现下约 `O(m)`；
- 搜索平均复杂度：通常优于朴素匹配，经验上接近线性；
- 理论最坏复杂度：经典 BM 在某些构造下可达 `O(nm)`（工程上常用变体可保证线性）。

空间复杂度为 `O(m + |Sigma_used|)`，本实现按模式中出现字符建表。

## R07

优点：

- 典型输入下比较次数少，速度快；
- 预处理一次模式后可复用于多段文本；
- 算法思想清晰，适合教学和工程实现。

局限：

- 好后缀预处理实现细节较多，易写错；
- 对极端短模式、极小文本时优势不明显；
- 经典 BM 并非所有情形都保证线性最坏界。

## R09

适用场景：

- 单模式精确匹配；
- 文本较长、模式中字符分布有区分度；
- 需要可解释、可自定义的搜索流程。

不适用场景：

- 多模式同时匹配（更适合 Aho-Corasick）；
- 近似匹配/编辑距离匹配；
- 大规模在线流匹配且需严格最坏线性保证（可考虑 KMP 或工程变体）。

## R10

正确性直觉：

1. 每次对齐比较时，若整段匹配则记录当前起点；
2. 失配时，坏字符位移保证不会跳过任何可能匹配位置；
3. 好后缀位移基于“已匹配后缀仍可对齐”的必要条件，也不会丢失解；
4. 取两者最大位移后继续，直到扫描完 `text` 可对齐区间。

因此输出集合与朴素穷举匹配一致。

## R11

鲁棒性与边界处理：

- `pattern == ""`：按通用定义返回 `0..n` 全部位置；
- `len(pattern) > len(text)`：直接返回空；
- 使用 Python `str`，天然支持 Unicode 字符；
- 通过随机对照朴素算法做回归检查，减少实现错误风险。

## R14

常见失效模式：

1. 好后缀表下标错位（`j` 与 `j+1` 混用）导致漏报；
2. 坏字符位移未做 `max(1, ...)` 导致死循环；
3. 忽略空模式边界导致行为不一致；
4. 把 Python 切片当作算法本体，失去 BM 的流程可解释性。

本实现通过函数拆分与随机回归测试降低上述风险。

## R15

工程化建议：

- 加入性能基准（不同文本分布、不同模式长度）；
- 若追求最坏线性，可扩展到 Boyer-Moore-Horspool、Turbo BM 或与 KMP 混合策略；
- 在生产环境加入超长文本分块与监控统计。

## R16

相关算法对比：

- **Naive Search**：实现最简单，最坏 `O(nm)`；
- **KMP**：利用前缀函数，最坏线性，跳步更保守；
- **Rabin-Karp**：哈希过滤候选，适合多模式批量哈希；
- **Boyer-Moore-Horspool**：BM 的简化常用变体，仅用坏字符规则。

BM 的优势在于常见文本中的较大跳跃能力。

## R18

`demo.py` 的源码级算法流程（8 步）：

1. `build_bad_character_table(pattern)` 扫描模式串，记录每个字符最后一次出现下标。  
2. `build_good_suffix_shift(pattern)` 用 `shift` 与 `bpos` 两个数组先做 strong suffix 预处理，再做 case2 兜底。  
3. `boyer_moore_search` 初始化对齐偏移 `s=0`，每轮从 `j=m-1` 向左比较。  
4. 若 `pattern[j] == text[s+j]`，继续左移 `j`；若 `j < 0`，说明整段匹配，记录 `s`。  
5. 全匹配后依据 `good_suffix[0]` 做下一次跳转（至少右移 1），支持重叠匹配。  
6. 发生失配时，计算坏字符位移 `bad_shift = j - last[text[s+j]]`（未出现按 `-1` 处理）。  
7. 同时读取好后缀位移 `good_shift = good_suffix[j+1]`，并取 `max(1, bad_shift, good_shift)` 更新 `s`。  
8. 循环直到 `s > n-m`，返回全部匹配位置；`run_fixed_case` 与 `run_randomized_checks` 用朴素算法校验结果一致性。

本实现未调用第三方字符串搜索黑盒函数，预处理与搜索位移逻辑均在源码中可追踪。
